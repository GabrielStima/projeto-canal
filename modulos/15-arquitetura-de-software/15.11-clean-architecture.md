# Clean Architecture

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar a ideia central da Clean Architecture;
- diferenciar regras centrais de detalhes externos;
- entender a direção das dependências;
- aplicar a ideia em uma API pequena sem exagero.

## Onde Esta Aula Entra no Curso

- Módulo: 15. Arquitetura de Software
- Aula anterior: Monolito
- Próxima aula: DDD
- Pré-requisitos: Policy vs detail, Dependency Inversion Principle, camadas, backend, testes e design de software.

## Ideia Central

Clean Architecture organiza o sistema para que regras importantes não dependam de detalhes como banco, framework, UI ou serviços externos. A regra de dependência aponta para dentro: detalhes externos podem conhecer casos de uso e contratos, mas a política central não deve conhecer detalhes externos.

O objetivo não é criar muitas pastas. É proteger o que é importante de coisas que mudam por motivos técnicos.

## Tópicos da Aula

- Entidades e regras centrais.
- Casos de uso.
- Adaptadores.
- Infraestrutura.
- Regra de dependência.
- Aplicação prática com moderação.

## Exemplo Prático

```text
CompleteLessonUseCase depende de:
- LessonProgressRepository (contrato)
- Clock (contrato)

Infraestrutura implementa:
- PrismaLessonProgressRepository
- SystemClock

O caso de uso não conhece Prisma, Express ou HTTP.
```

## Analogias e Intuições

- O centro da arquitetura deve saber pouco sobre o mundo externo.
- Framework é ferramenta; regra de negócio é promessa do sistema.

## Erros Comuns

- Criar pastas de Clean Architecture sem entender dependências.
- Colocar regra de negócio no adapter.
- Criar interface para tudo mecanicamente.
- Usar entidades anêmicas sem comportamento.
- Exagerar em sistemas pequenos.

## Checklist de Entendimento

- Que parte é regra central?
- Que parte é detalhe externo?
- A dependência aponta para dentro?
- A regra pode ser testada sem banco e HTTP?
- A estrutura está protegendo algo real?

## Prática Recomendada

- Escolha um fluxo de API.
- Escreva um caso de uso sem Express e sem ORM.
- Defina um contrato de repositório.
- Implemente um adapter simples para persistência.
- Teste o caso de uso isoladamente.
