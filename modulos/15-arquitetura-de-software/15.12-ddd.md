# DDD

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar a ideia central de Domain-Driven Design;
- entender dominio, modelo, linguagem ubiqua e bounded context;
- reconhecer quando DDD ajuda;
- evitar transformar DDD em excesso de cerimonia.

## Onde Esta Aula Entra no Curso

- Modulo: 15. Arquitetura de Software
- Aula anterior: Clean Architecture
- Proxima aula: Padroes Arquiteturais
- Pre-requisitos: Model-driven design, boundaries, Clean Architecture, requisitos, backend, APIs e bancos de dados.

## Ideia Central

DDD e uma abordagem para organizar software em torno do dominio do problema. Em vez de deixar banco, framework ou tela definirem o sistema, DDD busca criar modelos que expressem regras, linguagem e limites reais do negocio.

O ponto mais importante para comecar nao e decorar todos os padroes taticos. E aprender a conversar sobre dominio, separar contextos e evitar um modelo unico tentando representar tudo.

## Topicos da Aula

- Dominio.
- Modelo de dominio.
- Linguagem ubiqua.
- Bounded context.
- Entidades, value objects e agregados em nivel introdutorio.
- Quando usar e quando evitar.

## Exemplo Pratico

```text
Termo: aula

No contexto de catalogo:
- aula tem titulo, descricao, modulo e ordem.

No contexto de progresso:
- aula tem status, data de conclusao e relacao com aluno.

Mesmo nome, modelos diferentes.
```

## Analogias e Intuicoes

- DDD tenta fazer o codigo falar a lingua do problema.
- Um bounded context e uma fronteira onde uma palavra tem significado consistente.

## Erros Comuns

- Usar DDD em CRUD simples sem necessidade.
- Criar nomes sofisticados para regras simples.
- Ignorar conversa com produto e negocio.
- Forcar um modelo unico para contextos diferentes.
- Confundir entidade de dominio com tabela do banco.

## Checklist de Entendimento

- Qual e o dominio principal?
- Que palavras precisam de definicao clara?
- Existe mais de um significado para o mesmo termo?
- Onde esta o boundary do contexto?
- O modelo expressa regra ou so dados?

## Pratica Recomendada

- Escolha um fluxo da plataforma de estudos.
- Liste termos importantes do dominio.
- Identifique dois contextos onde um termo muda de significado.
- Desenhe um bounded context simples.
- Escreva uma regra em linguagem do dominio.
