# DDD

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar a ideia central de Domain-Driven Design;
- entender domínio, modelo, linguagem ubíqua e bounded context;
- reconhecer quando DDD ajuda;
- evitar transformar DDD em excesso de cerimônia.

## Onde Esta Aula Entra no Curso

- Módulo: 15. Arquitetura de Software

- Próxima aula: Padrões Arquiteturais

## Ideia Central

DDD é uma abordagem para organizar software em torno do domínio do problema. Em vez de deixar banco, framework ou tela definirem o sistema, DDD busca criar modelos que expressem regras, linguagem e limites reais do negócio.

O ponto mais importante para começar não é decorar todos os padrões táticos. É aprender a conversar sobre domínio, separar contextos e evitar um modelo único tentando representar tudo.

## Tópicos da Aula

- Domínio.
- Modelo de domínio.
- Linguagem ubíqua.
- Bounded context.
- Entidades, value objects e agregados em nível introdutório.
- Quando usar e quando evitar.

## Exemplo Prático

```text
Termo: aula

No contexto de catálogo:
- aula tem título, descrição, módulo e ordem.

No contexto de progresso:
- aula tem status, data de conclusão e relação com aluno.

Mesmo nome, modelos diferentes.
```

## Analogias e Intuições

- DDD tenta fazer o código falar a língua do problema.
- Um bounded context é uma fronteira onde uma palavra tem significado consistente.

## Erros Comuns

- Usar DDD em CRUD simples sem necessidade.
- Criar nomes sofisticados para regras simples.
- Ignorar conversa com produto e negócio.
- Forçar um modelo único para contextos diferentes.
- Confundir entidade de domínio com tabela do banco.

## Checklist de Entendimento

- Qual é o domínio principal?
- Que palavras precisam de definição clara?
- Existe mais de um significado para o mesmo termo?
- Onde está o boundary do contexto?
- O modelo expressa regra ou só dados?

## Prática Recomendada

- Escolha um fluxo da plataforma de estudos.
- Liste termos importantes do domínio.
- Identifique dois contextos onde um termo muda de significado.
- Desenhe um bounded context simples.
- Escreva uma regra em linguagem do domínio.
