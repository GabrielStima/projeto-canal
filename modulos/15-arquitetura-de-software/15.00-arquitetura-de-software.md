# Arquitetura de Software

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que arquitetura de software organiza;
- diferenciar arquitetura de design de código;
- reconhecer os temas que serão estudados no módulo;
- entender por que arquitetura é uma conversa sobre trade-offs.

## Ideia Central

Arquitetura de software é o conjunto de decisões que organiza um sistema além de arquivos, funções e classes. Ela lida com limites, componentes, dependências, comunicação, deploy, operação, evolução e custos de mudança.

Uma boa arquitetura não é a mais sofisticada. É a que protege as decisões importantes, deixa o sistema evoluir no ritmo certo e torna explícitos os trade-offs que o time aceitou.

## Tópicos da Aula

- O que arquitetura organiza.
- Diferença entre design, arquitetura e infraestrutura.
- Componentes, boundaries e dependências.
- Estilos e padrões arquiteturais.
- Distribuição, mensageria e operação.
- Trade-offs como centro da decisão.

## Exemplo Prático

```text
Sistema: plataforma de estudos

Decisões arquiteturais:
- progresso do aluno fica no mesmo serviço das aulas?
- conclusão de aula é regra central ou detalhe de UI?
- notificação acontece no request ou em processamento assíncrono?
- relatórios usam o mesmo modelo de escrita?
- vale separar serviços agora ou manter monólito?
```

## Analogias e Intuições

- Arquitetura é o mapa de circulação de uma cidade: não define cada prédio, mas influencia como tudo se conecta.
- Uma decisão arquitetural ruim costuma aparecer como uma mudança simples que vira reforma grande.

## Erros Comuns

- Confundir arquitetura com estrutura de pastas.
- Escolher padrão antes de entender problema e restrições.
- Copiar arquitetura de empresa grande para projeto pequeno.
- Distribuir um sistema antes de entender seus limites.
- Ignorar operação, testes e observabilidade.

## Checklist de Entendimento

- Que decisões são difíceis de mudar depois?
- Que partes do sistema mudam por motivos diferentes?
- Onde estão os limites naturais?
- Que dependência deve apontar para quem?
- Qual trade-off está sendo aceito?

## Prática Recomendada

- Pegue uma API pequena que você já criou.
- Liste suas partes principais.
- Marque regras de negócio, detalhes de framework, banco, transporte e integrações.
- Escreva três decisões arquiteturais que esse sistema já tomou, mesmo sem perceber.
