# Arquitetura de Software

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o que arquitetura de software organiza;
- diferenciar arquitetura de design de codigo;
- reconhecer os temas que serao estudados no modulo;
- entender por que arquitetura e uma conversa sobre trade-offs.

## Onde Esta Aula Entra no Curso

- Modulo: 15. Arquitetura de Software
- Aula anterior: Projeto Pratico: Refatorando uma API para Melhor Design
- Proxima aula: Do Design de Software a Arquitetura
- Pre-requisitos: Backend, APIs, bancos de dados, seguranca, qualidade, observabilidade, performance e design de software.

## Ideia Central

Arquitetura de software e o conjunto de decisoes que organiza um sistema alem de arquivos, funcoes e classes. Ela lida com limites, componentes, dependencias, comunicacao, deploy, operacao, evolucao e custos de mudanca.

Uma boa arquitetura nao e a mais sofisticada. E a que protege as decisoes importantes, deixa o sistema evoluir no ritmo certo e torna explicitos os trade-offs que o time aceitou.

## Topicos da Aula

- O que arquitetura organiza.
- Diferenca entre design, arquitetura e infraestrutura.
- Componentes, boundaries e dependencias.
- Estilos e padroes arquiteturais.
- Distribuicao, mensageria e operacao.
- Trade-offs como centro da decisao.

## Exemplo Pratico

```text
Sistema: plataforma de estudos

Decisoes arquiteturais:
- progresso do aluno fica no mesmo servico das aulas?
- conclusao de aula e regra central ou detalhe de UI?
- notificacao acontece no request ou em processamento assincrono?
- relatorios usam o mesmo modelo de escrita?
- vale separar servicos agora ou manter monolito?
```

## Analogias e Intuicoes

- Arquitetura e o mapa de circulacao de uma cidade: nao define cada predio, mas influencia como tudo se conecta.
- Uma decisao arquitetural ruim costuma aparecer como uma mudanca simples que vira reforma grande.

## Erros Comuns

- Confundir arquitetura com estrutura de pastas.
- Escolher padrao antes de entender problema e restricoes.
- Copiar arquitetura de empresa grande para projeto pequeno.
- Distribuir um sistema antes de entender seus limites.
- Ignorar operacao, testes e observabilidade.

## Checklist de Entendimento

- Que decisoes sao dificeis de mudar depois?
- Que partes do sistema mudam por motivos diferentes?
- Onde estao os limites naturais?
- Que dependencia deve apontar para quem?
- Qual trade-off esta sendo aceito?

## Pratica Recomendada

- Pegue uma API pequena que voce ja criou.
- Liste suas partes principais.
- Marque regras de negocio, detalhes de framework, banco, transporte e integracoes.
- Escreva tres decisoes arquiteturais que esse sistema ja tomou, mesmo sem perceber.
