# Twelve-Factor Apps

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o objetivo dos Twelve-Factor Apps;
- entender fatores importantes para apps implantáveis e operáveis;
- conectar configuração, logs, processos e dependências com arquitetura;
- preparar a entrada nos módulos de cloud e infraestrutura.

## Onde Esta Aula Entra na Formação

- Módulo: 15. Arquitetura de Software
- Aula anterior: Serverless
- Próxima aula: Service Mesh
- Pré-requisitos: Backend, variáveis de ambiente, deploy, logging, processos, APIs e observabilidade.

## Ideia Central

Twelve-Factor Apps é um conjunto de princípios para construir aplicações que rodam bem em ambientes modernos de deploy. Eles tratam de código, dependências, configuração, serviços externos, build, release, processos, logs e portabilidade.

No contexto desta formação, eles ajudam a conectar arquitetura de software com operação e cloud sem entrar ainda em Kubernetes ou infraestrutura profunda.

## Tópicos da Aula

- Código e dependências.
- Configuração no ambiente.
- Serviços externos como recursos anexados.
- Build, release e run.
- Processos stateless.
- Logs como fluxo de eventos.

## Exemplo Prático

Regras de Ouro no Deploy do PetCare:

Fator III - Configurações no ambiente: Você nunca deve chumar senhas, links de API do banco no código-fonte do PetCare. Use variáveis `.env` ou Vaults.
Fator VI - Processos Stateless: Sua API Node não pode salvar a sessão do login na memória RAM dela, senão quando reiniciar, os médicos são deslogados. Jogue o state num Redis.

## Analogias e Intuições

- 12-Factor é o manual de boas maneiras para os apps jantarem na Nuvem sem fazer sujeira.

## Erros Comuns

- Colocar configuração sensível no código.
- Depender de estado local no servidor.
- Tratar logs como arquivo manual.
- Misturar build e runtime sem critério.
- Ignorar diferença entre ambiente local e produção.

## Checklist de Entendimento

- A configuração está fora do código?
- A aplicação é stateless?
- Logs são consumíveis pela plataforma?
- Serviços externos são declarados como dependências?
- Build, release e run estão separados?

## Prática Recomendada

- Identifique no código atual do seu sistema onde o backend viola a regra do Stateless. Ex: Upload de imagens é feito na pasta `/uploads` dentro do container Node?
