## Ideia Central

MVC é um padrão de organização que separa modelo, interface e controle de fluxo. A ideia central é impedir que a tela, a regra e a coordenação fiquem completamente misturadas.

Em aplicações modernas, MVC aparece de formas diferentes. Em backend web, controller costuma receber request e coordenar resposta. Em frontend, componentes, estado e renderização podem ocupar papéis parecidos, mas nem sempre com os mesmos nomes.

## Exemplo Prático

O MVC Tradicional e APIs:

O MVC nasceu para telas (Model guarda os Pets, View é o HTML renderizado do paciente, Controller recebe o click no botão).
Nas APIs REST de hoje (PetCare backend), o conceito mudou: A 'View' é um JSON enviado de volta ao FrontEnd (React). O Controller recebe HTTP e delega pro Model (Regras e Banco).

## Analogias e Intuições

- O Model é a Cozinha (faz o prato/dado). A View é o Prato Empratado na Mesa (Interface). O Controller é o Garçom (leva o pedido e traz o prato).

## Erros Comuns

- Colocar regra de negócio pesada no controller.
- Tratar model como tabela do banco apenas.
- Misturar View com regra central.
- Achar que MVC resolve todos os limites.
- Usar nomes de MVC sem responsabilidades reais.

## Checklist de Entendimento

- Quem recebe a entrada?
- Quem representa a regra ou dado principal?
- Quem apresenta a saída?
- O controller está coordenando ou fazendo tudo?
- MVC é suficiente para este contexto?
