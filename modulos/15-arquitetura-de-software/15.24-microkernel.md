# Microkernel

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o padrão Microkernel;
- entender núcleo mínimo e plugins;
- reconhecer quando extensibilidade justifica esse padrão;
- evitar criar mecanismo de plugin cedo demais.

## Onde Esta Aula Entra no Curso

- Módulo: 15. Arquitetura de Software
- Aula anterior: Blackboard
- Próxima aula: Projeto Prático: Arquitetando uma API para Evolução
- Pré-requisitos: Padrões arquiteturais, interfaces, abstrações, composition over inheritance, boundaries e modularidade.

## Ideia Central

Microkernel organiza o sistema em um núcleo pequeno e estável, com funcionalidades variáveis adicionadas por plugins ou extensões. O núcleo define contratos e ciclo de vida; os plugins implementam capacidades específicas.

Esse padrão é útil quando extensibilidade é requisito central. Se a variação ainda não existe, criar um microkernel pode ser arquitetura demais.

## Exemplo Prático

```text
Sistema de avaliação de exercícios:

Núcleo:
- recebe submissão;
- coordena execução;
- registra resultado.

Plugins:
- avaliador de JavaScript;
- avaliador de SQL;
- avaliador de HTML/CSS;
- gerador de feedback.
```

## Analogias e Intuições

- Microkernel é como um editor com extensões: o núcleo abre caminho para capacidades extras.
- Plugin system é produto, não detalhe barato.

## Erros Comuns

- Criar sistema de plugins sem demanda real.
- Fazer o núcleo conhecer detalhes dos plugins.
- Definir contrato instável demais.
- Não isolar falhas de extensões.
- Transformar tudo em extensão.

## Checklist de Entendimento

- Extensibilidade é requisito real?
- O núcleo pode permanecer pequeno?
- O contrato de plugin é claro?
- Como lidar com falha de extensão?
- O custo compensa a variação esperada?

## Prática Recomendada

- Escolha um sistema com variações reais.
- Defina o que ficaria no núcleo.
- Desenhe dois plugins.
- Escreva um contrato mínimo de extensão.
