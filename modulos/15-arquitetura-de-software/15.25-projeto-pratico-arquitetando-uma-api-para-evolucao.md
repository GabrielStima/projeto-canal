# Projeto Pratico: Arquitetando uma API para Evolucao

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- analisar uma API pequena com olhar arquitetural;
- identificar componentes, boundaries e dependencias;
- escolher uma arquitetura proporcional ao contexto;
- registrar trade-offs e plano de evolucao.

## Onde Esta Aula Entra no Curso

- Modulo: 15. Arquitetura de Software
- Aula anterior: Microkernel
- Proxima aula: Dados em Tempo Real e Comunicacao Assincrona
- Pre-requisitos: Todo o modulo 15, design de software, backend, APIs, bancos de dados, testes, observabilidade e seguranca.

## Ideia Central

O projeto do modulo junta as decisoes arquiteturais em uma API realista. A meta nao e usar todos os padroes estudados. E escolher limites claros, proteger regras importantes, reduzir acoplamento, manter operacao simples e preparar evolucao sem reescrever o sistema.

Uma boa entrega arquitetural deve explicar o que foi escolhido, por que foi escolhido, o que ficou de fora e quais sinais indicariam uma mudanca futura.

## Topicos da Aula

- Diagnostico da API atual.
- Identificacao de componentes.
- Boundaries e policy vs detail.
- Escolha de estilo arquitetural.
- Trade-offs.
- Plano incremental de evolucao.

## Exemplo Pratico

```text
API: plataforma de estudos

Fluxos:
- listar aulas;
- concluir aula;
- acompanhar progresso;
- emitir certificado;
- enviar notificacao.

Decisoes:
- monolito modular como base;
- progresso como boundary proprio;
- notificacao assincrona como evolucao possivel;
- regras em casos de uso testaveis;
- banco ainda compartilhado, com acesso controlado.
```

## Analogias e Intuicoes

- Projeto arquitetural bom parece uma decisao bem explicada, nao uma colecao de diagramas bonitos.
- Evoluir arquitetura e trocar uma ponte enquanto o transito ainda existe.

## Erros Comuns

- Usar todos os padroes do modulo no mesmo projeto.
- Projetar para escala imaginaria.
- Ignorar testes e observabilidade.
- Separar servicos sem boundaries internos.
- Nao registrar motivos e trade-offs.

## Checklist de Entendimento

- Quais sao os componentes principais?
- Que boundaries protegem regras importantes?
- Que detalhes ficaram nas bordas?
- Qual estilo arquitetural foi escolhido?
- Que trade-offs foram aceitos?
- Qual mudanca futura justificaria outra arquitetura?

## Pratica Recomendada

- Escolha uma API criada nos modulos anteriores.
- Desenhe a arquitetura atual.
- Liste problemas de acoplamento e coesao.
- Proponha um monolito modular ou outra estrutura proporcional.
- Defina boundaries e contratos internos.
- Escreva um ADR curto registrando a decisao.
