# Projeto Prático: Arquitetando uma API para Evolução

## Objetivo

Ao final desta aula, você deve ser capaz de:

- analisar uma API pequena com olhar arquitetural;
- identificar componentes, boundaries e dependências;
- escolher uma arquitetura proporcional ao contexto;
- registrar trade-offs e plano de evolução.

## Onde Esta Aula Entra na Formação

- Módulo: 15. Arquitetura de Software
- Aula anterior: Microkernel
- Próxima aula: Dados em Tempo Real e Comunicação Assíncrona
- Pré-requisitos: Todo o módulo 15, design de software, backend, APIs, bancos de dados, testes, observabilidade e segurança.

## Ideia Central

O projeto do módulo junta as decisões arquiteturais em uma API realista. A meta não é usar todos os padrões estudados. É escolher limites claros, proteger regras importantes, reduzir acoplamento, manter operação simples e preparar evolução sem reescrever o sistema.

Uma boa entrega arquitetural deve explicar o que foi escolhido, por que foi escolhido, o que ficou de fora e quais sinais indicariam uma mudança futura.

## Tópicos da Aula

- Diagnóstico da API atual.
- Identificação de componentes.
- Boundaries e policy vs detail.
- Escolha de estilo arquitetural.
- Trade-offs.
- Plano incremental de evolução.

## Exemplo Prático

marco arquitetural - Preparando a Nuvem:

No projeto prático, você atua como o Staff Engineer do PetCare OS. A clínica virou rede nacional.
Você documentará (ADR - Architecture Decision Records) as mudanças e extrairá a funcionalidade mais instável do monólito, implementando Comunicação Assíncrona via RabbitMQ/BULL-MQ ou Cloud.
A Aplicação se tornará um Monólito Modular robusto que delega tarefas densas (como OCR de pdf e IA de exames) ao Service Worker.
O seu backend se livra da dor da lentidão.

## Analogias e Intuições

- Você para de ser o pedreiro e vira o engenheiro da cidade. Você define o tráfego pesado das vias expressas assíncronas do Hospital.

## Erros Comuns

- Usar todos os padrões do módulo no mesmo projeto.
- Projetar para escala imaginária.
- Ignorar testes e observabilidade.
- Separar serviços sem boundaries internos.
- Não registrar motivos e trade-offs.

## Checklist de Entendimento

- Quais são os componentes principais?
- Que boundaries protegem regras importantes?
- Que detalhes ficaram nas bordas?
- Qual estilo arquitetural foi escolhido?
- Que trade-offs foram aceitos?
- Qual mudança futura justificaria outra arquitetura?
