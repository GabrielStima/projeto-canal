# Princípios de Componentes

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é um componente em arquitetura;
- avaliar coesão e estabilidade de componentes;
- entender dependências entre componentes;
- reconhecer quando uma divisão ajuda ou atrapalha.

## Onde Esta Aula Entra na Formação

- Módulo: 15. Arquitetura de Software
- Aula anterior: Acoplamento e Coesão em Nível Arquitetural
- Próxima aula: Boundaries
- Pré-requisitos: Módulos, pacotes, acoplamento, coesão, SOLID e testes.

## Ideia Central

Um componente é uma unidade maior de organização: pode ser um pacote, módulo, biblioteca interna, serviço ou parte bem delimitada do sistema. Ele deve ter uma responsabilidade compreensível, uma interface de uso e dependências explícitas.

Princípios de componentes ajudam a decidir o que deve ficar junto, o que deve ficar separado e em que direção as dependências devem apontar.

## Tópicos da Aula

- Componente como unidade de mudança.
- Coesão de pacote.
- Estabilidade e dependências.
- Interface pública do componente.
- Dependências cíclicas.
- Divisão prematura.

## Exemplo Prático

```text
Componentes possíveis:
- catálogo de aulas;
- progresso do aluno;
- assinaturas;
- notificações;
- relatórios.

Cada componente precisa responder:
- o que ele oferece?
- do que ele depende?
- quem pode chamá-lo?
- que mudança o afetaria?
```

## Analogias e Intuições

- Um componente bom parece uma ferramenta com entrada e saída claras.
- Se você precisa abrir o componente para saber como usar, a interface pública está fraca.

## Erros Comuns

- Separar por tipo técnico em vez de responsabilidade.
- Permitir dependências cíclicas.
- Exportar tudo de um componente.
- Criar componente sem comportamento próprio.
- Confundir componente com pasta.

## Checklist de Entendimento

- O componente tem responsabilidade clara?
- Sua interface pública é pequena?
- Suas dependências são explícitas?
- Existe ciclo entre componentes?
- A divisão reduz ou aumenta trabalho?

## Prática Recomendada

- Escolha uma API existente.
- Agrupe arquivos por responsabilidade de negócio.
- Defina uma interface pública para cada grupo.
- Identifique dependências cíclicas ou desnecessárias.
