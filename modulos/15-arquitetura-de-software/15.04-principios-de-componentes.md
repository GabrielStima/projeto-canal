# Principios de Componentes

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o que e um componente em arquitetura;
- avaliar coesao e estabilidade de componentes;
- entender dependencias entre componentes;
- reconhecer quando uma divisao ajuda ou atrapalha.

## Onde Esta Aula Entra no Curso

- Modulo: 15. Arquitetura de Software
- Aula anterior: Acoplamento e Coesao em Nivel Arquitetural
- Proxima aula: Boundaries
- Pre-requisitos: Modulos, pacotes, acoplamento, coesao, SOLID e testes.

## Ideia Central

Um componente e uma unidade maior de organizacao: pode ser um pacote, modulo, biblioteca interna, servico ou parte bem delimitada do sistema. Ele deve ter uma responsabilidade compreensivel, uma interface de uso e dependencias explicitas.

Principios de componentes ajudam a decidir o que deve ficar junto, o que deve ficar separado e em que direcao as dependencias devem apontar.

## Topicos da Aula

- Componente como unidade de mudanca.
- Coesao de pacote.
- Estabilidade e dependencias.
- Interface publica do componente.
- Dependencias ciclicas.
- Divisao prematura.

## Exemplo Pratico

```text
Componentes possiveis:
- catalogo de aulas;
- progresso do aluno;
- assinaturas;
- notificacoes;
- relatorios.

Cada componente precisa responder:
- o que ele oferece?
- do que ele depende?
- quem pode chama-lo?
- que mudanca o afetaria?
```

## Analogias e Intuicoes

- Um componente bom parece uma ferramenta com entrada e saida claras.
- Se voce precisa abrir o componente para saber como usar, a interface esta fraca.

## Erros Comuns

- Separar por tipo tecnico em vez de responsabilidade.
- Permitir dependencias ciclicas.
- Exportar tudo de um componente.
- Criar componente sem comportamento proprio.
- Confundir componente com pasta.

## Checklist de Entendimento

- O componente tem responsabilidade clara?
- Sua interface publica e pequena?
- Suas dependencias sao explicitas?
- Existe ciclo entre componentes?
- A divisao reduz ou aumenta trabalho?

## Pratica Recomendada

- Escolha uma API existente.
- Agrupe arquivos por responsabilidade de negocio.
- Defina uma interface publica para cada grupo.
- Identifique dependencias ciclicas ou desnecessarias.
