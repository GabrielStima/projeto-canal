# Do Design de Software a Arquitetura

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar a diferenca entre design de codigo e arquitetura;
- perceber quando uma decisao local vira decisao arquitetural;
- conectar principios do modulo anterior com sistemas maiores;
- identificar limites iniciais em uma aplicacao.

## Onde Esta Aula Entra no Curso

- Modulo: 15. Arquitetura de Software
- Aula anterior: Arquitetura de Software
- Proxima aula: Principios Arquiteturais
- Pre-requisitos: Todo o modulo 14, backend com Node.js, APIs, bancos de dados e testabilidade.

## Ideia Central

Design de software organiza comportamento dentro do codigo. Arquitetura organiza o sistema como um conjunto de partes que precisam evoluir, conversar, operar e falhar de maneira compreensivel.

A ponte entre os dois esta nas dependencias. Uma funcao acoplada a um detalhe tecnico e um problema de design. Um sistema inteiro dependente de detalhes tecnicos no lugar errado vira problema arquitetural.

## Topicos da Aula

- Design como decisao local.
- Arquitetura como decisao estrutural.
- Responsabilidades que crescem.
- Dependencias que atravessam o sistema.
- Testabilidade como sinal arquitetural.
- Evolucao incremental.

## Exemplo Pratico

```text
No design:
- extrair uma funcao de validacao;
- separar handler HTTP de caso de uso.

Na arquitetura:
- decidir se casos de uso dependem de repositorios abstratos;
- decidir se relatorios ficam no mesmo modulo de escrita;
- decidir se eventos de dominio precisam existir.
```

## Analogias e Intuicoes

- Design e arrumar uma sala; arquitetura e decidir a planta da casa.
- Um detalhe repetido em muitos lugares deixa de ser detalhe e vira restricao estrutural.

## Erros Comuns

- Tratar arquitetura como algo separado do codigo real.
- Achar que arquitetura so aparece em sistemas grandes.
- Ignorar sinais de testabilidade ruim.
- Chamar qualquer pasta de camada.
- Fazer uma reescrita quando uma evolucao incremental resolveria.

## Checklist de Entendimento

- Essa decisao afeta so um arquivo ou varias partes do sistema?
- O sistema consegue trocar um detalhe tecnico sem tocar regra de negocio?
- Que mudanca futura ficaria facil ou dificil?
- Que teste denuncia acoplamento demais?
- Que limite ja existe no dominio?

## Pratica Recomendada

- Pegue uma refatoracao feita no modulo 14.
- Liste quais melhorias eram design de codigo.
- Liste quais melhorias mudariam a arquitetura.
- Escreva uma regra simples para diferenciar os dois.
