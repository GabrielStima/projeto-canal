# Do Design de Software à Arquitetura

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar a diferença entre design de código e arquitetura;
- perceber quando uma decisão local vira decisão arquitetural;
- conectar princípios do módulo anterior com sistemas maiores;
- identificar limites iniciais em uma aplicação.

## Onde Esta Aula Entra na Formação

- Módulo: 15. Arquitetura de Software
- Aula anterior: Arquitetura de Software
- Próxima aula: Princípios Arquiteturais
- Pré-requisitos: Todo o módulo 14, backend com Node.js, APIs, bancos de dados e testabilidade.

## Ideia Central

Design de software organiza comportamento dentro do código. Arquitetura organiza o sistema como um conjunto de partes que precisam evoluir, conversar, operar e falhar de maneira compreensível.

A ponte entre os dois está nas dependências. Uma função acoplada a um detalhe técnico é um problema de design. Um sistema inteiro dependente de detalhes técnicos no lugar errado vira problema arquitetural.

## Tópicos da Aula

- Design como decisão local.
- Arquitetura como decisão estrutural.
- Responsabilidades que crescem.
- Dependências que atravessam o sistema.
- Testabilidade como sinal arquitetural.
- Evolução incremental.

## Exemplo Prático

Escalando o PetCare:

- Nível de Design: Decidir usar a classe `CalculadoraImposto` isolada (SRP/OCP) para calcular os custos.
- Nível de Arquitetura: Decidir se a `CalculadoraImposto` roda no mesmo servidor web que o `Portal do Tutor`, ou se ela vira um módulo isolado que processa dados da Receita Federal todo dia à meia-noite.

## Analogias e Intuições

- Design: Decidir como organizar as ferramentas dentro da caixa.
- Arquitetura: Decidir em quais andares do prédio cada oficina ficará e os elevadores que ligam elas.

## Erros Comuns

- Tratar arquitetura como algo separado do código real.
- Achar que arquitetura só aparece em sistemas grandes.
- Ignorar sinais de testabilidade ruim.
- Chamar qualquer pasta de camada.
- Fazer uma reescrita quando uma evolução incremental resolveria.

## Checklist de Entendimento

- Essa decisão afeta só um arquivo ou várias partes do sistema?
- O sistema consegue trocar um detalhe técnico sem tocar regra de negócio?
- Que mudança futura ficaria fácil ou difícil?
- Que teste denuncia acoplamento demais?
- Que limite já existe no domínio?

## Prática Recomendada

- Descreva um problema no seu código atual do PetCare OS que pode ser resolvido com Design (Refatoração de classe) e um problema que só seria resolvido com Arquitetura (Mover para Background Jobs/Filas).
