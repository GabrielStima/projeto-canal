# Padrões Arquiteturais

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que são padrões arquiteturais;
- diferenciar padrões arquiteturais de padrões de design;
- usar padrões como vocabulário de decisão;
- preparar o estudo dos padrões específicos do módulo.

## Onde Esta Aula Entra no Curso

- Módulo: 15. Arquitetura de Software
- Aula anterior: DDD
- Próxima aula: SOA
- Pré-requisitos: Estilos arquiteturais, boundaries, componentes, DDD, backend e APIs.

## Ideia Central

Padrões arquiteturais são soluções recorrentes para problemas recorrentes de organização de sistemas. Eles aparecem em nível maior que classes e objetos: serviços, comunicação, deploy, extensibilidade, dados, eventos e limites.

O papel de um padrão arquitetural não é substituir raciocínio. É dar nome a uma alternativa para que o time consiga discutir custos, benefícios e riscos com mais clareza.

## Tópicos da Aula

- Padrões como vocabulário.
- Diferença entre estilo e padrão.
- Padrões estruturais.
- Padrões distribuídos.
- Padrões orientados a eventos.
- Riscos de aplicar padrão por moda.

## Exemplo Prático

```text
Problema: separar leitura pesada da escrita principal.

Padrões candidatos:
- CQRS;
- fila para atualizar projeções;
- cache;
- relatório separado;
- réplica de leitura.

Cada escolha paga um custo diferente.
```

## Analogias e Intuições

- Padrão arquitetural é uma solução com histórico, não uma lei.
- Saber o nome do padrão ajuda a pesquisar, comparar e explicar.

## Erros Comuns

- Escolher padrão antes do problema.
- Misturar padrões incompatíveis sem critério.
- Ignorar custo operacional.
- Aplicar padrão sofisticado para CRUD simples.
- Achar que nome bonito melhora arquitetura ruim.

## Checklist de Entendimento

- Qual problema o padrão resolve?
- Que custo ele adiciona?
- Que alternativa mais simples existe?
- O time consegue operar isso?
- O padrão protege uma decisão importante?

## Prática Recomendada

- Escolha um problema arquitetural.
- Liste três padrões ou alternativas.
- Compare custo, complexidade, testabilidade e evolução.
- Escolha a alternativa mais simples que resolve o problema real.
