# TypeScript: O Que É e Como Pensar em Tipos

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar TypeScript como JavaScript com checagem estática de tipos;
- diferenciar tempo de desenvolvimento e runtime;
- entender que TypeScript não muda a natureza do JavaScript executado;
- reconhecer o papel dos tipos como documentação e proteção;
- preparar a base para inferência e anotações.

## Onde Esta Aula Entra no Curso

- Módulo: Programação com TypeScript
- Aula anterior: Promises e async/await
- Próxima aula: Inferência de tipos

## Ideia Central

TypeScript adiciona tipos ao JavaScript para encontrar problemas antes da execução. Ele ajuda o editor, o compilador e a pessoa desenvolvedora a entenderem quais valores são esperados em cada parte do código.

O ponto mais importante: TypeScript checa o código durante o desenvolvimento, mas o que roda no fim ainda é JavaScript. Por isso, aprender TypeScript também exige entender JavaScript.

## Tópicos da Aula

- TypeScript como superset de JavaScript
- Checagem estática
- Runtime vs tempo de checagem
- Tipos como contrato
- Tipos como documentação
- O que TypeScript consegue e não consegue garantir
- Por que isso importa em projetos maiores

## Exemplo Prático

```typescript
function criarSaudacao(nome: string) {
  return `Olá, ${nome}`;
}

criarSaudacao("Ana");
criarSaudacao(42);
```

Neste exemplo, TypeScript aponta o problema antes da execução: a função espera uma `string`, mas recebeu um `number`.

## Analogias e Intuições

- TypeScript é uma revisão antecipada do código.
- Tipos são combinados sobre o formato esperado dos dados.
- O runtime ainda precisa ser entendido, porque TypeScript não executa a aplicação por você.

## Erros Comuns

- Achar que TypeScript evita todo bug.
- Pensar que TypeScript é uma linguagem totalmente separada de JavaScript.
- Anotar tudo sem aproveitar inferência.
- Confundir erro de tipo com erro de regra de negócio.
- Ignorar que dados externos ainda precisam ser validados em runtime.
