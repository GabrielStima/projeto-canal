# Projeto Pratico: Tornando uma API Testavel, Observavel e Mensuravel

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- integrar os conceitos do modulo em uma API pequena;
- criar uma estrategia de testes para fluxos criticos;
- adicionar sinais basicos de observabilidade;
- medir performance de um fluxo importante;
- relacionar failure modes com testes, logs, metricas e alertas.

## Onde Esta Aula Entra no Curso

- Modulo: 13. Qualidade, Testes, Observabilidade e Performance
- Aula anterior: Failure Modes na Pratica
- Proxima aula: Design de Software
- Pre-requisitos: Todas as aulas do modulo 13, projeto de API dos modulos anteriores, API design, autenticacao, autorizacao, seguranca, banco de dados, logging e profiling.

## Ideia Central

O projeto pratico transforma o modulo em uma evolucao concreta da API do painel de estudos. A API ja tem contrato, autenticacao, autorizacao e seguranca basica. Agora ela precisa ganhar uma rede minima de confianca: testes bem escolhidos, sinais investigaveis e medicao de performance.

O foco nao e instalar a maior quantidade de ferramentas. O foco e criar decisoes explicaveis: por que este teste existe, que falha ele cobre, que log ajuda a investigar, que metrica mostra saude e que alerta indicaria impacto real.

## Topicos da Aula

- Escolha de fluxos criticos.
- Estrategia de testes.
- Testes unitarios, integracao, contrato e E2E.
- Regressao e CI.
- Profiling de um endpoint.
- Logs, metricas, traces, monitoramento e alertas.
- Checklist final de failure modes.

## Exemplo Pratico

```text
Fluxos escolhidos:
1. login;
2. consultar aulas;
3. atualizar progresso;
4. rota administrativa de usuario.

Para cada fluxo:
- cenarios de teste;
- failure modes;
- logs relevantes;
- metricas;
- trace textual;
- alerta possivel.
```

## Analogias e Intuicoes

- O projeto e uma revisao aplicada: cada aula vira uma decisao visivel.
- Confiabilidade cresce quando comportamento, falha e investigacao sao tratados juntos.

## Erros Comuns

- Comecar instalando ferramenta sem estrategia.
- Criar testes apenas para caminho feliz.
- Logar eventos sem contexto.
- Medir performance sem baseline.
- Criar alerta sem acao esperada.

## Checklist de Entendimento

Antes de seguir, voce deve conseguir responder:

- Quais fluxos sao criticos?
- Que riscos cada fluxo tem?
- Que testes protegem esses riscos?
- Que sinais ajudam a investigar falhas?
- Que metrica indicaria degradacao?
- Que alerta exigiria acao?

## Pratica Recomendada

- Escolha quatro fluxos da API do painel de estudos.
- Monte uma matriz com: risco, teste, log, metrica, trace e alerta.
- Escreva testes unitarios para regras pequenas.
- Escreva testes de integracao para endpoints com banco.
- Defina pelo menos um contract test.
- Defina um fluxo E2E critico.
- Crie um teste de regressao para uma falha conhecida ou imaginada.
- Rode os testes em um fluxo de CI conceitual ou real.
- Faca profiling textual de um endpoint lento.
- Desenhe um painel de monitoramento minimo.
