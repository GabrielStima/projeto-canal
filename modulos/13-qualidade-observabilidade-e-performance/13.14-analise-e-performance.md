# Análise e Performance

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar performance como comportamento medido;
- diferenciar sintoma, hipótese e gargalo;
- reconhecer métricas iniciais de performance;
- evitar otimização antes de diagnóstico.

## Onde Esta Aula Entra no Curso

## Ideia Central

Performance não é uma sensação genérica de rapidez. Ela precisa ser analisada a partir de sintomas observaveis: latência alta, timeout, uso excessivo de CPU, consumo de memória, query lenta, fila crescendo ou experiência ruim para o usuário.

Antes de otimizar, você precisa medir, formular hipóteses e comparar depois da mudança. Sem isso, performance vira palpite.

## Tópicos da Aula

- Sintoma de performance.
- Latência e throughput.
- CPU, memória e I/O em alto nível.
- Banco e N+1 queries.
- Hipótese e medição.
- Comparação antes e depois.

## Exemplo Prático

```text
Sintoma: GET /lessons demora 1800 ms.

Hipóteses:
- query sem indice;
- N+1 queries para tags;
- serialização grande;
- dependência externa lenta;
- event loop bloqueado.

Próximo passo: coletar dados para confirmar ou descartar cada hipótese.
```

## Analogias e Intuições

- Análise de performance é investigação, não adivinhação.
- Otimizar sem medir é trocar peça sem abrir o diagnóstico.

## Erros Comuns

- Otimizar o trecho errado.
- Medir apenas com poucos dados locais.
- Confundir latência média com experiência real.
- Ignorar banco de dados.
- Culpar framework antes de olhar evidências.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual sintoma está sendo observado?
- Que dado prova que existe problema?
- Quais hipóteses explicam o sintoma?
- Como medir cada hipótese?
- Como comparar antes e depois?

## Prática Recomendada

- Escolha um endpoint lento imaginário.
- Liste cinco hipóteses de gargalo.
- Defina uma medição para cada hipótese.
- Escolha uma melhoria e diga como validar resultado.
