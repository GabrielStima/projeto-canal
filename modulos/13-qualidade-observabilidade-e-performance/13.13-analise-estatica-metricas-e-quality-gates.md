# Análise Estática, Métricas e Quality Gates

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que a análise estática consegue verificar sem executar o sistema;
- interpretar métricas de cobertura, complexidade, duplicação e manutenibilidade;

- diferenciar qualidade mensurável de qualidade real;
- reconhecer quando uma decisão ainda depende de revisão humana.

## Ideia Central

Parte da qualidade de um código pode ser analisada automaticamente. Ferramentas conseguem encontrar padrões perigosos, vulnerabilidades conhecidas, duplicação, complexidade excessiva, baixa cobertura e violações de regras antes que uma mudança seja integrada.

Um quality gate reúne esses resultados em condições objetivas para aprovar ou reprovar uma mudança. Isso reduz dependência de verificações manuais repetitivas, mas não prova que o código resolve o problema certo, tem uma boa arquitetura ou é fácil de entender.

## Exemplo Prático

```text
Quality gate conceitual para código novo:

1. nenhum problema crítico novo;
2. cobertura mínima de 80%;
3. duplicação abaixo de 3%;
4. nenhuma vulnerabilidade de alta severidade;
5. todos os checks obrigatórios aprovados.

Resultado:
- passou: a mudança pode seguir no fluxo;
- falhou: a mudança precisa ser corrigida ou receber uma exceção justificada.
```

Os números são exemplos, não regras universais. Uma política deve considerar risco, contexto, maturidade do sistema e custo de manutenção.

## Analogias e Intuições

- Um quality gate funciona como uma catraca: verifica condições objetivas antes de permitir a passagem.
- Métricas são sinais do sistema, não uma avaliação completa da qualidade.
- Automatizar uma regra evita esquecimento, mas uma regra ruim continua sendo uma regra ruim.

## Erros Comuns

- Tratar cobertura alta como prova de bons testes.
- Otimizar métricas sem melhorar o código.
- Bloquear toda mudança com thresholds arbitrários.
- Aplicar a mesma política a qualquer projeto.
- Ignorar falsos positivos sem revisar as regras.
- Usar quality gates como substitutos de code review.
- Medir apenas o código inteiro e impedir melhorias graduais em sistemas legados.

## Prática Recomendada

- Explique por que cada condição deveria bloquear ou apenas alertar.
- Compare uma política para código novo com uma política para todo o projeto.
- Analise um caso em que uma métrica passa, mas o código continua ruim.
- Desenhe um processo para aprovar exceções sem desativar a política.
