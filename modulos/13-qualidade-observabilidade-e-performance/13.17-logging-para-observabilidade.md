# Logging para Observabilidade

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar logs como sinal de observabilidade;
- diferenciar log útil de ruído;
- escolher eventos relevantes para registrar;
- incluir contexto sem vazar dados sensíveis.

## Onde Esta Aula Entra na Formação

- Módulo: 13. Qualidade, Testes, Observabilidade é Performance
- Aula anterior: Testes de Carga

- Pré-requisitos: Logging no backend, erros padronizados, autenticação, segurança em APIs, dados sensíveis, failure modes e análise de performance.

## Ideia Central

Logging para observabilidade não é espalhar mensagens aleatórias no código. É registrar eventos que ajudam a entender o comportamento do sistema quando algo acontece longe dos seus olhos.

Bons logs têm nível, evento, contexto de requisição, identificadores relevantes, duração e cuidado com dados sensíveis. Eles ajudam a investigar falhas, auditoria e comportamento operacional.

## Tópicos da Aula

- Log como sinal.
- Evento relevante.
- Nível de log.
- Contexto de requisição.
- Logs estruturados.
- Dados sensíveis.

## Exemplo Prático

```json
{
  "level": "warn",
  "event": "progress_update_denied",
  "requestId": "req_123",
  "userId": "user_1",
  "targetUserId": "user_2",
  "reason": "forbidden"
}
```

O log ajuda a investigar uma tentativa negada sem registrar senha, token ou dados desnecessários.

## Analogias e Intuições

- Log útil é uma pista com contexto.
- Log demais sem critério vira neblina operacional.

## Erros Comuns

- Registrar dados sensíveis.
- Logar apenas erros inesperados.
- Esquecer requestId ou correlação.
- Usar mensagens soltas impossíveis de filtrar.
- Criar logs que não respondem perguntas de investigação.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que evento merece log?
- Qual nível faz sentido?
- Que contexto ajuda a investigar?
- Que dado não deve aparecer?
- Como esse log seria filtrado depois?

## Prática Recomendada

- Escolha três endpoints.
- Liste eventos de sucesso, aviso e erro.
- Escreva um log estruturado para cada endpoint.
- Remova qualquer dado sensível ou inútil.
