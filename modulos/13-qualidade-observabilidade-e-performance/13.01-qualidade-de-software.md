# Qualidade de Software

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar qualidade de software em termos práticos;
- diferenciar qualidade percebida, interna e operacional;
- relacionar qualidade com risco, manutenção e confiabilidade;
- reconhecer decisões que aumentam ou reduzem qualidade.

## Onde Esta Aula Entra no Curso

- Módulo: 13. Qualidade, Testes, Observabilidade é Performance
- Aula anterior: Qualidade, Testes, Observabilidade é Performance
- Próxima aula: Estratégia de Testes
- Pré-requisitos: Projeto prático de backend, API design, segurança, validação, erros padronizados, logging e failure modes.

## Ideia Central

Qualidade de software não é uma propriedade única. Uma aplicação pode parecer bonita, mas ser difícil de testar. Pode ter muitos testes, mas falhar em produção sem deixar pistas. Pode ser rápida localmente, mas lenta com dados reais.

Neste módulo, qualidade significa reduzir risco de forma consciente: comportamento correto, código modificável, feedback rápido, falhas investigáveis e performance medida.

## Tópicos da Aula

- Qualidade externa e interna.
- Qualidade operacional.
- Risco e impacto.
- Manutenção e evolução.
- Critérios de aceite como base.
- Trade-offs de qualidade.

## Exemplo Prático

```text
Funcionalidade: login

Qualidade envolve:
- aceitar credenciais validas;
- rejeitar credenciais invalidas;
- não vazar detalhes sensíveis;
- registrar tentativa relevante;
- responder em tempo aceitável;
- permitir mudanças futuras sem quebrar o fluxo.
```

## Analogias e Intuições

- Qualidade é a diferença entre funcionar uma vez e continuar funcionando quando muda.
- Um sistema confiável deixa pistas quando algo da errado.

## Erros Comuns

- Confundir qualidade com ausência momentanea de bugs.
- Tratar teste como a única prática de qualidade.
- Ignorar experiência do usuário em cenários de erro.
- Melhorar código sem entender risco real.
- Adiar qualidade para o fim do projeto.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual problema esta qualidade tentando reduzir?
- Quem percebe a falha?
- Como a falha seria detectada?
- Como a falha seria investigada?
- O código permite mudança segura?

## Prática Recomendada

- Escolha três funcionalidades do projeto.
- Para cada uma, liste uma qualidade externa, uma interna e uma operacional.
- Liste um risco que um teste poderia reduzir.
- Liste um risco que observabilidade poderia reduzir.
