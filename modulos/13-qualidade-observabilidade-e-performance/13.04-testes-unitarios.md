# Testes Unitários

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que um teste unitário valida;
- escolher regras pequenas para testar;
- escrever cenários de entrada, processamento e saída;
- evitar testes acoplados a detalhes internos.

## Onde Esta Aula Entra na Formação

## Ideia Central

Teste unitário valida uma unidade pequena de comportamento. Em TypeScript, isso costuma ser uma função, método ou regra de domínio que pode ser exercitada com entradas controladas e saídas esperadas.

O melhor teste unitário é fácil de entender, rápido de rodar e claro quando falha. Ele deve proteger comportamento, não congelar a implementação.

## Tópicos da Aula

- Unidade de comportamento.
- Arrange, act, assert.
- Casos de sucesso e erro.
- Funções puras.
- Regras de domínio.
- Testes frágeis.

## Exemplo Prático

```ts
type User = { id: string; role: "student" | "admin" };

function canEditUser(currentUser: User, targetUserId: string) {
  return currentUser.role === "admin" || currentUser.id === targetUserId;
}
```

Essa regra pode ser testada sem servidor, banco ou framework.

## Analogias e Intuições

- Teste unitário examina uma peça fora da máquina.
- Quanto menor o teste, mais precisa deve ser a pergunta.

## Erros Comuns

- Testar implementação em vez de comportamento.
- Criar um teste por linha de código.
- Ignorar casos negativos.
- Usar banco real para regra que poderia ser pura.
- Criar nomes de teste que não explicam o cenário.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual comportamento está sendo testado?
- Quais entradas importam?
- Qual saída ou erro é esperado?
- O teste roda rápido?
- O teste continuaria valido após refatoração interna?

## Prática Recomendada

- Escolha uma regra de autorização.
- Liste três cenários: permitido, negado e papel especial.
- Escreva os nomes dos testes antes do código.
- Separe regra pura de acesso a banco.
