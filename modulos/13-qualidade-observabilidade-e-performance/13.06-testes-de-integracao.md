# Testes de Integração

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que um teste de integração valida;
- escolher integrações importantes para testar;
- diferenciar integração real de teste unitário com mock;
- reconhecer cuidados com banco, estado e isolamento.

## Onde Esta Aula Entra no Curso

- Módulo: 13. Qualidade, Testes, Observabilidade é Performance
- Aula anterior: Mocks, Stubs e Fakes
- Próxima aula: Contract Tests
- Pré-requisitos: Testes unitários, doubles de teste, backend com Node.js, APIs, banco de dados, ORMs, transactions e erros padronizados.

## Ideia Central

Teste de integração verifica se partes do sistema trabalham juntas. Em uma API, isso pode envolver rota, validação, regra de negócio, banco de dados e formato de resposta.

Ele custa mais que um teste unitário, mas oferece uma confiança diferente: não apenas que a regra isolada funciona, mas que as pecas realmente se conectam.

## Tópicos da Aula

- Fronteiras de integração.
- API e banco.
- Estado inicial e limpeza.
- Dados de teste.
- Fluxos de erro.
- Custo e confiança.

## Exemplo Prático

```text
Teste: PATCH /progress/:lessonId

Preparar:
- usuário autenticado;
- aula existente;
- progresso inicial.

Executar:
- chamar endpoint.

Verificar:
- status code;
- corpo da resposta;
- registro atualizado no banco.
```

## Analogias e Intuições

- Teste unitário verifica pecas; integração verifica encaixes.
- Um encaixe ruim pode quebrar mesmo quando cada peça parece correta.

## Erros Comuns

- Chamar qualquer teste com banco de "unitário".
- Compartilhar estado entre testes.
- Testar apenas sucesso e ignorar erro.
- Usar dados irreais demais.
- Criar integração tão ampla que vira E2E lento.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quais partes estão sendo integradas?
- Qual estado o teste precisa preparar?
- Como o teste se isola dos demais?
- Que falha de contrato interno ele detecta?
- O custo desse teste vale a confiança que ele traz?

## Prática Recomendada

- Escolha um endpoint com banco.
- Escreva um cenário de sucesso e dois de erro.
- Defina dados iniciais e resultado esperado.
- Liste o que deve ser limpo ou resetado depois do teste.
