# SCA

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é Software Composition Analysis;
- entender como SCA amplia dependency scanning;
- reconhecer riscos de vulnerabilidade, licença, manutenção e procedência;
- usar SCA para priorizar ação em componentes de terceiros.

- Módulo: 21. Supply Chain Security e Secure SDLC

## Ideia Central

SCA significa Software Composition Analysis. A ideia é analisar a composição do software: quais componentes de terceiros existem, quais versões estão em uso, quais vulnerabilidades são conhecidas, quais licenças aparecem e quais riscos de manutenção existem.

Dependency scanning costuma focar vulnerabilidades conhecidas. SCA olha de forma mais ampla para a dependência como parte da composição do produto.

Em aplicações modernas, software não é só o código que o time escreveu. É também o conjunto de bibliotecas, pacotes, imagens, ferramentas e componentes que entram no artefato final.

## Tópicos da Aula

- Composição de software.
- Vulnerabilidades conhecidas.
- Licenças e obrigações.
- Pacotes abandonados ou pouco mantidos.
- Dependências diretas e transitivas.
- Priorização por impacto.

## Exemplo Prático

Software Composition Analysis (SCA):

SCA vai além do Dependency Scan.
O PetCare instalou um gerador PDF no NPM. O Scanners de SCA além de procurar se o gerador tem falhas hackers, ele audita a LICENÇA.
Ele grita no Pipeline: *'O pacote de PDF usa Licença GPLv3 (Copyleft). Se você lançar o PetCare OS vendendo na internet, a licença força o código inteiro do PetCare OS a virar Open-Source Gratuito legalmente'*. O advogado agradece e bloqueia o pacote.

## Analogias e Intuições

- O SCA é a união do Inspetor Sanitário com o Advogado. O sanitário vê se o milho do bolo não tá podre. O advogado vê se você roubou a patente da marca do milho.

## Erros Comuns

- Achar que SCA é só lista de CVEs.
- Ignorar licença porque o projeto é técnico.
- Tratar dependência de desenvolvimento como sempre inofensiva.
- Não saber quais componentes entram no artefato final.
- Confundir popularidade com segurança.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- O que compõe o software além do código próprio?
- Como SCA se relaciona com dependency scanning?
- Por que licença também é risco?
- Como manutenção de pacote afeta segurança?
- Como priorizar achados de SCA?

## Prática Recomendada

- Diferencie drasticamente uma licença Permissiva (Ex: MIT, Apache 2.0) de uma licença Copyleft agressiva/viral (Ex: GPLv3) do ponto de vista de Risco de Compliance (SCA).
