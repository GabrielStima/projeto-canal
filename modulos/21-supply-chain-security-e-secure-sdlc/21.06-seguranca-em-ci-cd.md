# Segurança em CI/CD

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer o pipeline como superfície de ataque;
- identificar riscos em jobs, runners, secrets, branches e artefatos;
- propor controles básicos para pipelines de build e deploy;
- relacionar CI/CD seguro com menor privilégio e ambientes.

## Ideia Central

CI/CD automatiza tarefas poderosas: instala dependências, executa scripts, acessa secrets, constrói artefatos, publica imagens e pode fazer deploy.

Por isso, pipeline não é só infraestrutura de produtividade. Ele é uma parte crítica da segurança. Um atacante que controla um job, uma dependência ou uma permissão do pipeline pode conseguir acesso maior do que teria explorando uma rota da aplicação.

Segurança em CI/CD significa controlar quem dispara o pipeline, o que cada etapa pode acessar, quais artefatos são produzidos e quando uma mudança pode chegar em ambiente sensível.

## Tópicos da Aula

- Pipeline como superfície de ataque.
- Runners e ambiente de execução.
- Secrets em pull requests e branches.
- Permissões por job.
- Artefatos, logs e cache.
- Deploy protegido.

## Exemplo Prático

A Corrupção da Esteira:

O Dev comete um erro num Pull Request. O Robô do Github Actions pega o código, testa e faz Deploy.
O Atacante não foca na API. Ele tenta hackear o *Github Actions*.
O atacante edita o YAML do pipeline (`.github/workflows`) no repositório de testes, manda rodar e pede pro Github imprimir as Senhas Secretas (Vault/OIDC) nos logs abertos.
Se o Github Actions não for blindado, o Atacante domina a máquina inteira antes mesmo da API nascer.

## Analogias e Intuições

- A esteira de CI/CD é o caminhão blindado que leva o dinheiro do banco. A API é o banco. É mais fácil assaltar o caminhão no meio da estrada (CI/CD) do que explodir o cofre na agência (Produção).

## Erros Comuns

- Dar secrets de produção para todos os jobs.
- Confiar em código de pull request como se já estivesse revisado.
- Reutilizar cache sem pensar em contaminação.
- Publicar artefatos sem rastrear origem.
- Fazer deploy automático sem proteção de ambiente.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quem pode disparar cada pipeline?
- Quais jobs acessam secrets?
- Quais branches podem publicar ou fazer deploy?
- Quais artefatos são gerados e onde ficam?
- O que acontece se uma dependência executar código durante instalação?
