# DAST

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é DAST;
- diferenciar DAST de SAST;
- entender por que DAST precisa de uma aplicação em execução;
- usar DAST como parte de uma estratégia, não como garantia absoluta.

## Ideia Central

DAST significa Dynamic Application Security Testing. Diferente do SAST, que olha para código estático, DAST testa uma aplicação rodando.

Ele tenta observar comportamentos perigosos a partir de fora: headers ausentes, endpoints expostos, respostas inseguras, falhas de configuração, entradas mal tratadas e problemas que aparecem apenas em runtime.

DAST é útil, mas tem limites. Ele enxerga o que consegue alcançar. Se o ambiente, autenticação ou dados de teste forem ruins, os resultados também serão limitados.

## Exemplo Prático

Dynamic Application Security Testing (DAST):

O SAST não achou erro lendo o texto do código. Tudo parece bem.
Na esteira de homologação, o sistema DAST (por exemplo, OWASP ZAP) inicia a API real da clínica e envia milhares de requisições HTTP malformadas e conhecidamente perigosas para detectar vulnerabilidades.
O DAST pode descobrir, por exemplo, que inserir `<script>alert('hack')</script>` no campo nome do pet causa uma falha de XSS na tela principal.
Ele encontra vulnerabilidades apenas quando a aplicação está em execução.

## Analogias e Intuições

- O SAST analisa o código-fonte procurando padrões inseguros. O DAST testa a aplicação em execução, enviando entradas maliciosas para descobrir vulnerabilidades reais expostas na interface.

## Erros Comuns

- Rodar DAST contra produção sem planejamento.
- Achar que DAST cobre todo o código.
- Ignorar autenticação e testar só rotas públicas.
- Bloquear deploy por alerta sem triagem.
- Usar ambiente de teste diferente demais de produção.
