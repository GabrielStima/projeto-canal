# Projeto Pratico: Operando uma API em Producao

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- aplicar os conceitos do modulo em um cenario completo;
- definir sinais, objetivos, alertas e procedimentos operacionais;
- simular resposta a incidente e postmortem;
- saber o que revisar antes de avancar.

## Onde Esta Aula Entra no Curso

- Modulo: 20. SRE, Operacao e Incidentes
- Aula anterior: Postmortems
- Proxima aula: Supply Chain Security e Secure SDLC
- Pre-requisitos: Todas as aulas anteriores do modulo 20, observabilidade, cloud, CI/CD, containers e Kubernetes.

## Ideia Central

Este projeto junta o modulo inteiro em uma situacao pratica: uma API que precisa ser operada como se estivesse em producao.

O objetivo nao e criar uma infraestrutura enorme. O objetivo e praticar pensamento operacional: quais fluxos importam, quais sinais mostram saude, quais objetivos sao realistas, como responder quando algo falha e como aprender depois.

Ao final, voce deve ter um pequeno pacote de operacao: SLIs, SLOs, alertas, probes, plano de rollback, runbook, simulacao de incidente e postmortem.

## Topicos da Aula

- Escolha de fluxo critico
- Definicao de SLIs e SLOs
- Alertas acionaveis
- Readiness, liveness e degradacao
- Rollback e runbook
- Simulacao de incidente
- Postmortem e acoes de melhoria

## Exemplo Pratico

Use uma API simples, como uma API de estudos com login, listagem de aulas e registro de progresso.

Durante o projeto:

1. escolha o fluxo mais importante;
2. defina SLIs de sucesso, latencia e erro;
3. proponha SLO para 30 dias;
4. descreva o error budget;
5. defina quando alertar;
6. crie criterios de readiness e liveness;
7. simule uma nova versao com aumento de erro;
8. decida rollback ou degradacao;
9. escreva um runbook;
10. produza um postmortem curto.

## Analogias e Intuicoes

- O projeto e um ensaio controlado antes de enfrentar producao real.
- Operar bem e conectar sinais, decisoes e aprendizado.

## Erros Comuns

- Escolher metricas demais e nao saber agir sobre nenhuma.
- Criar SLO sem fluxo critico claro.
- Simular incidente sem registrar linha do tempo.
- Escrever runbook depois do postmortem sem validar se ele ajuda.
- Tratar o projeto como checklist burocratico, nao como treino de decisao.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quais fluxos da API sao mais importantes?
- Quais SLIs representam experiencia real?
- Qual SLO seria realista para o fluxo escolhido?
- Quando o time deveria fazer rollback?
- O que o postmortem ensinou sobre o sistema?

## Pratica Recomendada

- Monte uma tabela com fluxo, SLI, SLO, alerta e acao esperada.
- Escreva um runbook para o principal alerta.
- Simule um incidente em texto, com horario, sinais, decisao, mitigacao e resultado.
- Escreva um postmortem com no maximo uma pagina.
