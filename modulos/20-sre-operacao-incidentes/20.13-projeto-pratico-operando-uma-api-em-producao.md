# Projeto Prático: Operando uma API em Produção

## Objetivo

Ao final desta aula, você deve ser capaz de:

- aplicar os conceitos do módulo em um cenário completo;
- definir sinais, objetivos, alertas e procedimentos operacionais;
- simular resposta a incidente e postmortem;
- saber o que revisar antes de avançar.

## Onde Esta Aula Entra na Formação

- Módulo: 20. SRE, Operação e Incidentes
- Aula anterior: Postmortems
- Próxima aula: Supply Chain Security e Secure SDLC
- Pré-requisitos: Todas as aulas anteriores do módulo 20, observabilidade, cloud, CI/CD, containers e Kubernetes.

## Ideia Central

Este projeto junta o módulo inteiro em uma situação prática: uma API que precisa ser operada como se estivesse em produção.

O objetivo não é criar uma infraestrutura enorme. O objetivo é praticar pensamento operacional: quais fluxos importam, quais sinais mostram saúde, quais objetivos são realistas, como responder quando algo falha e como aprender depois.

Ao final, você deve ter um pequeno pacote de operação: SLIs, SLOs, alertas, probes, plano de rollback, runbook, simulação de incidente e postmortem.

## Tópicos da Aula

- Escolha de fluxo crítico
- Definição de SLIs e SLOs
- Alertas acionáveis
- Readiness, liveness e degradação
- Rollback e runbook
- Simulação de incidente
- Postmortem e ações de melhoria

## Exemplo Prático

Use uma API simples, como uma API de estudos com login, listagem de aulas e registro de progresso.

Durante o projeto:

1. escolha o fluxo mais importante;
2. defina SLIs de sucesso, latência e erro;
3. proponha SLO para 30 dias;
4. descreva o error budget;
5. defina quando alertar;
6. crie critérios de readiness e liveness;
7. simule uma nova versão com aumento de erro;
8. decida rollback ou degradação;
9. escreva um runbook;
10. produza um postmortem curto.

## Analogias e Intuições

- O projeto é um ensaio controlado antes de enfrentar a produção real.
- Operar bem é conectar sinais, decisões e aprendizado.

## Erros Comuns

- Escolher métricas demais e não saber agir sobre nenhuma.
- Criar SLO sem fluxo crítico claro.
- Simular incidente sem registrar linha do tempo.
- Escrever runbook depois do postmortem sem validar se ele ajuda.
- Tratar o projeto como checklist burocrático, não como treino de decisão.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quais fluxos da API são mais importantes?
- Quais SLIs representam experiência real?
- Qual SLO seria realista para o fluxo escolhido?
- Quando o time deveria fazer rollback?
- O que o postmortem ensinou sobre o sistema?

## Prática Recomendada

- Monte uma tabela com fluxo, SLI, SLO, alerta e ação esperada.
- Escreva um runbook para o principal alerta.
- Simule um incidente em texto, com horário, sinais, decisão, mitigação e resultado.
- Escreva um postmortem com no máximo uma página.
