# Degradação Graciosa

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar degradação graciosa;
- reconhecer quando entregar valor parcial é melhor que falhar completamente;
- propor estratégias simples de degradação em uma aplicação;
- saber o que estudar em seguida.

## Onde Esta Aula Entra na Formação

- Módulo: 20. SRE, Operação e Incidentes
- Aula anterior: Readiness e Liveness
- Próxima aula: Rollback
- Pré-requisitos: Failure modes, APIs, filas, dependências externas, cache, feature flags, observabilidade e confiabilidade.

## Ideia Central

Degradação graciosa é a capacidade de um sistema continuar oferecendo parte do valor quando alguma dependência, recurso ou funcionalidade falha.

Em vez de responder "tudo ou nada", o sistema pode reduzir escopo: esconder um recurso secundário, usar cache, enfileirar trabalho, limitar uma operação pesada, desligar uma feature temporariamente ou mostrar uma mensagem honesta para o usuário.

O ponto não é mascarar problema. O ponto é reduzir dano enquanto a equipe investiga, mitiga ou recupera o sistema.

## Tópicos da Aula

- Falha total versus valor parcial
- Dependências críticas e secundárias
- Cache, fallback e filas
- Feature flags e kill switches
- Comunicação com o usuário durante degradação

## Exemplo Prático

Uma plataforma de estudos usa uma API externa para gerar sugestões personalizadas. Se essa API externa falha, o sistema ainda pode:

- mostrar a lista de aulas normalmente;
- esconder a sugestão personalizada;
- usar uma sugestão em cache;
- registrar o erro com contexto;
- avisar que recomendações estão temporariamente indisponíveis.

Assim, a parte principal do produto continua funcionando.

## Analogias e Intuições

- Degradar graciosamente é manter o essencial funcionando quando o ideal não está disponível.
- Nem toda falha precisa virar tela branca.

## Erros Comuns

- Tratar dependência secundária como obrigatória.
- Criar fallback que retorna dado incorreto sem aviso.
- Não medir quando o sistema está em modo degradado.
- Esconder falhas a ponto de ninguém investigar.
- Degradar funcionalidade crítica sem critério.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- O que é degradação graciosa?
- Quando ela ajuda o usuário?
- Quando ela pode ser perigosa?
- Qual a diferença entre fallback e mascarar erro?
- Como feature flags podem ajudar em incidentes?

## Prática Recomendada

- Escolha uma funcionalidade com dependência externa e proponha um fallback.
- Liste quais sinais indicariam que o sistema está degradado.
- Escreva uma mensagem simples para o usuário em caso de recurso temporariamente indisponível.
