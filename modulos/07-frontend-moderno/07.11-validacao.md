# Validação

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que validar entradas no frontend;
- diferenciar validação no cliente e no servidor;
- mostrar mensagens de erro úteis;
- criar validações simples com TypeScript e estado.

## Ideia Central

Validação no frontend melhora a experiência do usuário, mas não substitui validação no servidor. O browser pode avisar rapidamente que um campo está vazio, que um email parece invalido ou que uma senha não atende a critérios mínimos.

O servidor ainda precisa validar de novo, porque qualquer pessoa pode alterar ou ignorar regras do frontend.

Boas mensagens de erro explicam o problema de forma clara e aparecem perto do campo relacionado.

## Exemplo Prático

```tsx
function validarEmail(email: string) {
  if (!email) return "Email é obrigatório.";
  if (!email.includes("@")) return "Informe um email válido.";
  return "";
}

function validarNomePet(nome: string) {
  if (!nome) return "O nome do pet é obrigatório.";
  if (nome.length < 2) return "O nome deve ter pelo menos 2 caracteres.";
  return "";
}

Essas funções não resolvem todos os casos possíveis. Elas mostram o princípio: regra clara, retorno claro e mensagem útil para a UI. No PetCare OS, validação aparece no cadastro de pet, no formulário de triagem e na aprovação de orçamento.

## Analogias e Intuições

- Validação no frontend é um aviso rápido na porta.
- Validação no servidor é a verificação oficial antes de aceitar os dados.

## Erros Comuns

- Confiar apenas na validação do cliente.
- Mostrar mensagens genéricas como "erro".
- Validar tarde demais e deixar o usuário perdido.
- Criar regras tão rígidas que bloqueiam entradas validas.
