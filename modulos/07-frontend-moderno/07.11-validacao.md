# Validacao

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar por que validar entradas no frontend;
- diferenciar validacao no cliente e no servidor;
- mostrar mensagens de erro uteis;
- criar validacoes simples com TypeScript e estado.

## Onde Esta Aula Entra no Curso

- Modulo: 06. Frontend Moderno
- Aula anterior: Formularios no frontend moderno
- Proxima aula: Testes em frontend
- Pre-requisitos: Formularios HTML, React, estado, eventos, TypeScript e chamadas de API.

## Ideia Central

Validacao no frontend melhora a experiencia do usuario, mas nao substitui validacao no servidor. O browser pode avisar rapidamente que um campo esta vazio, que um email parece invalido ou que uma senha nao atende a criterios minimos.

O servidor ainda precisa validar de novo, porque qualquer pessoa pode alterar ou ignorar regras do frontend.

Boas mensagens de erro explicam o problema de forma clara e aparecem perto do campo relacionado.

## Topicos da Aula

- Tipos para representar dados validos e invalidos.

## Exemplo Pratico

```tsx
function validarEmail(email: string) {
  if (!email) return "Email e obrigatorio.";
  if (!email.includes("@")) return "Informe um email valido.";
  return "";
}
```

Essa funcao nao resolve todos os casos possiveis de email. Ela mostra o principio: regra clara, retorno claro e mensagem util para a UI.

## Analogias e Intuicoes

- Validacao no frontend e um aviso rapido na porta.
- Validacao no servidor e a verificacao oficial antes de aceitar os dados.

## Erros Comuns

- Confiar apenas na validacao do cliente.
- Mostrar mensagens genericas como "erro".
- Validar tarde demais e deixar o usuario perdido.
- Criar regras tao rigidas que bloqueiam entradas validas.

## Checklist de Entendimento

Antes de seguir, voce deve conseguir responder:

- Por que validar no frontend?
- Por que o servidor precisa validar tambem?
- O que torna uma mensagem de erro boa?
- Quando usar validacao HTML nativa?
- Como representar erros no estado da UI?
