# REST

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar REST em nível prático;
- pensar APIs em recursos;
- usar métodos HTTP com intenção;
- reconhecer limites entre introducao a REST e API design profissional.

## Onde Esta Aula Entra no Curso

- Módulo: 10. Backend com Node.js
- Aula anterior: JSON APIs
- Próxima aula: SOAP
- Pré-requisitos: HTTP, APIs, JSON APIs, status codes e servidor backend.

## Ideia Central

REST é um estilo arquitetural para desenhar sistemas distribuídos. Em APIs HTTP do dia a dia, o uso prático costuma aparecer na ideia de recursos, URLs estáveis, métodos HTTP e respostas padronizadas.

Nesta aula, o foco e a base: entender que `GET /aulas` representa uma coleção, `GET /aulas/10.01` representa um recurso específico e métodos como `POST`, `PUT`, `PATCH` e `DELETE` comunicam intenção. O aprofundamento em versionamento, paginação, filtros, erros e governança fica para o módulo 11.

## Tópicos da Aula

- Recurso.
- Coleção e item.
- Metodos HTTP.
- Status codes comuns.
- Corpo de requisição e resposta.
- REST na prática vs REST formal.
- Limites deste módulo.

## Exemplo Prático

```text
GET /aulas
GET /aulas/10.01
POST /aulas
PATCH /aulas/10.01
DELETE /aulas/10.01
```

Esses caminhos falam sobre recursos do dominio, não sobre nomes de funções internas do backend.

## Analogias e Intuições

- Recursos são coisas que o sistema conhece.
- Metodos HTTP são verbos com significado compartilhado.

## Erros Comuns

- Usar sempre `POST` para tudo.
- Colocar verbo desnecessario no caminho, como `/createLesson`.
- Retornar `200` para erro de validação.
- Pensar endpoints diretamente como tabelas do banco.
- Tentar resolver todo API design em uma aula introdutória.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O que é recurso?
- Qual a diferença entre coleção e item?
- Quando usar `GET`?
- Quando usar `POST`?
- Por que REST não é só nome de URL?

## Prática Recomendada

- Pegue uma entidade do painel de estudos.
- Desenhe cinco endpoints REST iniciais.
- Escolha métodos HTTP para cada endpoint.
- Defina status codes de sucesso e erro.
- Liste o que ainda precisaria ser aprofundado no módulo 11.
