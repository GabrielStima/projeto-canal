# Profiling Performance

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é profiling;
- diferenciar chute de medição;
- identificar gargalos simples em backend;
- entender a relação entre performance, banco, event loop e logs.

## Onde Esta Aula Entra no Curso

- Módulo: 10. Backend com Node.js
- Aula anterior: Threads e Workers
- Próxima aula: Message Brokers

## Ideia Central

Profiling é observar uma aplicação para descobrir onde tempo, CPU, memória ou recursos estão sendo consumidos. Sem medição, performance vira opinião. Com medição, você pode comparar antes e depois de uma mudança.

No backend, gargalos podem estar em query lenta, processamento pesado, serializacao de JSON, dependência externa, excesso de logs, falta de índice, N+1 queries ou bloqueio do event loop.

## Tópicos da Aula

- O que é profiling.
- Latencia.
- CPU e memória em alto nível.
- Query lenta.
- N+1 queries.
- Event loop bloqueado.
- Medir antes de otimizar.

## Exemplo Prático

```text
Endpoint lento:
- tempo total: 1200 ms;
- validação: 5 ms;
- query de aulas: 80 ms;
- query de tags repetida 100 vezes: 1000 ms;
- montagem da resposta: 20 ms.

Gargalo provavel: N+1 queries.
```

## Analogias e Intuições

- Profiling é acender a luz antes de mexer nos móveis.
- Performance boa nasce de medição, não de palpite.

## Erros Comuns

- Otimizar antes de medir.
- Culpar Node.js sem olhar banco.
- Ignorar queries repetidas.
- Medir apenas no ambiente local com poucos dados.
- Confundir rapidez percebida com ausencia de gargalo.
