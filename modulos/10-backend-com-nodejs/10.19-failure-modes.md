# Failure Modes

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que são failure modes;
- listar falhas comuns em APIs backend;
- pensar em timeout, indisponibilidade e dados inválidos;
- preparar o estudo de resiliência, observabilidade e testes nos módulos seguintes.

## Onde Esta Aula Entra na Formação

- Módulo: 10. Backend com Node.js
- Aula anterior: ORMs
- Próxima aula: Threads e Workers
- Pré-requisitos: APIs, banco de dados, logging, tratamento de erros e ORMs.

## Ideia Central

Failure mode é uma forma pela qual um sistema pode falhar. Em backend, falhas não aparecem apenas como bug de código. Banco pode ficar indisponível, uma query pode demorar, um serviço externo pode responder erro, uma requisição pode vir malformada, uma operação pode ser repetida e uma dependência pode devolver dados inesperados.

Pensar em failure modes muda a postura: em vez de perguntar "funciona?", você pergunta "como isso falha e o que acontece quando falha?".

## Tópicos da Aula

- Falhas de entrada.

- Relação com logs.

## Exemplo Prático

```text
Endpoint: POST /progress

Possíveis falhas:
- usuário não existe;
- aula não existe;
- banco indisponível;
- requisição duplicada;
- payload invalido;
- timeout na escrita;
- erro inesperado no servidor.
```

## Analogias e Intuições

- Failure modes são perguntas de "e se?" aplicadas ao sistema.
- Um backend maduro não evita toda falha, mas falha de forma mais previsível.

## Erros Comuns

- Tratar todo erro como `500`.
- Não configurar timeout.
- Repetir operação sem pensar em duplicidade.
- Esconder erro sem log.
- Expor detalhe interno para o cliente.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O que é failure mode?
- Que falhas comuns existem em APIs?
- Qual a diferença entre erro do cliente e erro do servidor?
- Por que timeout importa?
- Como logs ajudam a investigar falhas?

## Prática Recomendada

- Escolha três endpoints.
- Liste cinco failure modes para cada um.
- Defina resposta esperada para cada falha.
- Defina o que deveria ser logado.
- Identifique quais casos precisam de teste no futuro.
