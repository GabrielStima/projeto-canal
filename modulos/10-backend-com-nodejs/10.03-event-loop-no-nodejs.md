# Event Loop no Node.js

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que o event loop importa no backend;
- diferenciar operações de I/O e trabalho pesado de CPU;
- entender como callbacks, promises e timers entram no fluxo;
- evitar bloquear o processo com código pesado.

## Ideia Central

Node.js consegue lidar bem com muitas operações de entrada e saída porque não precisa ficar parado esperando cada resposta terminar. O event loop coordena tarefas assíncronas e permite que o processo continue trabalhando enquanto espera banco, rede, arquivos ou timers.

Isso não significa que qualquer código será rápido. Se você coloca um cálculo pesado e síncrono na thread principal, o processo fica ocupado e deixa de responder outras requisições.

## Exemplo Prático

```javascript
console.log("início");

setTimeout(() => {
  console.log("timer");
}, 0);

Promise.resolve().then(() => {
  console.log("promise");
});

console.log("fim");
```

O resultado mostra que a ordem escrita no arquivo não é sempre a ordem final de execução das tarefas assíncronas.

## Analogias e Intuições

- O event loop parece uma pessoa organizando fila de tarefas: algumas podem esperar resposta, outras precisam de atencao imediata.
- I/O é como pedir comida e continuar trabalhando; CPU pesada é como cozinhar sozinho sem poder atender ninguém.

## Erros Comuns

- Achar que `async` torna qualquer coisa automaticamente paralela.
- Fazer loops pesados dentro de uma rota.
- Ignorar que JSON grande também pode custar processamento.
- Confundir concorrencia com paralelismo.
- Pensar que Node.js nunca usa threads.
