# Frameworks Backend

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que frameworks backend existem;
- reconhecer tarefas repetitivas de um servidor;
- comparar frameworks em alto nível sem escolher por moda;
- entender que framework não substitui fundamento.

## Onde Esta Aula Entra na Formação

- Módulo: 10. Backend com Node.js
- Aula anterior: Servidor HTTP com Node.js Puro
- Próxima aula: APIs
- Pré-requisitos: Servidor HTTP com Node.js, módulos, HTTP e TypeScript.

## Ideia Central

Frameworks backend existem para organizar e reduzir trabalho repetitivo: rotas, middlewares, tratamento de erros, parsing de corpo, validação, plugins, configuração e integração com outras ferramentas.

Eles aceleram o desenvolvimento, mas também trazem convenções. Antes de escolher um framework, é preciso entender o tamanho do projeto, a equipe, o estilo de arquitetura e o quanto de estrutura você realmente precisa.

## Tópicos da Aula

- O que frameworks abstraem.
- Rotas e handlers.
- Middlewares.
- Plugins e ecossistema.
- Minimalismo vs estrutura.
- Express, Fastify e NestJS em alto nível.

## Exemplo Prático

```text
Sem framework:
- verificar URL manualmente;
- montar resposta manualmente;
- tratar corpo da requisição manualmente;
- repetir validações.

Com framework:
- declarar rotas;
- usar middlewares;
- padronizar erros;
- integrar validação, logs e plugins.
```

## Analogias e Intuições

- Framework é uma oficina organizada: você ainda precisa saber consertar, mas as ferramentas estão no lugar.
- Quanto mais opinativo o framework, mais ele guia seu caminho.

## Erros Comuns

- Escolher framework antes de entender o problema.
- Achar que framework resolve design ruim.
- Usar estrutura pesada para API muito pequena.
- Ignorar convenções do framework escolhido.
- Trocar de framework para fugir de fundamentos.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que tarefas um framework backend facilita?
- O que é middleware?
- O que significa um framework ser opinativo?
- Quando minimalismo ajuda?
- Quando estrutura ajuda?

## Prática Recomendada

- Liste tarefas repetitivas do servidor HTTP puro.
- Compare como dois frameworks atuais lidam com rotas, validação e erros.
- Escolha um deles para a API do módulo e justifique pela documentação, tamanho do projeto e curva de aprendizagem.
- Registre também uma razão para não escolher o outro.
- Seguindo a documentação oficial, recrie somente a rota `/health` no framework escolhido.
- Compare o handler com a versão feita em `node:http`.
- Evite mudar de framework durante os exercícios seguintes sem uma limitação concreta.
