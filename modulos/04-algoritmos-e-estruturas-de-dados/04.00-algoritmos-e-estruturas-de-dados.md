# Algoritmos e Estruturas de Dados

## Objetivo

Ao final desta aula, você deve ser capaz de:
- entender por que estruturas de dados e algoritmos importam na prática, além de entrevistas;
- reconhecer que a escolha de estrutura de dados afeta desempenho e legibilidade do código;
- relacionar complexidade de algoritmos com consequências reais em sistemas;
- saber o que será aprofundado ao longo do módulo.

## Ideia Central

Todo software manipula dados. A forma como esses dados são organizados — e os algoritmos que os processam — determina se um sistema escala para 10 usuários ou para 10 milhões. Escolher um array onde uma hash table seria melhor, ou uma busca linear onde uma busca binária seria possível, são decisões que aparecem no código do dia a dia.

Este módulo desenvolve o raciocínio computacional necessário para fazer essas escolhas conscientemente. Você aprenderá a analisar complexidade com Big-O, a escolher estruturas adequadas ao problema e a implementar algoritmos clássicos entendendo o porquê de cada etapa.

O objetivo não é decorar algoritmos — é desenvolver o instinto de perguntar "qual é a operação mais frequente aqui?" antes de escolher como organizar os dados.

## Tópicos da Aula

- Estruturas de dados: arrays, linked lists, stacks e queues.
- Hash tables: busca em O(1) e colisões.
- Complexidade de algoritmos: Big-O, tempo vs espaço.
- Algoritmos de busca: linear e binária.
- Algoritmos de ordenação: bubble, selection, insertion, merge, quick e heap sort.
- Árvores: árvore binária e árvore de busca binária (BST).
- Heaps e priority queues.
- Recursão e divisão e conquista.

## Analogias e Intuições

- Uma stack é como uma pilha de fichas de atendimento: você sempre pega a de cima. Uma queue é a fila do veterinário: quem chegou primeiro é atendido primeiro.
- Big-O é como prever quanto tempo uma receita leva: dobrar o número de ingredientes (N) pode dobrar o tempo (O(N)) ou quadruplicar (O(N²)), dependendo do preparo.
- Recursão é como espelhos se olhando: cada reflexo é uma versão menor do problema original, até chegar em um reflexo que não precisa mais de espelho.

## Erros Comuns

- Usar Big-O apenas para calcular e não para tomar decisões de implementação.
- Ignorar complexidade de espaço e focar só em tempo de execução.
- Implementar ordenação manual em contextos onde `.sort()` com comparador resolve corretamente.
- Não identificar o caso base em funções recursivas, causando stack overflow.
- Escolher uma linked list por achá-la "mais eficiente" sem considerar que arrays modernos têm boa localidade de cache.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual a diferença entre stack e queue e quando usar cada uma?
- O que O(n log n) significa na prática?
- Por que hash tables têm busca O(1) na maioria dos casos?
- Quando recursão é uma boa escolha e quando pode ser um problema?
- Como uma priority queue difere de uma queue comum?

## Prática Recomendada

- Implemente uma `PriorityQueue<T>` em TypeScript com `enqueue` (com prioridade) e `dequeue`.
- Use-a para simular a fila de triagem do PetCare OS com três níveis de urgência.
- Implemente busca binária para encontrar um paciente em uma lista ordenada por ID.
- Analise a complexidade de cada operação implementada e justifique suas escolhas.
