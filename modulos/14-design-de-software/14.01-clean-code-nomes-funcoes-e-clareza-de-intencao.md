# Clean Code: Nomes, Funções e Clareza de Intenção

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar Clean Code sem tratá-lo como dogma;
- escolher nomes que revelem intenção e vocabulário de domínio;
- dividir funções por decisões e responsabilidades compreensíveis;
- reconhecer fragmentação excessiva e abstrações pequenas sem valor.

- Módulo: 14. Design de Software

## Ideia Central

Clean Code reúne práticas para tornar código compreensível e modificável. Nomes bons reduzem a tradução mental entre o problema e a implementação. Funções focadas ajudam quando cada uma representa uma decisão ou etapa com significado.

Essas práticas não são regras mecânicas. Transformar cada linha em função, usar nomes enormes ou esconder um fluxo simples atrás de várias chamadas pode piorar a leitura. A pergunta central é: a mudança revelou intenção e reduziu surpresa?

## Tópicos da Aula

- Intenção clara e vocabulário do domínio.
- Nomes de ações, conceitos e predicados.
- Funções com uma ideia principal.
- Parâmetros demais como sinal de conceito escondido.
- Comentários que explicam decisões, não sintaxe.
- Tratamento de erros legível.
- Limites e exageros do Clean Code.

## Exemplo Prático

Compare:

```ts
if (p.st === 1 && t > 0) {
  process(p, t)
}
```

com:

```ts
if (patient.isReadyForDischarge() && outstandingBalance > 0) {
  requestPaymentBeforeDischarge(patient, outstandingBalance)
}
```

O segundo trecho revela mais intenção, mas ainda merece perguntas. `requestPaymentBeforeDischarge` representa uma decisão relevante ou apenas esconde uma linha? `Patient` conhece uma regra clínica real ou virou uma classe cerimonial? Clean Code melhora a comunicação quando os nomes refletem limites verdadeiros.

## Analogias e Intuições

**Etiquetas em uma farmácia**

Uma etiqueta precisa identificar o medicamento e reduzir a chance de confusão sob pressão. Nomes no código cumprem papel parecido: orientam decisões rápidas sem exigir reconstruir todo o contexto.

*Limite: nomes não substituem regras, tipos e testes que garantem o comportamento.*

**Etapas de um procedimento**

Separar preparação, execução e registro ajuda a equipe a compreender o fluxo. Dividir uma função é útil quando cada parte representa uma etapa reconhecível, não quando apenas troca uma linha por outra chamada.

*Limite: funções podem compartilhar dados e efeitos de formas que um procedimento físico não representa.*

## Erros Comuns

- Usar nomes genéricos como `data`, `item`, `handle` e `process`, obrigando o leitor a investigar a implementação.
- Criar nomes longos para compensar uma unidade com responsabilidades confusas.
- Extrair funções minúsculas que não acrescentam vocabulário nem reduzem complexidade.
- Remover comentários que registram contexto ou trade-offs porque “código bom não precisa de comentários”.
- Aceitar renomeações sugeridas por IA sem verificar se elas correspondem à linguagem usada pelo domínio.

## Checklist de Entendimento

- O nome revela intenção e resultado esperado?
- A função possui uma ideia principal?
- Os parâmetros formam um conceito que deveria ser modelado?
- O comentário explica por que uma decisão existe?
- A extração melhorou leitura e teste sem esconder o fluxo?
- O vocabulário é consistente com o restante do sistema?
