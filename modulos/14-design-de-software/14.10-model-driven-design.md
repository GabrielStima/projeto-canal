# Model-Driven Design

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar model-driven design em nivel de codigo;
- diferenciar modelo de dominio de modelo de banco;
- usar linguagem do problema nos nomes;
- evitar modelos que so refletem tabela.

## Onde Esta Aula Entra no Curso

- Modulo: 14. Design de Software
- Aula anterior: Acoplamento e Coesao no Codigo
- Proxima aula: Principios de Design de Software
- Pre-requisitos: Requisitos, modelagem de dados, APIs, POO, TypeScript e testes.

## Ideia Central

Model-driven design organiza o codigo em torno dos conceitos do problema. Em vez de deixar framework, banco ou payload mandarem em todos os nomes, o modelo tenta expressar o dominio: matricula, progresso, aula, conclusao, assinatura, permissao.

Isso nao significa criar uma camada complexa. Significa evitar que as regras importantes fiquem escondidas em objetos genericos, tabelas ou handlers.

## Topicos da Aula

- Linguagem do dominio.
- Modelo de dominio vs DTO.
- Modelo de banco vs comportamento.
- Regras perto dos conceitos.
- Limites do modelo.
- Ponte com DDD no modulo 15.

## Exemplo Pratico

```text
Payload:
{ lessonId, userId }

Modelo de dominio:
StudyProgress.completeLesson(lessonId)
```

## Analogias e Intuicoes

- O modelo e o vocabulário que o codigo usa para conversar sobre o problema.
- Quando o dominio some, o codigo vira manipulacao anonima de dados.

## Erros Comuns

- Tratar entidade de banco como modelo completo.
- Criar modelo rico para CRUD simples demais.
- Usar nomes tecnicos onde existe nome de negocio.
- Misturar DTO, entidade, view model e regra sem criterio.

## Checklist de Entendimento

- Quais conceitos existem no problema?
- Que regras pertencem a cada conceito?
- O nome veio do dominio ou da tecnologia?
- O banco esta mandando no design?
- O modelo ajuda a escrever testes?

## Pratica Recomendada

- Escolha um fluxo da API.
- Liste palavras do dominio.
- Separe payload, entidade de banco e modelo de regra.
- Renomeie uma funcao usando linguagem do problema.
