# Paradigmas e Recursos de Linguagem em TypeScript

## Objetivo

Ao final desta aula, você deve ser capaz de:

- comparar os paradigmas procedural, funcional e orientado a objetos;
- separar paradigma de recurso oferecido pela linguagem;
- reconhecer como TypeScript permite combinar estilos;
- escolher funções, objetos, classes, closures e interfaces conforme o problema.

## Ideia Central

Paradigmas são formas de organizar pensamento em código. Eles influenciam onde ficam dados, comportamento, estado e efeitos colaterais. Recursos de linguagem, como funções de primeira classe, closures, classes, interfaces e módulos, são mecanismos usados para expressar esses estilos.

TypeScript é multiparadigma. Uma solução profissional pode usar objetos para proteger invariantes, funções puras para cálculos e módulos para organizar capacidades. O objetivo não é defender uma identidade, mas selecionar restrições que deixem a regra mais clara.

## Exemplo Prático

O cálculo de uma fatura pode ser expresso de formas diferentes:

- procedural: percorre itens e altera um acumulador;
- funcional: transforma os itens e reduz o resultado sem mutar a entrada;
- orientado a objetos: a `Invoice` protege seu estado e expõe operações de domínio.

Uma implementação também pode combinar estilos:

```ts
class Invoice {
  constructor(private readonly items: InvoiceItem[]) {}

  totalFor(category: string): number {
    return this.items
      .filter((item) => item.category === category)
      .reduce((total, item) => total + item.price, 0)
  }
}
```

A classe organiza o conceito e as operações de coleção realizam uma transformação local. Não há conflito em misturar paradigmas quando cada escolha tem função clara.

## Analogias e Intuições

**Formas de organizar uma cozinha**

Uma cozinha pode seguir uma sequência fixa de preparo, dividir trabalho por estações ou organizar profissionais responsáveis por pratos completos. Cada organização favorece tipos diferentes de problema.

*Limite: paradigmas podem coexistir no mesmo trecho de software com mais flexibilidade que equipes físicas.*

**Ferramentas e método de trabalho**

Possuir faca, forno e liquidificador não define sozinho como uma receita será preparada. Da mesma forma, a existência de `class`, `map` ou generics não determina o paradigma nem garante bom design.

*Limite: recursos de linguagem também impõem restrições técnicas que utensílios não representam.*

## Erros Comuns

- Defender um paradigma como solução universal.
- Confundir o uso de `class` com orientação a objetos bem modelada.
- Usar `map`, `filter` e `reduce` apenas para parecer funcional, mesmo quando um laço seria mais claro.
- Introduzir imutabilidade ou generics sem avaliar custo de leitura e cópia.
- Pedir à IA para converter todo o projeto para um paradigma sem formular quais problemas a mudança deveria resolver.

## Checklist de Entendimento

- Onde estão dados, estado e comportamento?
- Que efeitos colaterais existem?
- Que paradigma torna a regra mais explícita?
- Que recurso de TypeScript está sendo usado e com qual propósito?
- Uma combinação de estilos simplificaria o problema?
- A escolha facilita teste e mudança?
