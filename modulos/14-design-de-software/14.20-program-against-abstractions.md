# Program Against Abstractions

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o que significa programar contra abstracoes;
- criar contratos pequenos e expressivos;
- diferenciar abstracao util de abstracao vazia;
- conectar este principio com DIP e testes.

## Onde Esta Aula Entra no Curso

- Modulo: 14. Design de Software
- Aula anterior: Dependency Inversion Principle
- Proxima aula: Composition Over Inheritance
- Pre-requisitos: Interfaces, DIP, testes, backend e acoplamento.

## Ideia Central

Programar contra abstracoes significa depender de capacidades e contratos, nao de detalhes concretos. O caso de uso precisa de algo que salve progresso, nao necessariamente de uma classe especifica do ORM.

Abstracao boa tem nome claro, tamanho pequeno e nasce de uma necessidade real. Abstracao ruim apenas troca uma dependencia concreta por uma interface sem significado.

## Topicos da Aula

- Capacidade vs implementacao.
- Contratos pequenos.
- Interfaces de entrada e saida.
- Testes com fakes.
- Abstracao prematura.
- Relacao com DIP.

## Exemplo Pratico

```ts
interface ProgressSaver {
  save(progress: StudyProgress): Promise<void>
}
```

## Analogias e Intuicoes

- O codigo deve pedir "transporte", nao "um carro vermelho especifico".
- Contrato bom revela o que importa para o consumidor.

## Erros Comuns

- Criar interface com o mesmo nome da classe concreta.
- Expor metodos demais no contrato.
- Abstrair detalhe estavel sem motivo.
- Usar abstracao para esconder design confuso.

## Checklist de Entendimento

- Que capacidade o consumidor precisa?
- O contrato e menor que a implementacao?
- O nome comunica papel?
- Existe ganho de teste ou evolucao?
- A abstracao tem motivo real?

## Pratica Recomendada

- Pegue um caso de uso com dependencia concreta.
- Crie uma interface minima baseada no consumidor.
- Substitua a dependencia no teste por um fake.
