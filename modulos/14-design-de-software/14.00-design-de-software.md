# Design de Software

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que design de software significa no nível do código;
- diferenciar código que funciona de código sustentável;
- reconhecer os temas que serão estudados no módulo;
- entender a ponte entre testabilidade e arquitetura.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Projeto Prático: Tornando uma API Testável, Observável e Mensurável
- Próxima aula: Código Simples, Legível e Sustentável
- Pré-requisitos: TypeScript, backend, APIs, bancos de dados, qualidade, testes, failure modes e testabilidade.

## Ideia Central

Design de software é o conjunto de escolhas que torna o código mais fácil de entender, testar, mudar e remover. Ele aparece antes da arquitetura grande: em nomes, funções, objetos, dependências, responsabilidades e limites.

O objetivo não é decorar princípios. É aprender a perceber quando o código está juntando coisas demais, escondendo dependências, criando abstrações cedo demais ou dificultando mudanças simples.

## Tópicos da Aula

- Código como decisão de design.
- Simplicidade, legibilidade e sustentabilidade.
- Paradigmas e POO.
- Princípios de design.
- SOLID e princípios relacionados.
- Padrões como vocabulário.

## Exemplo Prático

Decisão de Design no PetCare OS:
A funcionalidade de "Alta Médica" envolve checar se o pet pagou as contas, se tem dono registrado, mudar status e enviar e-mail.

Pergunta de Design:
- O Controller Express deve ter as 200 linhas dessa lógica?
- Devemos criar uma classe `AltaMedicaService`?
- Como isolar o envio de email (infraestrutura) da regra "só dar alta se pagou" (domínio)?

## Analogias e Intuições

- Design de Software é projetar a planta elétrica do Hospital Veterinário: Se você puxar o fio do ar condicionado junto com o do bisturi, quando o ar quebrar, a cirurgia para.

## Erros Comuns

- Confundir design com arquitetura corporativa.
- Usar padrões antes de entender o problema.
- Criar abstrações para possibilidades que ainda não existem.
- Misturar regra de negócio com detalhes de framework.

## Checklist de Entendimento

- O que torna código difícil de mudar?
- Qual responsabilidade está misturada?
- Que dependência deveria estar explícita?
- Que teste ficou caro por causa do design?
- O que este módulo vai aprofundar?
