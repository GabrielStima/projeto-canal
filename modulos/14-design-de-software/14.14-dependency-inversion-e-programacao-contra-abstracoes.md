# Dependency Inversion e Programação Contra Abstrações

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar inversão de dependência entre regra e infraestrutura;
- modelar contratos a partir da capacidade exigida pelo consumidor;
- usar injeção de dependência para testar regras isoladamente;
- diferenciar abstração útil de interface vazia ou prematura.

## Ideia Central

Dependency Inversion Principle diz que regras de alto nível não deveriam depender diretamente de detalhes instáveis. Programar contra abstrações transforma essa direção em prática: o consumidor define a capacidade de que precisa, e o detalhe implementa esse contrato.

Uma abstração boa é menor que a implementação, usa vocabulário do consumidor e cria ganho concreto de teste ou evolução. Uma interface que apenas copia todos os métodos de uma classe concreta troca o tipo da dependência sem melhorar seu design.

## Exemplo Prático

```ts
interface AppointmentRepository {
  save(appointment: Appointment): Promise<void>
}

class ScheduleAppointment {
  constructor(
    private readonly appointments: AppointmentRepository,
    private readonly notifier: AppointmentNotifier,
  ) {}
}
```

O caso de uso pede capacidades de salvar e notificar. Ele não conhece Prisma, SQL, `fetch` ou a marca do provedor. As implementações concretas adaptam esses detalhes ao contrato.

O benefício não é poder trocar de banco por esporte. É manter a regra compreensível e testável com um fake pequeno, além de limitar o impacto de detalhes externos.

## Analogias e Intuições

**Tomada padronizada**

Um equipamento depende das características elétricas e do formato acordado, não da usina específica que produz energia. O contrato permite que detalhes se adaptem ao consumidor.

*Limite: contratos de software incluem comportamento, erros e efeitos que uma tomada não representa.*

**Solicitação por capacidade**

Uma equipe pede “transporte refrigerado”, não um modelo específico de veículo, quando o que importa é preservar a carga. O nome do contrato deve descrever a capacidade necessária.

*Limite: às vezes a implementação concreta possui características relevantes que precisam aparecer na decisão.*

## Erros Comuns

- Criar uma interface para cada classe automaticamente.
- Nomear o contrato com base na tecnologia, como `IPrismaRepository`.
- Expor métodos demais porque a implementação concreta os possui.
- Esconder dependências num container mágico, service locator ou singleton global.
- Aceitar abstrações geradas por IA sem verificar qual consumidor as define e qual ganho produzem.

## Checklist de Entendimento

- Qual parte representa regra e qual representa detalhe?
- Que capacidade o consumidor realmente exige?
- O contrato é menor e mais estável que a implementação?
- A dependência está explícita no construtor ou função?
- Um fake simples consegue substituir o detalhe no teste?
- A indireção paga seu custo?
