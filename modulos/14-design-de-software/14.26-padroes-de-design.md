# Padrões de Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar padrões de design como soluções recorrentes;
- diferenciar padrão de receita obrigatória;
- reconhecer quando um padrão nomeia algo já necessário;
- preparar o estudo de GoF e POSA.

## Onde Esta Aula Entra no Curso

- Módulo: 14. Design de Software
- Aula anterior: Princípio de Hollywood
- Próxima aula: GoF: Padrões Criacionais, Estruturais e Comportamentais
- Pré-requisitos: Princípios de design, POO, composição, interfaces, SOLID e testes.

## Ideia Central

Padrões de design são nomes para formas recorrentes de resolver problemas de design. Eles ajudam a conversar, reconhecer trade-offs e evitar reinventar estruturas conhecidas.

O perigo é estudar padrões como catálogo para encaixar em todo código. Primeiro vem o problema; depois, talvez, o padrão.

## Tópicos da Aula

- Problema recorrente.
- Contexto.
- Solução.
- Consequências.
- Nome compartilhado.
- Padrões vs overengineering.

## Exemplo Prático

```text
Problema:
várias formas de calcular preço.

Possível padrão:
Strategy, se a variação for real e precisar ser trocada com clareza.
```

## Analogias e Intuições

- Padrão é vocabulário, não decoração.
- Reconhecer um padrão é diferente de forçar um padrão.

## Erros Comuns

- Procurar onde encaixar Factory antes de haver problema.
- Decorar nomes sem entender consequências.
- Usar padrão para parecer sofisticado.
- Ignorar soluções simples.

## Checklist de Entendimento

- Qual problema recorrente existe?
- O contexto combina?
- Que custo o padrão adiciona?
- Que variação ele organiza?
- Uma solução simples bastaria?

## Prática Recomendada

- Escolha uma dor de design real.
- Descreva problema, contexto e consequência.
- Só depois pesquise se existe padrão relacionado.
