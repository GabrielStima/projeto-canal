# Projeto Prático: Refatorando uma API para Melhor Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diagnosticar problemas de design em uma API pequena e propor pelo menos duas soluções alternativas;
- escolher entre padrões de design com justificativa de trade-offs (custo, testabilidade, simplicidade);
- refatorar com passos seguros protegidos por testes;
- registrar decisões de design em formato de ADR (Architecture Decision Record) justificando a rejeição de alternativas.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Desafio: Escolhendo Padrões de Design
- Próxima aula: Princípios Arquiteturais
- Pré-requisitos: Todo o módulo 14, backend com Node.js, bancos de dados, API design, testes e testabilidade.

## Ideia Central

O projeto do módulo junta os conceitos em uma refatoração realista. A meta não é criar a arquitetura perfeita. É pegar um fluxo que já funciona, propor alternativas de design, escolher uma com critério e justificar por que rejeitou as outras.

Refatorar bem significa mudar estrutura sem mudar comportamento observado. Mas refatorar com maturidade significa também **registrar por que esta estrutura foi escolhida e não outra**. O ADR (Architecture Decision Record) é a prova de que você não decorou o padrão — você o avaliou.

> **Regra deste projeto:** toda decisão de design que envolva um padrão deve ser acompanhada de uma nota curta no formato ADR: "Consideramos X e Y. Escolhemos X porque Z. Rejeitamos Y porque W."

## Tópicos da Aula

- Escolha do fluxo com problema real de design.
- Diagnóstico de responsabilidades e proposta de alternativas.
- Avaliação de trade-offs entre padrões candidatos.
- Testes de proteção antes da refatoração.
- Separação de regra e infraestrutura.
- Abstrações pequenas com justificativa.
- Registro de decisões em formato ADR.
- Revisão: a escolha ainda faz sentido após a implementação?

## Exemplo Prático

marco de design - A Refatoração do PetCare OS:

Você tem a API testada e observável do Módulo 13, mas ela foi construída de forma rígida.
Sua missão é criar camadas lógicas (Controllers, Services/UseCases, Repositories). O Controller chama UseCases, o UseCase aplica as regras de Domínio puro e chama as abstrações (Interfaces) dos Repositories.
A Injeção de Dependências substitui todos os Imports estáticos que travavam seus testes de unidade.

## Analogias e Intuições

- Refatorar não é mudar o que o sistema faz por fora. É reformar a fundação do hospital sem precisar fechar a entrada do pronto-socorro.

## Erros Comuns

- Refatorar é adicionar feature ao mesmo tempo.
- Criar camadas demais.
- Não escrever teste antes de mexer.
- Trocar nomes sem melhorar responsabilidade.
- Não registrar trade-offs.

## Checklist de Entendimento

- O comportamento externo continuou igual?
- Que responsabilidades foram separadas?
- Que dependência ficou explícita?
- Que teste protege a regra principal?
- Que decisão de design foi tomada?
