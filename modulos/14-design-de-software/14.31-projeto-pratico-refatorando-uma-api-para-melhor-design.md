# Projeto Prático: Refatorando uma API para Melhor Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diagnosticar problemas de design em uma API pequena e propor pelo menos duas soluções alternativas;
- escolher entre padrões de design com justificativa de trade-offs (custo, testabilidade, simplicidade);
- refatorar com passos seguros protegidos por testes;
- registrar decisões de design em formato de ADR (Architecture Decision Record) justificando a rejeição de alternativas.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Desafio: Escolhendo Padrões de Design
- Próxima aula: Princípios Arquiteturais
- Pré-requisitos: Todo o módulo 14, backend com Node.js, bancos de dados, API design, testes e testabilidade.

## Ideia Central

O projeto do módulo junta os conceitos em uma refatoração realista. A meta não é criar a arquitetura perfeita. É pegar um fluxo que já funciona, propor alternativas de design, escolher uma com critério e justificar por que rejeitou as outras.

Refatorar bem significa mudar estrutura sem mudar comportamento observado. Mas refatorar com maturidade significa também **registrar por que esta estrutura foi escolhida e não outra**. O ADR (Architecture Decision Record) é a prova de que você não decorou o padrão — você o avaliou.

> **Regra deste projeto:** toda decisão de design que envolva um padrão deve ser acompanhada de uma nota curta no formato ADR: "Consideramos X e Y. Escolhemos X porque Z. Rejeitamos Y porque W."

## Tópicos da Aula

- Escolha do fluxo com problema real de design.
- Diagnóstico de responsabilidades e proposta de alternativas.
- Avaliação de trade-offs entre padrões candidatos.
- Testes de proteção antes da refatoração.
- Separação de regra e infraestrutura.
- Abstrações pequenas com justificativa.
- Registro de decisões em formato ADR.
- Revisão: a escolha ainda faz sentido após a implementação?

## Exemplo Prático

```text
Fluxo sugerido:
POST /appointments (criação de consulta no PetCare OS)

Código original com problemas de design:
- handler HTTP faz validação, regra de negócio e persistência;
- notificações são enviadas diretamente com fetch;
- não há separação entre caso de uso e infraestrutura.

Alternativa A — Refatoração com princípios SOLID:
- extrair validação para objeto de valor;
- extrair persistência para repositório com interface;
- extrair notificações para serviço de domínio.

Alternativa B — Refatoração com padrão Strategy + Observer:
- Strategy para políticas de preço (rotina, emergência, teleconsulta);
- Observer para notificar tutores, atualizar agenda e gerar fatura
  sem acoplar o caso de uso aos efeitos colaterais.

Avaliação de trade-offs:
- Alternativa A é mais simples, direta e fácil de testar.
- Alternativa B desacopla melhor, mas introduz indireção e
  torna o fluxo menos explícito.

ADR de exemplo:
"Decisão: escolhemos Alternativa A para este fluxo.
Consideramos Strategy + Observer (Alternativa B), mas rejeitamos
porque o PetCare OS hoje tem apenas 3 tipos de consulta e
2 canais de notificação. A indireção do Observer prejudicaria
a rastreabilidade sem trazer benefício proporcional.
Reavaliaremos se o número de tipos de consulta ultrapassar 5
ou se novos canais de notificação forem adicionados."

Refatorar:
- handler HTTP;
- validação de entrada;
- caso de uso;
- modelo de domínio;
- repositório;
- resposta padronizada;
- testes de proteção;
- ADR registrando decisões.
```

## Analogias e Intuições

- Refatoração boa troca a estrutura enquanto mantém a promessa externa.
- Teste é o cinto de segurança da mudança.

## Erros Comuns

- Refatorar é adicionar feature ao mesmo tempo.
- Criar camadas demais.
- Não escrever teste antes de mexer.
- Trocar nomes sem melhorar responsabilidade.
- Não registrar trade-offs.

## Checklist de Entendimento

- O comportamento externo continuou igual?
- Que responsabilidades foram separadas?
- Que dependência ficou explícita?
- Que teste protege a regra principal?
- Que decisão de design foi tomada?

## Prática Recomendada

- Escolha um fluxo da API do PetCare OS.
- Escreva testes do comportamento atual.
- Liste problemas de design.
- Proponha pelo menos duas soluções alternativas (uma pode ser "sem padrão formal").
- Avalie trade-offs: complexidade, testabilidade, evolução, legibilidade, operação.
- Escolha uma solução e justifique por que rejeitou as outras.
- Refatore em passos pequenos, protegidos por testes.
- Compare antes e depois.
- Escreva um ADR curto (máximo 1 parágrafo) registrando:
  - o problema,
  - as alternativas consideradas,
  - a escolha e a justificativa,
  - as condições que fariam reconsiderar a decisão.
