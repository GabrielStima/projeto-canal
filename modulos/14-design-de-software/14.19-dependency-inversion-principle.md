# Dependency Inversion Principle

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar dependência de abstrações;
- separar regra de detalhe de infraestrutura;
- usar injeção de dependência de forma simples;
- evitar interfaces sem motivo.

- Módulo: 14. Design de Software

## Ideia Central

DIP diz que código de alto nível não deveria depender diretamente de detalhes instáveis. Uma regra de negócio não precisa conhecer SQL, ORM, HTTP client ou fila em detalhe.

A inversão acontece quando a regra define o que precisa e o detalhe implementa esse contrato. Isso melhora teste e reduz impacto de mudanças técnicas.

## Tópicos da Aula

- Alto nível vs detalhe.
- Abstração como contrato.
- Injeção de dependência.
- Repositórios e gateways.
- Testes com fakes.
- Custo da indireção.

## Exemplo Prático

```ts
class CompleteLessonUseCase {
  constructor(private progressRepository: ProgressRepository) {}
}
```

## Analogias e Intuicoes

- A regra pede uma capacidade, não uma marca específica de ferramenta.
- Inverter dependência é fazer o detalhe se adaptar ao caso de uso.

## Erros Comuns

- Criar interface para tudo automaticamente.
- Esconder dependência em container mágico cedo demais.
- Colocar detalhe de banco no caso de uso.
- Trocar dependência direta por indireção sem ganho de teste ou mudança.

## Checklist de Entendimento

- Qual parte é regra de negócio?
- Qual parte é detalhe externo?
- O contrato tem nome de capacidade?
- A injeção melhora o teste?
- A indireção paga o custo?

## Prática Recomendada

- Escolha um caso de uso que acessa banco.
- Defina contrato mínimo do repositório.
- Injete uma implementação fake no teste.
- Mantenha a implementação real fora da regra.
