# Dependency Inversion Principle

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar dependencia de abstracoes;
- separar regra de detalhe de infraestrutura;
- usar injecao de dependencia de forma simples;
- evitar interfaces sem motivo.

## Onde Esta Aula Entra no Curso

- Modulo: 14. Design de Software
- Aula anterior: Interface Segregation Principle
- Proxima aula: Program Against Abstractions
- Pre-requisitos: Interfaces, backend, banco de dados, testes, mocks e SOLID.

## Ideia Central

DIP diz que codigo de alto nivel nao deveria depender diretamente de detalhes instaveis. Uma regra de negocio nao precisa conhecer SQL, ORM, HTTP client ou fila em detalhe.

A inversao acontece quando a regra define o que precisa e o detalhe implementa esse contrato. Isso melhora teste e reduz impacto de mudancas tecnicas.

## Topicos da Aula

- Alto nivel vs detalhe.
- Abstracao como contrato.
- Injecao de dependencia.
- Repositorios e gateways.
- Testes com fakes.
- Custo da indirecao.

## Exemplo Pratico

```ts
class CompleteLessonUseCase {
  constructor(private progressRepository: ProgressRepository) {}
}
```

## Analogias e Intuicoes

- A regra pede uma capacidade, nao uma marca especifica de ferramenta.
- Inverter dependencia e fazer o detalhe se adaptar ao caso de uso.

## Erros Comuns

- Criar interface para tudo automaticamente.
- Esconder dependencia em container magico cedo demais.
- Colocar detalhe de banco no caso de uso.
- Trocar dependencia direta por indirecao sem ganho de teste ou mudanca.

## Checklist de Entendimento

- Qual parte e regra de negocio?
- Qual parte e detalhe externo?
- O contrato tem nome de capacidade?
- A injecao melhora o teste?
- A indirecao paga o custo?

## Pratica Recomendada

- Escolha um caso de uso que acessa banco.
- Defina contrato minimo do repositorio.
- Injete uma implementacao fake no teste.
- Mantenha a implementacao real fora da regra.
