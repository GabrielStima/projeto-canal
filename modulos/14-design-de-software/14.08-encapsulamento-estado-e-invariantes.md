# Encapsulamento, Estado e Invariantes

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar encapsulamento além de `private`;
- identificar invariantes de uma regra;
- proteger estado contra mudanças inválidas;
- escrever testes para comportamento, não para detalhe interno.

## Onde Esta Aula Entra no Curso

- Módulo: 14. Design de Software
- Aula anterior: Princípios Primários de POO
- Próxima aula: Acoplamento e Coesão no Código
- Pré-requisitos: Classes, objetos, testes unitários, regras de negócio e tratamento de erros.

## Ideia Central

Encapsular é controlar como o estado pode mudar. Uma invariante é uma regra que precisa continuar verdadeira para o objeto fazer sentido.

Quando qualquer parte do sistema altera dados diretamente, a regra fica espalhada. Quando o objeto oferece operações significativas, ele protege seu próprio estado.

## Tópicos da Aula

- Estado válido.
- Invariantes.
- Métodos de comportamento.
- Evitar setters automáticos.
- Erros de domínio.
- Testes de comportamento.

## Exemplo Prático

```ts
class Enrollment {
  cancel() {
    if (this.status === "completed") throw new Error("Cannot cancel")
    this.status = "cancelled"
  }
}
```

## Analogias e Intuições

- Invariante é uma promessa que o objeto nunca deve quebrar.
- Encapsulamento bom reduz lugares onde uma regra pode vazar.

## Erros Comuns

- Criar setter para todo campo.
- Validar regra apenas no endpoint.
- Expor arrays internos para alteração externa.
- Testar campo privado em vez de comportamento público.

## Checklist de Entendimento

- Que estados são válidos?
- Quem pode mudar o estado?
- Que operação representa a mudança?
- Que erro deve impedir estado inválido?
- O teste protege a regra?

## Prática Recomendada

- Escolha um objeto com status.
- Liste estados válidos e transições proibidas.
- Implemente métodos para mudar estado.
- Teste uma transição inválida.
