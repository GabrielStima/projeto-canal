# Encapsulamento, Estado e Invariantes

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar encapsulamento além de `private`;
- identificar invariantes de uma regra;
- proteger estado contra mudanças inválidas;
- escrever testes para comportamento, não para detalhe interno.

## Ideia Central

Encapsular é controlar como o estado pode mudar. Uma invariante é uma regra que precisa continuar verdadeira para o objeto fazer sentido.

Quando qualquer parte do sistema altera dados diretamente, a regra fica espalhada. Quando o objeto oferece operações significativas, ele protege seu próprio estado.

## Exemplo Prático

Invariante do Agendamento:

Invariante é uma regra inquebrável. "Uma consulta não pode terminar antes de começar".
Se `dataFim` e `dataInicio` forem atributos públicos (`public`), qualquer dev pode quebrar a regra. Se forem privados (`private`), a classe `Consulta` garante no método `finalizar(data)` que `data > dataInicio` ou lança um erro.

## Analogias e Intuições

- Encapsular o estado é como o cofre do banco. Você não deixa o dinheiro solto pra ser retirado; você fornece um caixa eletrônico (método) que só entrega o dinheiro se a senha bater.

## Erros Comuns

- Criar setter para todo campo.
- Validar regra apenas no endpoint.
- Expor arrays internos para alteração externa.
- Testar campo privado em vez de comportamento público.

## Checklist de Entendimento

- Que estados são válidos?
- Quem pode mudar o estado?
- Que operação representa a mudança?
- Que erro deve impedir estado inválido?
- O teste protege a regra?
