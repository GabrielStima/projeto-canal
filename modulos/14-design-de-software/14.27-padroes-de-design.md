# Padrões de Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- avaliar quando um padrão de design resolve um problema real e quando adiciona complexidade desnecessária;
- comparar trade-offs entre padrões candidatos para um mesmo problema;
- justificar a rejeição de um padrão em favor de uma solução mais simples;
- preparar critérios de escolha para o estudo de GoF e POSA.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Princípio de Hollywood
- Próxima aula: GoF: Padrões Criacionais, Estruturais e Comportamentais
- Pré-requisitos: Princípios de design, POO, composição, interfaces, SOLID e testes.

## Ideia Central

Padrões de design são nomes para formas recorrentes de resolver problemas de design. Eles ajudam a conversar, reconhecer trade-offs e evitar reinventar estruturas conhecidas.

O perigo é estudar padrões como catálogo para encaixar em todo código. Primeiro vem o problema; depois, talvez, o padrão. Mas o passo que separa quem decora padrões de quem os usa com critério é a **avaliação consciente**: este padrão é realmente melhor que uma solução direta? Qual custo ele introduz? Quando ele prejudica mais do que ajuda?

## Tópicos da Aula

- Problema recorrente vs. solução forçada.
- Contexto e variações que mudam a escolha.
- Trade-offs entre padrões candidatos.
- Consequências de curto e longo prazo.
- Nome compartilhado como vocabulário de equipe.
- Padrões vs. overengineering: quando rejeitar.
- Critérios de decisão: simplicidade, testabilidade, evolução.

## Exemplo Prático

```text
Problema:
várias formas de calcular preço de consulta no PetCare OS
(rotina, emergência, teleconsulta, plano de saúde pet).

Candidatos:
- Strategy: cada política é uma estratégia intercambiável.
- Chain of Responsibility: cada regra tenta aplicar o desconto
  e passa para a próxima se não couber.
- Solução simples: switch/case ou lookup table.

Trade-offs a avaliar:
- Strategy exige interface + classes, mas isola mudanças.
- Chain permite combinar regras, mas cria indireção difícil de testar.
- Switch é direto, mas cresce com cada novo tipo de consulta.

Decisão depende de:
- Quantos tipos existem hoje e quantos virão.
- Se regras precisam ser combinadas ou são mutuamente exclusivas.
- Se testes de unidade precisam cobrir cada política isoladamente.
```

## Analogias e Intuições

- Padrão é vocabulário, não decoração.
- Reconhecer um padrão é diferente de forçar um padrão.
- Escolher padrão é como escolher ferramenta: a melhor é a que resolve o problema com o menor custo de manutenção.
- Um médico não receita antibiótico para gripe: o tratamento deve ser proporcional ao problema.

## Erros Comuns

- Procurar onde encaixar Factory antes de haver problema.
- Decorar nomes sem entender consequências.
- Usar padrão para parecer sofisticado.
- Ignorar soluções simples.
- Escolher padrão sem comparar alternativas viáveis.
- Aplicar padrão que aumenta acoplamento em vez de reduzi-lo.
- Rejeitar padrão útil apenas por aversão a complexidade.

## Checklist de Entendimento

- Qual problema recorrente existe?
- O contexto combina?
- Que padrões alternativos poderiam resolver o mesmo problema?
- Quais são os custos de cada alternativa (complexidade, testabilidade, indireção)?
- Que variação o padrão organiza e que variação ele esconde?
- Uma solução simples bastaria? Por que sim ou por que não?
- Se eu precisar mudar esse código daqui a 6 meses, qual alternativa será mais fácil de entender?

## Prática Recomendada

- Escolha uma dor de design real no PetCare OS ou em código próprio.
- Descreva problema, contexto e consequência.
- Liste pelo menos duas soluções possíveis (uma com padrão, uma sem).
- Compare trade-offs: complexidade, testabilidade, evolução, legibilidade.
- Escolha uma solução e justifique por que rejeitou a outra.
- Só depois pesquise se existe padrão relacionado — e avalie se ele realmente se encaixa.

```text
Estou estudando Padrões de Design em uma trilha de desenvolvimento fullstack JavaScript/TypeScript.

Meu objetivo é aprender a avaliar e escolher padrões com critério, não apenas reconhecê-los.

Contexto da aula:
- Descrição da aula: Padrões de design são nomes para formas recorrentes de resolver problemas de design. O foco desta aula é desenvolver critérios de escolha: comparar alternativas, avaliar trade-offs e saber quando rejeitar um padrão.
- Descrição do módulo: O módulo ensina design de código, Clean Code, paradigmas, POO, model-driven design, princípios de design, SOLID, composição, abstrações e padrões para escrever software mais simples, legível, testável e sustentável.
- Itens já estudados: Princípios de design, POO, composição, interfaces, SOLID e testes.

Crie uma sessão de estudo com:

Regras:
- Não entregue respostas completas antes de eu tentar responder.
- Não diga qual padrão é "o certo" antes de eu justificar minha escolha.

- Avalie separadamente: (a) acurácia técnica da escolha, (b) solidez da justificativa de rejeição, (c) awareness de custos.

```
