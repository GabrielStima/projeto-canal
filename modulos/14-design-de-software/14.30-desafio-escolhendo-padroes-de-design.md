# Desafio: Escolhendo Padrões de Design

## Objetivo

Ao final deste desafio, você deve ser capaz de:

- escolher entre padrões de design com justificativa baseada em trade-offs;
- reconhecer quando um padrão ajuda e quando adiciona complexidade desnecessária;
- aplicar pelo menos um padrão GoF em código real do PetCare OS;
- articular por que rejeitou alternativas viáveis.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: POSA e Padrões Além do GoF
- Próxima aula: Projeto Prático: Refatorando uma API para Melhor Design
- Pré-requisitos: Padrões GoF (criacionais, estruturais e comportamentais), princípios de design, SOLID, composição e abstrações.

## Ideia Central

Reconhecer padrões é diferente de escolher entre eles com critério. Este desafio apresenta três cenários do PetCare OS onde mais de um padrão poderia ser aplicado. Sua tarefa é escolher, implementar e justificar — incluindo o motivo pelo qual as alternativas foram descartadas.

O objetivo não é acertar a resposta "certa". É exercitar o raciocínio de trade-off que diferencia quem decorou padrões de quem os usa.

## Os Cenários

### Cenário 1 — Notificações com múltiplos canais

O PetCare OS precisa notificar tutores em diferentes momentos do ciclo de vida de uma consulta: agendamento, confirmação, lembrete 24h antes e cancelamento. Os canais disponíveis são e-mail, SMS e push notification. Canais novos podem surgir.

**Pergunta:** Qual padrão GoF você usaria para organizar os canais de notificação? Considerando Observer, Strategy e Chain of Responsibility — qual escolheria e por quê? Qual rejeitaria e por quê?

**Restrição:** A solução deve permitir adicionar um novo canal sem modificar o código que dispara a notificação.

---

### Cenário 2 — Criação de consultas com regras variáveis

Consultas no PetCare OS podem ser de três tipos: rotina, emergência e teleconsulta. Cada tipo tem validações e comportamentos ligeiramente diferentes na criação (campos obrigatórios distintos, janelas de horário diferentes, notificações diferentes).

**Pergunta:** Para encapsular a criação de cada tipo de consulta, você escolheria Factory Method, Abstract Factory ou Builder? Justifique para cada um: por que serve ou por que não.

**Restrição:** A lógica de qual tipo criar vem de um parâmetro recebido pelo handler HTTP. A lógica interna de criação de cada tipo não pode vazar para o handler.

---

### Cenário 3 — Integração com sistemas de pagamento externos

O PetCare OS se integra com dois gateways de pagamento (ex: Stripe e PagSeguro). A interface de cada gateway é diferente. O restante do sistema precisa processar pagamentos sem saber qual gateway está em uso.

**Pergunta:** Você usaria Adapter, Facade ou Bridge para resolver esse problema? Qual a diferença de intenção entre os três neste contexto?

**Restrição:** Deve ser possível trocar de gateway com mudança mínima. Os testes do serviço de pagamento não podem depender da implementação concreta do gateway.

---

## O Desafio

Para cada cenário:

1. **Escolha** um padrão e explique em uma frase por que ele se encaixa.
2. **Implemente** um esboço em TypeScript — não precisa ser código completo, mas deve mostrar a estrutura do padrão.
3. **Justifique a rejeição** das alternativas mencionadas: por que elas funcionariam menos bem aqui?

## Critérios de Avaliação

- A escolha usa vocabulário do padrão (intenção, participantes, consequências)?
- A implementação reflete a estrutura correta do padrão?
- A justificativa da rejeição vai além de "não sei usar bem"?
- O código resultante é mais fácil de testar do que sem o padrão?
- A solução não é over-engineered para o tamanho do problema?
