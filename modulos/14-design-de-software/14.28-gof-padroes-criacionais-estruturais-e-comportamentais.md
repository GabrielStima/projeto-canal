# GoF: Padrões Criacionais, Estruturais e Comportamentais

## Objetivo

Ao final desta aula, você deve ser capaz de:

- comparar padrões GoF candidatos para um mesmo problema e justificar a escolha;
- avaliar quando um padrão clássico é excessivo para um cenário TypeScript moderno;
- identificar conflitos entre padrões e princípios SOLID no mesmo design;
- decidir entre solução simples e padrão formal com base em trade-offs reais.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Padrões de Design
- Próxima aula: POSA e Padrões Além do GoF
- Pré-requisitos: POO, interfaces, composição, SOLID e princípios de design.

## Ideia Central

GoF organiza padrões clássicos em três grupos. Criacionais tratam da criação de objetos. Estruturais tratam de composição entre objetos. Comportamentais tratam de colaboração e distribuição de responsabilidades.

Nesta trilha, o objetivo não é memorizar implementações. É **escolher entre padrões com critério**: quando Strategy é melhor que State? Quando Factory Method é preferível a Abstract Factory? Quando um padrão clássico é overengineering em TypeScript moderno?

## Tópicos da Aula

- Criacionais: trade-offs entre Factory Method, Abstract Factory e Builder.
- Estruturais: quando Adapter, Decorator ou Composite competem pelo mesmo problema.
- Comportamentais: Strategy vs. State vs. Observer — escolha por intenção, não por estrutura.
- Conflitos entre padrões e princípios SOLID.
- TypeScript moderno: quando padrões clássicos são desnecessários.
- Cuidado com catálogo: reconhecer não basta, é preciso avaliar.

## Exemplo Prático

```text
Problema do PetCare OS:
o status de uma consulta muda e diferentes ações
são executadas em cada mudança (notificar tutor,
atualizar agenda, gerar fatura).

Candidatos comportamentais:
- Strategy: cada status tem uma estratégia de ação.
- State: a consulta muda de comportamento conforme
  seu próprio estado evolui.
- Observer: outros serviços "ouvem" a mudança de
  status e reagem independentemente.

Trade-offs:
- Strategy é simples, mas o chamador precisa saber
  qual estratégia usar a cada transição.
- State encapsula a transição dentro do objeto
  consulta, mas cria acoplamento entre estados.
- Observer desacopla completamente, mas torna o
  fluxo implícito e difícil de rastrear.

Decisão depende de:
- Se a lógica de transição pertence à consulta
  (State) ou ao orquestrador externo (Strategy).
- Se outros domínios precisam reagir à mudança
  (Observer) ou se a reação é local.
- Se rastreabilidade do fluxo é mais importante
  que desacoplamento.
```

## Analogias e Intuições

- As categorias mostram o tipo de problema: criar, montar ou coordenar.
- Em TypeScript, muitos padrões podem ser mais leves do que em linguagens clássicas.
- Escolher padrão é como decidir entre rotas de navegação: todas chegam ao destino, mas uma é mais rápida, outra mais segura, outra tem menos pedágio.
- Um bom designer não sabe mais padrões; sabe quando um padrão custa mais do que vale.

## Erros Comuns

- Implementar padrão clássico de forma pesada demais.
- Usar singleton para esconder estado global.
- Criar factory para construtor simples.
- Chamar qualquer objeto de strategy.
- Escolher padrão sem comparar com alternativas do mesmo grupo.
- Aplicar Strategy quando uma simples função resolveria.
- Usar Observer em fluxo que precisa de ordem garantida e rastreabilidade.
- Ignorar que TypeScript (interfaces, tipos, funções de primeira classe) pode
  reduzir a cerimônia de padrões clássicos de POO.

## Checklist de Entendimento

- O problema é de criação, estrutura ou comportamento?
- Quais padrões do mesmo grupo poderiam resolver este problema?
- Por que escolhi este padrão e não o outro?
- O padrão reduz acoplamento ou apenas move a complexidade?
- A implementação ficou proporcional ao tamanho do problema?
- Existe teste para o contrato?
- O nome ajuda a equipe a entender a intenção?
- Se eu remover o padrão e usar uma solução direta, o que perco e o que ganho?

## Prática Recomendada

- Pegue uma integração externa no PetCare OS.
- Modele um Adapter simples.
- Depois, liste pelo menos uma alternativa (ex: Facade, ou simples wrapper function)
  e compare: qual é mais simples? Qual protege melhor contra mudanças?
- Pegue uma regra variável (ex: cálculo de preço de consulta).
- Modele uma Strategy pequena.
- Depois, compare com uma solução usando lookup table ou função pura.
- Justifique por que a escolha vence a alternativa mais simples.
