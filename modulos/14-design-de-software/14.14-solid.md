# SOLID

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar SOLID como conjunto de principios de design;
- entender o problema central de cada letra;
- reconhecer que SOLID nao exige POO pesada;
- preparar as aulas especificas de cada principio.

## Onde Esta Aula Entra no Curso

- Modulo: 14. Design de Software
- Aula anterior: YAGNI
- Proxima aula: Single Responsibility Principle
- Pre-requisitos: POO, acoplamento, coesao, DRY, YAGNI, testes e TypeScript.

## Ideia Central

SOLID ajuda a criar codigo mais facil de mudar quando responsabilidades, extensoes, substituicoes, contratos e dependencias ficam problematicas.

Ele nao deve ser aplicado como ornamento. Cada principio responde a uma dor: classes com motivos demais para mudar, modificacoes arriscadas, subtipos que quebram contrato, interfaces grandes demais e dependencias em detalhes instaveis.

## Topicos da Aula

- S: responsabilidade unica.
- O: aberto para extensao.
- L: substituicao segura.
- I: interfaces especificas.
- D: dependencia de abstracoes.
- Limites de SOLID.

## Exemplo Pratico

```text
Um service que valida payload, aplica regra, acessa banco, envia email e formata resposta provavelmente viola mais de um principio.
```

## Analogias e Intuicoes

- SOLID e um conjunto de perguntas para investigar dor de mudanca.
- O objetivo e reduzir surpresa, nao aumentar cerimonia.

## Erros Comuns

- Criar uma interface para cada classe sem necessidade.
- Usar SOLID para justificar excesso de camadas.
- Aplicar em CRUD simples sem dor real.
- Decorar definicoes sem reconhecer sintomas.

## Checklist de Entendimento

- Que letra responde a qual problema?
- O problema existe neste codigo?
- A solucao adiciona custo?
- O teste ficou mais simples?
- A mudanca ficou mais localizada?

## Pratica Recomendada

- Escolha um service grande.
- Marque possiveis dores relacionadas a cada letra.
- Nao refatore ainda; apenas diagnostique.
