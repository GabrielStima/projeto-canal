# YAGNI

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar YAGNI em decisões de design;
- identificar complexidade criada por previsão;
- separar extensibilidade real de imaginada;
- manter código preparado sem antecipar demais.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: DRY
- Próxima aula: SOLID
- Pré-requisitos: Produto, requisitos, Clean Code, testes, backend e refatoração básica.

## Ideia Central

YAGNI lembra que você provavelmente não vai precisar de muitas flexibilidades imaginadas. Design bom aceita mudança, mas não precisa construir todas as possibilidades antes de existir demanda.

O ponto não é escrever código descartável. É evitar camadas, configurações, interfaces e padrões que ainda não pagam o custo que adicionam.

## Tópicos da Aula

- Previsão vs evidência.
- Extensibilidade com custo.
- Código simples e refatorável.
- Sinais de exagero.
- Relação com requisitos.
- Relação com testes.

## Exemplo Prático

You Aren't Gonna Need It (Você não vai precisar disso):

O PetCare OS atende apenas cães e gatos localmente.
O desenvolvedor cria a classe abstrata `MultitenantGlobalAnimalFederationStrategy` prevendo o dia em que exportarão elefantes para a Europa.
Isso é overengineering. Implemente apenas a entidade `Pet` hoje.

## Analogias e Intuições

- YAGNI é não comprar cadeirinhas de bebê para o seu carro conversível sendo que você nem conheceu ninguém para casar ainda.

## Erros Comuns

- Usar YAGNI como desculpa para código bagunçado.
- Criar interfaces para implementação única sem ganho.
- Projetar para cenários hipotéticos demais.
- Ignorar mudança provável e já conhecida.

## Checklist de Entendimento

- Qual necessidade real justifica esta abstração?
- Existe segunda implementação?
- O requisito já pede variação?
- O teste ficou mais difícil?
- A mudança futura continuará possível?

## Prática Recomendada

- Liste uma funcionalidade que você queria fazer 'genérica demais' na API, percebeu que era cedo, e recuou mantendo-a simples.
