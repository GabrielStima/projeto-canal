# Interface Segregation Principle

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar interfaces especificas para consumidores especificos;
- reconhecer contratos grandes demais;
- dividir interfaces por uso real;
- evitar fragmentar contratos sem ganho.

## Onde Esta Aula Entra no Curso

- Modulo: 14. Design de Software
- Aula anterior: Liskov Substitution Principle
- Proxima aula: Dependency Inversion Principle
- Pre-requisitos: Interfaces, TypeScript, SOLID, acoplamento, testes e backend.

## Ideia Central

ISP diz que consumidores nao deveriam depender de metodos que nao usam. Interfaces grandes criam acoplamento desnecessario e forcam implementacoes falsas.

Em TypeScript, interfaces pequenas podem representar papéis: algo que le, algo que salva, algo que publica evento, algo que calcula.

## Topicos da Aula

- Contratos por consumidor.
- Interfaces gordas.
- Implementacoes falsas.
- Papéis pequenos.
- Testes com dependencias menores.
- Excesso de fragmentacao.

## Exemplo Pratico

```ts
interface LessonReader {
  findById(id: string): Promise<Lesson | null>
}
```

## Analogias e Intuicoes

- Um consumidor deve receber a chave da sala que precisa, nao do predio inteiro.
- Interface pequena reduz o que o teste precisa simular.

## Erros Comuns

- Criar `Repository` com todos os metodos possiveis.
- Fazer classe implementar metodo que nunca deveria existir nela.
- Dividir interface sem olhar consumidores.
- Criar nomes genericos demais.

## Checklist de Entendimento

- Quem consome esta interface?
- Quais metodos esse consumidor realmente usa?
- Alguma implementacao esta fingindo suporte?
- O teste ficou menor?
- O nome representa um papel claro?

## Pratica Recomendada

- Pegue uma interface grande.
- Liste consumidores.
- Separe uma interface minima para um caso de uso.
- Ajuste o teste para depender do contrato menor.
