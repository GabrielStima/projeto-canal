# Interface Segregation Principle

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar interfaces específicas para consumidores específicos;
- reconhecer contratos grandes demais;
- dividir interfaces por uso real;
- evitar fragmentar contratos sem ganho.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Liskov Substitution Principle
- Próxima aula: Dependency Inversion Principle
- Pré-requisitos: Interfaces, TypeScript, SOLID, acoplamento, testes e backend.

## Ideia Central

ISP diz que consumidores não deveriam depender de métodos que não usam. Interfaces grandes criam acoplamento desnecessário e forçam implementações falsas.

Em TypeScript, interfaces pequenas podem representar papéis: algo que lê, algo que salva, algo que publica evento, algo que calcula.

## Tópicos da Aula

- Contratos por consumidor.
- Interfaces gordas.
- Implementações falsas.
- Papéis pequenos.
- Testes com dependências menores.
- Excesso de fragmentação.

## Exemplo Prático

```ts
interface LessonReader {
  findById(id: string): Promise<Lesson | null>
}
```

## Analogias e Intuições

- Um consumidor deve receber a chave da sala que precisa, não do prédio inteiro.
- Interface pequena reduz o que o teste precisa simular.

## Erros Comuns

- Criar `Repository` com todos os métodos possíveis.
- Fazer classe implementar método que nunca deveria existir nela.
- Dividir interface sem olhar consumidores.
- Criar nomes genéricos demais.

## Checklist de Entendimento

- Quem consome esta interface?
- Quais métodos esse consumidor realmente usa?
- Alguma implementação está fingindo suporte?
- O teste ficou menor?
- O nome representa um papel claro?

## Prática Recomendada

- Pegue uma interface grande.
- Liste consumidores.
- Separe uma interface mínima para um caso de uso.
- Ajuste o teste para depender do contrato menor.
