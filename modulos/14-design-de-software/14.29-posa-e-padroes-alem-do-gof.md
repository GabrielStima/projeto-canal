# POSA e Padrões Além do GoF

## Objetivo

Ao final desta aula, você deve ser capaz de:

- avaliar se um problema demanda padrão de design local ou padrão arquitetural;
- comparar trade-offs de escolha entre padrões de diferentes famílias (GoF vs. POSA vs. simples);
- justificar quando um padrão POSA é inadequado para escopo pequeno;
- decidir o limite correto entre design de código e arquitetura de software.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: GoF: Padrões Criacionais, Estruturais e Comportamentais
- Próxima aula: Desafio: Escolhendo Padrões de Design
- Pré-requisitos: Padrões de design, princípios de design, backend, mensageria introdutória e testabilidade.

## Ideia Central

POSA e outras famílias de padrões ampliam a conversa para estruturas maiores: camadas, brokers, pipes, microkernel, concorrência e sistemas distribuídos. Esses assuntos encostam no módulo 15, mas aqui entram apenas como mapa.

O limite importante: neste módulo, padrões servem para design de código e transição conceitual. Arquitetura será aprofundada depois. Mas o designer maduro precisa saber **onde está o limite**: usar um broker (POSA) para resolver um problema de adaptação de API (GoF) é aplicar canhão para matar mosquito. Usar um Adapter (GoF) para organizar uma arquitetura de mensageria distribuída é subdimensionar o problema.

## Tópicos da Aula

- GoF vs POSA: escala de aplicação e custo de adoção.
- Padrões locais: quando o problema cabe em classe ou módulo.
- Padrões arquiteturais: quando o problema exige estrutura de sistema.
- Critérios de decisão: tamanho do problema, equipe, tempo de vida, evolução.
- Vocabulário para o módulo 15: mapa, não aprofundamento.
- Cuidado com salto de escopo: avaliar antes de escalar.
- Trade-offs: simplicidade local vs. preparação para escala.

## Exemplo Prático

```text
Problema no PetCare OS: integrar com múltiplos gateways de pagamento.

Opção A — Design local (GoF):
Adapter para cada gateway. O serviço de pagamento usa a interface
unificada. Simples, testável, resolve o problema hoje.

Opção B — Arquitetura (POSA):
Broker de mensagens com filas de pagamento, retry automático,
circuit breaker, saga para compensação. Prepara para escala,
mas adiciona infraestrutura, operação e complexidade de deploy.

Avaliação de escala:
- Se o PetCare OS processa 100 pagamentos/dia, Adapter resolve.
- Se processa 100.000 pagamentos/dia com múltiplos gateways
  falhando independentemente, Broker se torna necessário.
- A escolha não é técnica apenas: é econômica e operacional.

Decisão correta:
Comece com Adapter. Monitore falhas e volume. Evolua para
Broker quando os dados mostrarem que a complexidade é justificada.
```

## Analogias e Intuições

- GoF costuma olhar peças menores; POSA ajuda a olhar estruturas maiores.
- O tamanho do problema muda o tipo de padrão.
- Usar POSA em problema pequeno é como contratar arquiteto para reformar banheiro: o resultado é bom, mas o custo é desproporcional.
- O limite entre design e arquitetura é fluido: o que é arquitetura para um time de 3 pessoas pode ser design local para um time de 30.

## Erros Comuns

- Chamar decisão local de arquitetura completa.
- Usar padrão arquitetural para problema pequeno.
- Misturar design de classe com estilo arquitetural.
- Entrar em microsserviços cedo demais.
- Avaliar padrão apenas pela elegância técnica, sem considerar custo operacional.
- Rejeitar padrão POSA por aversão a complexidade quando os dados justificam.
- Ignorar que o limite entre design e arquitetura depende de escala e equipe.

## Checklist de Entendimento

- O problema é local ou sistêmico?
- Qual é o menor padrão que resolve este problema hoje?
- Qual seria o custo operacional de adotar um padrão POSA agora?
- O padrão organiza objetos ou partes maiores?
- O módulo 14 deve aprofundar isso ou apenas apresentar?
- Que assunto fica para arquitetura?
- O exemplo ainda cabe em código pequeno?
- Se o volume de negócio dobrar nos próximos 6 meses, qual padrão escolheria?
