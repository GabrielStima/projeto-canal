# Programacao Orientada a Objetos no Design

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar POO como forma de organizar comportamento e estado;
- diferenciar classe de objeto conceitual;
- reconhecer quando POO ajuda no design;
- evitar POO cerimonial.

## Onde Esta Aula Entra no Curso

- Modulo: 14. Design de Software
- Aula anterior: Features de Paradigmas
- Proxima aula: Principios Primarios de POO
- Pre-requisitos: Classes em TypeScript, objetos, interfaces, modulos e testes.

## Ideia Central

POO nao e apenas criar classes. A ideia central e aproximar estado e comportamento quando isso protege regras, reduz dispersao e cria um modelo mais expressivo.

Em TypeScript, POO pode conviver com funcoes e modulos. O ponto e usar objetos quando eles representam conceitos com comportamento proprio, nao quando queremos apenas agrupar funcoes.

## Topicos da Aula

- Objeto, classe e instancia.
- Estado e comportamento.
- Identidade.
- Colaboracao entre objetos.
- POO em TypeScript.
- POO sem excesso de cerimonia.

## Exemplo Pratico

```ts
class StudyProgress {
  completeLesson(lessonId: string) {
    // protege regra de conclusao em um lugar
  }
}
```

## Analogias e Intuicoes

- Um objeto bom sabe proteger suas proprias regras.
- Classe vazia de comportamento costuma ser apenas estrutura de dados.

## Erros Comuns

- Criar getters e setters para tudo.
- Separar dados e comportamento sem motivo.
- Usar heranca cedo demais.
- Criar classes que apenas chamam outras classes.

## Checklist de Entendimento

- Este conceito tem comportamento proprio?
- O estado precisa ser protegido?
- A classe revela uma regra do dominio?
- Uma funcao simples resolveria melhor?
- O teste ficou mais claro?

## Pratica Recomendada

- Escolha uma regra com estado.
- Modele com objeto.
- Escreva um teste para comportamento publico.
- Evite testar detalhes internos.
