# Programação Orientada a Objetos no Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar POO como forma de organizar comportamento e estado;
- diferenciar classe de objeto conceitual;
- reconhecer quando POO ajuda no design;
- evitar POO cerimonial.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Features de Paradigmas
- Próxima aula: Princípios Primários de POO
- Pré-requisitos: Classes em TypeScript, objetos, interfaces, módulos e testes.

## Ideia Central

POO não é apenas criar classes. A ideia central é aproximar estado e comportamento quando isso protege regras, reduz dispersão e cria um modelo mais expressivo.

Em TypeScript, POO pode conviver com funções e módulos. O ponto é usar objetos quando eles representam conceitos com comportamento próprio, não quando queremos apenas agrupar funções.

## Tópicos da Aula

- Objeto, classe e instância.
- Estado e comportamento.
- Identidade.
- Colaboração entre objetos.
- POO em TypeScript.
- POO sem excesso de cerimônia.

## Exemplo Prático

```ts
class StudyProgress {
  completeLesson(lessonId: string) {
    // protege regra de conclusão em um lugar
  }
}
```

## Analogias e Intuições

- Um objeto bom sabe proteger suas próprias regras.
- Classe vazia de comportamento costuma ser apenas estrutura de dados.

## Erros Comuns

- Criar getters e setters para tudo.
- Separar dados e comportamento sem motivo.
- Usar herança cedo demais.
- Criar classes que apenas chamam outras classes.

## Checklist de Entendimento

- Este conceito tem comportamento próprio?
- O estado precisa ser protegido?
- A classe revela uma regra do domínio?
- Uma função simples resolveria melhor?
- O teste ficou mais claro?

## Prática Recomendada

- Escolha uma regra com estado.
- Modele com objeto.
- Escreva um teste para comportamento público.
- Evite testar detalhes internos.
