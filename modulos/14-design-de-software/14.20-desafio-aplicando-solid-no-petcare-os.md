# Desafio: Aplicando SOLID no PetCare OS

## Objetivo

Ao final deste desafio, você deve ser capaz de:

- identificar violações dos cinco princípios SOLID em código real;
- refatorar um trecho de código aplicando os princípios estudados;
- justificar cada decisão de design com vocabulário técnico;
- perceber a diferença entre código que reconhece SOLID e código que o aplica.

## Onde Esta Aula Entra na Formação

- Módulo: 14. Design de Software
- Aula anterior: Dependency Inversion Principle
- Próxima aula: Program Against Abstractions
- Pré-requisitos: DRY, YAGNI e os cinco princípios SOLID (14.12–14.19).

## Ideia Central

Este é um desafio de fixação. Até aqui você estudou os princípios individualmente. Agora o objetivo é aplicá-los juntos sobre um cenário real do PetCare OS.

Lembrar os princípios é diferente de usá-los com critério. Este desafio existe para cruzar essa fronteira antes de avançar para padrões de design.

## O Cenário

O PetCare OS possui um módulo de agendamento de consultas. Um trecho de código simplificado foi escrito com pressa e acumula problemas de design:

```ts
class AppointmentService {
  async schedule(data: any) {
    if (!data.petId || !data.vetId || !data.date) {
      throw new Error('Dados inválidos')
    }

    if (data.date < new Date()) {
      throw new Error('Data no passado')
    }

    const result = await db.query(
      `INSERT INTO appointments (pet_id, vet_id, date, status)
       VALUES ($1, $2, $3, 'pending')
       RETURNING *`,
      [data.petId, data.vetId, data.date]
    )

    await fetch('https://notify.petcareos.internal/email', {
      method: 'POST',
      body: JSON.stringify({
        to: data.ownerEmail,
        subject: 'Consulta agendada',
        body: `Consulta para ${data.petName} em ${data.date}`
      })
    })

    await fetch('https://notify.petcareos.internal/sms', {
      method: 'POST',
      body: JSON.stringify({ phone: data.ownerPhone, message: `Consulta em ${data.date}` })
    })

    return result.rows[0]
  }

  async cancel(appointmentId: string, reason: string) {
    await db.query(
      `UPDATE appointments SET status = 'cancelled', reason = $2 WHERE id = $1`,
      [appointmentId, reason]
    )

    await fetch('https://notify.petcareos.internal/email', {
      method: 'POST',
      body: JSON.stringify({ subject: 'Consulta cancelada' })
    })
  }

  async reschedule(appointmentId: string, newDate: Date) {
    if (newDate < new Date()) {
      throw new Error('Data no passado')
    }

    await db.query(
      `UPDATE appointments SET date = $2 WHERE id = $1`,
      [appointmentId, newDate]
    )
  }
}
```

## Desafio

**Etapa 1 — Diagnóstico (não escreva código ainda)**

Leia o código acima e responda por escrito:

1. Quais responsabilidades distintas essa classe carrega?
2. Qual princípio SOLID é violado pelo fato de a validação de data aparecer em dois métodos?
3. Qual princípio SOLID é violado pela dependência direta do `db` e do `fetch`?
4. Se você precisasse adicionar notificação por push, em quantos lugares mexeria? Qual princípio isso viola?
5. Se você criasse uma subclasse `UrgentAppointmentService`, quais partes do comportamento você conseguiria substituir com segurança?

**Etapa 2 — Refatoração**

Refatore o código aplicando os princípios identificados. Não existe uma resposta única correta — o que importa é que cada mudança tenha justificativa.

Sugestões de pontos de partida (escolha um ou mais):

- Extraia a validação de data para um objeto ou função com responsabilidade única.
- Crie uma abstração para envio de notificação que não dependa diretamente do `fetch`.
- Separe a persistência da regra de negócio usando um repositório com contrato mínimo.
- Organize as responsabilidades de forma que adicionar um novo canal de notificação não exija mudar `AppointmentService`.

**Etapa 3 — Registro de decisões**

Após a refatoração, escreva um parágrafo curto respondendo:

- Que princípio foi mais difícil de aplicar e por quê?
- Em algum momento SOLID criou mais complexidade do que resolveu? Se sim, onde e por quê.
- Que parte do código você escolheu *não* refatorar e qual foi o critério?

## Critérios de Avaliação

- Consegue nomear o princípio violado antes de refatorar?
- A refatoração melhora testabilidade sem criar indireção gratuita?
- As abstrações têm nome claro e tamanho pequeno?
- O código refatorado ainda seria legível para outro desenvolvedor?
- A justificativa de cada decisão usa vocabulário dos princípios estudados?
