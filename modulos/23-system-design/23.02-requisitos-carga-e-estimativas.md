# Requisitos, Carga e Estimativas

## Objetivo

Ao final desta aula, você deve ser capaz de:

- levantar requisitos de um sistema;
- estimar tráfego, armazenamento e throughput;
- usar números aproximados para orientar decisões;
- identificar gargalos prováveis antes da implementação.

## Onde Esta Aula Entra na Formação

- Módulo: 23. System Design
- Aula anterior: 23.01 Como Pensar em Design de Sistemas
- Próxima aula: 23.03 Disponibilidade vs Consistência
- Pré-requisitos: APIs, bancos de dados, performance, observabilidade, capacidade e saturação.

## Ideia Central

Estimativas em system design não existem para acertar o número exato. Elas existem para evitar decisões cegas. Uma arquitetura para cem usuários por dia é diferente de uma arquitetura para cem mil requisições por segundo.

O objetivo é chegar a uma ordem de grandeza: quantos usuários ativos, quantas leituras, quantas escritas, quanto dado por registro, quanto armazenamento por mês, qual latência aceitável e quais fluxos são críticos.

Com esses números, você consegue decidir se uma solução simples basta ou se precisa pensar em cache, filas, particionamento, réplicas, CDN, autoscaling ou outras estratégias.

## Exemplo Prático

A Matemática de Padaria (Back-of-the-envelope):

O Chefe fala: 'Vamos guardar as radiografias de todos os cães'.
O Arquiteto pensa: 1.000 clínicas * 50 radiografias/dia * 5MB por foto.
Isso dá `250 GB` por dia. Em um ano, são `91 Terabytes` de armazenamento de fotos inativas.
Se tentarmos guardar isso no disco de um Postgres local, o servidor explode em 2 meses. Conclusão: Precisamos usar *Object Storage* (AWS S3) com camadas de Cold Storage (Glacier) desde o dia zero.

## Analogias e Intuições

- Estimativa de Carga é calcular quantos quilos de comida você precisa comprar pro Buffet de um casamento de 5.000 pessoas. Se você tentar estimar na base do 'achismo' no meio da festa, os noivos passarão fome.

## Erros Comuns

- Não separar leitura de escrita.
- Usar usuários cadastrados como se todos fossem ativos.
- Ignorar picos de tráfego.
- Estimar armazenamento sem considerar logs, índices, backups e retenção.
- Fingir precisão com números inventados.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual é o fluxo mais acessado?
- Qual operação escreve mais?
- Qual dado cresce mais rápido?
- Qual latência é aceitável?
- O que muda se o tráfego crescer dez vezes?
