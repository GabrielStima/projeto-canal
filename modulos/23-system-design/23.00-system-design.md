## Ideia Central

System design é a prática de desenhar sistemas considerando requisitos, restrições, escala, confiabilidade, segurança, custo, dados e operação. O foco não é decorar uma arquitetura famosa. O foco é aprender a pensar: qual problema estamos resolvendo, quais suposições estamos fazendo, quais componentes entram, quais trade-offs aceitamos e como o sistema falha.

Um sistema pequeno pode funcionar com uma aplicação, um banco e um deploy simples. Conforme o produto cresce, aparecem perguntas novas: como distribuir tráfego, como reduzir latência, como manter dados consistentes, como lidar com falhas parciais, como observar produção, como proteger dados e como evoluir sem reescrever tudo.

Este módulo junta muitos assuntos que já apareceram na formação. A diferença é que agora eles serão usados como decisões de desenho, não como tópicos isolados.

## Exemplo Prático

Imagine uma plataforma de aulas online. No começo, ela pode ter:

- frontend;
- API;
- banco relacional;
- autenticação;
- storage para vídeos;
- logs básicos.

Quando o uso cresce, surgem decisões: usar CDN para vídeos, cache para páginas públicas, filas para processamento, banco gerenciado com réplica, rate limiting em APIs, monitoramento por fluxo crítico e estratégia de rollback.

System design é o raciocínio que conecta essas decisões.

## Analogias e Intuições

- System design é como projetar uma cidade: ruas, energia, água e segurança precisam trabalhar juntos.
- Trade-off é uma escolha com custo: melhorar uma propriedade quase sempre piora ou complica outra.
- Arquitetura é uma hipótese sobre como o sistema deve crescer e falhar.

## Erros Comuns

- Começar escolhendo ferramentas antes de entender requisitos.
- Copiar arquitetura de empresa grande para problema pequeno.
- Ignorar operação, custo e segurança.
- Desenhar somente o caminho feliz.
- Confundir escalabilidade com complexidade.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- O que system design tenta resolver?
- Por que requisitos não funcionais importam?
- O que é um trade-off técnico?
- Por que sistemas maiores falham de formas diferentes?
- Por que uma arquitetura simples pode ser a melhor primeira versão?

## Prática Recomendada

- Escolha uma aplicação que você já criou e desenhe sua arquitetura atual.
- Liste três limitações dessa arquitetura se o tráfego crescesse dez vezes.
- Liste uma melhoria simples e uma melhoria que talvez fosse exagerada.
