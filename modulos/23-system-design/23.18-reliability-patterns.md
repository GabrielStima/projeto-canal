# Reliability Patterns

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer padrões de confiabilidade;
- usar timeout, retry, backoff, circuit breaker e fallback;
- entender bulkhead e isolamento de falhas;
- evitar padrões aplicados de forma perigosa.

## Ideia Central

Sistemas falham. Reliability patterns ajudam a limitar dano, recuperar melhor e manter partes úteis funcionando quando dependências estão lentas ou indisponíveis.

Timeout evita espera infinita. Retry tenta novamente falhas transitórias. Backoff reduz pressão. Circuit breaker para de chamar uma dependência quebrada por um tempo. Fallback oferece alternativa. Bulkhead isola recursos para uma falha não consumir tudo.

Esses padrões só funcionam bem com limites, idempotência e observabilidade.

## Exemplo Prático

Uma API consulta um serviço externo de recomendação. Se esse serviço demora, a página inteira não deveria travar.

O sistema pode usar timeout curto, circuit breaker e fallback para recomendações populares. O usuário recebe uma experiência menor, mas ainda útil.

## Analogias e Intuições

- Timeout é parar de esperar para proteger o sistema.
- Circuit breaker é dar tempo para a dependência respirar.
- Bulkhead é impedir que uma área alagada afunde o navio inteiro.

## Erros Comuns

- Retry sem backoff.
- Retry em operação não idempotente.
- Timeout maior que o tempo aceitável do usuário.
- Fallback que esconde falha crítica.
- Circuit breaker sem métrica e alerta.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual dependência pode ficar lenta?
- Qual timeout faz sentido?
- A operação pode ser repetida?
- Qual fallback é aceitável?
- Como saber que o circuito abriu?

## Prática Recomendada

- Escolha três dependências externas de uma API.
- Defina timeout, retry e fallback para cada uma.
- Explique onde retry seria perigoso.
