# Padrões de Design em Cloud

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer padrões comuns de arquitetura em cloud;
- combinar serviços gerenciados com desenho de sistema;
- avaliar custo, operação e lock-in;
- escolher padrões proporcionais ao problema.

## Onde Esta Aula Entra no Curso

- Módulo: 23. System Design
- Aula anterior: 23.19 Disponibilidade, Alta Disponibilidade e Resiliência
- Próxima aula: 23.21 Design e Implementação em Cloud
- Pré-requisitos: cloud fundamentals, IaC, containers, Kubernetes, serverless, CDN, filas, bancos gerenciados, SRE e custos.

## Ideia Central

Cloud oferece blocos prontos: CDN, load balancer, compute, serverless, bancos gerenciados, filas, storage, secrets, observabilidade e rede. Padrões de design em cloud mostram formas recorrentes de combinar esses blocos.

Mas cloud não elimina system design. Ela muda as perguntas: qual serviço gerenciado reduz operação, qual aumenta custo, qual cria dependência, qual simplifica disponibilidade e qual exige limites claros?

O bom desenho usa cloud para resolver problemas reais, não para empilhar serviços.

## Tópicos da Aula

- Arquitetura web gerenciada.
- Event-driven em cloud.
- Serverless para workloads específicos.
- Storage + CDN.
- Managed databases e filas.
- Multi-conta, ambientes e isolamento.
- Custo, limites e lock-in.

## Exemplo Prático

Uma aplicação de uploads pode usar API para autorização, object storage para arquivos, fila para processamento, worker para gerar thumbnails e CDN para entrega.

Esse padrão evita que a aplicação principal carregue arquivos pesados e permite processar em segundo plano.

## Analogias e Intuições

- Cloud oferece peças prontas, mas não escolhe prioridades.
- Serviço gerenciado reduz trabalho operacional, mas adiciona limites próprios.
- Arquitetura boa em cloud é simples de operar e clara de justificar.

## Erros Comuns

- Usar serviço novo sem entender limite e custo.
- Criar dependência forte sem necessidade.
- Ignorar IAM e isolamento.
- Misturar ambientes de forma arriscada.
- Desenhar para escala imaginária.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual problema o serviço resolve?
- Qual custo cresce com uso?
- Qual limite operacional existe?
- Como controlar acesso?
- Como migrar ou reduzir lock-in se necessário?

## Prática Recomendada

- Escolha um sistema de upload e processamento.
- Desenhe com serviços cloud gerenciados.
- Liste custos, limites, riscos e alternativa mais simples.
