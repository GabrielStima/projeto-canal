# Projeto Prático: Desenhando um Sistema Escalável e Confiável

## Objetivo

Ao final desta aula, você deve ser capaz de:

- conduzir um exercício completo de system design;
- declarar requisitos, suposições e estimativas;
- desenhar uma arquitetura inicial e sua evolução;
- justificar decisões de escala, dados, confiabilidade, cloud e segurança.

## Ideia Central

Este projeto fecha o módulo juntando o raciocínio completo. A proposta é escolher um sistema e desenhá-lo de forma incremental: primeiro uma versão simples que funciona, depois uma evolução para mais tráfego, mais confiabilidade e melhor operação.

O importante não é criar o desenho mais sofisticado. É explicar cada escolha: por que esse banco, por que esse cache, por que essa fila, por que esse SLO, por que essa estratégia de segurança e o que fica fora do escopo.

System design bem feito mostra maturidade porque separa problema, restrição, decisão e consequência.

## Tópicos da Aula

- Escolha do produto e escopo.
- Requisitos funcionais e não funcionais.
- Estimativas de carga e armazenamento.
- Arquitetura inicial.
- Evolução para escala.
- Dados, cache e mensageria.
- Confiabilidade, monitoramento e segurança.
- Apresentação dos trade-offs.

## Exemplo Prático

Nesta atividade final você desenha a arquitetura do PetCare OS para um cenário de crescimento:

1. A dona da clínica pede um sistema onde tutores possam acompanhar o status do pet e receber atualizações em tempo real, com previsão de crescimento para várias unidades.
2. Você declara requisitos funcionais, não funcionais e estimativas de carga (usuários, requisições, armazenamento).
3. Usando Excalidraw, Lucidchart ou Mermaid, desenhe a arquitetura inicial e uma evolução para escala:
   - DNS e CDN para conteúdo estático.
   - Load balancer distribuindo requisições entre instâncias stateless da API Node.
   - Cache para leituras frequentes.
   - Filas para processamento assíncrono de eventos.
   - Banco relacional para dados transacionais e, se justificado, banco secundário para logs ou atividades.
4. Documente trade-offs: por que cada componente foi escolhido, quais alternativas foram descartadas e quais são os próximos gargalos.

## Analogias e Intuições

- System design é como projetar uma cidade: antes de escolher prédios, estradas e pontes, você precisa saber quantas pessoas vão morar ali, onde elas precisam ir e quanto pode custar. O desenho muda conforme os números e as restrições.

## Erros Comuns

- Criar arquitetura grande sem requisitos.
- Não estimar carga.
- Não separar versão inicial de evolução.
- Ignorar dados, segurança e operação.
- Não declarar trade-offs.

## Checklist de Entendimento

Antes de concluir, a pessoa deve conseguir responder:

- Qual problema o sistema resolve?
- Quais fluxos são críticos?
- Quais números orientam o desenho?
- Onde o sistema pode falhar?
- Quais trade-offs foram aceitos?
