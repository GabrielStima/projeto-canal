# Mensageria em Cloud

## Objetivo

Ao final desta aula, você deve ser capaz de:

- entender serviços de mensageria em cloud como decisão de arquitetura;
- diferenciar fila, tópico e pub/sub;
- reconhecer DLQ, retries, ordering e backpressure;
- desenhar consumidores confiáveis.

## Ideia Central

Serviços de mensageria em cloud ajudam sistemas a desacoplar componentes, absorver picos e processar trabalho de forma assíncrona. Uma fila entrega trabalho para consumidores. Um tópico publica eventos para múltiplos interessados.

Esses serviços reduzem carga operacional, mas exigem desenho cuidadoso: tamanho de mensagem, tempo de visibilidade, retries, DLQ, ordenação, deduplicação, permissões, custo e monitoramento.

Mensageria melhora resiliência quando o consumidor é preparado para falhas.

## Exemplo Prático

Um sistema de pedidos publica um evento `PedidoCriado`. Um consumidor envia e-mail, outro atualiza analytics e outro reserva estoque.

Se o consumidor de e-mail falhar, os outros podem continuar. Se a mensagem falhar repetidamente, ela vai para DLQ para análise, sem travar todo o fluxo.

## Analogias e Intuições

- Fila distribui trabalho.
- Tópico anuncia fato para vários interessados.
- DLQ é uma área de quarentena para mensagens problemáticas.

## Erros Comuns

- Ignorar mensagens duplicadas.
- Não configurar DLQ.
- Não monitorar idade da mensagem.
- Colocar payload sensível sem necessidade.
- Tratar ordering como garantido sem confirmar.
