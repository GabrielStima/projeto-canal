# Padrões de Disponibilidade

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar disponibilidade em termos práticos;
- reconhecer pontos únicos de falha;
- entender redundância, failover e degradação;
- escolher padrões proporcionais ao risco.

## Onde Esta Aula Entra no Curso

- Módulo: 23. System Design
- Aula anterior: 23.04 Padrões de Consistência
- Próxima aula: 23.06 DNS no System Design
- Pré-requisitos: SRE, SLI, SLO, cloud, regiões, zonas, health checks, rollback e degradação graciosa.

## Ideia Central

Disponibilidade não é "nunca cair". É a capacidade de manter o serviço útil dentro de um objetivo definido, mesmo com falhas. Para isso, sistemas usam redundância, health checks, failover, isolamento, degradação graciosa e recuperação.

O nível certo depende do produto. Um blog pessoal pode aceitar horas fora do ar. Um checkout, um sistema médico ou uma autenticação central exigem outro cuidado.

Cada aumento de disponibilidade traz custo e complexidade. Por isso, disponibilidade precisa ser desenhada a partir de impacto, não de vaidade técnica.

## Tópicos da Aula

- Pontos únicos de falha.
- Redundância.
- Health checks e failover.
- Degradação graciosa.
- Multi-zona e multi-região.
- Custo de disponibilidade.

## Exemplo Prático

Uma API roda em uma única instância. Se ela falhar, o sistema sai do ar. Uma melhoria inicial é rodar múltiplas instâncias atrás de um load balancer com health checks.

Depois, pode fazer sentido distribuir entre zonas de disponibilidade. Multi-região só deve entrar quando o risco, o público e o custo justificarem.

## Analogias e Intuições

- Disponibilidade é continuidade de serviço, não ausência de falha.
- Redundância sem health check pode só duplicar problema.
- Alta disponibilidade custa dinheiro, operação e simplicidade.

## Erros Comuns

- Declarar "alta disponibilidade" sem SLO.
- Criar redundância sem testar failover.
- Depender de um banco único sem plano de recuperação.
- Achar que Kubernetes resolve disponibilidade sozinho.
- Ignorar dependências externas.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual é o SLO do sistema?
- Quais são os pontos únicos de falha?
- O que acontece se uma zona cair?
- O sistema degrada ou para totalmente?
- O failover foi testado?

## Prática Recomendada

- Desenhe uma API simples e marque pontos únicos de falha.
- Proponha uma melhoria de disponibilidade barata.
- Proponha uma melhoria cara e explique quando ela seria justificável.
