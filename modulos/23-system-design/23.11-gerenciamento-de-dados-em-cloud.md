# Gerenciamento de Dados em Cloud

## Objetivo

Ao final desta aula, você deve ser capaz de:

- entender decisões de dados em ambientes cloud;
- avaliar bancos gerenciados, backups, restore e regiões;
- reconhecer impactos de custo, segurança e governança;
- desenhar dados pensando em operação.

## Onde Esta Aula Entra na Formação

## Ideia Central

Em cloud, dados não são apenas tabelas e documentos. Eles envolvem serviços gerenciados, regiões, zonas, replicação, snapshots, backups, restore, criptografia, acesso, auditoria, retenção e custo.

Um banco gerenciado reduz trabalho operacional, mas não elimina decisões de arquitetura. Você ainda precisa escolher tamanho, rede, permissões, política de backup, retenção, monitoramento, migração e recuperação.

Gerenciar dados em cloud é pensar no ciclo de vida: criar, acessar, proteger, replicar, arquivar, restaurar e descartar.

## Exemplo Prático

NoSQL vs Relacional sob carga brutal:

A Clínica quer armazenar relatórios estruturados financeiros consistentes (PostgreSQL).
Mas e se a clínica botar sensores de IoT (Internet das coisas) na coleira de 1 milhão de pets, enviando a temperatura do cão a cada segundo? O PostgreSQL vai ser dizimado pelos LOCKs de transação.
No System Design, a gente roteia cargas de leitura absurdas sem consistência perfeita para um banco NoSQL tipo Cassandra/DynamoDB (High Write Throughput). Cada dado pro seu balde correto.

## Analogias e Intuições

- Banco Relacional é o tabelionato do governo onde cada vírgula importa e demora. Banco NoSQL Column-Family é o bloco de anotações do repórter frenético onde ele joga palavras soltas a 200 km/h sem se preocupar em revisar.

## Erros Comuns

- Não testar restore.
- Dar permissões amplas demais.
- Ignorar custo de transferência entre regiões.
- Guardar dados para sempre sem necessidade.
- Não separar acesso humano de acesso da aplicação.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Onde os dados ficam?
- Quem acessa os dados?
- Como restaurar depois de falha?
- Quanto tempo os dados devem existir?
- Quais métricas mostram risco operacional?

## Prática Recomendada

- Relacione a estrutura clássica de Chave-Valor (Key-Value) com a velocidade alucinante de leitura provida por um Memcached ou Redis na camada corporativa.
