# Database no System Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- escolher banco considerando requisitos do sistema;
- relacionar modelo de dados, consultas e escala;
- entender índices, transações, replicação e particionamento como decisões de design;
- reconhecer quando SQL, NoSQL ou combinação fazem sentido.

## Ideia Central

Banco de dados é uma das decisões mais importantes de system design. Ele guarda o estado durável do sistema e influência consistência, latência, custo, disponibilidade, evolução e operação.

Não existe "melhor banco" isolado do problema. Um sistema transacional com relações fortes pode se beneficiar de banco relacional. Um catálogo com leitura massiva e estrutura flexível pode usar outro modelo. Um sistema de busca talvez precise de motor especializado além do banco principal.

A decisão precisa começar pelo acesso aos dados: que entidades existem, quais consultas são críticas, quais escritas acontecem, que consistência é necessária e como o dado cresce.

## Exemplo Prático

Quando o Monolito chora (Sharding e Replicas):

O PetCare tem 1 Banco de Dados PostgreSQL. Ele encheu.
1. A primeira solução Sênior: Separar em *Read-Replicas*. O Mestre (Master) só recebe gravações de dados (cirurgias). Os clientes leem os relatórios em 3 bancos secundários (Slaves/Replicas) apenas-leitura que ficam copiando o mestre.
2. Se nem isso der conta (O Banco tem trilhões de linhas), fazemos *Sharding*. Os cachorros de `A a M` salvam num Servidor. Os de `N a Z` em outro servidor. Quebramos o banco ao meio na horizontal. Complexidade absurda, escalabilidade infinita.

## Analogias e Intuições

- Replicas de leitura é ter 1 livro oficial que o escritor escreve (mestre), e imprimir 5 mil cópias pra galera ler. Sharding é rasgar a Enciclopédia em 30 volumes: quem quer a Letra C, vai procurar no Prédio C.

## Erros Comuns

- Escolher banco por moda.
- Ignorar padrão de consulta.
- Criar índice sem entender custo.
- Usar NoSQL para evitar modelagem.
- Não planejar backup, restore e migração.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual dado é fonte de verdade?
- Quais consultas são mais frequentes?
- Quais escritas precisam de transação?
- Como o dado cresce?
- Como recuperar o sistema se o banco falhar?
