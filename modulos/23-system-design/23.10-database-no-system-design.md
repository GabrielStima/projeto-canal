# Database no System Design

## Objetivo

Ao final desta aula, você deve ser capaz de:

- escolher banco considerando requisitos do sistema;
- relacionar modelo de dados, consultas e escala;
- entender índices, transações, replicação e particionamento como decisões de design;
- reconhecer quando SQL, NoSQL ou combinação fazem sentido.

## Ideia Central

Banco de dados é uma das decisões mais importantes de system design. Ele guarda o estado durável do sistema e influencia consistência, latência, custo, disponibilidade, evolução e operação.

Não existe "melhor banco" isolado do problema. Um sistema transacional com relações fortes pode se beneficiar de banco relacional. Um catálogo com leitura massiva e estrutura flexível pode usar outro modelo. Um sistema de busca talvez precise de motor especializado além do banco principal.

A decisão precisa começar pelo acesso aos dados: que entidades existem, quais consultas são críticas, quais escritas acontecem, que consistência é necessária e como o dado cresce.

## Tópicos da Aula

- Banco como fonte de verdade.
- SQL vs NoSQL como trade-off.
- Modelo de dados orientado por consultas.
- Índices e custo de escrita.
- Transações e consistência.
- Réplicas, particionamento e backup.

## Exemplo Prático

Em uma loja online, pedidos, pagamentos e estoque pedem consistência forte e transações. Produtos e catálogo podem exigir leitura rápida, busca e filtros. Analytics pode ser processado em outro armazenamento.

Uma arquitetura madura pode combinar banco relacional para pedidos, cache para catálogo, motor de busca para pesquisa e pipeline analítico para métricas.

## Analogias e Intuições

- Banco é memória de longo prazo do sistema.
- Índice acelera leitura, mas cobra custo em escrita e armazenamento.
- Modelo de dados ruim aparece como consulta cara.

## Erros Comuns

- Escolher banco por moda.
- Ignorar padrão de consulta.
- Criar índice sem entender custo.
- Usar NoSQL para evitar modelagem.
- Não planejar backup, restore e migração.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual dado é fonte de verdade?
- Quais consultas são mais frequentes?
- Quais escritas precisam de transação?
- Como o dado cresce?
- Como recuperar o sistema se o banco falhar?

## Prática Recomendada

- Modele dados de uma plataforma de cursos.
- Liste três consultas críticas.
- Escolha banco e índices iniciais justificando trade-offs.
