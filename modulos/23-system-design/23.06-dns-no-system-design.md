## Ideia Central

DNS é uma das primeiras peças do caminho de uma requisição. Antes de chegar ao load balancer, CDN ou aplicação, o cliente precisa descobrir para onde enviar tráfego.

Em system design, DNS importa porque influência disponibilidade, roteamento, latência, failover, migrações e recuperação. Um TTL muito alto pode tornar mudanças lentas. Uma dependência mal desenhada pode derrubar acesso mesmo que a aplicação esteja saudável.

DNS não resolve todos os problemas de tráfego, mas é parte essencial da borda do sistema.

## Exemplo Prático

O Agente de Trânsito Global (DNS):

O cliente digita `petcare-os.com`.
Antes de bater no seu Servidor Node.js, ele bate no DNS (Domain Name System).
O DNS avançado (Ex: AWS Route 53) não só converte nome pra IP. Ele faz *Geo-Routing*.
Se o cliente está em Portugal, o DNS responde com o IP do servidor na Europa. Se o cliente está no Brasil, ele responde com o IP do Brasil. Ele faz balanceamento de tráfego na borda da internet.

## Analogias e Intuições

- O DNS não é só a lista telefônica (Converter nome para número de telefone). O DNS avançado é o serviço de Táxi: Se você ligar de SP, ele manda um táxi de SP. Se ligar do RJ, ele roteia pro motorista do RJ.

## Erros Comuns

- Esquecer TTL antes de migrações.
- Usar DNS como único mecanismo de balanceamento.
- Não monitorar expiração ou configuração de domínio.
- Confundir DNS failover com recuperação instantânea.
- Ignorar dependência do provedor de DNS.

- Como DNS se relaciona com CDN?

## Prática Recomendada
