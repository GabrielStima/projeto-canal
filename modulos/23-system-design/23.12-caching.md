# Caching

## Ideia Central

Cache guarda uma cópia de dados para responder mais rápido, reduzir carga e melhorar experiência. Ele pode existir no browser, CDN, aplicação, memória local, cache distribuído ou banco.

O benefício vem com risco: dado antigo, invalidação difícil, inconsistência, vazamento de dados e picos quando muitas chaves expiram ao mesmo tempo.

Cache deve ser usado onde há leitura repetida, custo alto de recomputação ou latência relevante. Não deve esconder problema de modelagem sem entendimento.

## Exemplo Prático

O Escudo de Memória (Redis):

O banco de dados relacional (Postgres) demora `15ms` para achar a Ficha da Cirurgia no disco rígido. Se tivermos 10.000 requests, o banco trava.
O Caching introduz um escudo de memória RAM rápida (Redis) na frente.
Quando a clínica pede a Ficha, a API olha pro Redis primeiro. Se estiver lá (Cache Hit), ele responde em `1ms` (velocidade da luz) sem nunca tocar no banco lento. Se a ficha mudou, nós invalidamos o Cache.

## Analogias e Intuições

- Cache é você lembrar o número de telefone da sua mãe de cabeça. Banco de Dados é você pegar o seu celular e procurar na agenda de contatos demoradamente toda vez que quiser ligar pra ela.

## Erros Comuns

- Cachear dado personalizado como se fosse público.
- Não medir cache hit ratio.
- Usar TTL arbitrário.
- Esquecer invalidação em atualização.
- Cachear erro por tempo demais.
