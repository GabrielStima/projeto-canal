# Caching

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que cache existe;
- diferenciar cache local, distribuído, HTTP e CDN;
- entender TTL, invalidação e cache stampede;
- decidir quando cache ajuda ou atrapalha.

## Ideia Central

Cache guarda uma cópia de dados para responder mais rápido, reduzir carga e melhorar experiência. Ele pode existir no browser, CDN, aplicação, memória local, cache distribuído ou banco.

O benefício vem com risco: dado antigo, invalidação difícil, inconsistência, vazamento de dados e picos quando muitas chaves expiram ao mesmo tempo.

Cache deve ser usado onde há leitura repetida, custo alto de recomputação ou latência relevante. Não deve esconder problema de modelagem sem entendimento.

## Tópicos da Aula

- Cache aside.

- Cache stampede.
- Dados públicos, privados e sensíveis.

## Exemplo Prático

Uma API de catálogo consulta produtos populares milhares de vezes por minuto. Colocar esses produtos em cache reduz carga no banco.

Mas se preço ou estoque mudam, o sistema precisa decidir se invalida imediatamente, usa TTL curto ou aceita atraso. Para checkout, talvez seja necessário confirmar o valor final no banco antes de fechar pedido.

## Analogias e Intuições

- Cache é um atalho com prazo de validade.
- Invalidação é decidir quando o atalho ficou perigoso.
- Cache bom reduz trabalho; cache ruim esconde erro.

## Erros Comuns

- Cachear dado personalizado como se fosse público.
- Não medir cache hit ratio.
- Usar TTL arbitrário.
- Esquecer invalidação em atualização.
- Cachear erro por tempo demais.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- O dado pode ficar velho?
- Quanto atraso é aceitável?
- Quem pode ver esse dado?
