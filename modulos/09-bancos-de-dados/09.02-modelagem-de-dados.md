# Modelagem de Dados

## Objetivo

Ao final desta aula, você deve ser capaz de:

- identificar entidades, atributos e relacionamentos;
- transformar uma necessidade de produto em estrutura de dados;
- perceber decisões ruins de modelagem cedo;
- preparar uma base para SQL e NoSQL.

## Onde Esta Aula Entra na Formação

- Módulo: 09. Bancos de Dados
- Aula anterior: SQL e NoSQL
- Próxima aula: SQL Básico
- Pré-requisitos: SQL e NoSQL, objetos, arrays, formulários e noções de aplicação.

## Ideia Central

Modelagem de dados é decidir como representar informações importantes para um sistema. Antes de pensar em comando, tabela ou collection, você precisa entender quais coisas existem no domínio, quais dados cada coisa possui e como elas se relacionam.

Uma boa modelagem não tenta prever o universo inteiro. Ela representa o problema atual com clareza e deixa espaço para evoluir sem quebrar tudo.

## Tópicos da Aula

- Entidades.
- Atributos.
- Relacionamentos.
- Cardinalidade.
- Dados obrigatórios e opcionais.
- Identificadores.
- Modelo conceitual, lógico e físico em nível introdutório.

## Exemplo Prático

Para o Portal do Tutor:

```text
Entidades:
- Tutor
- Pet
- Atendimento
- Orcamento
- ItemOrcamento
- Servico

Relacionamentos:
- Tutor possui muitos Pets
- Pet possui muitos Atendimentos
- Atendimento pode gerar um Orcamento
- Orcamento possui muitos ItensOrcamento
- ItemOrcamento referencia um Servico
```

Esse desenho vem antes de criar tabelas ou documentos.

## Analogias e Intuições

- Modelagem é o mapa antes da obra.
- Uma tabela ou collection é uma consequência da decisão de modelagem, não o começo da conversa.

## Erros Comuns

- Começar pelo comando antes de entender o dado.
- Guardar tudo em uma estrutura gigante.
- Separar dados demais sem necessidade.
- Não definir identificadores claros.
- Ignorar regras do domínio.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quais entidades existem no problema?
- Quais atributos pertencem a cada entidade?
- Quais relações são um-para-um, um-para-muitos ou muitos-para-muitos?
- Que dados são obrigatórios?
- Que regras precisam ser protegidas?
