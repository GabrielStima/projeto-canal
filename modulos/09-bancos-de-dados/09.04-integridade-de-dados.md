## Ideia Central

Integridade de dados é a confiança de que as informações armazenadas continuam coerentes. Validação na interface e no backend ajuda, mas o banco também precisa proteger regras importantes.

Constraints como `PRIMARY KEY`, `FOREIGN KEY`, `NOT NULL`, `UNIQUE` e `CHECK` impedem muitos estados inválidos antes que eles virem problema em produção.

## Exemplo Prático

```sql
CREATE TABLE orcamentos (
  id INT PRIMARY KEY,
  atendimento_id INT NOT NULL UNIQUE,
  status VARCHAR(20) NOT NULL,
  desconto_percentual NUMERIC(5, 2) NOT NULL DEFAULT 0,
  FOREIGN KEY (atendimento_id) REFERENCES atendimentos(id),
  CHECK (desconto_percentual BETWEEN 0 AND 100)
);
```

A regra `UNIQUE (atendimento_id)` evita dois orçamentos para o mesmo atendimento nesta versão do produto. A foreign key impede um orçamento sem atendimento válido, e o `CHECK` bloqueia percentuais impossíveis.

## Analogias e Intuições

- Validação na aplicação é uma porta com aviso.
- Constraint no banco é uma trava na estrutura.

## Erros Comuns

- Confiar apenas no frontend para validar dados.
- Criar tabelas sem chave primária.
- Permitir `NULL` em campos que deveriam ser obrigatórios.
- Duplicar dados que deveriam ser únicos.
- Usar chave estrangeira sem entender a relação.
