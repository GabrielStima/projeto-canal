# Projeto Prático: Persistência de Dados em uma Aplicação Fullstack

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os principais conceitos do módulo em um projeto pequeno;
- verificar uma camada relacional construída progressivamente;
- registrar uma comparação proporcional entre SQL e MongoDB;
- preparar a transição para backend com Node.js.

## Onde Esta Aula Entra na Formação

- Módulo: 09. Bancos de Dados
- Aula anterior: Security em NoSQL e MongoDB
- Próxima aula: 10. Backend com Node.js
- Pré-requisitos: Todas as aulas anteriores do módulo 09.

## Ideia Central

Você já construiu o Portal do Tutor em Next.js e, ao longo deste módulo, produziu modelo relacional, schema, seed, consultas, transaction, índices, migrations e decisões de segurança e recuperação.

O fechamento não começa outro projeto. Ele organiza e verifica essa camada de dados para que o módulo de backend consiga conectá-la a uma API Node.js. O estudo documental permanece uma comparação: MongoDB só deve entrar na implementação se os requisitos justificarem a complexidade adicional.

## Tópicos da Aula

- Escopo do projeto.
- Entidades principais.
- Modelo relacional.
- Consultas SQL.
- Constraints.
- Índices.
- Migration inicial.
- Backup e restore.
- Modelo MongoDB.
- Queries e aggregation.
- Segurança básica.
- Comparação final.

## Exemplo Prático

```text
Aplicação: Portal do Tutor do PetCare OS

Trabalho já produzido:
- tutores, pets, atendimentos, orçamentos, itens e serviços;
- schema e dados iniciais;
- consultas do portal;
- transaction de aprovação;
- índices e migrations;
- recuperação, acesso e decisão de persistência.

Verificação:
- executar schema e seed em um banco vazio;
- executar consultas;
- provocar uma violação de constraint;
- demonstrar commit e rollback;
- aplicar migrations em outro banco vazio e comparar a estrutura;
- documentar a ordem de execução.
```

## Analogias e Intuições

- Este projeto é uma ponte entre aprender sobre bancos e conectá-los ao backend.
- O objetivo é tomar decisões e justificar, não decorar a resposta perfeita.

## Erros Comuns

- Recriar tabelas no encerramento e abandonar os arquivos dos exercícios.
- Querer usar ORM antes de entender o banco.
- Documentar uma ordem de execução que não funciona.
- Declarar que MongoDB é obrigatório sem um requisito que o justifique.
- Esquecer de testar constraint, rollback e migration.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quais dados a aplicação precisa persistir?
- Qual é o modelo relacional?
- Quais consultas principais a aplicação faz?
- Quais constraints protegem regras importantes?
- Que índices parecem necessários?
- Como o schema evoluiria?
- Qual seria o modelo documental?
- Que riscos de segurança existem?
- O que preciso revisar antes de conectar isso a Node.js?
