# Migrations

## Objetivo

## Ideia Central

Migration é uma mudança versionada na estrutura do banco. Ela registra como o schema evolui ao longo do tempo: criar tabela, adicionar coluna, alterar constraint, criar índice ou remover estrutura antiga.

Sem migrations, cada ambiente pode ter um banco diferente. Com migrations, a evolução do banco acompanha o projeto de forma rastreável.

- Criar tabela.

- Alterar constraint.
- Criar índice.

- Riscos em produção.

## Exemplo Prático

```sql
ALTER TABLE aulas
ADD COLUMN publicado_em TIMESTAMP;
```

Como comando isolado, isso muda uma tabela. Como migration, essa mudança ganha nome, ordem e histórico.

## Analogias e Intuições

- Migration é o histórico de reformas da estrutura do banco.
- Versionar schema evita depender de memória e comandos manuais.

## Erros Comuns

- Alterar banco manualmente e esquecer de registrar.
- Remover coluna usada pela aplicação.
- Fazer migration destrutiva sem plano.
- Não pensar em dados existentes.
- Confundir migration com backup.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que mudança essa migration faz?
- Ela afeta dados existentes?
- A aplicação atual ainda funciona depois dela?
- Existe caminho de rollback?
- Essa mudança foi testada em ambiente local?
