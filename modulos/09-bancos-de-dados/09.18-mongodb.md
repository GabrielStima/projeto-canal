# MongoDB

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é MongoDB;
- entender por que ele é chamado de banco de documentos;
- reconhecer documentos e collections;
- comparar a experiência inicial com bancos relacionais.

## Ideia Central

MongoDB é um banco de dados orientado a documentos. Em vez de organizar tudo em tabelas e linhas, ele armazena documentos flexíveis, parecidos com objetos JSON, agrupados em collections.

Essa proximidade com estruturas usadas em JavaScript ajuda no aprendizado, mas não elimina decisões importantes de modelagem, consulta, consistência e segurança.

## Tópicos da Aula

- Documentos JSON-like.

- Quando MongoDB pode fazer sentido.

## Exemplo Prático

```javascript
{
  _id: "audit-2026-001",
  acao: "orcamento_aprovado",
  entidade: { tipo: "orcamento", id: 310 },
  realizadoPor: { usuarioId: 27, papel: "tutor" },
  alteracoes: [
    { campo: "status", anterior: "pendente", novo: "aprovado" }
  ],
  ocorridoEm: "2026-06-12T14:30:00Z"
}
```

Esse documento guarda o evento e suas alterações na mesma estrutura. Ele serve para estudar o modelo documental; não prova, sozinho, que o projeto precise adotar MongoDB.

## Analogias e Intuições

- Uma collection lembra uma pasta de documentos parecidos.
- Documento lembra um objeto persistido, mas com responsabilidade de banco.

## Erros Comuns

- Achar que MongoDB é apenas JSON salvo em disco.
- Não modelar porque o schema é flexível.
- Embutir tudo em um documento enorme.
- Criar documentos com formatos inconsistentes sem necessidade.
- Ignorar índices e padrões de consulta.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O que é um documento?
- O que é uma collection?
- Para que serve `_id`?
- Por que flexibilidade não elimina modelagem?
- Quando um documento pode ser mais natural que várias tabelas?
