## Ideia Central

Toda aplicação precisa guardar dados. A forma como você os organiza define o que é possível buscar, o quão rápido isso acontece e o quão difícil é manter a integridade ao longo do tempo. Bancos de dados não são apenas "onde a informação fica" — são a base sobre a qual regras de negócio se sustentam.

Bancos relacionais como PostgreSQL garantem integridade via esquemas, constraints e transactions ACID. Bancos não relacionais como MongoDB oferecem flexibilidade de esquema e escalabilidade horizontal para dados menos estruturados. A escolha não é ideológica: é sobre o problema que precisa ser resolvido.

Este módulo cobre desde SQL básico até MongoDB, passando por modelagem, índices, transactions, migrations e segurança, com foco em tomar decisões conscientes sobre persistência.

## Analogias e Intuições

- Um banco relacional é como um arquivo de prontuários físicos com fichas vinculadas: tudo referenciado, nada duplicado, ordem garantida.
- Um índice é como o índice de um livro: não muda o conteúdo, mas torna a busca drasticamente mais rápida.
- Uma transaction é como um contrato médico assinado: ou tudo é confirmado junto, ou nada entra no sistema.

## Erros Comuns

- Modelar tudo como uma tabela única para "simplificar" e criar redundância.
- Não usar índices em colunas que aparecem frequentemente em WHERE e JOIN.
- Tratar bancos relacionais e não relacionais como intercambiáveis sem considerar o modelo de consistência.
- Fazer migrations sem testar em staging e quebrar dados em produção.
- Armazenar senhas em texto puro no banco de dados.
