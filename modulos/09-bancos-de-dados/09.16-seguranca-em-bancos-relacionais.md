# Segurança em Bancos Relacionais

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer riscos comuns em bancos relacionais;
- entender o papel de permissões;
- relacionar SQL injection com entrada não confiável;
- preparar uma base para o módulo de segurança.

## Onde Esta Aula Entra na Formação

- Módulo: 09. Bancos de Dados
- Aula anterior: Backup e Restore
- Próxima aula: NoSQL Básico
- Pré-requisitos: SQL básico, validação, variáveis de ambiente e noções de backend-for-frontend.

## Ideia Central

Segurança em banco começa com uma postura simples: dados são valiosos e toda entrada externa precisa ser tratada como não confiável.

Mesmo antes de estudar segurança em profundidade, você precisa entender permissões mínimas, segredo de conexão, cuidado com queries dinâmicas, backups protegidos e separação entre ambientes.

## Tópicos da Aula

- Princípio do menor privilégio.
- Usuários e permissões.
- Segredos de conexão.
- SQL injection em nível introdutório.
- Queries parametrizadas em nível conceitual.
- Separação de ambientes.
- Proteção de backups.

## Exemplo Prático

```text
Risco:
- usuário digita texto em um formulário;
- aplicação monta SQL concatenando string;
- entrada maliciosa altera o sentido da query.

Defesa conceitual:
- validar entrada;
- usar queries parametrizadas;
- limitar permissões do usuário do banco.
```

## Analogias e Intuições

- Permissão de banco deve ser chave de trabalho, não chave mestra.
- Entrada do usuário nunca deve virar comando sem tratamento.

## Erros Comuns

- Usar o usuário administrador na aplicação.
- Colocar senha de banco no código.
- Montar SQL por concatenação de strings.
- Usar o mesmo banco para teste e produção.
- Deixar backups sem proteção.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual usuário a aplicação usa para se conectar?
- Esse usuário tem permissões mínimas?
- Onde ficam os segredos?
- A aplicação concatena SQL com entrada do usuário?
- Backup também é dado sensível?

## Prática Recomendada

- Liste permissões que uma aplicação comum realmente precisa.
- Explique por que não usar um usuário administrador no código.
- Descreva um exemplo conceitual de SQL injection.
- Escreva três regras de segurança para ambiente local, teste e produção.
