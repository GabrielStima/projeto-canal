# SQL e NoSQL

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar a diferença entre SQL, bancos relacionais e NoSQL;
- reconhecer quando um modelo relacional ajuda;
- reconhecer quando um modelo NoSQL pode fazer sentido;
- evitar escolher um banco de dados apenas por moda.

## Onde Esta Aula Entra na Formação

- Módulo: 09. Bancos de Dados
- Aula anterior: Bancos de Dados
- Próxima aula: Modelagem de Dados
- Pré-requisitos: HTTP, estado, objetos, arrays, TypeScript e noções de aplicação fullstack.

## Ideia Central

SQL é uma linguagem usada para consultar e manipular dados em bancos relacionais. Bancos relacionais organizam dados em tabelas, linhas, colunas, relações e regras de integridade.

NoSQL é uma família ampla de bancos que não seguem obrigatoriamente o modelo relacional. Existem bancos de documentos, chave-valor, grafos e colunas. O ponto não é dizer que um é melhor que o outro, mas entender que cada modelo favorece certos tipos de dados, consultas e evolução.

## Tópicos da Aula

- O que é persistência.
- O que é banco relacional.
- O que é SQL.
- O que significa NoSQL.

- Trade-offs de consistência, flexibilidade e consulta.

## Exemplo Prático

O Portal do Tutor precisa guardar tutores, pets, atendimentos e orçamentos.

```text
Modelo relacional:
- tutores
- pets
- atendimentos
- orcamentos
- itens_orcamento

Modelo documental:
- preferências de notificação embutidas no tutor
- registros de auditoria com detalhes da mudança
- snapshots de leitura quando o padrão de acesso justificar
```

No relacional, relações explícitas e integridade ajudam muito nos dados principais. No documental, alguns dados que normalmente são lidos juntos podem ficar próximos. Isso não significa que a primeira versão precise usar os dois modelos.

## Analogias e Intuições

- Um banco relacional parece uma planilha bem disciplinada, com regras fortes entre abas.
- Um banco de documentos parece guardar objetos completos, mais próximos do formato usado pela aplicação.

## Erros Comuns

- Achar que SQL é o banco, e não a linguagem.
- Achar que NoSQL significa "sem regras".
- Escolher MongoDB porque "não precisa modelar".
- Usar banco relacional sem pensar nas relações.
- Comparar ferramentas sem comparar o problema.
