## Ideia Central

Performance em SQL não começa com truques. Ela começa com boas perguntas: qual query está lenta, com quantos dados, com qual filtro, com quais relações e com qual frequência?

Otimizar é medir, entender e mudar com intenção. Às vezes a solução é um índice. Às vezes é ajustar modelagem, limitar retorno, paginar, evitar consulta repetida ou simplificar uma leitura.

## Exemplo Prático

```sql
SELECT *
FROM aulas
ORDER BY criado_em DESC;
```

Essa consulta pode parecer inocente, mas em uma tabela grande ela busca todas as colunas e todos os registros. Uma versão melhor normalmente seleciona colunas específicas, filtra quando possível e pagina resultados.

## Analogias e Intuições

- Performance é investigação, não adivinhação.
- Uma query rápida com cem linhas pode virar problema com dez milhões.

## Erros Comuns

- Otimizar sem medir.
- Usar `SELECT *` por padrão.
- Não paginar listas grandes.
- Criar índices em excesso.
- Culpar o banco antes de olhar a query.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual query está lenta?
- Quantas linhas ela precisa analisar?
- Quais colunas realmente preciso retornar?
- Existe filtro ou paginação?
- Algum índice combina com essa consulta?

## Prática Recomendada

- Pegue três consultas do módulo.
- Reescreva cada uma selecionando apenas colunas necessárias.
- Adicione paginação conceitual a uma lista.
- Sugira um índice para uma consulta e explique o custo.
