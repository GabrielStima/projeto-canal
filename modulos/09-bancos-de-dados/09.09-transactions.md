# Transactions

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é uma transaction;
- entender por que algumas operações precisam acontecer juntas;
- reconhecer a ideia de commit e rollback;
- relacionar transactions com consistência.

## Ideia Central

Uma transaction agrupa operações para que elas sejam tratadas como uma unidade. Se tudo der certo, as mudanças são confirmadas. Se algo der errado, as mudanças podem ser desfeitas.

Isso é essencial quando uma ação de negócio depende de mais de uma escrita. Sem transaction, o banco pode ficar em um estado pela metade.

## Tópicos da Aula

- Operações que precisam acontecer juntas.
- `BEGIN`, `COMMIT` e `ROLLBACK`.
- Estado parcial.
- Consistência.
- Introdução à ideia de ACID.
- Exemplos de falha durante escrita.

## Exemplo Prático

```sql
BEGIN;

UPDATE orcamentos
SET status = 'aprovado'
WHERE id = 10
  AND status = 'pendente';

UPDATE atendimentos
SET status = 'em_procedimento'
WHERE id = (
  SELECT atendimento_id
  FROM orcamentos
  WHERE id = 10
);

COMMIT;
```

A aprovação do orçamento e a mudança de status do atendimento precisam permanecer coerentes. O exemplo é simplificado: uma implementação real também deve verificar quantas linhas foram alteradas antes do `COMMIT`.

## Analogias e Intuições

- Transaction é um pacote de mudanças que entra inteiro ou não entra.
- `ROLLBACK` é desfazer uma tentativa que não pode ficar pela metade.

## Erros Comuns

- Fazer múltiplas escritas relacionadas sem transaction.
- Achar que transaction corrige regra de negócio mal definida.
- Manter transactions abertas por tempo demais.
- Ignorar o que acontece quando uma operação falha no meio.
