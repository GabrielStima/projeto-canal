# CORS

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar Same-Origin Policy e CORS;
- entender quando o browser bloqueia uma resposta;
- configurar origens permitidas com cuidado;
- relacionar CORS com cookies, tokens e APIs.

## Ideia Central

CORS é um mecanismo do browser para controlar quando uma página de uma origem pode ler respostas de outra origem. Ele não é uma proteção genérica da API contra qualquer cliente. Ele protege o ambiente do browser.

Configurar CORS bem exige saber quais frontends podem chamar a API, quais métodos e headers são aceitos e se credenciais como cookies devem ser enviadas.

## Tópicos da Aula

- Origem: scheme, host e porta.
- Same-Origin Policy.
- Requisições simples.
- Preflight.
- `Access-Control-Allow-Origin`.
- `Access-Control-Allow-Credentials`.
- Headers permitidos.
- CORS e credenciais.

## Exemplo Prático

```http
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Headers: Content-Type, Authorization
```

Quando credenciais entram no fluxo, permitir qualquer origem é uma decisão perigosa e, muitas vezes, inválida.

## Analogias e Intuições

- CORS é uma regra do browser sobre quem pode ler a resposta.
- Uma API pública ainda precisa de autenticação, autorização e rate limiting.

## Erros Comuns

- Achar que CORS protege contra `curl` ou servidor malicioso.
- Usar `*` sem pensar.
- Liberar credenciais para origens demais.
- Confundir erro de CORS com erro da API.
- Resolver tudo desabilitando proteções do browser.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O que é origem?
- O que o browser bloqueia em CORS?
- O que é preflight?
- Quando `credentials` entra?
- Por que CORS não substitui autenticação?

## Prática Recomendada

- Escreva uma configuração CORS restrita.
- Explique o impacto de cookies cross-origin.
- Diferencie problema de CORS de erro `401`.
