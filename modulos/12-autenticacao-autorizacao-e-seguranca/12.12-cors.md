## Ideia Central

CORS é um mecanismo do browser para controlar quando uma página de uma origem pode ler respostas de outra origem. Ele não é uma proteção genérica da API contra qualquer cliente. Ele protege o ambiente do browser.

Configurar CORS bem exige saber quais frontends podem chamar a API, quais métodos e headers são aceitos e se credenciais como cookies devem ser enviadas.

## Exemplo Prático

Configuração de API Multi-Origem:

```http
Access-Control-Allow-Origin: https://app.petcareos.com.br
Access-Control-Allow-Credentials: true
Access-Control-Allow-Methods: GET, POST, PATCH, DELETE
```

A API do PetCare OS rejeitará o acesso do navegador de um tutor se ele estiver acessando um domínio não reconhecido, como `http://localhost:3000` (quando a política não permite) ou um site de terceiros que tenta injetar chamadas via fetch.

## Analogias e Intuições

- CORS é um filtro imposto pela portaria (navegador). Se o entregador do restaurante A (origem A) tenta entrar no condomínio B (origem B), o porteiro verifica as regras do condomínio. Se não for autorizado, o porteiro barra na porta do apartamento.
*Limite: O CORS não impede ferramentas como cURL ou Postman de fazerem a requisição, ele restringe ações do browser.*

## Erros Comuns

- Achar que CORS protege contra `curl` ou servidor malicioso.
- Usar `*` sem pensar.
- Liberar credenciais para origens demais.
- Confundir erro de CORS com erro da API.
- Resolver tudo desabilitando proteções do browser.
