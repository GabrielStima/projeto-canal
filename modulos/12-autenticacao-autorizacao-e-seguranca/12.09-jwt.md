## Ideia Central

JWT é um formato de token compacto composto por três partes: header, payload e assinatura. Ele permite que o servidor valide a integridade do token e leia claims sem consultar um storage central a cada requisição.

Essa conveniência tem trade-offs. JWT mal usado pode dificultar revogação, carregar dados demais, expor informações no payload e criar confusão entre autenticação e autorização.

- `exp`, `iat`, `iss`, `aud` e `sub`.
- Validação.
- Revogação.
- Dados sensíveis no payload.

## Exemplo Prático

JWT do Veterinário:

```json
{
  "sub": "vet_456",
  "role": "CLINICO",
  "unidade": "Filial_Centro",
  "iss": "auth.petcareos.com.br",
  "exp": 1780689600
}
```

O módulo clínico não precisa fazer uma query no banco a cada clique para descobrir o cargo do veterinário. O payload do JWT entrega isso e a assinatura garante que o JWT não foi adulterado. Porém, não é possível forçar o token a expirar mais cedo de forma simples se o veterinário for demitido.

## Analogias e Intuições

- JWT é um passaporte diplomático: ele traz seus dados carimbados por uma autoridade. A fronteira lê o carimbo (assinatura) e confia sem precisar ligar para o país de origem.
*Limite: se o país quiser cancelar o passaporte antes do vencimento, é difícil avisar todas as fronteiras imediatamente (o desafio da revogação).*

## Erros Comuns

- Colocar dados sensíveis no payload.
- Não validar algoritmo, emissor, audiência e expiração.
- Usar JWT quando sessão simples resolveria melhor.
- Criar JWT sem plano de revogação.
- Confundir claim com permissão sempre atualizada.
