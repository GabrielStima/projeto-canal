# Algoritmos de Hashing

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diferenciar hash genérico de algoritmo para senha;
- explicar por que velocidade pode ser ruim em senhas;
- reconhecer critérios para escolher um algoritmo;
- evitar decisões perigosas como criar criptografia própria.

## Onde Esta Aula Entra na Formação

- Módulo: 12. Autenticação, Autorização e Segurança
- Aula anterior: Hashing de Senhas
- Próxima aula: Autenticação Básica

## Ideia Central

Nem todo hash serve para senha. Alguns algoritmos são feitos para integridade e velocidade. Senhas precisam do contrário: uma função com salt e custo ajustável, para dificultar ataques offline quando hashes vazam.

Em projetos reais, a escolha deve seguir bibliotecas maduras e recomendações atuais. O desenvolvedor precisa entender o critério, não reinventar o algoritmo.

## Tópicos da Aula

- Hash para integridade.
- Hash para senha.
- Ataque offline.
- Salt único por senha.
- Custo computacional.
- Algoritmos adaptativos.
- Migração de hashes antigos.
- Bibliotecas confiáveis.

## Exemplo Prático

Cenário de migração no PetCare OS:
- a clínica operava com senhas hasheadas em bcrypt custo 10;
- a equipe decide aumentar a segurança para custo 12;
- o tutor faz login usando a senha correta;
- backend verifica o hash (que foi gerado com custo 10) e aceita o login;
- backend percebe que o custo do hash salvo está desatualizado;
- backend imediatamente gera um novo hash (custo 12) com a senha informada em memória;
- backend substitui o hash antigo pelo novo no banco sem o tutor notar.

## Analogias e Intuições

- Custo do hash é como o peso de uma porta blindada. Deve ser pesado o suficiente para impedir que um atacante abra 10.000 portas por segundo, mas leve o suficiente para que o cliente não desista de empurrar ao fazer login.
*Limite: servidores aumentam de capacidade ao longo do tempo, então o 'peso' precisa ser reajustado (migração de custo).*

## Erros Comuns

- Usar MD5 ou SHA simples para senha.
- Reutilizar o mesmo salt para todos os usuários.
- Salvar parâmetros do algoritmo fora do hash de forma confusa.
- Nunca revisar custo com o passar dos anos.
- Misturar segredo de aplicação com senha sem entender o objetivo.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Por que bibliotecas maduras importam?

## Prática Recomendada

- Compare um hash MD5 (rápido) com bcrypt (lento por design) e explique por que a lentidão protege a senha.
- Defina uma estratégia para a clínica migrar os hashes sem forçar o reset de senha de todos os clientes.
- Explique os perigos de a equipe de engenharia tentar criar seu próprio algoritmo de embaralhamento de senhas.
