# Algoritmos de Hashing

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diferenciar hash genérico de algoritmo para senha;
- explicar por que velocidade pode ser ruim em senhas;
- reconhecer critérios para escolher um algoritmo;
- evitar decisões perigosas como criar criptografia própria.

## Onde Esta Aula Entra no Curso

- Módulo: 12. Autenticação, Autorização e Segurança

- Próxima aula: Autenticação Básica

## Ideia Central

Nem todo hash serve para senha. Alguns algoritmos são feitos para integridade e velocidade. Senhas precisam do contrário: uma função com salt e custo ajustável, para dificultar ataques offline quando hashes vazam.

Em projetos reais, a escolha deve seguir bibliotecas maduras e recomendações atuais. O desenvolvedor precisa entender o critério, não reinventar o algoritmo.

## Tópicos da Aula

- Hash para integridade.
- Hash para senha.
- Ataque offline.
- Salt único por senha.
- Custo computacional.
- Algoritmos adaptativos.
- Migração de hashes antigos.
- Bibliotecas confiáveis.

## Exemplo Prático

```text
Cenário:
- usuário antigo tem hash com custo baixo;
- usuário faz login com sucesso;
- backend verifica o hash;
- backend percebe que o parâmetro está antigo;
- backend recalcula o hash com parâmetros atuais.
```

## Analogias e Intuições

- Hash rápido é bom para conferir arquivo, mas ruim para proteger senha.
- Custo é uma trava: deve incomodar o atacante sem derrubar o usuário legítimo.

## Erros Comuns

- Usar MD5 ou SHA simples para senha.
- Reutilizar o mesmo salt para todos os usuários.
- Salvar parâmetros do algoritmo fora do hash de forma confusa.
- Nunca revisar custo com o passar dos anos.
- Misturar segredo de aplicação com senha sem entender o objetivo.
