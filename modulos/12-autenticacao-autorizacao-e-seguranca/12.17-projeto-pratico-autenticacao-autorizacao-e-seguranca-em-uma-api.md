# Projeto Pratico: Autenticacao, Autorizacao e Seguranca em uma API

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- integrar os conceitos do modulo em uma API pequena;
- implementar um fluxo seguro de cadastro e login;
- proteger rotas por autenticacao e autorizacao;
- aplicar mitigacoes basicas contra riscos comuns;
- criar um checklist de seguranca para evolucao do projeto.

## Onde Esta Aula Entra no Curso

- Modulo: 12. Autenticacao, Autorizacao e Seguranca
- Aula anterior: Seguranca em APIs
- Proxima aula: Qualidade, Testes, Observabilidade e Performance
- Pre-requisitos: Todas as aulas do modulo 12, API design profissional, backend com Node.js, bancos de dados, validacao, erros padronizados e logging.

## Ideia Central

O projeto pratico transforma o modulo em um fluxo completo. A API do painel de estudos passa a ter usuarios reais, senha com hash, login, sessao ou token, rotas protegidas, regras de autorizacao, protecoes contra abuso e um checklist de seguranca.

O foco nao e criar a autenticacao mais sofisticada possivel. O foco e montar uma base clara, explicavel e evolutiva.

## Topicos da Aula

- Modelo de usuario.
- Cadastro.
- Login.
- Hash de senha.
- Sessao, cookie ou token.
- Middleware de autenticacao.
- Regras de autorizacao.
- Rate limiting.
- Revisao contra CSRF, XSS e SQL injection.
- Checklist final.

## Exemplo Pratico

```text
Fluxo minimo:
1. POST /register
2. POST /login
3. GET /me
4. GET /progress
5. PATCH /progress/:lessonId
6. POST /admin/users/:id/role
```

Cada endpoint deve ter contrato, validacao, regra de autenticacao, regra de autorizacao e resposta de erro padronizada.

## Analogias e Intuicoes

- O projeto e uma revisao aplicada: cada aula anterior vira uma decisao concreta.
- Seguranca boa deixa rastro: regra escrita, erro padronizado, log util e limite claro.

## Erros Comuns

- Comecar por ferramenta antes de desenhar o fluxo.
- Implementar login sem hash de senha.
- Proteger rota, mas esquecer permissao por recurso.
- Nao testar casos negativos.
- Deixar checklist separado do contrato da API.

## Checklist de Entendimento

Antes de seguir, voce deve conseguir responder:

- Como usuario se cadastra?
- Como senha e armazenada?
- Como a API reconhece usuario autenticado?
- Quem pode alterar progresso?
- Que endpoints precisam de rate limiting?
- Que riscos de CSRF, XSS e SQL injection existem no projeto?

## Pratica Recomendada

- Crie o modelo de usuario e credencial.
- Implemente cadastro e login com hash.
- Escolha sessao/cookie ou token e justifique.
- Proteja `GET /me` e rotas de progresso.
- Adicione regra de role para rota administrativa.
- Adicione rate limiting no login.
- Revise queries e renderizacao de dados.
- Escreva um checklist de seguranca por endpoint.
