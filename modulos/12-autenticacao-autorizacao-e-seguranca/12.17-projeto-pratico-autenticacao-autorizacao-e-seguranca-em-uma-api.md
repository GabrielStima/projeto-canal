# Projeto Prático: Autenticação, Autorização e Segurança em uma API

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os conceitos do módulo em uma API pequena;
- implementar um fluxo seguro de cadastro e login;
- proteger rotas por autenticação e autorização;
- aplicar mitigações básicas contra riscos comuns;
- criar um checklist de segurança para evolução do projeto.

## Onde Esta Aula Entra na Formação

- Módulo: 12. Autenticação, Autorização e Segurança
- Aula anterior: Segurança em APIs
- Próxima aula: Qualidade, Testes, Observabilidade e Performance
- Pré-requisitos: Todas as aulas do módulo 12, API design profissional, backend com Node.js, bancos de dados, validação, erros padronizados e logging.

## Ideia Central

O projeto prático transforma o módulo em um fluxo completo. A API do painel de estudos passa a ter usuários reais, senha com hash, login, sessão ou token, rotas protegidas, regras de autorização, proteções contra abuso e um checklist de segurança.

O foco não é criar a autenticação mais sofisticada possível. O foco é montar uma base clara, explicável e evolutiva.

## Tópicos da Aula

- Modelo de usuário.
- Cadastro.
- Login.
- Hash de senha.
- Sessão, cookie ou token.
- Middleware de autenticação.
- Regras de autorização.
- Rate limiting.
- Revisão contra CSRF, XSS e SQL injection.
- Checklist final.

## Exemplo Prático

Marco de Integração N4 - Segurança no PetCare OS:
1. Adicione a tabela de Usuários conectada aos Perfis (Clínico, Tutor, Admin) e defina permissões RBAC.
2. Substitua os logins de teste por rotas seguras `POST /register` e `POST /login` usando bcrypt para gerar hash de senhas.
3. Configure o envio do token/JWT seguro em um cookie `HttpOnly` com validações SameSite.
4. Mapeie todas as rotas de Prontuário, Orçamento e Agendamento para usarem middleware de autenticação.
5. Adicione Rate Limiting nas rotas críticas de autenticação.
6. Revise todas as rotas com banco contra injeções ou exposição excessiva.

## Analogias e Intuições

- Transformar a aplicação insegura em segura é como sair da fundação e subir os muros com controle de portaria da obra. O prédio já funcionava antes, mas agora tem acesso controlado rigorosamente.
*Limite: A segurança precisará ser mantida e monitorada sempre, não é um trabalho feito só uma vez.*

## Erros Comuns

- Começar por ferramenta antes de desenhar o fluxo.
- Implementar login sem hash de senha.
- Proteger rota, mas esquecer permissão por recurso.
- Não testar casos negativos.
- Deixar checklist separado do contrato da API.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Como usuário se cadastra?
- Como senha e armazenada?
- Como a API reconhece usuário autenticado?
- Quem pode alterar progresso?
- Que endpoints precisam de rate limiting?
- Que riscos de CSRF, XSS e SQL injection existem no projeto?
