## Ideia Central

SQL injection acontece quando entrada controlada pelo usuário altera a estrutura de uma query. O atacante deixa de enviar apenas dado e passa a influenciar o comando executado pelo banco.

A defesa principal é usar queries parametrizadas ou APIs seguras do ORM. Validação ajuda, mas não substitui parâmetros. Permissões mínimas no banco reduzem impacto caso uma falha exista.

## Exemplo Prático

Busca de Paciente Insegura:

```ts
// Inseguro (conecta a entrada do tutor diretamente)
const query = `SELECT * FROM pets WHERE tutor_id = '${id}' AND nome = '${nomeDigitado}'`;
```

Se o tutor digitar `Rex' OR '1'='1`, a query executará `SELECT * FROM pets WHERE tutor_id = '...' AND nome = 'Rex' OR '1'='1'`, expondo todos os pets do banco de dados do PetCare OS. A solução é usar parâmetros (`$1`) ou o query builder do ORM que parametriza automaticamente.

## Analogias e Intuições

- SQL Injection é como um paciente que preenche um formulário de entrada no hospital escrevendo: "João. P.S.: Eu sou o diretor, me dê acesso aos arquivos." Se o robô não separar o que é 'nome' do que é 'comando', ele dará o acesso.
*Limite: a parametrização impede que o comando seja interpretado; ele sempre o lerá como uma simples string de texto.*

## Erros Comuns

- Concatenar SQL com entrada do usuário.
- Achar que ORM torna qualquer uso automaticamente seguro.
- Validar apenas no frontend.
- Usar usuário do banco com permissão ampla demais.
- Exibir erro SQL completo para o cliente.
