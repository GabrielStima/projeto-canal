# Autenticação, Autorização e Segurança

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diferenciar autenticação de autorização e explicar por que ambas importam;
- reconhecer os temas que serão estudados no módulo;
- identificar as principais ameaças de segurança web e como cada uma funciona;
- entender como JWT, OAuth, RBAC e proteções contra ataques se encaixam na arquitetura.

## Ideia Central

Autenticação responde "quem é você?". Autorização responde "o que você pode fazer?". São problemas distintos com soluções distintas — e confundi-los é a origem de boa parte das vulnerabilidades em aplicações reais.

Segurança não é uma feature adicionada no final: é uma perspectiva aplicada desde o design. Senhas mal armazenadas, tokens expostos, falta de validação de entrada e configurações erradas de CORS e cookies são problemas que custam caro quando descobertos em produção.

Este módulo cobre desde hashing de senhas até rate limiting, passando por sessões, JWT, OAuth, RBAC, CORS, XSS, CSRF e SQL Injection, construindo um repertório para identificar e mitigar riscos em aplicações fullstack.

## Exemplo Prático

Fluxo: O clínico acessa o prontuário de um pet recém-chegado, enquanto o recepcionista apenas aprova a entrada e não tem permissão para alterar diagnósticos.

Perguntas que este módulo vai responder:
- como garantir que um tutor não consiga ver os dados de outro tutor ao acessar o Portal do Tutor?
- como impedir que um recepcionista acesse funcionalidades exclusivas do Módulo Financeiro?
- como armazenar a senha do usuário sem que vazamentos do banco as exponham?
- como proteger o formulário de login contra ataques de força bruta?
- como configurar CORS para que o frontend do portal consiga chamar a API com segurança?

## Analogias e Intuições

- Autenticação é a recepção do hospital: verifica sua identidade e documento antes de deixar você entrar.
- Autorização é o crachá de acesso: ele define que a porta da farmácia não abre para você, mesmo que você esteja dentro do hospital.
*Limite: esta analogia não captura como as sessões expiram digitalmente, diferente de um crachá físico que você devolve na saída.*
- Hashing é como transformar a senha em uma impressão digital irreversível.

## Erros Comuns

- Armazenar senhas com MD5 ou SHA1 em vez de bcrypt ou Argon2.
- Colocar o token JWT no localStorage e expô-lo a ataques XSS.
- Usar o mesmo token para autenticação e autorização sem verificar escopo.
- Não validar entrada do usuário antes de montar queries SQL.
- Configurar CORS com `*` em produção, permitindo qualquer origem.

## Checklist de Entendimento

- Qual a diferença entre sessão baseada em cookie e autenticação via JWT?
- Por que MD5 não é adequado para armazenar senhas?
- O que é CSRF e como cookies SameSite ajudam a mitigá-lo?
- Como RBAC define o que cada papel pode fazer na aplicação?
- O que este módulo vai aprofundar?

## Prática Recomendada

- Mapeie os papéis do PetCare OS (Clínico, Recepcionista, Administrador, Tutor) e defina duas ações exclusivas para cada um.
- Desenhe o fluxo de login de um recepcionista, desde a senha digitada até o token devolvido pela API.
- Identifique três endpoints da API do Módulo de Estoque que precisariam de autorização.
- Escreva um exemplo de SQL Injection que quebraria a tela de login se os parâmetros não fossem tratados.
