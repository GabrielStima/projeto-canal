# Projeto Prático: Desenhando a Infraestrutura Cloud de uma Aplicação

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os principais blocos de cloud em um desenho coerente;
- justificar escolhas de região, IAM, rede, storage, banco, fila, CDN, serverless e secrets;
- identificar riscos de custo, indisponibilidade, exposição e perda de dados;
- preparar o terreno para Infrastructure as Code.

## Onde Esta Aula Entra no Curso

- Módulo: 17. Cloud Fundamentals
- Aula anterior: Fundamentos de FinOps
- Próxima aula: Infrastructure as Code
- Pré-requisitos: Regiões, zonas, IAM, VPC, object storage, CDN, managed databases, filas gerenciadas, serverless, secrets, backup, disaster recovery, custos e FinOps.

## Ideia Central

O projeto final do módulo transforma conceitos separados em uma arquitetura cloud inicial. A ideia não é provisionar recursos ainda. Isso ficará para o módulo de Infrastructure as Code. Aqui, o objetivo é desenhar, justificar e revisar uma infraestrutura antes de automatizá-la.

Uma boa solução cloud começa com perguntas simples: quem usa, onde está, quais dados são críticos, quais partes precisam ser públicas, quais devem ficar privadas, quem tem permissão, o que pode falhar, quanto pode custar e como recuperar.

O resultado esperado é um desenho com decisões explícitas, não um diagrama bonito sem raciocínio.

## Tópicos da Aula

- Escolha da aplicação base.
- Região e disponibilidade.
- IAM para pessoas, aplicação, deploy e workers.
- Rede pública e privada.
- Storage, CDN, banco, fila e serverless.
- Secrets, backup, disaster recovery e custo.
- Registro de decisões e trade-offs.

## Exemplo Prático

```text
Aplicação: plataforma de cursos

Componentes:
- frontend estático entregue por CDN;
- object storage para assets, imagens e certificados;
- API em ambiente privado exposta por load balancer;
- banco gerenciado privado;
- fila gerenciada para processamento assíncrono;
- worker ou função serverless para gerar certificados;
- secret manager para credenciais;
- backups do banco e versionamento de objetos importantes;
- alertas de custo e métricas de operação.
```

## Analogias e Intuições

- O projeto é uma planta baixa antes da construção.
- Cada seta no desenho precisa ter motivo.
- Cada recurso público precisa se justificar.
- Cada custo precisa ter dono.

## Erros Comuns

- Desenhar serviço demais antes de entender o fluxo.
- Colocar banco ou fila acessível publicamente.
- Não separar identidade de aplicação, deploy e pessoa.
- Esquecer backups e restore.
- Ignorar custos de logs, tráfego e armazenamento.
- Pular trade-offs e apresentar escolhas como óbvias.

## Checklist de Entendimento

- O desenho mostra quem acessa o quê?
- Recursos públicos e privados estão claros?
- IAM está separado por responsabilidade?
- Dados críticos têm backup e plano de restore?
- Há estratégia para arquivos, CDN, banco, fila e secrets?
- Os principais custos foram identificados?
- O desenho está pronto para virar IaC depois?

## Prática Recomendada

- Escolha uma aplicação fullstack criada ou imaginada no curso.
- Faça um desenho simples da infraestrutura cloud.
- Escreva uma lista de recursos necessários.
- Para cada recurso, justifique por que ele existe.
- Liste riscos e trade-offs.
- Defina próximos passos para transformar o desenho em Infrastructure as Code.
