# Regiões, Zonas e Modelo Global de Cloud

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar a diferença entre região e zona de disponibilidade;
- entender como localização afeta latência, disponibilidade, custo e regras de dados;
- reconhecer por que algumas arquiteturas usam múltiplas zonas ou regiões;
- escolher uma estratégia inicial simples para uma aplicação.

## Ideia Central

Cloud parece abstrata, mas seus recursos continuam existindo em lugares físicos. Uma região representa uma área geográfica onde o provedor opera datacenters. Uma zona de disponibilidade é uma parte isolada dentro de uma região, pensada para reduzir o impacto de falhas locais.

Essa escolha importa porque usuários estão em lugares diferentes, leis de dados podem exigir localização específica, falhas podem atingir uma zona inteira e custos variam por região. Começar simples geralmente significa escolher uma região próxima dos usuários e usar recursos distribuídos entre zonas quando o serviço exigir disponibilidade maior.

Multi-região é poderoso, mas traz complexidade de dados, consistência, deploy, observabilidade e custo. Não é o ponto de partida automático.

## Tópicos da Aula

## Exemplo Prático

Física Importa no PetCare:

Se o hospital veterinário fica em São Paulo, e o seu servidor AWS for escolhido em `us-east-1` (Virgínia - EUA), toda foto de raio-X enviada viaja 7 mil quilômetros no cabo submarino.
A latência é alta. Se usarmos `sa-east-1` (São Paulo), a latência cai de 150ms para 15ms. E, dentro de SP, dividimos a aplicação na Zona A e Zona B. Se um bairro inteiro cair sem luz, o PetCare não cai.

## Analogias e Intuições

- Região é a Cidade (São Paulo). Availability Zone (AZ) é o bairro. Se chover muito e inundar o Bairro A (Zona A), o Bairro B (Zona B) na mesma cidade ainda tem luz e segura a carga do seu app.

## Erros Comuns

- Ignorar latência para o usuário principal.
- Achar que usar cloud automaticamente significa alta disponibilidade.
- Ativar multi-região sem entender consistência de dados.
- Colocar serviços dependentes em regiões diferentes sem motivo.
- Esquecer que dados podem ter exigências legais de localização.

## Checklist de Entendimento

- Onde meus usuários estão?
- O serviço precisa sobreviver à falha de uma zona?
- O sistema precisa sobreviver à falha de uma região?
- Meus dados podem sair de determinada localização?
- Qual complexidade estou aceitando para ganhar disponibilidade?

## Prática Recomendada

- Responda: Se a lei de proteção de dados (LGPD) exige que prontuários médicos não saiam do Brasil, onde a sua Região do banco de dados obrigatoriamente deve ficar?
