## Ideia Central

Uma fila gerenciada permite desacoplar quem solicita trabalho de quem executa trabalho. A aplicação publica uma mensagem, e um consumidor processa depois. O provedor cuida de parte da infraestrutura da fila, como armazenamento das mensagens, disponibilidade, escala e entrega.

Mesmo assim, a aplicação continua responsável por semântica, idempotência, tratamento de erro, retries, observabilidade e desenho do fluxo. Uma mensagem pode ser entregue mais de uma vez, atrasar, falhar ou parar em uma fila de erro.

Filas gerenciadas são úteis quando você quer resposta rápida ao usuário, absorver picos ou isolar trabalho pesado.

## Exemplo Prático

```text
Funcionalidade: gerar certificado

1. API recebe pedido.
2. API salva solicitação pendente.
3. API envia mensagem GenerateCertificate para a fila.
4. Worker consome a mensagem.
5. Worker verifica idempotência.
6. Worker gera certificado e atualiza status.
7. Mensagens com falha repetida vão para análise.
```

## Analogias e Intuições

- Fila é uma esteira de tarefas.
- Produtor coloca trabalho na esteira.
- Consumidor pega trabalho da esteira.
- Dead letter queue é a área de itens que não conseguiram ser processados.

## Erros Comuns

- Assumir que mensagem será processada uma única vez.
- Não registrar estado antes de publicar mensagem.
- Não criar idempotência no consumidor.
- Ignorar mensagens presas ou falhando em loop.
- Usar fila para esconder lentidão sem entender a causa.

## Checklist de Entendimento

- Quem publica a mensagem?
- Quem consome?
- O que acontece se a mensagem duplicar?
- O que acontece se o consumidor falhar?
- Como vou saber que a fila está atrasando?
- Existe estratégia para mensagens problemáticas?

## Prática Recomendada

- Modele um fluxo assíncrono de uma aplicação conhecida.
- Defina produtor, mensagem, fila e consumidor.
- Escreva uma chave de idempotência.
- Liste métricas mínimas.
- Explique o que vai para uma fila de erro.
