# Secrets e Configuração Sensível

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diferenciar configuração comum de segredo;
- explicar por que segredos não devem ficar no código;
- entender secret managers em alto nível;
- reconhecer práticas básicas de rotação, acesso mínimo e auditoria.

## Ideia Central

Segredo é qualquer dado que concede acesso ou poder sensível: senha de banco, token de API, chave privada, credencial de integração, webhook secret ou certificado. Configuração comum descreve comportamento. Segredo abre portas.

Em aplicações cloud, segredos precisam ser armazenados, entregues e auditados com cuidado. O ideal é que o código não contenha segredos e que cada aplicação acesse apenas o que precisa por meio de identidade e permissão.

Secret manager é um serviço para armazenar e recuperar segredos com controle de acesso, histórico, auditoria e, em alguns casos, rotação.

## Exemplo Prático

Protegendo os Dados do Cartão:

A chave da API do Gateway de Pagamento Pagar.me JAMAIS pode estar no código fonte (`git push`) e nem chumbada no Dockerfile.
A infraestrutura Cloud oferece serviços de gerenciamento como AWS Secrets Manager ou Azure Key Vault. A aplicação, na hora do Boot, faz um request seguro na Cloud pegando a senha, injeta na memória RAM e deleta a referência física.

## Analogias e Intuições

- Secrets Manager é o cofre digital do hospital. A chave mestra não fica largada no balcão (Repositório de Código), ela fica no cofre, e só quem tem a senha-mestra (Role do IAM) consegue extrair o código de dentro.

## Erros Comuns

- Commitar `.env` com credenciais reais.
- Expor segredo em log de erro.
- Usar o mesmo segredo para todos os ambientes.
- Dar acesso a todos os segredos para toda aplicação.
- Nunca rotacionar credenciais.

## Checklist de Entendimento

- Esse valor concede acesso sensível?
- Onde ele está armazenado?
- Quem pode ler?
- Ele aparece em logs?
- Como seria rotacionado?
- O ambiente de desenvolvimento usa segredo diferente de produção?

## Prática Recomendada
