# IAM: Identidade, Permissões e Princípio do Menor Privilégio

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o papel de IAM em ambientes cloud;
- diferenciar identidade humana, identidade de aplicação e permissão;
- aplicar o princípio do menor privilégio em exemplos simples;
- reconhecer riscos de permissões amplas demais.

## Ideia Central

IAM significa Identity and Access Management. Em cloud, quase tudo depende de identidade e permissão: pessoas acessam painéis, pipelines fazem deploy, aplicações leem segredos, funções acessam filas, APIs escrevem em bancos e serviços conversam entre si.

O erro perigoso é tratar permissões como obstáculo e liberar acesso amplo demais para "fazer funcionar". Isso aumenta o impacto de vazamentos, bugs, credenciais expostas e automações mal configuradas.

O princípio do menor privilégio diz que cada identidade deve ter somente as permissões necessárias para cumprir sua função, pelo menor escopo possível e pelo menor tempo necessário.

## Exemplo Prático

```text
Fluxo: API gera certificados

A API precisa:
- ler templates no object storage;
- gravar certificados gerados;
- enviar mensagem para uma fila;
- ler um segredo de configuração.

A API não precisa:
- apagar qualquer bucket;
- criar usuários;
- mudar permissões;
- acessar bancos de outros sistemas;
- administrar a conta cloud inteira.
```

## Analogias e Intuições

- IAM é o sistema de crachás e chaves da infraestrutura.
- Uma política diz quais portas uma identidade pode abrir.
- Menor privilégio é entregar só a chave necessária, não o chaveiro inteiro.

## Erros Comuns

- Usar conta administrativa para deploy automático.
- Compartilhar credenciais entre pessoas e aplicações.
- Dar acesso total para resolver erro de permissão.
- Não revisar permissões antigas.
- Guardar credenciais permanentes onde credenciais temporárias seriam melhores.

## Checklist de Entendimento

- Quem ou o que precisa acessar este recurso?
- Qual ação exata precisa ser permitida?
- Qual recurso exato precisa ser acessado?
- A permissão precisa ser permanente?
- Como vou auditar quem fez o quê?

## Prática Recomendada

- Pegue uma aplicação com API, banco, storage e fila.
- Liste as identidades envolvidas: desenvolvedor, pipeline, API e worker.
- Para cada identidade, escreva somente as ações necessárias.
- Marque permissões que seriam perigosas se fossem amplas demais.
