# VPC e Networking em Cloud

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o papel de uma rede virtual privada em cloud;
- diferenciar recurso público, privado e controlado por regras de tráfego;
- entender subnets, roteamento e exposição em nível conceitual;
- reconhecer riscos comuns de networking em aplicações cloud.

## Ideia Central

VPC é uma rede virtual isolada onde recursos cloud podem existir e se comunicar. Ela ajuda a separar o que deve ficar acessível pela internet do que deve permanecer privado, como bancos de dados, filas internas e serviços administrativos.

Networking em cloud é uma das áreas onde pequenos erros têm impacto grande. Uma API precisa receber tráfego público, mas um banco normalmente não deveria. Um worker pode precisar acessar a internet para chamar um serviço externo, mas isso não significa que ele precise receber conexões públicas.

O objetivo aqui não é decorar todos os componentes de rede de um provedor. É entender a pergunta principal: quem precisa falar com quem, por qual caminho e com quais restrições?

## Tópicos da Aula

- VPC como rede privada lógica.
- Subnets públicas e privadas.
- Entrada e saída de tráfego.
- Regras de firewall em alto nível.
- Load balancer e exposição controlada.
- Banco privado, API pública e workers internos.

## Exemplo Prático

```text
Aplicação: e-commerce simples

Público:
- CDN;
- load balancer;
- frontend estático;
- endpoint público da API.

Privado:
- banco de dados;
- fila;
- worker de processamento;
- serviços administrativos.

Regra central:
- usuário acessa frontend e API;
- API acessa banco e fila;
- usuário nunca acessa banco diretamente.
```

## Analogias e Intuições

- VPC é um condomínio privado.
- Subnets são áreas internas com regras diferentes.
- Load balancer é a portaria pública para serviços que aceitam tráfego.
- Firewall é a lista de quem pode conversar com quem.

## Erros Comuns

- Deixar banco de dados público por conveniência.
- Misturar tudo em uma rede sem entender fluxo de tráfego.
- Liberar acesso de qualquer origem para portas sensíveis.
- Confundir saída para internet com entrada pública.
- Tentar resolver problema de aplicação abrindo rede demais.

## Checklist de Entendimento

- Quais recursos precisam ser públicos?
- Quais recursos devem ser privados?
- Quem inicia a conexão?
- Que portas e protocolos são necessários?
- O que aconteceria se esse recurso ficasse exposto?

## Prática Recomendada

- Desenhe uma aplicação com frontend, API, banco, fila e worker.
- Marque quais recursos são públicos e quais são privados.
- Desenhe setas indicando quem fala com quem.
- Para cada seta, explique se ela precisa ser permitida.
- Identifique uma exposição perigosa.
