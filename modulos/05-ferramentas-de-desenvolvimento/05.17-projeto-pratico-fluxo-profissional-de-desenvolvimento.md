# Projeto Pratico: Fluxo Profissional de Desenvolvimento

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- aplicar as ferramentas do modulo em um projeto pequeno;
- organizar setup, scripts, Git, GitHub e documentacao;
- simular um fluxo de branch, pull request e review;
- saber o que estudar em seguida.

## Onde Esta Aula Entra no Curso

- Modulo: Ferramentas de Desenvolvimento
- Aula anterior: Code review
- Proxima aula: Frontend Moderno
- Pre-requisitos: todas as aulas anteriores do modulo 05.

## Ideia Central

Esta aula junta o modulo inteiro em uma pratica unica. A meta nao e criar uma aplicacao complexa. A meta e transformar um projeto simples em algo que outra pessoa conseguiria abrir, instalar, rodar, entender e revisar.

Um bom fluxo profissional cria previsibilidade: comandos claros, historico limpo, configuracao documentada e revisao objetiva.

## Topicos da Aula

- Escolha de um projeto pequeno.
- Setup com gerenciador de pacote.
- Scripts minimos.
- `.env.example`.
- Git e commits pequenos.
- Repositorio remoto.
- Pull request e self-review.
- README final.

## Exemplo Pratico

Projeto sugerido: uma mini aplicacao de tarefas em TypeScript.

Fluxo:

```text
1. Criar estrutura de pastas.
2. Inicializar package.json.
3. Adicionar scripts de desenvolvimento e validacao.
4. Criar README.
5. Criar .env.example.
6. Inicializar Git.
7. Fazer commits pequenos.
8. Publicar no GitHub.
9. Abrir branch de melhoria.
10. Criar pull request.
11. Fazer self-review.
12. Mergear a mudanca.
```

## Analogias e Intuicoes

- O projeto pratico e um ensaio geral antes de frameworks e backend.
- Ferramentas bem configuradas fazem o projeto parecer menos misterioso para quem chega depois.

## Erros Comuns

- Tentar fazer uma aplicacao grande demais.
- Configurar ferramentas sem documentar comandos.
- Fazer um unico commit com tudo.
- Publicar segredo por acidente.
- Abrir pull request sem explicar contexto.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Outra pessoa consegue rodar meu projeto seguindo o README?
- Meus scripts tem nomes claros?
- Meus commits contam uma historia compreensivel?
- Meu `.env.example` documenta as variaveis esperadas?
- Meu pull request explica o que mudou e como testar?

## Pratica Recomendada

- Execute o fluxo completo em um projeto pequeno.
- Grave os problemas encontrados durante o setup.
- Ajuste o README ate o fluxo ficar repetivel.
- Faca uma revisao final olhando apenas para clareza e risco.
