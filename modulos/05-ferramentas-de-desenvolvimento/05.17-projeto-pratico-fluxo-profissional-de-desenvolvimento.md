# Projeto Prático: Fluxo Profissional de Desenvolvimento

## Objetivo

Ao final desta aula, você deve ser capaz de:

- aplicar as ferramentas do módulo em um projeto pequeno;
- organizar setup, scripts, Git, GitHub e documentação do PetCare OS;
- produzir um repositório base que será evoluído nos próximos módulos;
- simular um fluxo de branch, pull request e review;
- saber o que estudar em seguida.

## Onde Esta Aula Entra na Formação

- Módulo: Ferramentas de Desenvolvimento
- Aula anterior: Code review
- Próxima aula: Frontend Moderno
- Pré-requisitos: todas as aulas anteriores do módulo 05.

## Ideia Central

Esta aula junta o módulo inteiro em uma prática única. A meta não é criar uma aplicação complexa. A meta é transformar a ideia do PetCare OS em um repositório organizado que outra pessoa consiga abrir, instalar, rodar, entender e revisar.

Um bom fluxo profissional cria previsibilidade: comandos claros, histórico limpo, configuração documentada e revisão objetiva.

## O que você já sabe

Você estudou editor, terminal, shell, ambientes de desenvolvimento, variáveis de ambiente, versionamento semântico, gerenciadores de pacote, Git, GitHub, linters, formatters, module bundlers, documentação mínima e code review. Você também praticou variáveis de ambiente, versionamento com Git, revisão de mudanças e scripts de qualidade em exercícios menores.

## O que você vai produzir

Um repositório local chamado `petcare-os-base` com:

- estrutura de pastas clara;
- `package.json` com scripts de desenvolvimento e qualidade;
- `.env.example` com variáveis relevantes para o PetCare OS;
- `.gitignore` ignorando `node_modules`, `.env` e `dist/`;
- `README.md` com instruções mínimas de instalação e uso;
- histórico Git com pelo menos três commits bem descritos;
- branch `feature/readme-inicial` criada, usada e mergeada.

## Como praticar

Siga a [Atividade Final do Módulo — Repositório Base do PetCare OS](exercicios/atividade-final-modulo.md). Ela detalha a estrutura, os critérios de correção e o prompt para estudar com IA.

Resumo do fluxo:

1. Crie a pasta do projeto.
2. Inicialize o `package.json` com um gerenciador de pacote.
3. Adicione scripts `dev`, `build`, `check` e `format`.
4. Crie `src/index.ts` com uma função simples.
5. Crie `.env.example` e `.gitignore`.
6. Escreva o `README.md`.
7. Inicialize Git e faça commits pequenos.
8. Crie a branch `feature/readme-inicial`, melhore o README e faça merge.
9. Se possível, publique no GitHub e abra um pull request de revisão.

## Como corrigir seu trabalho com IA

Use o prompt de correção da atividade final. Ele pede para a IA avaliar estrutura, scripts, variáveis de ambiente, `.gitignore`, README, histórico de commits e branch. Cole sua resposta no marcador indicado.

## Onde esse trabalho será retomado

O repositório base criado aqui será evoluído nos próximos módulos. Em Frontend Moderno e Next.js você adicionará componentes e páginas. Em Backend e Banco de Dados você expandirá a estrutura com serviços e modelos. Guarde esse repositório com carinho: ele é a semente do PetCare OS.

## Exemplo Prático

Projeto sugerido: o repositório base do PetCare OS.

Fluxo:

```text
1. Criar estrutura de pastas.
2. Inicializar package.json.
3. Adicionar scripts de desenvolvimento e validação.
4. Criar README.
5. Criar .env.example.
6. Criar .gitignore.
7. Inicializar Git.
8. Fazer commits pequenos.
9. Publicar no GitHub.
10. Abrir branch de melhoria.
11. Criar pull request.
12. Fazer self-review.
13. Mergear a mudança.
```

## Analogias e Intuições

- O projeto prático é um ensaio geral antes de frameworks e backend.
- Ferramentas bem configuradas fazem o projeto parecer menos misterioso para quem chega depois.

## Erros Comuns

- Tentar fazer uma aplicação grande demais.
- Configurar ferramentas sem documentar comandos.
- Fazer um único commit com tudo.
- Publicar segredo por acidente.
- Abrir pull request sem explicar contexto.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Outra pessoa consegue rodar meu projeto seguindo o README?
- Meus scripts tem nomes claros?
- Meus commits contam uma historia compreensivel?
- Meu `.env.example` documenta as variáveis esperadas?
- Meu `.gitignore` protege arquivos sensíveis e gerados?
- Meu pull request explica o que mudou e como testar?

## Prática Recomendada

- Execute o fluxo completo da [Atividade Final do Módulo](exercicios/atividade-final-modulo.md).
- Grave os problemas encontrados durante o setup.
- Ajuste o README até o fluxo ficar repetível.
- Faça uma revisão final olhando apenas para clareza e risco.
