# Code Review

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o objetivo de uma code review;
- revisar mudancas pequenas com criterio;
- receber comentarios sem transformar a revisao em disputa pessoal;
- saber o que estudar em seguida.

## Onde Esta Aula Entra no Curso

- Modulo: Ferramentas de Desenvolvimento
- Aula anterior: Documentacao minima de projeto
- Proxima aula: Projeto pratico: fluxo profissional de desenvolvimento
- Pre-requisitos: Git, GitHub, pull requests, documentacao minima e scripts de qualidade.

## Ideia Central

Code review e uma pratica de avaliacao tecnica de mudancas antes de integra-las ao projeto. O objetivo nao e provar quem sabe mais. O objetivo e reduzir risco, melhorar clareza, compartilhar conhecimento e proteger a saude do codigo.

Uma boa review olha comportamento, legibilidade, testes, escopo, consistencia e impacto. Comentarios devem ser especificos e acionaveis.

## Topicos da Aula

- O que revisar.
- Como abrir um pull request revisavel.
- Comentarios claros e respeitosos.
- Aprovar, comentar ou pedir alteracoes.
- Self-review antes de pedir review.
- Tamanho ideal de mudancas.

## Exemplo Pratico

Antes de pedir review, responda:

```text
- O que mudou?
- Como testei?
- Essa mudanca tem risco?
- O README ou exemplo precisa atualizar?
- Os commits e arquivos fazem sentido?
```

Durante a review, prefira comentarios como:

```text
Este nome pode esconder que a funcao tambem valida entrada. Que tal separar validacao de transformacao?
```

## Analogias e Intuicoes

- Review e uma segunda leitura tecnica antes de publicar uma decisao.
- Pull request pequeno e como um paragrafo bem delimitado: mais facil revisar e melhorar.

## Erros Comuns

- Revisar apenas formatacao.
- Fazer comentarios vagos como "melhorar isso".
- Levar pedido de alteracao como ataque pessoal.
- Abrir pull request gigante sem contexto.
- Aprovar sem rodar ou ler o que importa.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual o objetivo de uma code review?
- O que faz um comentario ser acionavel?
- Por que pull requests pequenos ajudam?
- O que verificar antes de pedir review?

## Pratica Recomendada

- Abra um pull request ficticio com uma mudanca pequena.
- Escreva descricao, risco e como testar.
- Faca uma self-review.
- Escreva tres comentarios de review: um sobre bug, um sobre clareza e um sobre documentacao.
