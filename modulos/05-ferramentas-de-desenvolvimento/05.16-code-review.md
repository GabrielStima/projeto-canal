# Code Review

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o objetivo de uma code review;
- revisar mudanças pequenas com critério;
- receber comentários sem transformar a revisão em disputa pessoal;
- saber o que estudar em seguida.

## Onde Esta Aula Entra na Formação

- Módulo: Ferramentas de Desenvolvimento
- Aula anterior: Documentação mínima de projeto
- Próxima aula: Projeto prático: fluxo profissional de desenvolvimento
- Pré-requisitos: Git, GitHub, pull requests, documentação mínima e scripts de qualidade.

## Ideia Central

Code review é uma prática de avaliação técnica de mudanças antes de integrá-las ao projeto. O objetivo não é provar quem sabe mais. O objetivo é reduzir risco, melhorar clareza, compartilhar conhecimento e proteger a saúde do código.

Uma boa review olha comportamento, legibilidade, testes, escopo, consistência e impacto. Comentários devem ser específicos e acionáveis.

## Tópicos da Aula

- O que revisar.
- Como abrir um pull request revisável.
- Comentários claros e respeitosos.
- Aprovar, comentar ou pedir alterações.
- Self-review antes de pedir review.
- Tamanho ideal de mudanças.

## Exemplo Prático

Antes de pedir review, responda:

```text
- O que mudou?
- Como testei?
- Essa mudança tem risco?
- O README ou exemplo precisa atualizar?
- Os commits e arquivos fazem sentido?
```

Durante a review, prefira comentários como:

```text
Este nome pode esconder que a função também valida entrada. Que tal separar validação de transformação?
```

## Analogias e Intuições

- Review é uma segunda leitura técnica antes de publicar uma decisão.
- Pull request pequeno é como um parágrafo bem delimitado: mais fácil revisar e melhorar.

## Erros Comuns

- Revisar apenas formatação.
- Fazer comentários vagos como "melhorar isso".
- Levar pedido de alteracao como ataque pessoal.
- Abrir pull request gigante sem contexto.
- Aprovar sem rodar ou ler o que importa.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual o objetivo de uma code review?
- O que faz um comentário ser acionável?
- Por que pull requests pequenos ajudam?
- O que verificar antes de pedir review?

## Prática Recomendada

- Abra um pull request fictício com uma mudança pequena.
- Escreva descrição, risco e como testar.
- Faça uma self-review.
- Escreva três comentários de review: um sobre bug, um sobre clareza e um sobre documentação.
