# Linter e Formatters

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar a diferença entre linter e formatter;
- entender por que padronizacao reduz discussão inutil;
- reconhecer scripts comuns de qualidade;
- saber o que estudar em seguida.

## Onde Esta Aula Entra no Curso

- Módulo: Ferramentas de Desenvolvimento
- Aula anterior: GitHub
- Próxima aula: Module bundlers
- Pré-requisitos: gerenciadores de pacote, scripts e projeto JS/TS.

## Ideia Central

Formatter cuida de estilo automático: espacos, quebras de linha, aspas e organização visual. Linter analisa código procurando problemas de padrão, possíveis bugs e regras combinadas pelo projeto.

Essas ferramentas ajudam o time a gastar menos energia com preferências individuais e mais energia com comportamento, clareza e arquitetura.

## Tópicos da Aula

- Formatter.
- Linter.
- Regras automáticas vs decisões humanas.
- Scripts como `format`, `lint` e `check`.
- Integração com editor.
- Cuidados com regras excessivas.

## Exemplo Prático

Um projeto pode expor scripts assim:

```json
{
  "scripts": {
    "format": "prettier . --write",
    "lint": "eslint .",
    "check": "npm run lint"
  }
}
```

A aula não precisa configurar ferramentas do zero ainda. O foco e entender o papel de cada uma.

## Analogias e Intuições

- Formatter arruma a mesa.
- Linter aponta habitos suspeitos antes que virem problema maior.

## Erros Comuns

- Confundir formatação com qualidade de software.
- Colocar regras demais sem maturidade do time.
- Ignorar warnings até o projeto ficar barulhento.
- Debater estilo manualmente quando a ferramenta poderia decidir.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- O que um formatter faz?
- O que um linter faz?
- Por que scripts ajudam o time?
- O que uma ferramenta automática não consegue avaliar?

## Prática Recomendada

- Pegue um arquivo mal formatado e descreva o que um formatter corrigiria.
- Liste três problemas que um linter poderia apontar.
- Escreva scripts fictícios para `format`, `lint` e `check`.
- Explique por que code review não deve gastar tempo com espacos e aspas.
