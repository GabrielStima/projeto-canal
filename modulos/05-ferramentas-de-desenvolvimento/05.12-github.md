# GitHub

Ao final desta aula, você deve ser capaz de:

- explicar o papel do GitHub em um fluxo de desenvolvimento;
- criar um repositório remoto em alto nível;
- entender branches, pull requests e reviews no GitHub;
- saber o que estudar em seguida.

## Ideia Central

GitHub é uma plataforma de colaboração em torno de repositórios Git. Ele permite hospedar código, abrir issues, discutir mudanças, criar pull requests, revisar código e integrar automações.

Para iniciantes, o fluxo mais importante e: criar branch, fazer commits, enviar para o remoto, abrir pull request, receber revisão, ajustar e fazer merge quando estiver aprovado.

## Tópicos da Aula

- Repositórios no GitHub.
- Branch principal e branches de trabalho.
- Pull request como proposta de mudança.
- Reviews: comentar, aprovar ou solicitar alterações.
- Issues em alto nível.
- Boas práticas antes de publicar código.

## Exemplo Prático

Fluxo básico no PetCare OS:

```bash
git switch -c feature/melhora-readme
git add README.md
git commit -m "docs: melhora instrucoes de setup do projeto"
git push origin feature/melhora-readme
```

Depois disso, abra um pull request no GitHub comparando a branch `feature/melhora-readme` com a branch principal. Na descrição, explique o que mudou e como testar.

## Analogias e Intuições

- Pull request é um pedido organizado para incluir uma mudança.
- Review é uma conversa técnica sobre qualidade, risco e clareza.

## Erros Comuns

- Abrir pull request com descrição vazia.
- Enviar arquivos sensíveis para um repositório público.
- Tratar comentário de review como ataque pessoal.
- Misturar mudanças sem relação no mesmo pull request.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Para que serve o GitHub além de guardar código?
- O que é um pull request?
- O que significa aprovar ou pedir alterações em uma review?
- Por que uma branch pequena facilita revisão?

## Prática Recomendada

- Publique um projeto pequeno no GitHub.
- Crie uma branch para alterar o README.
- Abra um pull request com descrição clara.
- Revise o próprio pull request antes de mergear.
