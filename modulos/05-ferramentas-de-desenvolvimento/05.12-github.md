# GitHub

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o papel do GitHub em um fluxo de desenvolvimento;
- criar um repositorio remoto em alto nivel;
- entender branches, pull requests e reviews no GitHub;
- saber o que estudar em seguida.

## Onde Esta Aula Entra no Curso

- Modulo: Ferramentas de Desenvolvimento
- Aula anterior: VCS hosting
- Proxima aula: Linter e formatters
- Pre-requisitos: Git, repositorio remoto, commits e branches.

## Ideia Central

GitHub e uma plataforma de colaboracao em torno de repositorios Git. Ele permite hospedar codigo, abrir issues, discutir mudancas, criar pull requests, revisar codigo e integrar automacoes.

Para iniciantes, o fluxo mais importante e: criar branch, fazer commits, enviar para o remoto, abrir pull request, receber revisao, ajustar e fazer merge quando estiver aprovado.

## Topicos da Aula

- Repositorios no GitHub.
- Branch principal e branches de trabalho.
- Pull request como proposta de mudanca.
- Reviews: comentar, aprovar ou solicitar alteracoes.
- Issues em alto nivel.
- Boas praticas antes de publicar codigo.

## Exemplo Pratico

Fluxo basico:

```bash
git switch -c melhora-readme
git add README.md
git commit -m "melhora instrucoes do projeto"
git push origin melhora-readme
```

Depois disso, abra um pull request no GitHub comparando a branch de trabalho com a branch principal.

## Analogias e Intuicoes

- Pull request e um pedido organizado para incluir uma mudanca.
- Review e uma conversa tecnica sobre qualidade, risco e clareza.

## Erros Comuns

- Abrir pull request com descricao vazia.
- Enviar arquivos sensiveis para um repositorio publico.
- Tratar comentario de review como ataque pessoal.
- Misturar mudancas sem relacao no mesmo pull request.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Para que serve o GitHub alem de guardar codigo?
- O que e um pull request?
- O que significa aprovar ou pedir alteracoes em uma review?
- Por que uma branch pequena facilita revisao?

## Pratica Recomendada

- Publique um projeto pequeno no GitHub.
- Crie uma branch para alterar o README.
- Abra um pull request com descricao clara.
- Revise o proprio pull request antes de mergear.
