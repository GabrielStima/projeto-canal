# Ferramentas de Desenvolvimento

## Objetivo

Ao final desta aula, você deve ser capaz de:
- reconhecer as ferramentas que compõem o fluxo de trabalho de um desenvolvedor profissional;
- entender por que cada ferramenta existe e qual problema ela resolve;
- relacionar editor, terminal, Git, linters e gerenciadores de pacote como partes de um fluxo integrado;
- saber o que será aprofundado ao longo do módulo.

## Ideia Central

Escrever código é apenas uma parte do trabalho. Um desenvolvedor profissional também precisa navegar pelo terminal, versionar o código, instalar dependências com segurança, garantir que o estilo do código seja consistente e documentar o suficiente para que outros (e o próprio desenvolvedor no futuro) entendam o projeto.

Essas ferramentas não são luxo — são o que torna possível trabalhar em equipe, manter histórico de mudanças, compartilhar ambiente de desenvolvimento e entregar código que outras pessoas podem revisar e continuar. Sem elas, projetos crescem e ficam impossíveis de manter.

Este módulo cobre o fluxo básico de trabalho: do editor ao pull request. Ao final, você terá um ambiente configurado e saberá operar esse fluxo com confiança.

## Tópicos da Aula

- Editor, IDE e produtividade: atalhos, extensões e configurações.
- Terminal e linha de comando: navegação e automação básica.
- Shell básico: pipes, redirecionamentos e scripts simples.
- Ambientes de desenvolvimento e variáveis de ambiente.
- Versionamento semântico e gerenciadores de pacote (npm, pnpm, yarn).
- VCS/Git: histórico, branches e merges.
- VCS hosting/GitHub: pull requests, code review e colaboração.
- Linters e formatters: qualidade e consistência automática.
- Module bundlers: como o código é preparado para o browser ou Node.js.
- Documentação mínima de projeto.

## Analogias e Intuições

- Git é uma máquina do tempo do código: você pode voltar a qualquer ponto da história, criar linhas alternativas (branches) e depois juntar as linhas de volta.
- Linters são como revisores ortográficos do código: pegam inconsistências antes do leitor humano.
- Variáveis de ambiente são como credenciais no cofre: o código sabe onde buscar, mas o valor nunca fica escrito diretamente no arquivo.

## Erros Comuns

- Commitar diretamente na branch principal sem passar por pull request em projetos colaborativos.
- Versionar arquivos de ambiente (`.env`) com segredos reais no repositório.
- Usar linter mas desativar todas as regras que "atrapalham" — esvaziando o propósito.
- Escrever mensagens de commit como "fix" ou "ajustes" sem contexto útil.
- Instalar dependências globalmente quando deveriam ser locais ao projeto, quebrando portabilidade.
