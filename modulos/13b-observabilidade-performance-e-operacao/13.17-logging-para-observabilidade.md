# Logging para Observabilidade

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar logs como sinal de observabilidade;
- diferenciar log útil de ruído;
- escolher eventos relevantes para registrar;
- incluir contexto sem vazar dados sensíveis.

## Ideia Central

Logging para observabilidade não é espalhar mensagens aleatórias no código. É registrar eventos que ajudam a entender o comportamento do sistema quando algo acontece longe dos seus olhos.

Bons logs têm nível, evento, contexto de requisição, identificadores relevantes, duração e cuidado com dados sensíveis. Eles ajudam a investigar falhas, auditoria e comportamento operacional.

## Exemplo Prático

Log Estruturado no Faturamento:

Um log solto como `console.log('Erro no pagamento')` é inútil.

Log Operacional:
```json
{
  "level": "error",
  "event": "billing_gateway_timeout",
  "requestId": "1234-abcd",
  "tutorId": "tutor_99",
  "durationMs": 5003,
  "gateway": "Pagar.me",
  "message": "Gateway não respondeu na emissão da fatura do Rex."
}
```
Não salvamos o cartão de crédito (vazamento PIX/PCI-DSS) e garantimos um `requestId` para atrelar esse erro à navegação específica do cliente.

## Analogias e Intuições

- Logging em texto livre (printf) é escrever um diário poético. Logging estruturado (JSON) é preencher um formulário de boletim de ocorrência que permite filtro e busca em massa.

## Erros Comuns

- Registrar dados sensíveis.
- Logar apenas erros inesperados.
- Esquecer requestId ou correlação.
- Usar mensagens soltas impossíveis de filtrar.
- Criar logs que não respondem perguntas de investigação.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que evento merece log?
- Qual nível faz sentido?
- Que contexto ajuda a investigar?
- Que dado não deve aparecer?
- Como esse log seria filtrado depois?
