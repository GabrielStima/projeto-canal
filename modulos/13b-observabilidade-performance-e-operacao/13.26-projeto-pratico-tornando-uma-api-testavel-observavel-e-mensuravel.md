# Projeto Prático: Tornando uma API Testável, Observável e Mensurável

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os conceitos do módulo em uma API pequena;
- criar uma estratégia de testes para fluxos críticos;
- adicionar sinais básicos de observabilidade;
- medir performance de um fluxo importante;
- relacionar failure modes com testes, logs, métricas e alertas.

## Onde Esta Aula Entra na Formação

- Módulo: 13. Qualidade, Testes, Observabilidade é Performance
- Aula anterior: Failure Modes na Prática
- Próxima aula: Design de Software
- Pré-requisitos: Todas as aulas do módulo 13, projeto de API dos módulos anteriores, API design, autenticação, autorização, segurança, banco de dados, logging e profiling.

## Ideia Central

O projeto prático transforma o módulo em uma evolução concreta da API do painel de estudos. A API já tem contrato, autenticação, autorização e segurança básica. Agora ela precisa ganhar uma rede mínima de confiança: testes bem escolhidos, sinais investigáveis e medição de performance.

O foco não é instalar a maior quantidade de ferramentas. O foco é criar decisões explicáveis: por que este teste existe, que falha ele cobre, que log ajuda a investigar, que métrica mostra saúde e que alerta indicaria impacto real.

## Tópicos da Aula

- Escolha de fluxos críticos.
- Estratégia de testes.
- Testes unitários, integração, contrato e E2E.
- Regressão e CI.
- Profiling de um endpoint.
- Logs, métricas, traces, monitoramento e alertas.
- Checklist final de failure modes.

## Exemplo Prático

Marco operacional — sistema resiliente:

Aqui não tem código novo de regras de negócio. Você pega as APIs que construiu de Agendamento ou Cadastro de Pets.

1. Instale Pino.js ou similar. Substitua todos os Logs.
2. Injete um gerador de Correlação (`req.id` usando uuid/crypto) nos middlewares.
3. Escreva um script `load-test.js` (K6) que tente inserir 200 prontuários em 30 segundos no seu container docker local e gere o report de Latência e RPS (Requests per Second).
4. Produza o documento Runbook provando que sua aplicação avisa e sobrevive se derrubar o Redis/DB de cache parcial.

## Analogias e Intuições

- Projeto prático é o 'Crash Test' final na montadora de carros, validando se o motor suporta peso em subidas longas, se o cinto de segurança relata o impacto, e se o farol não desliga junto com o ar condicionado.

## Erros Comuns

- Comecar instalando ferramenta sem estratégia.
- Criar testes apenas para caminho feliz.
- Logar eventos sem contexto.
- Medir performance sem baseline.
- Criar alerta sem ação esperada.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quais fluxos são críticos?
- Que riscos cada fluxo tem?
- Que testes protegem esses riscos?
- Que sinais ajudam a investigar falhas?
- Que métrica indicaria degradação?
- Que alerta exigiria ação?
