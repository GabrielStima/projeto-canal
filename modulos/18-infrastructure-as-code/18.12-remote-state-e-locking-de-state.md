# Remote State e Locking de State

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que state local não é suficiente para times;
- entender remote state;
- entender locking de state;
- reconhecer riscos de aplicação concorrente.

## Ideia Central

Remote state coloca o state em um backend compartilhado e controlado, em vez de deixá-lo apenas na máquina de uma pessoa. Isso ajuda colaboração, automação, backup e controle de acesso.

Locking evita que duas execuções tentem alterar o mesmo state ao mesmo tempo. Sem lock, duas pessoas ou automações podem aplicar mudanças concorrentes e deixar o gerenciamento inconsistente.

Remote state não resolve tudo sozinho. Ele precisa de permissões corretas, proteção contra acesso indevido, versionamento ou recuperação e estratégia clara por ambiente.

## Exemplo Prático

Colisão de Engenheiros no PetCare:

A Engenheira A e o Engenheiro B estão trabalhando no PetCare.
Se o State ficar no laptop (Local), a Engenheira A cria o banco, o B cria a fila e ambos dão erro porque não conhecem os recursos um do outro.
Remote State: salvamos o `tfstate` em um backend compartilhado com locking. No backend S3 atual, `use_lockfile = true` ativa o arquivo de bloqueio. A Engenheira A roda o `apply` e obtém o lock; se o Engenheiro B tentar aplicar ao mesmo tempo, precisa aguardar a liberação. O locking baseado em DynamoDB está descontinuado e deve aparecer apenas em projetos legados durante migração.

## Analogias e Intuições

- Remote State é guardar o papel do inventário na parede da recepção, onde todos veem. Locking é colocar a caneta na mão de uma pessoa só; se alguém quiser escrever no papel, tem que esperar a caneta ser solta.

## Erros Comuns

- Cada pessoa manter um state local diferente.
- Permitir que qualquer usuário leia state de produção.
- Não configurar bloqueio contra execução concorrente.
- Misturar states de ambientes diferentes.
- Não planejar backup ou recuperação do state.

## Checklist de Entendimento

- Onde o state compartilhado fica?
- Quem pode ler e escrever nele?
- Existe locking?
- O state é separado por ambiente?
- Como recuperar uma versão anterior se necessário?
