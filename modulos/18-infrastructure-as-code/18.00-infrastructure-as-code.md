# Infrastructure as Code

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é Infrastructure as Code;
- entender por que infraestrutura precisa ser versionada e revisada;
- reconhecer o papel do Terraform no módulo;
- saber o que será aprofundado nas próximas aulas.

## Ideia Central

Infrastructure as Code é a prática de definir infraestrutura em arquivos versionados, em vez de depender apenas de cliques manuais em painéis de cloud.

A ideia não é transformar todo mundo em especialista de infraestrutura de uma vez. A ideia é criar uma forma mais segura de descrever, revisar, reproduzir e evoluir recursos como redes, permissões, bancos, buckets, filas e serviços.

Neste módulo, Terraform será a ferramenta principal porque ele trabalha com configuração declarativa, mantém state, calcula planos de mudança e integra vários provedores.

## Tópicos da Aula

- Infraestrutura como artefato versionado.
- Revisão, rastreabilidade e reprodução de ambientes.
- Terraform como ferramenta principal do módulo.
- O mapa das próximas aulas.

## Exemplo Prático

```text
Aplicação precisa de:
- rede;
- banco gerenciado;
- bucket para uploads;
- fila;
- permissões mínimas.

Sem IaC:
- alguém cria tudo manualmente;
- outra pessoa tenta lembrar os detalhes;
- produção e staging ficam diferentes.

Com IaC:
- recursos são descritos em arquivos;
- mudanças são revisadas;
- plano mostra o que será criado, alterado ou destruído;
- o histórico fica no Git.
```

## Analogias e Intuições

- IaC é uma planta baixa da infraestrutura, mas com capacidade de executar mudanças.
- O código de infraestrutura é uma receita auditável para criar e alterar ambientes.

## Erros Comuns

- Achar que IaC elimina a necessidade de entender cloud.
- Tratar `apply` como comando inocente.
- Versionar segredos junto com a configuração.
- Ignorar state e achar que Terraform só lê arquivos.
- Automatizar infraestrutura ruim sem revisar o desenho.

## Checklist de Entendimento

- Que problema IaC tenta resolver?
- Por que versionar infraestrutura?
- O que ainda precisa ser entendido sobre cloud?
- Que riscos aparecem quando mudanças de infraestrutura viram código?
- Por que Terraform precisa de state?

## Prática Recomendada

- Pegue a arquitetura desenhada no módulo anterior.
- Liste quais recursos poderiam virar código.
- Separe recursos, permissões, configurações e segredos.
- Escreva quais mudanças deveriam passar por revisão antes de produção.
