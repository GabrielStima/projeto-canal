# Projeto Prático: Provisionando a Infraestrutura de uma Aplicação

## Objetivo

Ao final desta aula, você deve ser capaz de:

- planejar uma infraestrutura simples para uma aplicação;
- escrever uma configuração Terraform organizada;
- revisar plano antes de aplicar;
- lidar com state, ambientes e segredos com cuidado;
- documentar decisões e riscos.

## Onde Esta Aula Entra na Formação

- Módulo: 18. Infrastructure as Code
- Aula anterior: Segurança e Segredos em IaC
- Próxima aula: CI/CD
- Pré-requisitos: Todas as aulas anteriores do módulo 18, Cloud Fundamentals, IAM, VPC, storage, bancos, filas, secrets, custos e Git.

## Ideia Central

O projeto consolida o módulo: transformar uma arquitetura cloud simples em configuração Terraform. O objetivo não é criar uma infraestrutura enorme, e sim praticar o ciclo completo com segurança: desenho, configuração, plano, aplicação controlada, state, outputs, ambientes e documentação.

A aplicação de referência pode ser uma API com banco, bucket de uploads, fila e permissões mínimas. O foco é tomar decisões explícitas e deixar claro o que é criado, o que é consultado, o que é sensível e como destruir sem surpresa.

## Tópicos da Aula

- Escopo da infraestrutura.
- Estrutura de arquivos.
- Providers, resources e data sources.
- Variables, locals e outputs.
- Remote state.
- Organização por ambiente.
- Checklist de segurança.
- Documentação operacional.

## Exemplo Prático

Laboratório de IaC do PetCare:

No projeto prático, você atua como Engenheiro de DevOps.
Você usará HCL para representar a arquitetura cloud em um laboratório local reproduzível, antes de assumir custos ou credenciais de um provedor.
Sua tarefa final:
1. Escrever HCL organizado em módulos e ambientes.
2. Representar os componentes com `terraform_data`.
3. Documentar como o state remoto seria protegido com locking atual, acesso mínimo e recuperação.
4. Demonstrar `fmt`, `validate`, `plan`, `apply`, um segundo plano sem mudanças e `destroy`.

## Analogias e Intuições

- O marco de IaC converte a arquitetura desenhada em arquivos de configuração que podem ser versionados, revisados e aplicados de forma reproduzível.

## Erros Comuns

- Tentar provisionar infraestrutura grande demais.
- Aplicar sem revisar plano.
- Ignorar custos de recursos criados.
- Expor segredos em outputs ou arquivos.
- Não documentar como destruir recursos de laboratório.
- Misturar produção com ambiente de estudo.

## Checklist de Entendimento

- O escopo do projeto está pequeno e claro?
- Cada recurso tem motivo para existir?
- O plano foi revisado?
- O state está protegido?
- Os segredos ficaram fora do código?
- Existe documentação para operar e destruir?
