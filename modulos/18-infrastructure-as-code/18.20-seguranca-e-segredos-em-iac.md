# Segurança e Segredos em IaC

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer riscos de segurança em IaC;
- evitar segredos em arquivos versionados;
- aplicar princípio do menor privilégio ao Terraform;
- revisar mudanças de infraestrutura com olhar de segurança.

## Onde Esta Aula Entra na Formação

- Módulo: 18. Infrastructure as Code
- Aula anterior: Workspaces e Suas Limitações
- Próxima aula: Projeto Prático: Provisionando a Infraestrutura de uma Aplicação
- Pré-requisitos: IAM, secrets, dados sensíveis no state, providers, remote state, ambientes e segurança em APIs.

## Ideia Central

IaC mexe com recursos reais. Isso inclui permissões, redes, bancos, buckets, segredos, logs e exposição pública. Segurança em IaC começa antes do `apply`: começa em como credenciais são usadas, como state é protegido, como mudanças são revisadas e como ambientes são separados.

Segredos não devem ficar no código. Credenciais do Terraform devem seguir menor privilégio. Planos precisam ser revisados não só para ver criação e destruição, mas também para perceber abertura de acesso, exposição pública e alteração de políticas.

## Tópicos da Aula

- Segredos fora do código.
- Credenciais temporárias ou controladas.
- Menor privilégio para providers.
- Revisão de políticas e redes.
- Proteção de state.
- Separação de ambientes.

## Exemplo Prático

```text
Checklist de mudança:
- abriu acesso público?
- alterou IAM?
- expôs output sensível?
- mudou backend ou state?
- usa segredo em arquivo versionado?
- aplica em produção com credencial adequada?
```

## Analogias e Intuições

- Terraform com permissão ampla é uma chave mestra automatizada.
- Revisar IaC é revisar poder de alteração sobre a infraestrutura.

## Erros Comuns

- Commitar `.tfvars` com segredo real.
- Usar credencial pessoal para produção.
- Dar permissão administrativa para automação simples.
- Não revisar políticas de acesso.
- Proteger código, mas deixar state exposto.

## Checklist de Entendimento

- Onde as credenciais ficam?
- O provider tem permissão mínima?
- O state está protegido?
- Alguma mudança abre acesso público?
- O plano altera IAM, rede ou dados sensíveis?

## Prática Recomendada

- Monte um checklist de code review para IaC.
- Liste três mudanças que exigem aprovação extra.
- Defina como segredos entram no processo sem serem versionados.
- Descreva permissões mínimas para um apply em dev.
