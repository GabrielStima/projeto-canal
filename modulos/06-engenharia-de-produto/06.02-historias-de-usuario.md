# Histórias de Usuário

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o papel de uma história de usuário;
- escrever histórias simples com usuário, ação e valor;
- diferenciar história de usuário de tarefa técnica;
- reconhecer histórias grandes ou vagas demais;
- saber o que estudar em seguida.

## Onde Esta Aula Entra na Formação

- Módulo: Engenharia de Produto
- Aula anterior: Requisitos
- Próxima aula: Critérios de aceite
- Pré-requisitos: requisitos funcionais, requisitos não funcionais, problema e contexto da funcionalidade.

## Ideia Central

História de usuário é uma forma curta de descrever uma necessidade do ponto de vista de quem usa ou se beneficia do sistema. Ela ajuda o time a manter o foco no valor entregue, não apenas na tarefa técnica.

Um formato comum é:

```text
Como [tipo de usuário],
quero [ação ou capacidade],
para [valor ou motivo].
```

Esse formato não é uma regra sagrada. Ele é uma ferramenta de clareza. Uma boa história deixa evidente quem precisa da funcionalidade, o que essa pessoa quer fazer e por que isso importa.

## Tópicos da Aula

- Usuário, ação e valor.
- História de usuário vs requisito.
- História de usuário vs tarefa técnica.
- Fatiamento de funcionalidades.
- Histórias grandes demais.
- Histórias sem valor claro.
- Perguntas para melhorar uma história.

## Exemplo Prático

Funcionalidade: favoritos em um catálogo.

História fraca:

```text
Criar favoritos.
```

História melhor:

```text
Como usuário autenticado,
quero favoritar um produto do catálogo,
para encontrar esse produto novamente com facilidade.
```

Possíveis histórias menores:

```text
- Como usuário autenticado, quero marcar um produto como favorito para salvá-lo na minha lista.
- Como usuário autenticado, quero remover um produto dos favoritos para manter minha lista atualizada.
- Como usuário autenticado, quero ver meus produtos favoritos em uma página própria para acessá-los rapidamente.
```

## Analogias e Intuições

- História de usuário é uma frase que lembra por que o código existe.
- Ela não substitui requisito, mas ajuda a organizar requisitos em entregas de valor.
- Se a história não mostra valor, talvez ela ainda seja só uma tarefa.

## Erros Comuns

- Escrever história do ponto de vista do sistema, não do usuário.
- Usar histórias para esconder tarefas técnicas sem valor visível.
- Criar histórias grandes demais para uma única entrega.
- Escrever "para o sistema funcionar" como justificativa genérica.
- Ignorar usuários diferentes com necessidades diferentes.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quem é o usuário desta história?
- Que capacidade essa pessoa ganha?
- Qual valor fica mais claro depois da entrega?
- Esta história está pequena o bastante para ser implementada e revisada?
- Quais requisitos precisam acompanhar esta história?

## Prática Recomendada

- Escolha uma funcionalidade pequena de um projeto.
- Escreva três histórias de usuário para ela.
- Reescreva cada história removendo termos técnicos desnecessários.
- Separe histórias que entregam valor de tarefas que apenas preparam a implementação.
- Identifique uma história grande demais e divida em duas menores.
