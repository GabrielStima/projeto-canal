# Requisitos

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é um requisito;
- diferenciar requisito funcional, não funcional e restrição;
- reconhecer requisitos incompletos em pedidos reais;
- escrever requisitos simples antes de implementar;
- saber o que estudar em seguida.

## Ideia Central

Requisito é uma necessidade, regra, comportamento ou restrição que orienta o que o software deve entregar. Ele ajuda a tirar uma ideia da conversa solta e colocar em um formato que pode ser discutido, implementado, testado e revisado.

O ponto principal não é escrever documentos enormes. O ponto é reduzir ambiguidade. Quando alguém diz "precisamos de login", ainda faltam muitas perguntas: quem pode entrar, como cria conta, o que acontece se errar senha, quais telas precisam estar protegidas, quais dados devem ser salvos e quais riscos existem.

## Tópicos da Aula

- Ideia, problema, solução e requisito.
- Requisitos funcionais.
- Requisitos não funcionais.
- Restrições técnicas, de negócio e de prazo.
- Perguntas para descobrir requisitos.
- Ambiguidade, suposição e risco.
- Como registrar requisitos em linguagem simples.

## Exemplo Prático

Pedido inicial:

```text
Quero que o usuário consiga favoritar produtos.
```

Requisitos possíveis:

```text
- O usuário autenticado deve conseguir marcar um produto como favorito.
- O usuário autenticado deve conseguir remover um produto dos favoritos.
- A tela de catálogo deve indicar visualmente quais produtos já foram favoritados.
- A lista de favoritos deve ser preservada ao recarregar a página.
- A ação de favoritar deve responder em até 1 segundo em condições normais.
```

Perguntas que ainda faltam:

```text
- Usuário anônimo pode favoritar?
- Existe limite de favoritos?
- Favoritos são privados?
- O que acontece se a API falhar?
- A funcionalidade precisa funcionar offline?
```

## Analogias e Intuições

- Requisito é a combinação entre pedido, regra e limite.
- Um requisito bom diminui o espaço para adivinhação.
- Requisitos não eliminam mudança, mas deixam a mudança mais visível.

## Erros Comuns

- Confundir requisito com solução técnica.
- Escrever requisito vago: "a tela deve ser fácil de usar".
- Ignorar requisitos não funcionais.
- Aceitar um pedido sem perguntar quem será afetado.
- Tratar restrições como detalhe irrelevante.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual problema este requisito tenta resolver?
- Quem precisa deste comportamento?
- O requisito descreve necessidade ou já escolhe uma solução?
- Como eu saberia que este requisito foi atendido?
- Quais restrições podem mudar a implementação?

## Prática Recomendada

- Pegue uma funcionalidade simples que você já criou.
- Escreva cinco requisitos funcionais.
- Escreva dois requisitos não funcionais.
- Liste perguntas abertas que você precisaria responder antes de implementar.
- Separe o que é necessidade do que é solução técnica.

```text
Estou estudando requisitos em uma trilha de desenvolvimento fullstack JavaScript/TypeScript.

Meu objetivo é entender o conceito, praticar e revisar sem receber respostas prontas cedo demais.

Contexto da aula:
- Descrição da aula: Esta aula ensina a transformar pedidos e ideias em requisitos funcionais, requisitos não funcionais, restrições e perguntas abertas.
- Descrição do módulo: O módulo Engenharia de Produto ensina a entender problema, contexto e entrega antes de transformar uma ideia em implementação.
- Itens já estudados: fundamentos da Web, pequenos projetos com TypeScript, Git, GitHub, documentação mínima e code review.

Crie uma sessão de estudo com:
1. uma explicação curta do conceito;
2. 5 perguntas progressivas;
3. 3 exercícios práticos;
4. 1 desafio um pouco mais difícil;
5. exemplos de aplicação em projetos reais;
6. um checklist para eu avaliar se entendi.

Regras:
- Não entregue respostas completas antes de eu tentar responder.
- Corrija minhas respostas com explicações.
- Se eu errar, dê uma dica antes de revelar a solução.
- Ao final, sugira o próximo assunto que devo estudar.
```
