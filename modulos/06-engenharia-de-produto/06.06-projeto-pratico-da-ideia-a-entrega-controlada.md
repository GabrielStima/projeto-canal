# Projeto Prático: Da Ideia a Entrega Controlada

## Objetivo

Ao final desta aula, você deve ser capaz de:

- aplicar os conceitos do módulo em uma funcionalidade pequena;
- transformar uma ideia em requisitos, histórias e critérios de aceite;
- planejar uma entrega controlada com feature flag;
- propor um experimento simples para validar valor;
- saber o que estudar em seguida.

## Onde Esta Aula Entra na Formação

- Módulo: Engenharia de Produto
- Aula anterior: Experimentos e A/B testing
- Próxima aula: Frontend Moderno
- Pré-requisitos: todas as aulas anteriores do módulo 06.

## Ideia Central

Esta aula junta o módulo inteiro em uma prática única. A meta não é construir uma aplicação grande. A meta é praticar o raciocínio que vem antes da implementação: entender problema, organizar requisitos, fatiar entrega, definir aceite, reduzir risco e planejar aprendizado.

Um bom plano de produto não precisa ser enorme. Ele precisa deixar claro o que será feito, por que será feito, como será avaliado e quais incertezas ainda existem.

## Tópicos da Aula

- Escolha de uma funcionalidade pequena.
- Problema e público.
- Requisitos funcionais e não funcionais.
- Histórias de usuário.
- Critérios de aceite.
- Plano de feature flag.
- Hipótese e métrica de experimento.
- Perguntas abertas e riscos.

## Exemplo Prático

Funcionalidade sugerida: favoritos em uma aplicação de catálogo.

Documento final esperado:

```text
1. Problema
   Usuários encontram produtos interessantes, mas não conseguem salvá-los para voltar depois.

2. Requisitos
   - Usuário autenticado pode favoritar produto.
   - Usuário autenticado pode remover favorito.
   - Favoritos continuam salvos ao recarregar a página.
   - A interface deve indicar quando a ação falhar.

3. História de usuário
   Como usuário autenticado,
   quero favoritar um produto,
   para encontrá-lo novamente com facilidade.

4. Critérios de aceite
   - Produto não favoritado pode ser marcado como favorito.
   - Produto favoritado pode ser removido dos favoritos.
   - Erro da API exibe mensagem e permite nova tentativa.

5. Feature flag
   - Nome: favorites_feature
   - Estado inicial: desligada para usuários finais.
   - Liberação: primeiro para ambiente local, depois para um grupo pequeno.
   - Remoção: após validação e estabilização.

6. Experimento
   - Hipótese: usuários que veem o botão "Salvar" retornam mais aos produtos.
   - Métrica: percentual de usuários autenticados que favoritam pelo menos um produto.
```

## Analogias e Intuições

- O projeto prático é um ensaio de conversa profissional antes do código.
- O plano não prende o time, mas revela o que o time está assumindo.
- Uma funcionalidade pequena bem pensada ensina mais do que uma funcionalidade grande escrita no impulso.

## Erros Comuns

- Escolher uma funcionalidade grande demais.
- Escrever requisitos depois de decidir toda a solução técnica.
- Criar histórias sem usuário ou sem valor.
- Ignorar critérios de erro.
- Usar feature flag sem plano de remoção.
- Criar experimento sem métrica clara.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual problema a funcionalidade resolve?
- Quais requisitos precisam ser atendidos?
- Quais histórias entregam valor real?
- Como saberei que cada história foi aceita?
- Como posso liberar com menos risco?
- O que quero aprender com a entrega?

## Prática Recomendada

- Escolha uma funcionalidade pequena para um projeto que você já conhece.
- Escreva um documento curto com problema, requisitos, histórias e critérios.
- Planeje uma feature flag simples.
- Escreva uma hipótese de experimento.
- Defina uma métrica principal.
- Revise o documento como se fosse abrir um pull request depois.
