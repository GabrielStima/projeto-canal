# Projeto Pratico: Da Ideia a Entrega Controlada

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- aplicar os conceitos do modulo em uma funcionalidade pequena;
- transformar uma ideia em requisitos, historias e criterios de aceite;
- planejar uma entrega controlada com feature flag;
- propor um experimento simples para validar valor;
- saber o que estudar em seguida.

## Onde Esta Aula Entra no Curso

- Modulo: Engenharia de Produto
- Aula anterior: Experimentos e A/B testing
- Proxima aula: Frontend Moderno
- Pre-requisitos: todas as aulas anteriores do modulo 06.

## Ideia Central

Esta aula junta o modulo inteiro em uma pratica unica. A meta nao e construir uma aplicacao grande. A meta e praticar o raciocinio que vem antes da implementacao: entender problema, organizar requisitos, fatiar entrega, definir aceite, reduzir risco e planejar aprendizado.

Um bom plano de produto nao precisa ser enorme. Ele precisa deixar claro o que sera feito, por que sera feito, como sera avaliado e quais incertezas ainda existem.

## Topicos da Aula

- Escolha de uma funcionalidade pequena.
- Problema e publico.
- Requisitos funcionais e nao funcionais.
- Historias de usuario.
- Criterios de aceite.
- Plano de feature flag.
- Hipotese e metrica de experimento.
- Perguntas abertas e riscos.

## Exemplo Pratico

Funcionalidade sugerida: favoritos em uma aplicacao de catalogo.

Documento final esperado:

```text
1. Problema
   Usuarios encontram produtos interessantes, mas nao conseguem salva-los para voltar depois.

2. Requisitos
   - Usuario autenticado pode favoritar produto.
   - Usuario autenticado pode remover favorito.
   - Favoritos continuam salvos ao recarregar a pagina.
   - A interface deve indicar quando a acao falhar.

3. Historia de usuario
   Como usuario autenticado,
   quero favoritar um produto,
   para encontra-lo novamente com facilidade.

4. Criterios de aceite
   - Produto nao favoritado pode ser marcado como favorito.
   - Produto favoritado pode ser removido dos favoritos.
   - Erro da API exibe mensagem e permite nova tentativa.

5. Feature flag
   - Nome: favorites_feature
   - Estado inicial: desligada para usuarios finais.
   - Liberacao: primeiro para ambiente local, depois para um grupo pequeno.
   - Remocao: apos validacao e estabilizacao.

6. Experimento
   - Hipotese: usuarios que veem o botao "Salvar" retornam mais aos produtos.
   - Metrica: percentual de usuarios autenticados que favoritam pelo menos um produto.
```

## Analogias e Intuicoes

- O projeto pratico e um ensaio de conversa profissional antes do codigo.
- O plano nao prende o time, mas revela o que o time esta assumindo.
- Uma funcionalidade pequena bem pensada ensina mais do que uma funcionalidade grande escrita no impulso.

## Erros Comuns

- Escolher uma funcionalidade grande demais.
- Escrever requisitos depois de decidir toda a solucao tecnica.
- Criar historias sem usuario ou sem valor.
- Ignorar criterios de erro.
- Usar feature flag sem plano de remocao.
- Criar experimento sem metrica clara.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual problema a funcionalidade resolve?
- Quais requisitos precisam ser atendidos?
- Quais historias entregam valor real?
- Como saberei que cada historia foi aceita?
- Como posso liberar com menos risco?
- O que quero aprender com a entrega?

## Pratica Recomendada

- Escolha uma funcionalidade pequena para um projeto que voce ja conhece.
- Escreva um documento curto com problema, requisitos, historias e criterios.
- Planeje uma feature flag simples.
- Escreva uma hipotese de experimento.
- Defina uma metrica principal.
- Revise o documento como se fosse abrir um pull request depois.
