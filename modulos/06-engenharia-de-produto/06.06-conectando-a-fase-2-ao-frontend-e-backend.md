# Conectando a Fase 2 ao Frontend e Backend

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar como os conceitos da fase 2 serão usados nos módulos de frontend e backend;
- reconhecer onde TypeScript, estruturas de dados e decisões de produto aparecem em uma aplicação fullstack;
- desenhar um mapa mental conectando requisitos, tipos, estruturas e componentes;
- saber o que esperar dos próximos módulos sem confundir frontend, backend e ferramentas.

## Onde Esta Aula Entra na Formação

- Módulo: 06. Engenharia de Produto
- Aula anterior: Experimentos e A/B Testing
- Próxima aula: Product Discovery
- Pré-requisitos: JavaScript moderno, TypeScript básico, estruturas de dados fundamentais, requisitos, histórias de usuário, critérios de aceite e feature flags.

## Ideia Central

Até aqui você estudou três áreas distintas: linguagem e lógica (módulos 03 e 04), ferramentas (módulo 05) e produto (módulo 06). Nos próximos módulos, essas áreas vão se misturar. O frontend moderno vai pedir componentes, estado e rotas. O backend vai pedir APIs, banco de dados e regras de negócio. As ferramentas vão garantir que o código seja versionado, testado e entregue.

Esta aula é uma ponte: ela mostra que o `type Pet` criado no módulo 03 não é apenas um exercício, mas um contrato que vai aparecer no React, no Node.js e no banco de dados. A fila de prioridade do módulo 04 não é apenas um algoritmo, mas uma forma de organizar atendimentos urgentes no PetCare OS. As histórias de usuário do módulo 06 não são apenas texto, mas direcionam o que será implementado em cada camada.

## Tópicos da Aula

- Revisão da fase 2: o que você já tem.
- TypeScript como contrato entre frontend e backend.
- Onde as estruturas de dados aparecem em uma aplicação real.
- Como requisitos e histórias viram componentes, rotas e endpoints.
- O papel de cada módulo da fase 3 e fase 4.
- O que muda de uma fase para outra.

## Exemplo Prático

História de usuário do PetCare OS:

```text
Como tutor,
quero acompanhar o status do meu pet em tempo real,
para que eu não precise ligar para a clínica toda hora.
```

Essa história toca várias camadas:

- **Produto:** define o problema, o público e o valor esperado.
- **Frontend:** uma tela no portal do tutor mostra o status atual do pet.
- **Backend:** um endpoint `/api/pets/:id/status` retorna o status do atendimento.
- **TypeScript:** o tipo `Atendimento` é usado tanto no frontend quanto no backend para garantir que ambos falem a mesma língua.
- **Estruturas de dados:** a fila de prioridade da triagem decide qual pet precisa de atenção primeiro.
- **Ferramentas:** Git versiona as mudanças, testes garantem que o fluxo não quebre, CI/CD entrega a funcionalidade.

## Analogias e Intuições

- A fase 2 é como aprender a ler, escrever e fazer contas. A fase 3 é como usar essas habilidades para escrever um artigo. A fase 4 é como publicar esse artigo em um jornal.
- TypeScript é a língua franca entre frontend e backend: se ambos usam os mesmos nomes e formatos, há menos mal-entendidos.
- Requisitos são o destino; código, banco e interface são os caminhos.

## Erros Comuns

- Achar que frontend é só "deixar bonito" e backend é só "salvar no banco".
- Começar a implementar sem olhar para os requisitos e histórias já escritos.
- Criar tipos diferentes no frontend e no backend para a mesma entidade.
- Ignorar que estruturas de dados simples, como filas e listas, resolvem problemas reais.
- Querer aprender todas as ferramentas ao mesmo tempo em vez de seguir a sequência.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quais habilidades da fase 2 serão usadas no frontend?
- Quais habilidades da fase 2 serão usadas no backend?
- Onde TypeScript atua como ponte entre as duas pontas?
- Como uma história de usuário se transforma em tarefas técnicas?
- Quais módulos vêm a seguir e qual o papel de cada um?

## Prática Recomendada

1. Pegue a documentação de produto que você produziu no módulo 06.
2. Escolha uma funcionalidade pequena (por exemplo, "visualizar status do atendimento").
3. Liste quais tipos do módulo 03 seriam necessários.
4. Liste quais estruturas de dados do módulo 04 poderiam ser usadas.
5. Indique quais partes ficariam no frontend e quais no backend.
6. Escreva uma breve explicação sobre como a funcionalidade flui de uma tela até o banco de dados.
