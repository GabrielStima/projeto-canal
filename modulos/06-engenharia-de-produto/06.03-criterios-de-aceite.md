# Criterios de Aceite

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o que sao criterios de aceite;
- transformar uma historia de usuario em condicoes verificaveis;
- escrever criterios objetivos para comportamento esperado;
- identificar criterios vagos ou impossiveis de testar;
- saber o que estudar em seguida.

## Onde Esta Aula Entra no Curso

- Modulo: Engenharia de Produto
- Aula anterior: Historias de usuario
- Proxima aula: Feature flags
- Pre-requisitos: requisitos e historias de usuario.

## Ideia Central

Criterios de aceite descrevem as condicoes que precisam ser verdadeiras para uma historia ser considerada pronta. Eles reduzem discussao subjetiva no final da entrega e ajudam desenvolvimento, teste, review e produto a olharem para a mesma definicao de sucesso.

Se a historia diz o valor, os criterios dizem como verificar esse valor. Eles nao precisam cobrir todos os detalhes internos da implementacao, mas precisam deixar claro o comportamento esperado para os casos principais, casos de erro e limites importantes.

## Topicos da Aula

- O que significa "aceitar" uma entrega.
- Criterio verificavel vs criterio subjetivo.
- Cenarios principais, alternativos e de erro.
- Relacao com testes e code review.
- Criterios antes da implementacao.
- Criterios que revelam requisitos faltantes.

## Exemplo Pratico

Historia:

```text
Como usuario autenticado,
quero favoritar um produto do catalogo,
para encontrar esse produto novamente com facilidade.
```

Criterios de aceite:

```text
- Dado que estou autenticado e vejo um produto nao favoritado,
  quando clico em favoritar,
  entao o produto aparece como favorito.

- Dado que um produto ja esta favoritado,
  quando recarrego a pagina,
  entao o estado de favorito continua visivel.

- Dado que a acao de favoritar falha,
  quando o sistema recebe o erro,
  entao uma mensagem de erro e exibida e o usuario pode tentar novamente.
```

## Analogias e Intuicoes

- Criterio de aceite e o combinado que evita "mas eu achei que era outra coisa".
- Uma historia sem criterio de aceite pode virar uma entrega dificil de avaliar.
- Bons criterios ajudam o pull request a ter foco.

## Erros Comuns

- Escrever criterios vagos como "deve funcionar bem".
- Criar criterios que so falam da implementacao interna.
- Ignorar caso de erro.
- Escrever criterios depois que o codigo ja ficou pronto.
- Fazer criterios tao grandes que viram uma especificacao confusa.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Como eu testo se este criterio foi atendido?
- O criterio descreve comportamento observavel?
- Existe um caso de erro importante?
- O criterio esta conectado a uma historia de usuario?
- O criterio ajuda no code review?

## Pratica Recomendada

- Pegue uma historia de usuario criada na aula anterior.
- Escreva tres criterios de aceite.
- Inclua pelo menos um criterio para erro ou limite.
- Reescreva criterios subjetivos em linguagem verificavel.
- Use os criterios como checklist de review de uma implementacao imaginaria.
