# Critérios de Aceite

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que são critérios de aceite;
- transformar uma história de usuário em condições verificáveis;
- escrever critérios objetivos para comportamento esperado;
- identificar critérios vagos ou impossíveis de testar;
- saber o que estudar em seguida.

## Onde Esta Aula Entra na Formação

- Módulo: Engenharia de Produto
- Aula anterior: Histórias de usuário
- Próxima aula: Feature flags
- Pré-requisitos: requisitos e histórias de usuário.

## Ideia Central

Critérios de aceite descrevem as condições que precisam ser verdadeiras para uma história ser considerada pronta. Eles reduzem discussão subjetiva no final da entrega e ajudam desenvolvimento, teste, review e produto a olharem para a mesma definição de sucesso.

Se a história diz o valor, os critérios dizem como verificar esse valor. Eles não precisam cobrir todos os detalhes internos da implementação, mas precisam deixar claro o comportamento esperado para os casos principais, casos de erro e limites importantes.

## Tópicos da Aula

- O que significa "aceitar" uma entrega.
- Critério verificável vs critério subjetivo.
- Cenários principais, alternativos e de erro.
- Relação com testes e code review.
- Critérios antes da implementação.
- Critérios que revelam requisitos faltantes.

## Exemplo Prático

História:

```text
Como usuário autenticado,
quero favoritar um produto do catálogo,
para encontrar esse produto novamente com facilidade.
```

Critérios de aceite:

```text
- Dado que estou autenticado e vejo um produto não favoritado,
  quando clico em favoritar,
  então o produto aparece como favorito.

- Dado que um produto já está favoritado,
  quando recarrego a página,
  então o estado de favorito continua visível.

- Dado que a ação de favoritar falha,
  quando o sistema recebe o erro,
  então uma mensagem de erro é exibida e o usuário pode tentar novamente.
```

## Analogias e Intuições

- Critério de aceite é o combinado que evita "mas eu achei que era outra coisa".
- Uma história sem critério de aceite pode virar uma entrega difícil de avaliar.
- Bons critérios ajudam o pull request a ter foco.

## Erros Comuns

- Escrever critérios vagos como "deve funcionar bem".
- Criar critérios que só falam da implementação interna.
- Ignorar caso de erro.
- Escrever critérios depois que o código já ficou pronto.
- Fazer critérios tão grandes que viram uma especificação confusa.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Como eu testo se este critério foi atendido?
- O critério descreve comportamento observável?
- Existe um caso de erro importante?
- O critério está conectado a uma história de usuário?
- O critério ajuda no code review?

## Prática Recomendada

- Pegue uma história de usuário criada na aula anterior.
- Escreva três critérios de aceite.
- Inclua pelo menos um critério para erro ou limite.
- Reescreva critérios subjetivos em linguagem verificável.
- Use os critérios como checklist de review de uma implementação imaginária.
