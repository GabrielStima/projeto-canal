# Comunicação com Stakeholders

## Objetivo

Ao final desta aula, você deve ser capaz de:

- identificar quem são os stakeholders de um projeto de software;
- comunicar progresso, bloqueios e mudanças para pessoas não técnicas;
- negociar escopo usando o triângulo escopo–prazo–qualidade;
- documentar decisões de forma que técnicos e não técnicos possam revisar;
- escolher entre comunicação síncrona e assíncrona para cada situação.

## Ideia Central

Comunicação com stakeholders é a habilidade de alinhar expectativas, reportar progresso e negociar escopo com pessoas que têm interesse no projeto mas não necessariamente entende de código. Essa competência é o que separa um desenvolvedor que entrega código de um engenheiro que entrega valor — porque valor só existe se for compreendido e aceito pelas pessoas certas.

A maioria dos problemas de projeto não é técnica. É comunicação. Requisito mal entendido, mudança de escopo não documentada, bloqueio não reportado a tempo, prazo combinado sem considerar incerteza técnica. Um desenvolvedor que comunica bem reduz retrabalho, gera confiança e consegue proteger o time de pressão sem fundamento.

Comunicar para não técnicos não significa simplificar demais. Significa escolher o nível de detalhe certo, usar a linguagem do problema em vez da linguagem da solução, e deixar claro qual é a decisão que a outra pessoa precisa tomar — ou qual informação ela precisa para tomá-la.

## Tópicos da Aula

- Quem são os stakeholders de um projeto de software.
- Comunicar para técnicos vs comunicar para não técnicos.
- Reportar progresso: o que está feito, o que está em andamento, o que está bloqueado.
- Como apresentar impedimentos sem parecer reclamação.
- O triângulo escopo–prazo–qualidade e como usá-lo para negociar.
- Documentar decisões: por que, não só o quê.
- Como lidar com mudanças de requisito no meio da sprint.
- Comunicação síncrona vs assíncrona: quando usar cada uma.
- ADRs (Architecture Decision Records) como ferramenta de comunicação.

## Exemplo Prático

O módulo financeiro do PetCare OS vai atrasar uma semana. A causa técnica é real: uma inconsistência na modelagem do banco de dados foi descoberta durante a implementação — orçamentos podem ter múltiplas revisões, mas o modelo atual só suporta uma versão por orçamento. Corrigir agora custa 3 dias; corrigir depois custa muito mais.

**Como não comunicar:**
```text
"A entrega vai atrasar porque o schema do banco não suporta versionamento de orçamentos.
Precisamos de uma migration e refatorar as queries de busca."
```

**Como comunicar:**
```text
"Identificamos um problema durante a implementação do módulo financeiro.
O sistema atual não consegue guardar revisões de um orçamento — por exemplo,
se o veterinário ajusta o valor após o tutor pedir desconto, a versão original
seria perdida. Isso causa problemas para histórico e auditoria.

Temos duas opções:
1. Corrigir agora: entrega atrasada em 3 dias úteis, mas o problema fica resolvido definitivamente.
2. Lançar sem esse suporte: entrega no prazo, mas será necessário um segundo trabalho em breve,
   com risco maior de retrabalho nos dados.

Nossa recomendação é a opção 1. Você está de acordo?"
```

A segunda versão apresenta o problema na linguagem do negócio, oferece opções com consequências claras e pede uma decisão — não apenas comunica um atraso.

**ADR para a decisão:**

```markdown
## ADR-012: Versionamento de Orçamentos

**Decisão:** Adicionar suporte a múltiplas revisões por orçamento antes do lançamento.

**Contexto:** Descobrimos que o modelo inicial não suporta alterações pós-criação.
O custo de corrigir depois seria maior e com risco de inconsistência de dados.

**Consequências:** Atraso de 3 dias úteis. Lançamento reprogramado para [data].

**Aprovado por:** [nome da diretora], em [data].
```

## Analogias e Intuições

- Comunicar um bloqueio cedo é como freiar com antecedência — dói menos do que bater.
- ADR é a ata da decisão técnica que qualquer pessoa pode ler meses depois e entender.
- O triângulo escopo–prazo–qualidade não tem lado mágico: mexer em um afeta os outros.
- Relatório de progresso sem bloqueios explícitos não é relatório — é marketing.

## Erros Comuns

- Comunicar só boas notícias e esconder problemas até não ter mais como esconder.
- Usar jargão técnico em atualização para stakeholders não técnicos.
- Apresentar um problema sem opções — forçar a outra pessoa a fazer a pergunta "e aí, o que fazemos?".
- Aceitar mudança de requisito no meio da sprint sem registrar o impacto.
- Usar reunião quando um e-mail assíncrono seria mais eficiente.
- Não documentar decisões — o time muda, a memória vai embora, o código fica.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quem precisa saber desta informação e qual nível de detalhe cada pessoa precisa?
- Qual decisão esta comunicação está pedindo à outra parte?
- O bloqueio foi comunicado antes de virar crise?
- A mudança de requisito foi documentada com impacto em prazo e escopo?
- Existe registro de por que esta decisão técnica foi tomada?

## Prática Recomendada

- Escreva uma atualização de status para um projeto real ou imaginário: o que está feito, o que está em andamento, o que está bloqueado.
- Reescreva um bloqueio técnico em linguagem de negócio, com duas opções e suas consequências.
- Crie um ADR para uma decisão técnica que você tomou recentemente — mesmo que pequena.
- Simule uma negociação de escopo: uma funcionalidade precisa sair para o prazo ser mantido — como você comunica isso?
