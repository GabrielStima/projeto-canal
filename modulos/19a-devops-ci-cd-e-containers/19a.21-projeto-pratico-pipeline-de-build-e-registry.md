# Projeto Prático: Pipeline de Build e Registry

## Objetivo

Ao final desta aula, você deve ser capaz de:

- montar um pipeline de CI/CD que valide, build e publique uma aplicação;
- configurar credenciais temporárias para publicação segura em um registry;
- documentar tags de imagem e garantir rastreabilidade do artefato;
- decidir quando containers e Docker Compose já são suficientes antes de partir para orquestração.

## Onde Esta Aula Entra na Formação

- Módulo: 19A. DevOps, CI/CD e Containers

## Ideia Central

O projeto final consolida o módulo: uma alteração passa por validação automatizada, gera uma imagem Docker versionada e é publicada em um registry. O foco é montar um fluxo pequeno, rastreável e seguro, que entregue um artefato pronto para execução — seja localmente com Docker Compose, seja em um cluster no futuro.

## Exemplo Prático

Pipeline de entrega do PetCare OS:

1. Um desenvolvedor abre um pull request para a API.
2. O GitHub Actions executa lint, testes unitários e verificação de tipos.
3. Se tudo passar, o workflow builda a imagem Docker a partir de um Dockerfile otimizado.
4. A imagem recebe duas tags: o hash curto do commit e `latest`.
5. O workflow publica a imagem em um registry usando credenciais OIDC temporárias.
6. Localmente, outro desenvolvedor executa `docker compose pull && docker compose up` para rodar a última versão.

## Analogias e Intuições

- O pipeline é uma linha de montagem: cada estação faz uma verificação e, no final, o produto embalado (imagem Docker) está pronto para ser enviado.
- O registry é um depósito de artefatos: você guarda versões numeradas para poder voltar a qualquer uma delas.

## Erros Comuns

- Publicar imagens com a tag `latest` como única referência.
- Armazenar segredos de longa duração no pipeline.
- Rodar testes apenas no merge, em vez de no pull request.
- Ignorar cache de camadas do Dockerfile, tornando o build lento.
