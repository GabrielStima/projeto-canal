# Backward Compatibility

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar backward compatibility em APIs;
- reconhecer mudanças quebráveis e não quebráveis;
- evoluir contratos com menos risco;
- preparar a discussão sobre versionamento.

## Onde Esta Aula Entra na Formação

- Módulo: 11. API Design Profissional
- Aula anterior: OpenAPI na Prática
- Próxima aula: Versionamento de API
- Pré-requisitos: Contratos de API, OpenAPI, REST, JSON e consumidores de API.

## Ideia Central

Backward compatibility é a capacidade de uma API evoluir sem quebrar clientes existentes. Se um cliente antigo continua funcionando depois de uma mudança, a mudança provavelmente foi compatível.

Essa ideia vem antes de versionamento porque muitas APIs são versionadas cedo demais. Em muitos casos, dá para evoluir adicionando campos opcionais, criando novos endpoints, preservando nomes antigos por um tempo e comunicando depreciações.

## Tópicos da Aula

- O que significa "quebrar um cliente".
- Mudanças compatíveis.
- Mudanças incompatíveis.
- Campos obrigatórios e opcionais.
- Depreciação.
- Janelas de migração.
- Compatibilidade vs imobilidade.

## Exemplo Prático

Considere uma API de catálogo que já retorna:

```json
{
  "id": "item-1",
  "name": "Teclado",
  "price": 250
}
```

Adicionar `description` como campo opcional costuma ser compatível:

```json
{
  "id": "item-1",
  "name": "Teclado",
  "price": 250,
  "description": "Teclado mecânico compacto"
}
```

Clientes antigos que ignoram campos desconhecidos continuam funcionando.

Mudança quebrável:

```json
{
  "id": "item-1",
  "displayName": "Teclado",
  "price": "250.00"
}
```

Trocar `name` por `displayName` quebra clientes que esperam `name`. Mudar `price` de número para texto também altera o contrato.

Adicionar um valor a um enum exige cuidado: consumidores com tratamento exaustivo podem quebrar mesmo que o provedor considere a mudança aditiva.

## Analogias e Intuições

- Compatibilidade é trocar a fechadura sem impedir quem ainda tem a chave antiga de entrar durante a migração.
- Evoluir uma API é negociar com consumidores que talvez você nem conheça.
- Mudança compatível é como acrescentar uma seção opcional a um formulário sem alterar os campos que já existiam.

## Erros Comuns

- Remover campos sem janela de migração.
- Mudar tipo de campo.
- Tornar obrigatório um campo que antes era opcional.
- Reaproveitar status code com outro significado.
- Confundir "ninguém reclamou" com "ninguém quebrou".

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que clientes dependem deste contrato?
- Esta mudança remove ou renomeia algo?
- Algum campo opcional virou obrigatório?
- O tipo de algum campo mudou?
- Existe plano de depreciação?

## Prática Recomendada

- Pegue um contrato de uma API que você já consumiu ou construiu.
- Proponha três mudanças compatíveis.
- Proponha três mudanças quebráveis.
- Escreva uma mensagem de depreciação para um campo que será removido.
- Decida quando uma nova versão seria necessária.
