# Idempotência

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar idempotência em APIs;
- reconhecer operações em que a repetição é perigosa;
- desenhar uma estratégia simples com chave de idempotência;
- relacionar idempotência com retries e failure modes.

## Ideia Central

Uma operação idempotente pode ser repetida sem mudar o resultado além da primeira execução. Em APIs, isso importa quando o cliente não sabe se a requisição anterior chegou, falhou no caminho ou foi processada antes de a conexão cair.

Alguns métodos HTTP já têm semântica idempotente em muitos contextos, como `GET`, `PUT` e `DELETE`. Mas operações de criação ou pagamento, normalmente feitas com `POST`, podem precisar de uma chave de idempotência para evitar duplicidade.

## Exemplo Prático

```http
POST /progress-events
Idempotency-Key: progress-user-1-lesson-11-2026-06-05
```

Se o cliente enviar a mesma requisição duas vezes com a mesma chave, a API deve evitar criar dois eventos iguais ou deve retornar o mesmo resultado da primeira operação.

## Analogias e Intuições

- Idempotência é apertar o botão de elevador várias vezes sem chamar vários elevadores.
- Um retry seguro precisa de memória do que já foi processado.

## Erros Comuns

- Achar que todo `POST` pode ser repetido sem risco.
- Gerar chave de idempotência no servidor tarde demais.
- Não considerar concorrência.
- Não definir por quanto tempo a chave vale.
- Retornar respostas diferentes para a mesma chave sem regra clara.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O que acontece se esta requisição for repetida?
- Existe risco de duplicidade?
- Quem gera a chave de idempotência?
- Onde a API registra a chave?
- Qual resposta deve voltar em uma repetição?

## Prática Recomendada

- Escolha uma operação de criação de progresso.
- Descreva o risco de duplicidade.
- Defina uma chave de idempotência.
- Escreva respostas para a primeira chamada e para a repetição.
- Liste problemas de concorrência possíveis.
