# Contract Testing

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar contract testing em nível introdutório;
- diferenciar teste de contrato de teste unitário e teste end-to-end;
- validar respostas de API contra um contrato esperado;
- preparar o aprofundamento em qualidade e testes no módulo 13.

## Onde Esta Aula Entra no Curso

- Módulo: 11. API Design Profissional
- Aula anterior: API Governance
- Próxima aula: Projeto Prático: Design Profissional de uma API
- Pré-requisitos: Contratos de API, OpenAPI, erros padronizados, backward compatibility e testes em nível introdutório.

## Ideia Central

Contract testing verifica se o provedor e o consumidor continuam respeitando o mesmo contrato. O objetivo não é testar toda a regra de negócio, mas detectar quando uma mudança de API quebra aquilo que os consumidores esperam.

Neste módulo, o assunto aparece como validação de API design. O aprofundamento em estratégia de testes, pirâmide, mocks, integração, E2E, CI e flaky tests fica para o módulo 13.

## Tópicos da Aula

- Diferença para teste end-to-end.
- Validação de schema.
- Validação de exemplos.
- Teste contra OpenAPI.
- Limites do contract testing.

## Exemplo Prático

Contrato esperado:

```json
{
  "id": "string",
  "title": "string",
  "status": "draft | published"
}
```

Resposta real:

```json
{
  "id": "11.01",
  "name": "Design de APIs",
  "status": "draft"
}
```

O teste de contrato deve apontar quebra porque o campo esperado `title` não veio.

## Analogias e Intuições

- Contract testing é conferir se os dois lados ainda assinam o mesmo combinado.
- Ele não prova que o produto está correto, mas detecta cedo um contrato quebrado.

## Erros Comuns

- Testar apenas o status code e ignorar o corpo.
- Fazer contract test tão amplo que vira E2E disfarçado.
- Não atualizar o contrato quando uma mudança é aprovada.
- Validar exemplos irreais.
- Achar que contract testing substitui testes de integração.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual contrato está sendo validado?
- Quem depende desse contrato?
- O teste valida schema, exemplo ou interação?
- Que quebra ele deve detectar?
- O que ficará para o módulo de qualidade?

## Prática Recomendada

- Escolha um endpoint `GET /lessons/{lessonId}`.
- Escreva o schema esperado da resposta.
- Crie uma resposta válida e uma inválida.
- Liste quais mudanças deveriam quebrar o teste.
- Explique o que este teste não cobre.
