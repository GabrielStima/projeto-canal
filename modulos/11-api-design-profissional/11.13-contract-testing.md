# Contract Testing

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar contract testing em nível introdutório;
- diferenciar teste de contrato de teste unitário e teste end-to-end;
- validar respostas de API contra um contrato esperado;
- preparar o aprofundamento em qualidade e testes no módulo 13A.

## Ideia Central

Contract testing verifica se o provedor e o consumidor continuam respeitando o mesmo contrato. O objetivo não é testar toda a regra de negócio, mas detectar quando uma mudança de API quebra aquilo que os consumidores esperam.

Neste módulo, o assunto aparece como validação de API design. O aprofundamento em estratégia de testes, pirâmide, mocks, integração, E2E, CI e flaky tests fica para o módulo 13A.

Na `petcare-api`, contract testing pode garantir que o endpoint `GET /pets/{petId}` continua retornando os campos que o portal do tutor usa.

## Exemplo Prático

Contrato esperado para `GET /pets/{petId}`:

```json
{
  "id": "string",
  "name": "string",
  "species": "dog | cat | other",
  "tutorId": "string"
}
```

Resposta real:

```json
{
  "id": "pet-1",
  "fullName": "Thor",
  "species": "dog",
  "tutorId": "tutor-1"
}
```

O teste de contrato deve apontar quebra porque o campo esperado `name` não veio — veio `fullName`.

Outro exemplo: validação de status code e schema de erro.

```text
GET /pets/invalid-id
```

Esperado: `404` com corpo no formato `Error`.
Resposta real: `404` sem corpo.

O teste de contrato detecta que o consumidor não recebeu a estrutura de erro prometida.

Uma verificação executável pode começar com o test runner nativo do Node.js:

```js
import assert from "node:assert/strict";
import test from "node:test";

test("a resposta de pet preserva os campos obrigatórios", () => {
  const response = {
    id: "pet-1",
    name: "Thor",
    species: "dog",
    tutorId: "tutor-1",
  };

  assert.equal(typeof response.id, "string");
  assert.equal(typeof response.name, "string");
  assert.equal(typeof response.species, "string");
  assert.equal(typeof response.tutorId, "string");
});
```

Esse teste ainda não chama a API real, mas apresenta a mecânica mínima. A prática separada aplica as verificações ao comportamento observável do serviço.

## Analogias e Intuições

- Contract testing é conferir se os dois lados ainda assinam o mesmo combinado.
- Ele não prova que o produto está correto, mas detecta cedo um contrato quebrado.
- É como verificar se a fatura enviada para o tutor tem os mesmos campos que o sistema financeiro espera receber.

## Erros Comuns

- Testar apenas o status code e ignorar o corpo.
- Fazer contract test tão amplo que vira E2E disfarçado.
- Não atualizar o contrato quando uma mudança é aprovada.
- Validar exemplos irreais.
- Achar que contract testing substitui testes de integração.
