# Webhooks

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é um webhook;
- diferenciar um webhook de uma chamada comum de API;
- desenhar payloads e eventos;
- reconhecer riscos de entrega, duplicidade e segurança.

## Ideia Central

Webhook é uma chamada HTTP enviada por um sistema para avisar outro sistema sobre um evento. Em vez de o consumidor perguntar repetidamente "algo mudou?", o provedor chama uma URL cadastrada quando algo acontece.

Webhooks parecem simples, mas exigem design cuidadoso: tipos de evento, payload, assinatura, retries, ordem, duplicidade, logs e documentação. O consumidor precisa saber como validar, processar e responder.

Na `petcare-api`, um webhook poderia avisar um serviço externo de notificações quando um orçamento fosse aprovado. O browser do tutor não é o destinatário direto desse webhook. Neste módulo, o foco está no contrato; entrega assíncrona, filas e jobs serão aprofundados depois.

## Exemplo Prático

```json
{
  "id": "evt_123",
  "type": "budget.approved",
  "createdAt": "2026-06-05T12:00:00Z",
  "data": {
    "budgetId": "bgt-42",
    "petId": "pet-1",
    "tutorId": "tutor-1",
    "approvedAt": "2026-06-05T12:00:00Z"
  }
}
```

O consumidor deve tratar `id` como identificador do evento para evitar processá-lo mais de uma vez.

Outro exemplo: webhook para mudança de status de atendimento.

```json
{
  "id": "evt_124",
  "type": "appointment.status_changed",
  "createdAt": "2026-06-05T12:30:00Z",
  "data": {
    "appointmentId": "apt-99",
    "petId": "pet-1",
    "tutorId": "tutor-1",
    "previousStatus": "in_procedure",
    "currentStatus": "ready_for_discharge"
  }
}
```

## Analogias e Intuições

- Webhook é uma campainha: o sistema chama quando algo acontece.
- Um evento duplicado não deve virar uma ação duplicada.
- Consulta repetida é como o tutor ligando a cada cinco minutos para saber do pet; webhook é como a clínica ligando assim que há novidade.

## Erros Comuns

- Enviar payload sem identificador único de evento.
- Não documentar retries.
- Assumir que eventos chegam em ordem.
- Não validar assinatura.
- Não registrar tentativas e falhas de entrega.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que evento dispara o webhook?
- Qual payload será enviado?
- Como o consumidor valida a origem?
- O que acontece se falhar?
- Como evitar processar duplicado?
