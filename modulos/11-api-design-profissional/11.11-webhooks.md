# Webhooks

## Ideia Central

Webhook é uma chamada HTTP enviada por um sistema para avisar outro sistema sobre um evento. Em vez de o consumidor perguntar repetidamente "algo mudou?", o provedor chama uma URL cadastrada quando algo acontece.

Webhooks parecem simples, mas exigem design cuidadoso: tipos de evento, payload, assinatura, retries, ordem, duplicidade, logs e documentação. O consumidor precisa saber como validar, processar e responder.

## Exemplo Prático

```json
{
  "id": "evt_123",
  "type": "progress.completed",
  "createdAt": "2026-06-05T12:00:00Z",
  "data": {
    "userId": "user-1",
    "lessonId": "11.10",
    "completedAt": "2026-06-05T12:00:00Z"
  }
}
```

O consumidor deve tratar `id` como identificador do evento para evitar processar duplicado.

## Analogias e Intuições

- Webhook é uma campainha: o sistema chama quando algo acontece.
- Evento duplicado não deve virar ação duplicada.

## Erros Comuns

- Enviar payload sem identificador único de evento.
- Não documentar retries.
- Assumir que eventos chegam em ordem.
- Não validar assinatura.
- Não registrar tentativas e falhas de entrega.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que evento dispara o webhook?
- Qual payload será enviado?
- Como o consumidor valida a origem?
- O que acontece se falhar?
- Como evitar processar duplicado?

## Prática Recomendada

- Desenhe um webhook de progresso concluído.
- Defina tipo de evento e payload.
- Escreva status codes esperados do consumidor.
- Liste regra de retry.
- Explique como lidar com evento duplicado.
