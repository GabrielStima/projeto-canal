# OpenAPI na Prática

## Objetivo

Ao final desta aula, você deve ser capaz de:

- estruturar uma especificação OpenAPI pequena;
- descrever paths, operations, parameters, request bodies, responses e schemas;
- reutilizar componentes;
- entender como OpenAPI ajuda documentação, contrato e governança.

## Ideia Central

OpenAPI é uma forma padronizada de descrever APIs HTTP. Na prática, ela transforma decisões de contrato em um documento que pessoas e ferramentas conseguem ler.

Uma especificação OpenAPI costuma declarar metadados da API, servidores, paths, operações, parâmetros, request bodies, responses, schemas, componentes reutilizáveis, segurança e tags. O valor não está no arquivo YAML em si, mas no alinhamento que ele cria entre backend, frontend, QA, produto e documentação.

## Tópicos da Aula

- Objeto raiz: `openapi`, `info`, `servers` e `paths`.
- Operações por método HTTP.
- `parameters` em path e query.
- `requestBody` para entrada.
- `responses` para sucesso e erro.
- `schemas` para formatos de dados.
- `components` para reutilização.
- `tags`, exemplos e documentação.
- Limites de OpenAPI.

## Exemplo Prático

```yaml
openapi: 3.1.0
info:
  title: Study Panel API
  version: 1.0.0
paths:
  /lessons/{lessonId}:
    get:
      tags:
        - Lessons
      summary: Busca uma aula por ID
      parameters:
        - name: lessonId
          in: path
          required: true
          schema:
            type: string
      responses:
        "200":
          description: Aula encontrada
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/Lesson"
        "404":
          description: Aula não encontrada
components:
  schemas:
    Lesson:
      type: object
      required:
        - id
        - title
      properties:
        id:
          type: string
        title:
          type: string
```

## Analogias e Intuições

- OpenAPI é a planta do contrato, não o prédio funcionando.
- `components` é uma biblioteca de peças reutilizáveis da especificação.

## Erros Comuns

- Criar a spec depois que tudo já foi decidido sem discutir contrato.
- Documentar apenas `200`.
- Não reutilizar schemas e responses comuns.
- Escrever exemplos que não batem com o backend.
- Achar que OpenAPI garante bom design automaticamente.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O que fica no objeto raiz da especificação?
- O que uma operation descreve?
- Quando usar `parameters` e quando usar `requestBody`?
- Para que servem `schemas` e `components`?
- Como a spec pode ajudar testes e governança?

## Prática Recomendada

- Escolha dois endpoints do painel de estudos.
- Escreva uma especificação OpenAPI mínima.
- Inclua uma resposta de sucesso e uma de erro.
- Crie pelo menos um schema reutilizável.
- Compare a spec com o comportamento real esperado.
