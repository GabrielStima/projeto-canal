# API Design Profissional

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que diferencia uma API funcional de uma API bem desenhada;
- reconhecer os temas que serão estudados no módulo;
- entender como contratos, consistência, versionamento e documentação se conectam;
- identificar onde o design da API impacta quem consome e quem mantém.

## Onde Esta Aula Entra na Formação

- Módulo: 11. API Design Profissional

- Próxima aula: Design de APIs

## Ideia Central

Criar endpoints que funcionam é diferente de desenhar APIs que as pessoas conseguem usar, entender e evoluir sem quebrar quem já depende delas. API design profissional é sobre tomar decisões explícitas: nomes, formatos de erro, paginação, versionamento, idempotência e governança.

Uma API é um contrato. Quem consome não vê o código interno — vê apenas o que você expõe. Mudar esse contrato sem cuidado quebra clientes em produção. Documentar bem reduz atrito e suporte. Padronizar erros torna debugging previsível.

Este módulo cobre desde princípios de design até contract testing, passando por OpenAPI na prática, backward compatibility, idempotência, webhooks e governança, preparando o aluno para entregar APIs que duram e evoluem.

## Exemplo Prático

```text
Fluxo: integração do sistema de faturamento com o portal do tutor

Perguntas que este módulo vai responder:
- como garantir que reenviar um orçamento duas vezes não gere cobrança duplicada?
- como paginar a lista de serviços disponíveis para o frontend carregar aos poucos?
- como documentar o endpoint de aprovação de orçamento para que outro time consiga integrá-lo?
- como versionar a API quando a estrutura de resposta do atendimento precisar mudar?
- como padronizar os erros para que o frontend consiga exibir mensagens úteis em todos os fluxos?
```

## Analogias e Intuições

- Uma API bem desenhada é como um formulário médico padronizado: qualquer profissional consegue preencher e interpretar, independente de quem criou.
- Idempotência é como um botão de elevador: apertar duas vezes não muda o resultado.
- Backward compatibility é como ampliar um hospital sem fechar os corredores existentes: quem já usa o sistema não pode ser impedido de continuar funcionando.

## Erros Comuns

- Retornar sempre 200 OK e colocar o erro dentro do corpo da resposta.
- Mudar o contrato da API sem versionar, quebrando clientes em produção.
- Ignorar idempotência em endpoints de criação ou cobrança.
- Não paginar listas grandes, causando timeout e sobrecarga no servidor.
- Gerar documentação OpenAPI apenas no final, quando o design já está difícil de mudar.

## Checklist de Entendimento

- O que é backward compatibility e por que ela importa para quem consome a API?
- Qual a diferença entre paginação por offset e por cursor?
- Quando um endpoint precisa ser idempotente?
- O que um contrato de API deve garantir ao consumidor?
- O que este módulo vai aprofundar?

## Prática Recomendada

- Revise três endpoints da API do PetCare OS e verifique se os códigos HTTP estão corretos.
- Escreva a estrutura de erro padronizado que o PetCare OS usaria em todos os fluxos.
- Adicione paginação ao endpoint de listagem de atendimentos e defina os parâmetros de query.
- Identifique quais endpoints do fluxo de faturamento precisam de chave de idempotência.
