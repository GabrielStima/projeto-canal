# Design de APIs

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que significa desenhar uma API;
- diferenciar um endpoint funcionando de uma API bem desenhada;
- pensar em recursos, operações e consumidores;
- reconhecer decisões que afetam manutenção e evolução.

## Onde Esta Aula Entra na Formação

- Módulo: 11. API Design Profissional
- Aula anterior: API Design Profissional
- Próxima aula: Contratos de API
- Pré-requisitos: HTTP, REST, JSON APIs, OpenAPI introdutório e projeto prático de backend com Node.js.

## Ideia Central

Design de API é o trabalho de transformar uma capacidade do sistema em uma interface clara para outros sistemas ou pessoas. A pergunta não é só "qual rota eu crio?", mas "que promessa esta API faz para quem vai consumi-la?".

Uma API bem desenhada parece previsível. Recursos têm nomes consistentes, métodos HTTP comunicam intenção, erros têm formato conhecido, listagens seguem o mesmo padrão e mudanças futuras não quebram clientes sem necessidade.

## Tópicos da Aula

- API como interface de produto.
- Consumidor da API.
- Recursos e operações.
- Consistência de nomes.
- Payloads de entrada e saída.
- Decisões explícitas vs decisões acidentais.
- Trade-offs entre simplicidade e flexibilidade.

## Exemplo Prático

Compare duas formas de expor aulas do painel de estudos:

```text
POST /getLessons
POST /createNewLesson
POST /deleteLesson
```

```text
GET /lessons
POST /lessons
DELETE /lessons/{lessonId}
```

A segunda opção usa recursos e métodos HTTP para comunicar intenção. Ela não é automaticamente perfeita, mas cria uma base mais previsível.

## Analogias e Intuições

- API é um balcão de atendimento: a pessoa não precisa conhecer a cozinha para fazer um pedido.
- Um bom contrato reduz improviso entre equipes.

## Erros Comuns

- Criar endpoints conforme a função interna do backend.
- Desenhar a API como cópia direta das tabelas do banco.
- Usar nomes diferentes para ideias iguais.
- Pensar apenas no primeiro consumidor.
- Tratar documentação como substituta de design.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quem consome esta API?
- Qual recurso está sendo exposto?
- O método HTTP comunica a intenção correta?
- A resposta é previsível?
- Que decisão ficaria difícil de mudar depois?

## Prática Recomendada

- Pegue três endpoints do projeto do módulo 10.
- Reescreva os nomes usando recursos.
- Liste consumidores possíveis da API.
- Defina uma convenção simples para nomes de recursos.
- Identifique uma decisão que pode quebrar clientes no futuro.
