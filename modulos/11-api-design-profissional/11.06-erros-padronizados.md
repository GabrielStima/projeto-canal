# Erros Padronizados

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que erros precisam de padrão;
- desenhar respostas de erro úteis para consumidores;
- diferenciar erro técnico de erro de domínio;
- evitar respostas ambíguas.

## Onde Esta Aula Entra na Formação

## Ideia Central

Erro também faz parte do contrato. Quando uma API retorna formatos diferentes para cada problema, o consumidor precisa adivinhar o que aconteceu e como reagir.

Um erro padronizado deve responder a perguntas simples: qual foi o problema, qual código estável identifica o erro, que detalhe pode ajudar na correção, se a requisição pode ser tentada de novo e como correlacionar o erro em logs.

## Tópicos da Aula

- Informações sensíveis.

## Exemplo Prático

```json
{
  "error": {
    "code": "LESSON_TITLE_REQUIRED",
    "message": "O título da aula é obrigatório.",
    "details": [
      {
        "field": "title",
        "reason": "required"
      }
    ],
    "requestId": "req_123"
  }
}
```

Esse formato permite exibir mensagem, registrar logs e tratar casos específicos sem depender de texto solto.

## Analogias e Intuições

- Erro padronizado é um recibo do problema.
- Status code diz a categoria; o corpo explica o caso.

## Erros Comuns

- Retornar `200` com erro dentro do JSON.
- Usar mensagem como identificador de regra.
- Expor stack trace ou dados sensíveis.
- Misturar formatos de erro por endpoint.
- Não documentar erros na OpenAPI.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- O status code está correto?
- O consumidor tem um código estável para tratar?
- A mensagem ajuda sem vazar informação?
- Erros de validação são detalhados?
- O erro aparece no contrato da API?

## Prática Recomendada

- Liste cinco erros da API do painel de estudos.
- Escolha um status code para cada erro.
- Crie códigos estáveis.
- Escreva exemplos de JSON de erro.
- Documente pelo menos dois erros em OpenAPI.
