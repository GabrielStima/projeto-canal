# Projeto Prático: Design Profissional de uma API

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os principais conceitos do módulo em uma API pequena;
- evoluir endpoints existentes para contratos mais profissionais;
- documentar decisões de design;
- validar o contrato com testes introdutórios.

## Onde Esta Aula Entra na Formação

- Módulo: 11. API Design Profissional
- Aula anterior: Contract Testing
- Próxima aula: 12. Autenticação, Autorização e Segurança
- Pré-requisitos: Todas as aulas anteriores do módulo 11.

## Ideia Central

O projeto final do módulo deve pegar a `petcare-api` construída no módulo 10 e tratá-la como um produto de integração. O objetivo não é adicionar mil funcionalidades, mas melhorar o desenho da API: contrato, consistência, evolução, erros, listagens, idempotência, webhooks, governança e teste de contrato.

Essa prática prepara o próximo módulo porque autenticação, autorização e segurança ficam mais claras quando a API já tem contratos bem definidos.

## Tópicos da Aula

- Revisão do escopo da API.
- Recursos principais.
- Contratos de request e response.
- OpenAPI dos endpoints centrais.
- Backward compatibility.
- Decisão de versionamento.
- Erros padronizados.
- Paginação, filtros e ordenação.
- Idempotência em operação crítica.
- Webhook de notificação ao tutor.
- Checklist de governança.
- Contract testing introdutório.

## Exemplo Prático

```text
API: PetCare API

Recursos:
- pets;
- appointments (atendimentos);
- budgets (orçamentos);

Entregáveis:
- convenção de nomes;
- contrato de endpoints principais;
- spec OpenAPI parcial;
- formato padrão de erro;
- listagem paginada com filtros e ordenação;
- operação idempotente para aprovação de orçamento;
- decisão e contrato exploratório de webhook;
- checklist de API governance;
- teste de contrato para pelo menos uma resposta.
```

## Analogias e Intuições

- O projeto é uma revisão profissional da API, não uma reescrita completa.
- Uma API pequena também pode ter contrato sério.
- Melhorar contratos é como padronizar prontuários: o trabalho médico não muda, mas a comunicação fica mais confiável.

## Erros Comuns

- Tentar implementar autenticação completa antes do módulo 12.
- Criar versionamento sem mudança quebrável real.
- Documentar endpoints que não batem com o comportamento esperado.
- Padronizar erros apenas no caminho feliz.
- Fazer contract testing amplo demais para este momento.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quais recursos a API expõe?
- Quais contratos foram definidos?
- A OpenAPI está coerente com os exemplos?
- Que mudanças seriam backward compatible?
- Como os erros foram padronizados?
- Como as listagens usam paginação, filtros e ordenação?
- Qual operação precisa de idempotência?
- Que webhook foi desenhado?
- Que regra de governança evita inconsistência?
- Que contrato foi testado?

## Prática Recomendada

- Escolha três recursos da `petcare-api`.
- Desenhe endpoints e contratos.
- Escreva exemplos de request e response.
- Crie uma spec OpenAPI parcial.
- Padronize erros.
- Implemente listagem paginada, filtrável e ordenável.
- Defina a aprovação de orçamento como operação idempotente.
- Registre a decisão sobre um webhook e, se fizer sentido, desenhe somente seu contrato.
- Crie um checklist de governança.
- Escreva um teste de contrato introdutório.

A [Atividade Final do Módulo](exercicios/atividade-final-modulo.md) consolida todos esses pontos na `petcare-api`.
