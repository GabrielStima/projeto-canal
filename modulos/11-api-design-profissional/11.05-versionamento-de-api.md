# Versionamento de API

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que APIs são versionadas;
- diferenciar versionamento de compatibilidade;
- comparar estratégias comuns de versionamento;
- decidir quando uma nova versão faz sentido.

## Onde Esta Aula Entra no Curso

- Módulo: 11. API Design Profissional

- Próxima aula: Erros Padronizados

## Ideia Central

Versionamento de API é uma forma de separar contratos quando uma mudança relevante não deve afetar todos os consumidores ao mesmo tempo. Ele ajuda migração, mas também cria custo: documentação duplicada, manutenção de versões antigas, testes extras e decisões de descontinuação.

Uma API profissional não versiona tudo por medo. Ela primeiro tenta preservar compatibilidade. Quando a mudança realmente altera o contrato de forma quebrável ou muda a proposta da API, versionar pode ser a opção mais honesta.

## Exemplo Prático

```text
GET /v1/lessons/{lessonId}
GET /v2/lessons/{lessonId}
```

Na `v1`, a aula retorna `title`. Na `v2`, a API muda o modelo para suportar títulos por idioma:

```json
{
  "id": "lesson-1",
  "titles": {
    "pt-BR": "REST",
    "en-US": "REST"
  }
}
```

Essa mudança pode justificar uma nova versão porque altera o formato esperado pelo consumidor.

## Analogias e Intuições

- Versionamento é uma faixa nova na estrada, não uma desculpa para deixar a antiga abandonada.
- Uma versão antiga ainda é produto enquanto alguém depende dela.

## Erros Comuns

- Criar `v2` para qualquer campo novo.
- Manter versões antigas sem prazo ou ownership.
- Misturar comportamentos incompatíveis na mesma versão.
- Não documentar diferenças entre versões.
- Versionar no path e esquecer a spec.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- A mudança é realmente quebrável?
- Quem usa a versão antiga?
- Qual estratégia de versionamento será usada?
- Como o consumidor sabe que precisa migrar?
- Quando a versão antiga será descontinuada?

## Prática Recomendada

- Escolha uma mudança quebrável no recurso `lessons`.
- Desenhe como ficaria em `v1` e `v2`.
- Escreva uma nota curta de migração.
- Liste testes necessários para as duas versões.
- Defina um critério de descontinuação.
