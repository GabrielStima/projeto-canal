# Contratos de API

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é um contrato de API;
- identificar partes de um contrato;
- diferenciar contrato explícito de comportamento implícito;
- preparar o uso prático de OpenAPI.

## Onde Esta Aula Entra no Curso

- Módulo: 11. API Design Profissional
- Aula anterior: Design de APIs
- Próxima aula: OpenAPI na Prática
- Pré-requisitos: HTTP, JSON, REST, status codes, validação e design de APIs.

## Ideia Central

Contrato de API é a promessa observável entre provedor e consumidor. Ele descreve como chamar a API, que dados enviar, que respostas esperar, que erros podem acontecer e quais regras precisam ser respeitadas.

Mesmo quando ninguém escreveu uma documentação formal, o contrato existe. Se o frontend depende de `title`, `status` e `createdAt`, esses campos viram compromisso. O problema é que contratos implícitos quebram em silêncio.

## Tópicos da Aula

- Contrato como promessa.
- Método, caminho e parâmetros.
- Request body.
- Response body.
- Status codes.
- Headers importantes.
- Erros e regras de validação.
- Contrato escrito, testado e comunicado.

## Exemplo Prático

```text
GET /lessons/{lessonId}

Path params:
- lessonId: string, obrigatório

Resposta 200:
- id: string
- title: string
- module: string
- status: "draft" | "published"

Resposta 404:
- error.code: "LESSON_NOT_FOUND"
- error.message: string
```

Esse contrato já deixa claro o que o consumidor pode esperar e o que precisa tratar.

## Analogias e Intuições

- Contrato de API é um acordo de interface, não uma explicação do código interno.
- Um contrato bom evita conversas repetidas sobre o mesmo comportamento.

## Erros Comuns

- Documentar apenas o caso de sucesso.
- Mudar nomes de campos sem avisar.
- Retornar formatos diferentes em endpoints parecidos.
- Esconder regras importantes no código.
- Achar que contrato só importa para APIs públicas.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Que promessa esta API faz?
- Quais campos são obrigatórios?
- Quais erros podem acontecer?
- O consumidor sabe o que fazer com a resposta?
- O contrato está escrito em algum lugar?

## Prática Recomendada

- Escolha um endpoint de busca por ID.
- Escreva método, path, parâmetros e respostas.
- Inclua pelo menos dois erros.
- Marque campos obrigatórios e opcionais.
- Peça para outra pessoa consumir apenas pelo contrato escrito.
