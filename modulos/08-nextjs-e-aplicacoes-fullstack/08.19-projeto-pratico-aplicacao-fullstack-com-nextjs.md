# Projeto prático: aplicação fullstack com Next.js

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os principais assuntos do módulo em uma aplicação pequena;
- criar rotas, layouts, Server Components e Client Components;
- usar loading, erro, formulários, validação e API interna;
- preparar uma versão inicial para deploy.

## Onde Esta Aula Entra na Formação

- Módulo: 08. Next.js e Aplicações Fullstack no Frontend
- Aula anterior: Deploy de aplicações Next.js
- Próxima aula: 09. Bancos de Dados
- Pré-requisitos: Todas as aulas anteriores do módulo 08 e o projeto do módulo 07.

## Ideia Central

O projeto final transforma conceitos isolados em uma aplicação coerente. Nesta aula, você vai construir o **Portal do Tutor do PetCare OS**: um fluxo completo de Next.js onde o tutor vê seus pets, cadastra um novo pet, acompanha o status do último atendimento e simula a aprovação de um orçamento.

A ideia não é criar um produto grande nem antecipar banco de dados real. O foco é montar um fluxo de Next.js com dados simulados ou uma API local simples, reaproveitando os componentes e as regras que você construiu no módulo 07.

## Tópicos da Aula

- Escopo do Portal do Tutor.
- Árvore de rotas.
- Layout principal e navegação.
- Server Components: listagem e detalhes dos pets.
- Client Components: formulário de novo pet e aprovação de orçamento.
- Loading, erro e Suspense.
- Route Handler para simular aprovação.
- Validação e feedback.
- Simulação de sessão.
- Deploy.

## Exemplo Prático

```text
Aplicação: Portal do Tutor do PetCare OS

Rotas:
- /                         (página inicial com login simulado)
- /tutor/pets               (listagem de pets)
- /tutor/pets/[id]          (detalhes do pet e status do último atendimento)
- /tutor/pets/novo          (formulário de cadastro de pet)
- /tutor/orcamentos         (orçamentos pendentes de aprovação)
- /api/orcamentos/[id]      (Route Handler simples, dados em memória)

Funcionalidades:
- listar pets do tutor;
- ver detalhes de um pet e o status do último atendimento;
- cadastrar novo pet (nome, espécie, data de nascimento aproximada);
- listar orçamentos pendentes e simular aprovação;
- mostrar loading, erro e estados vazios;
- simular usuário logado;
- preparar deploy.
```

## Analogias e Intuições

- O projeto é um ensaio geral antes de banco de dados e backend completo.
- Escopo pequeno bem acabado ensina mais do que arquitetura grande incompleta.

## Erros Comuns

- Adicionar banco de dados antes de consolidar o fluxo.
- Transformar tudo em Client Component.
- Ignorar loading e erro.
- Criar autenticação falsa como se fosse segura.
- Fazer deploy sem testar rotas principais.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Quais rotas existem?
- Quais componentes rodam no servidor?
- Quais componentes precisam do cliente?
- Onde os dados são buscados?
- Como formulário e validação funcionam?
- O que precisa mudar quando entrar banco de dados?

## Prática Recomendada

- Reaproveite componentes do módulo 07 (como `PetCard`, `StatusBadge` e lista de orçamentos) adaptando-os para Server ou Client Components.
- Construa primeiro a navegação e o layout do portal.
- Adicione listagem e detalhes com dados simulados.
- Crie o formulário de cadastro de pet e a validação.
- Adicione um Route Handler simples para simular a aprovação de orçamento.
- Revise loading, erro e deploy.
