# Dados em Tempo Real e Comunicação Assíncrona

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que nem toda comunicação web cabe no request/response tradicional;
- diferenciar tempo real, comunicação assíncrona e processamento em background;
- reconhecer os temas que serão estudados no módulo;
- entender que essas escolhas sempre envolvem trade-offs.

## Onde Esta Aula Entra no Curso

- Módulo: 16. Dados em Tempo Real e Comunicação Assíncrona
- Aula anterior: Projeto Prático: Arquitetando uma API para Evolução
- Próxima aula: Do Request/Response ao Tempo Real
- Pré-requisitos: HTTP, APIs, JavaScript assíncrono, backend com Node.js, idempotência, observabilidade, sistemas distribuídos e messaging.

## Ideia Central

O modelo request/response é ótimo para muitas interações: o cliente pede algo, o servidor responde e a conversa termina. Mas alguns problemas não funcionam bem assim. Um chat precisa receber mensagens novas. Um painel precisa atualizar status. Um relatório pesado não deveria travar a requisição. Uma notificação pode ser enviada depois.

Dados em tempo real e comunicação assíncrona aparecem quando o sistema precisa lidar com tempo, espera, eventos, trabalho posterior ou múltiplas partes reagindo a uma mesma mudança.

O ponto central deste módulo não é decorar tecnologias. É aprender a escolher: polling, long polling, SSE, WebSocket, jobs, filas, broker, pub/sub ou simplesmente continuar com HTTP comum.

## Tópicos da Aula

- Limites do request/response tradicional.
- Tempo real como percepção de atualização rápida.
- Comunicação assíncrona como desacoplamento temporal.
- Polling, SSE e WebSocket em alto nível.
- Jobs, filas, brokers e pub/sub em alto nível.
- Falhas, duplicidade, atraso e idempotência.

## Exemplo Prático

```text
Fluxo: aluno conclui uma aula

Resposta imediata:
- salvar progresso;
- responder sucesso ao aluno.

Tempo real:
- atualizar dashboard aberto em outro dispositivo.

Processamento assíncrono:
- recalcular estatísticas;
- enviar notificação;
- verificar se certificado pode ser liberado.
```

## Analogias e Intuições

- Request/response é uma pergunta direta com resposta imediata.
- Tempo real é manter uma janela aberta para receber novidades.
- Processamento assíncrono é anotar uma tarefa para ser executada fora da conversa principal.

## Erros Comuns

- Usar WebSocket quando polling simples resolveria.
- Colocar tudo em fila sem entender o que precisa ser imediato.
- Chamar algo de tempo real quando alguns segundos de atraso seriam aceitáveis.
- Ignorar reconexão, duplicidade e reprocessamento.
- Tratar comunicação assíncrona como se fosse mais simples do que HTTP.

## Checklist de Entendimento

- Que parte do fluxo precisa responder agora?
- Que parte pode acontecer depois?
- O cliente precisa receber atualização do servidor?
- A comunicação precisa ser bidirecional?
- O sistema está preparado para duplicidade e falhas?

## Prática Recomendada

- Escolha três funcionalidades de uma aplicação que você conhece.
- Classifique cada uma como request/response, tempo real ou processamento assíncrono.
- Justifique uma escolha simples e uma escolha mais complexa.
- Liste uma falha possível em cada caso.
