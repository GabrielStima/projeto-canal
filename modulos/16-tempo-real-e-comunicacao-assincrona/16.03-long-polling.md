# Long Polling

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que é long polling;
- diferenciar long polling de short polling;
- reconhecer quando ele reduz requisições vazias;
- entender seus custos de conexão e timeout.

## Onde Esta Aula Entra no Curso

- Módulo: 16. Dados em Tempo Real e Comunicação Assíncrona
- Aula anterior: Short Polling
- Próxima aula: Server-Sent Events
- Pré-requisitos: HTTP, APIs, timeout, JavaScript assíncrono e short polling.

## Ideia Central

Long polling também usa HTTP, mas muda a dinâmica. O cliente faz uma requisição e o servidor segura essa requisição aberta até ter uma novidade ou até atingir um timeout. Quando a resposta chega, o cliente abre outra requisição.

Isso evita muitas respostas vazias quando as novidades são raras. Em troca, o servidor precisa lidar com requisições abertas por mais tempo, timeouts e reconexões.

Long polling é útil quando você quer atualização mais rápida que short polling, mas ainda não quer ou não pode usar SSE ou WebSocket.

## Tópicos da Aula

- Requisição mantida aberta.
- Resposta quando há evento ou timeout.
- Novo request após cada resposta.
- Diferença para short polling.
- Timeouts, proxies e reconexão.
- Custo de manter conexões penduradas.

## Exemplo Prático

```text
Tela: notificações do usuário

1. Cliente chama GET /notifications/wait.
2. Servidor espera até existir uma notificação nova.
3. Se existir, responde com a notificação.
4. Se não existir em 30 segundos, responde vazio.
5. Cliente abre uma nova requisição.
```

## Analogias e Intuições

- Short polling é bater na porta a cada poucos segundos.
- Long polling é ficar esperando na porta até alguém ter uma resposta ou pedir para você voltar depois.

## Erros Comuns

- Deixar requisições abertas sem limite de timeout.
- Não abrir nova requisição depois da resposta.
- Ignorar cancelamento quando o usuário sai da tela.
- Tratar timeout como erro real.
- Usar long polling para tráfego muito intenso e bidirecional.

## Checklist de Entendimento

- O servidor sabe quando existe novidade?
- Qual timeout será usado?
- O cliente reconecta quando recebe resposta?
- O que é resposta vazia?
- O custo de conexões abertas é aceitável?
