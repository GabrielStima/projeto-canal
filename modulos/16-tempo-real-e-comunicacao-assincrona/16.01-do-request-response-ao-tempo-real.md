# Do Request/Response ao Tempo Real

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o modelo request/response;
- reconhecer quando ele deixa de ser suficiente;
- diferenciar atualização periódica, atualização por evento e canal persistente;
- escolher um caminho inicial simples para dados em tempo real.

## Onde Esta Aula Entra no Curso

- Módulo: 16. Dados em Tempo Real e Comunicação Assíncrona
- Aula anterior: Dados em Tempo Real e Comunicação Assíncrona
- Próxima aula: Short Polling
- Pré-requisitos: HTTP, cliente/servidor, estado, APIs, cache e JavaScript assíncrono.

## Ideia Central

HTTP tradicional funciona muito bem quando o cliente sabe quando pedir uma informação. O problema começa quando a mudança acontece no servidor, mas o cliente precisa descobrir isso rapidamente.

Tempo real não significa necessariamente "instantâneo". Em muitos produtos, significa que o usuário percebe que a informação está atualizada sem precisar recarregar a página manualmente. Essa percepção pode ser construída com polling, long polling, SSE, WebSocket ou até uma combinação simples.

Antes de escolher tecnologia, pergunte qual experiência o usuário precisa e qual atraso é aceitável.

## Tópicos da Aula

- Request/response como conversa curta.
- Atualização iniciada pelo cliente.
- Atualização enviada pelo servidor.
- Latência aceitável para o usuário.
- Custo de conexão, custo de requisição e custo operacional.
- Critérios para escolher uma abordagem.

## Exemplo Prático

```text
Tela: status de processamento de vídeo

Perguntas:
- o usuário precisa ver cada etapa no mesmo instante?
- atualizar a cada 5 segundos é suficiente?
- o servidor precisa mandar mensagens variadas?
- o cliente também precisa enviar comandos durante o processamento?
```

Se atualizar a cada poucos segundos for suficiente, polling pode resolver. Se o servidor precisa empurrar eventos de progresso, SSE pode ser melhor. Se cliente e servidor precisam trocar mensagens o tempo todo, WebSocket pode fazer mais sentido.

## Analogias e Intuições

- Request/response é tocar a campainha e esperar alguém responder.
- Polling é ligar de tempos em tempos perguntando se há novidade.
- Tempo real orientado a eventos é receber aviso quando a novidade acontece.

## Erros Comuns

- Começar por WebSocket antes de entender a necessidade.
- Confundir baixa latência com necessidade real de tempo real.
- Esquecer que conexões persistentes também precisam ser operadas.
- Ignorar usuários em redes instáveis.
- Não definir quanto atraso é aceitável.

## Checklist de Entendimento

- Quem inicia a comunicação?
- O cliente precisa perguntar ou o servidor precisa avisar?
- Qual atraso é aceitável?
- A comunicação é unidirecional ou bidirecional?
- O ganho compensa a complexidade?

## Prática Recomendada

- Escolha uma tela de dashboard, chat ou processamento.
- Defina o atraso máximo aceitável.
- Compare polling, SSE e WebSocket para esse caso.
- Escreva qual seria sua primeira solução e por quê.
