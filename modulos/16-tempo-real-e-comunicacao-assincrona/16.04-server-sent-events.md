# Server-Sent Events

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o que são Server-Sent Events;
- entender o papel da API `EventSource`;
- reconhecer quando SSE é melhor que polling;
- diferenciar SSE de WebSocket.

## Onde Esta Aula Entra no Curso

- Módulo: 16. Dados em Tempo Real e Comunicação Assíncrona
- Aula anterior: Long Polling
- Próxima aula: WebSocket
- Pré-requisitos: HTTP, eventos no browser, JavaScript assíncrono, short polling e long polling.

## Ideia Central

Server-Sent Events, ou SSE, permitem que o servidor envie eventos para o cliente por uma conexão HTTP mantida aberta. No browser, a interface comum para consumir esse fluxo é `EventSource`.

SSE é unidirecional: os dados fluem do servidor para o cliente. Isso combina bem com status, notificações, feeds, progresso de tarefas e atualizações que o cliente precisa receber, mas sem precisar mandar mensagens frequentes pelo mesmo canal.

Quando o cliente também precisa enviar mensagens em tempo real pelo canal persistente, WebSocket tende a ser uma opção mais adequada.

## Tópicos da Aula

- Conexão HTTP aberta para eventos.
- Fluxo servidor -> cliente.
- `EventSource` no browser.
- Eventos padrão e eventos nomeados.
- Reconexão e identificação de eventos.
- Diferença entre SSE, polling e WebSocket.

## Exemplo Prático

```text
Tela: progresso de importação

Cliente abre:
- GET /imports/123/events

Servidor envia eventos:
- progress: 10%
- progress: 40%
- progress: 90%
- done
```

No cliente, a aplicação escuta mensagens e atualiza a interface conforme os eventos chegam.

## Analogias e Intuições

- SSE é como assinar um canal de avisos do servidor.
- O cliente abre a escuta; o servidor fala quando houver novidade.

## Erros Comuns

- Usar SSE quando o cliente precisa enviar mensagens constantes pelo mesmo canal.
- Não planejar reconexão.
- Enviar eventos grandes demais.
- Não identificar eventos quando o cliente precisa retomar de onde parou.
- Esquecer autenticação e autorização do fluxo.

## Checklist de Entendimento

- A comunicação é apenas servidor -> cliente?
- O cliente precisa receber eventos progressivos?
- O fluxo pode ser representado como mensagens pequenas?
- O que acontece se a conexão cair?
- Como o servidor sabe quais eventos o usuário pode receber?

## Prática Recomendada

- Defina três tipos de evento.
- Liste como o cliente reagiria à queda de conexão.
