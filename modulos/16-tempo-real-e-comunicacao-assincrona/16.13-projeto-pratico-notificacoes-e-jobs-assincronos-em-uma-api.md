# Projeto Prático: Notificações e Jobs Assíncronos em uma API

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar tempo real e processamento assíncrono em um fluxo único;
- separar resposta imediata, job, fila, evento e notificação;
- aplicar idempotência em consumidores;
- desenhar observabilidade mínima para operar o fluxo.

## Onde Esta Aula Entra na Formação

- Módulo: 16. Dados em Tempo Real e Comunicação Assíncrona
- Aula anterior: Operações Idempotentes em Fluxos Assíncronos
- Próxima aula: Cloud Fundamentals
- Pré-requisitos: Polling, SSE, WebSocket, processamento assíncrono, background jobs, filas, brokers, pub/sub, idempotência e observabilidade.

## Ideia Central

O projeto do módulo junta as decisões estudadas em um fluxo completo. A proposta é evoluir uma API para responder rápido ao usuário, processar trabalho pesado depois e avisar o cliente quando o resultado estiver pronto.

O objetivo não é usar a ferramenta mais sofisticada. É tomar decisões explícitas: o que acontece no request, o que vira job, o que entra na fila, que evento é publicado, como o cliente recebe atualização e como evitar duplicidade.

Esse projeto serve como ponte entre backend, arquitetura, observabilidade e os próximos módulos de cloud e infraestrutura.

## Tópicos da Aula

- Desenho do fluxo completo.
- Endpoint de solicitação.
- Registro de tarefa pendente.
- Job e fila.
- Evento de conclusão.
- Notificação em tempo real.
- Idempotência e observabilidade.

## Exemplo Prático

Marco de Sincronismo N7 - PetCare Assíncrono:

Sua API (que possui Arquitetura Limpa via Marco N6) vai virar bi-partida.
No projeto, você vai instalar um Broker (ex: RabbitMQ no Docker ou Redis para BullMQ).
1. Refatore a Rota de `POST /api/exames/lote` para apenas gravar os status preliminares, despachar o objeto pesadão pro Message Broker e retornar em 10ms.
2. Crie um arquivo `worker.ts` que escuta a fila de exames, imita um processamento pesado (Delay de 10s) e quando termina, ele joga o resultado em uma tabela limpa.
3. Implemente Server-Sent Events (SSE) ou Socket.io na UI do React para que a barra verde apareça dizendo 'O lote de Exames foi processado com sucesso' sem a Recepcionista apertar o botão Atualizar Página.

## Analogias e Intuições

- Esse Marco é como trocar a porta pesada de abrir do hospital por uma porta automática com sensor de movimento. A fricção acaba.

## Erros Comuns

- Fazer o request esperar todo o processamento.
- Não salvar estado pendente antes de publicar trabalho.
- Misturar comando e evento.
- Não proteger o consumer contra duplicidade.
- Não criar forma de investigar jobs travados ou falhos.

## Checklist de Entendimento

- O que acontece antes da resposta HTTP?
- Qual trabalho vira job?
- Qual mensagem entra na fila?
- Qual evento representa conclusão?
- Como o cliente recebe atualização?
- Como evitar duplicidade?
- Que logs, métricas e alertas são necessários?
