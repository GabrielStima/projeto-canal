# Logs com Dados Sensiveis

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- reconhecer dados pessoais e sensiveis em logs, traces e eventos;
- criar logs uteis para investigacao sem vazar payloads;
- aplicar mascaramento, allowlist e redacao de campos;
- definir retencao e acesso para logs.

## Ideia Central

Logs existem para entender sistemas, investigar falhas, auditar eventos e operar producao. Mas logs tambem podem virar uma copia perigosa dos dados mais sensiveis da aplicacao. Um `console.log(req.body)` em producao pode expor senha, token, documento, endereco, mensagem privada, dado de pagamento ou informacao de saude.

O objetivo nao e parar de logar. E logar com intencao: evento, nivel, correlacao, identificadores minimos, resultado e contexto suficiente. Em vez de registrar payload completo, prefira campos permitidos por allowlist e valores mascarados.

Logs tambem precisam de acesso controlado, retencao definida e cuidado em ferramentas externas de observabilidade.

## Topicos da Aula

- Dados sensiveis em logs de aplicacao, proxy, banco, CI/CD e observabilidade.
- Allowlist versus blocklist de campos.
- Mascaramento e redacao.
- Correlation ID, request ID e user ID com cuidado.
- Retencao, acesso e exportacao de logs.

## Exemplo Pratico

Evite:

```ts
logger.error("Payment failed", {
  body: req.body,
  headers: req.headers,
  error,
});
```

Prefira:

```ts
logger.error("Payment failed", {
  requestId: req.id,
  userId: req.user.id,
  orderId,
  paymentProvider: "example-pay",
  errorCode: error.code,
  statusCode: error.statusCode,
});
```

O segundo log ainda ajuda investigacao, mas evita registrar token de autorizacao, dados do cartao, endereco, documento ou payload completo.

## Analogias e Intuicoes

- Log e como camera de seguranca: util para investigar, perigoso se filmar areas privadas sem necessidade.
- Allowlist e mais segura que blocklist porque voce escolhe explicitamente o que pode sair.
- Correlation ID e como numero de protocolo: ajuda a achar o caso sem expor todo o conteudo.

## Erros Comuns

- Logar request e response completos.
- Registrar headers com `Authorization` ou cookies.
- Colocar dados pessoais em mensagens de erro.
- Enviar logs para ferramentas externas sem revisar campos.
- Reter logs por tempo indefinido.
- Dar acesso amplo a logs de producao.
- Mascarar no frontend, mas logar cru no backend.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Que campos nunca deveriam aparecer em logs?
- Como investigar sem payload completo?
- Por que allowlist costuma ser melhor que blocklist?
- Como retencao e acesso se aplicam a logs?
- Que logs deveriam virar eventos de auditoria?

## Pratica Recomendada

- Revise tres logs de uma API e remova dados desnecessarios.
- Crie uma funcao de redacao para campos como senha, token, documento e cartao.
- Defina uma politica simples de retencao para logs de aplicacao.
- Escreva um exemplo de log bom para falha de login e erro de pagamento.
