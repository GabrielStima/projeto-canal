# Logs com Dados Sensíveis

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer dados pessoais e sensíveis em logs, traces e eventos;
- criar logs úteis para investigação sem vazar payloads;
- aplicar mascaramento, allowlist e redação de campos;
- definir retenção e acesso para logs.

## Ideia Central

Logs existem para entender sistemas, investigar falhas, auditar eventos e operar produção. Mas logs também podem virar uma cópia perigosa dos dados mais sensíveis da aplicação. Um `console.log(req.body)` em produção pode expor senha, token, documento, endereço, mensagem privada, dado de pagamento ou informação de saúde.

O objetivo não é parar de logar. É logar com intenção: evento, nível, correlação, identificadores mínimos, resultado e contexto suficiente. Em vez de registrar payload completo, prefira campos permitidos por allowlist e valores mascarados.

Logs também precisam de acesso controlado, retenção definida e cuidado em ferramentas externas de observabilidade.

## Tópicos da Aula

- Dados sensíveis em logs de aplicação, proxy, banco, CI/CD e observabilidade.
- Allowlist versus blocklist de campos.
- Mascaramento e redação.
- Correlation ID, request ID e user ID com cuidado.
- Retenção, acesso e exportação de logs.

## Exemplo Prático

Evite:

```ts
logger.error("Payment failed", {
  body: req.body,
  headers: req.headers,
  error,
});
```

Prefira:

```ts
logger.error("Payment failed", {
  requestId: req.id,
  userId: req.user.id,
  orderId,
  paymentProvider: "example-pay",
  errorCode: error.code,
  statusCode: error.statusCode,
});
```

O segundo log ainda ajuda investigação, mas evita registrar token de autorização, dados do cartão, endereço, documento ou payload completo.

## Analogias e Intuicoes

- Log é como câmera de segurança: útil para investigar, perigoso se filmar áreas privadas sem necessidade.
- Allowlist é mais segura que blocklist porque você escolhe explicitamente o que pode sair.
- Correlation ID é como número de protocolo: ajuda a achar o caso sem expor todo o conteúdo.

## Erros Comuns

- Logar request e response completos.
- Registrar headers com `Authorization` ou cookies.
- Colocar dados pessoais em mensagens de erro.
- Enviar logs para ferramentas externas sem revisar campos.
- Reter logs por tempo indefinido.
- Dar acesso amplo a logs de produção.
- Mascarar no frontend, mas logar cru no backend.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Que campos nunca deveriam aparecer em logs?
- Como investigar sem payload completo?
- Por que allowlist costuma ser melhor que blocklist?
- Como retenção e acesso se aplicam a logs?
- Que logs deveriam virar eventos de auditoria?

## Prática Recomendada

- Revise três logs de uma API e remova dados desnecessários.
- Crie uma função de redação para campos como senha, token, documento e cartão.
- Defina uma política simples de retenção para logs de aplicação.
- Escreva um exemplo de log bom para falha de login e erro de pagamento.
