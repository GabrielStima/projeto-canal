# Políticas de Acesso a Dados

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar políticas de acesso a dados com princípio do menor privilégio;
- diferenciar autenticação, autorização e necessidade de acesso;
- desenhar regras por papel, recurso, escopo e contexto;
- evitar acesso administrativo amplo demais.

## Ideia Central

Privacidade também depende de quem pode ver dados. Não basta o usuário final estar autenticado. Pessoas do time, sistemas internos, jobs, dashboards, ferramentas de suporte, pipelines e integrações também precisam de limites.

Política de acesso a dados responde: quem pode acessar qual dado, em qual contexto, para qual finalidade e com qual nível de detalhe. Uma pessoa de suporte talvez precise ver status da conta, mas não dados de pagamento completos. Um job de analytics talvez precise de eventos agregados, mas não e-mail. Um administrador pode precisar de acesso temporário, registrado e justificado.

O princípio do menor privilégio vale para pessoas, serviços e consultas.

## Exemplo Prático

Em uma tela de suporte, em vez de retornar tudo:

```ts
type AdminUserView = {
  id: string;
  name: string;
  email: string;
  phone: string;
  cpf: string;
  paymentCardLast4: string;
  passwordHash: string;
};
```

Crie visões por necessidade:

```ts
type SupportUserView = {
  id: string;
  name: string;
  maskedEmail: string;
  accountStatus: "active" | "blocked" | "deleted";
  plan: string;
};
```

O backend deve impor essa regra. Ocultar no frontend ajuda a experiência, mas não é controle suficiente.

## Analogias e Intuicoes

- Política de acesso é como chave por cômodo: entrar no prédio não deveria abrir todas as portas.
- Menor privilégio é como ferramenta certa: você recebe o necessário para fazer a tarefa, não todo o arsenal.
- Mascaramento é como janela fosca: permite operar sem revelar detalhes desnecessários.

## Erros Comuns

- Criar papel `admin` que acessa tudo para sempre.
- Fazer controle apenas no frontend.
- Dar acesso ao banco de produção para depuração comum.
- Não separar acesso humano de acesso de serviço.
- Esquecer logs de quem consultou dados sensíveis.
- Não revisar permissões antigas depois de mudanças no time.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quem realmente precisa acessar cada tipo de dado?
- Que regra pertence ao backend, não ao frontend?
- Como diferenciar papel, recurso e contexto?
- Quando mascarar e quando bloquear totalmente?
- Como revisar acessos ao longo do tempo?

## Prática Recomendada

- Como o princípio de *Segregation of Duties* (Segregação de Funções) impediria um Desenvolvedor Sênior DBA de alterar sozinho a conta bancária para a qual a clínica paga os fornecedores?
