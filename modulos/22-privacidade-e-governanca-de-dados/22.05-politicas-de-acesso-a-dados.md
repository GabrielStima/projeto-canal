# Politicas de Acesso a Dados

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar politicas de acesso a dados com principio do menor privilegio;
- diferenciar autenticacao, autorizacao e necessidade de acesso;
- desenhar regras por papel, recurso, escopo e contexto;
- evitar acesso administrativo amplo demais.

## Onde Esta Aula Entra no Curso

- Modulo: 22. Privacidade e Governanca de Dados
- Aula anterior: 22.04 Consentimento e Finalidade de Uso
- Proxima aula: 22.06 Retencao, Descarte e Minimizacao de Dados
- Pre-requisitos: autenticacao, autorizacao, IAM, dados classificados, seguranca em APIs e auditoria em alto nivel.

## Ideia Central

Privacidade tambem depende de quem pode ver dados. Nao basta o usuario final estar autenticado. Pessoas do time, sistemas internos, jobs, dashboards, ferramentas de suporte, pipelines e integracoes tambem precisam de limites.

Politica de acesso a dados responde: quem pode acessar qual dado, em qual contexto, para qual finalidade e com qual nivel de detalhe. Uma pessoa de suporte talvez precise ver status da conta, mas nao dados de pagamento completos. Um job de analytics talvez precise de eventos agregados, mas nao e-mail. Um administrador pode precisar de acesso temporario, registrado e justificado.

O principio do menor privilegio vale para pessoas, servicos e consultas.

## Topicos da Aula

- Autenticacao, autorizacao e necessidade de acesso.
- RBAC, ABAC e autorizacao por recurso em alto nivel.
- Mascaramento e minimizacao em telas administrativas.
- Acesso de suporte, engenharia, analytics e automacoes.
- Revisao periodica de permissoes.

## Exemplo Pratico

Em uma tela de suporte, em vez de retornar tudo:

```ts
type AdminUserView = {
  id: string;
  name: string;
  email: string;
  phone: string;
  cpf: string;
  paymentCardLast4: string;
  passwordHash: string;
};
```

Crie visoes por necessidade:

```ts
type SupportUserView = {
  id: string;
  name: string;
  maskedEmail: string;
  accountStatus: "active" | "blocked" | "deleted";
  plan: string;
};
```

O backend deve impor essa regra. Ocultar no frontend ajuda a experiencia, mas nao e controle suficiente.

## Analogias e Intuicoes

- Politica de acesso e como chave por comodo: entrar no predio nao deveria abrir todas as portas.
- Menor privilegio e como ferramenta certa: voce recebe o necessario para fazer a tarefa, nao todo o arsenal.
- Mascaramento e como janela fosca: permite operar sem revelar detalhes desnecessarios.

## Erros Comuns

- Criar papel `admin` que acessa tudo para sempre.
- Fazer controle apenas no frontend.
- Dar acesso ao banco de producao para depuracao comum.
- Nao separar acesso humano de acesso de servico.
- Esquecer logs de quem consultou dados sensiveis.
- Nao revisar permissoes antigas depois de mudancas no time.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Quem realmente precisa acessar cada tipo de dado?
- Que regra pertence ao backend, nao ao frontend?
- Como diferenciar papel, recurso e contexto?
- Quando mascarar e quando bloquear totalmente?
- Como revisar acessos ao longo do tempo?

## Pratica Recomendada

- Escolha tres papeis de uma aplicacao e defina quais campos cada um pode ver.
- Crie uma resposta de API diferente para usuario comum e suporte.
- Liste quais acessos deveriam gerar evento de auditoria.
- Revise um endpoint administrativo e remova campos desnecessarios.
