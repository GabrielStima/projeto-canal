# Dados Enviados Para IA

## Objetivo

Ao final desta aula, você deve ser capaz de:

- reconhecer riscos de enviar dados pessoais para ferramentas de IA;
- aplicar minimização, redação e anonimização antes de prompts;
- diferenciar uso local, uso interno, API externa e ferramenta pública;
- criar uma política simples para uso de IA com dados de aplicação.

## Onde Esta Aula Entra na Formação

- Módulo: 22. Privacidade e Governança de Dados
- Aula anterior: 22.09 Auditoria e Rastreabilidade de Acesso a Dados
- Próxima aula: 22.11 Projeto Prático: Revisão de Privacidade de uma Aplicação Fullstack

## Ideia Central

Ferramentas de IA podem ajudar a resumir tickets, analisar logs, gerar respostas de suporte, revisar código e extrair insights. Mas prompts também podem carregar dados pessoais, segredos, tokens, logs, contratos, mensagens privadas e informações internas.

O risco não está apenas no modelo. Está no caminho inteiro: quem copia o dado, qual ferramenta recebe, qual política de retenção existe, se dados podem ser usados para treinamento, quem tem acesso ao histórico, se existe contrato, se há transferência internacional, se o prompt aparece em logs e se a resposta pode vazar algo para outra pessoa.

Para devs, a regra prática é minimizar antes de enviar: remova, mascare, agregue ou sintetize. Envie o problema técnico, não o dado real, sempre que possível.

## Topicos da Aula

- Dados pessoais, segredos e informações internas em prompts.
- Ferramenta pública, ferramenta corporativa, API externa e modelo local.
- Redação, mascaramento, sintetização e uso de dados fictícios.
- Política de uso de IA para suporte, logs, código e analytics.
- Auditoria e revisão de prompts sensíveis.

## Exemplo Prático

Prompt ruim:

```text
Analise este erro do usuário ana@example.com, CPF 123..., token Bearer eyJ..., pedido 987, endereço completo...
```

Prompt melhor:

```text
Analise este erro de checkout. Dados reais foram removidos.

Contexto:
- usuário autenticado
- pedido com pagamento recusado pelo provedor
- endpoint POST /orders/:id/pay
- status recebido do provedor: card_declined
- requestId: req_123

Quais hipóteses técnicas devo investigar?
```

A segunda versão preserva o problema técnico e reduz exposição.

## Analogias e Intuicoes

- Prompt é como e-mail para terceiro: pense no que você colocaria antes de enviar.
- Redação é como tarja em documento: remove o que não é necessário para a análise.
- Dados fictícios são bons para raciocinar sobre estrutura sem expor pessoas reais.

## Erros Comuns

- Colar logs completos com tokens, cookies e payloads.
- Enviar dados reais para depurar problema que poderia ser explicado com exemplo sintético.
- Usar ferramenta pessoal para dados da empresa ou de usuários.
- Esquecer histórico de conversa, plugins e integrações da ferramenta.
- Mandar segredos em prompts.
- Não revisar termos, retenção e configurações da ferramenta aprovada.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Que dados nunca devem entrar em prompt?
- Quando dados fictícios resolvem o mesmo problema?
- Como redigir logs antes de pedir ajuda a uma IA?
- Qual a diferença entre ferramenta pública e ambiente corporativo controlado?
- Que política mínima um time deve ter para uso de IA?

## Prompt Para Estudar Com IA

```text
Estou estudando Vazamentos em Modelos de IA (LLMs) em uma trilha de desenvolvimento fullstack JavaScript/TypeScript.

Meu objetivo é entender o conceito, praticar e revisar sem receber respostas prontas cedo demais.

Contexto da aula:
- Descrição da aula: A aula aborda riscos de vazamento de dados e propriedade intelectual ao usar modelos de IA de terceiros.
- Descrição do modulo: O módulo cobre privacidade, governança de dados, LGPD/GDPR para desenvolvedores, classificação de dados, retenção, anonimização, pseudonimização e logs sensíveis.
- Itens já estudados: DevSecOps, Modelagem de Banco de Dados, Logging (Observabilidade), Compliance de IAM.

Crie uma sessão de estudo com:
1. uma explicação curta do conceito;
2. 5 perguntas progressivas;
3. 3 exercícios práticos;
4. 1 desafio um pouco mais difícil;
5. exemplos de aplicação em projetos reais;
6. um checklist para eu avaliar se entendi.

Regras:
- Não entregue respostas completas antes de eu tentar responder.
- Corrija minhas respostas com explicações.
- Se eu errar, dê uma dica antes de revelar a solução.
- Ao final, sugira o próximo assunto que devo estudar.
```
