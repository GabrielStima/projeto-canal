# Dados Enviados Para IA

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- reconhecer riscos de enviar dados pessoais para ferramentas de IA;
- aplicar minimizacao, redacao e anonimização antes de prompts;
- diferenciar uso local, uso interno, API externa e ferramenta publica;
- criar uma politica simples para uso de IA com dados de aplicacao.

## Onde Esta Aula Entra no Curso

- Modulo: 22. Privacidade e Governanca de Dados
- Aula anterior: 22.09 Auditoria e Rastreabilidade de Acesso a Dados
- Proxima aula: 22.11 Projeto Pratico: Revisao de Privacidade de uma Aplicacao Fullstack
- Pre-requisitos: dados pessoais, consentimento, acesso, logs sensiveis, auditoria, secrets e Secure SDLC.

## Ideia Central

Ferramentas de IA podem ajudar a resumir tickets, analisar logs, gerar respostas de suporte, revisar codigo e extrair insights. Mas prompts tambem podem carregar dados pessoais, segredos, tokens, logs, contratos, mensagens privadas e informacoes internas.

O risco nao esta apenas no modelo. Esta no caminho inteiro: quem copia o dado, qual ferramenta recebe, qual politica de retencao existe, se dados podem ser usados para treinamento, quem tem acesso ao historico, se existe contrato, se ha transferencia internacional, se o prompt aparece em logs e se a resposta pode vazar algo para outra pessoa.

Para devs, a regra pratica e minimizar antes de enviar: remova, mascare, agregue ou sintetize. Envie o problema tecnico, nao o dado real, sempre que possivel.

## Topicos da Aula

- Dados pessoais, segredos e informacoes internas em prompts.
- Ferramenta publica, ferramenta corporativa, API externa e modelo local.
- Redacao, mascaramento, sintetizacao e uso de dados ficticios.
- Politica de uso de IA para suporte, logs, codigo e analytics.
- Auditoria e revisao de prompts sensiveis.

## Exemplo Pratico

Prompt ruim:

```text
Analise este erro do usuario ana@example.com, CPF 123..., token Bearer eyJ..., pedido 987, endereco completo...
```

Prompt melhor:

```text
Analise este erro de checkout. Dados reais foram removidos.

Contexto:
- usuario autenticado
- pedido com pagamento recusado pelo provedor
- endpoint POST /orders/:id/pay
- status recebido do provedor: card_declined
- requestId: req_123

Quais hipoteses tecnicas devo investigar?
```

A segunda versao preserva o problema tecnico e reduz exposicao.

## Analogias e Intuicoes

- Prompt e como e-mail para terceiro: pense no que voce colocaria antes de enviar.
- Redacao e como tarja em documento: remove o que nao e necessario para a analise.
- Dados ficticios sao bons para raciocinar sobre estrutura sem expor pessoas reais.

## Erros Comuns

- Colar logs completos com tokens, cookies e payloads.
- Enviar dados reais para depurar problema que poderia ser explicado com exemplo sintetico.
- Usar ferramenta pessoal para dados da empresa ou de usuarios.
- Esquecer historico de conversa, plugins e integrações da ferramenta.
- Mandar segredos em prompts.
- Nao revisar termos, retencao e configuracoes da ferramenta aprovada.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Que dados nunca devem entrar em prompt?
- Quando dados ficticios resolvem o mesmo problema?
- Como redigir logs antes de pedir ajuda a uma IA?
- Qual a diferenca entre ferramenta publica e ambiente corporativo controlado?
- Que politica minima um time deve ter para uso de IA?

## Pratica Recomendada

- Pegue um log ficticio e redija campos sensiveis antes de criar um prompt.
- Transforme um caso de suporte com dados reais em um exemplo sintetico.
- Crie uma checklist de "antes de enviar para IA".
- Defina tres tipos de dados proibidos em prompts do time.

## Prompt Para Estudar Com IA

```text
Estou estudando dados enviados para IA em uma trilha de desenvolvimento fullstack JavaScript/TypeScript.

Meu objetivo e aprender a usar IA com responsabilidade sem enviar dados pessoais, segredos, tokens ou informacoes desnecessarias.

Contexto da aula:
- Descricao da aula: A aula explica riscos de prompts com dados reais, minimizacao, redacao, dados ficticios, ferramentas externas, retencao e politica de uso.
- Descricao do modulo: O modulo ensina privacidade, governanca, LGPD/GDPR, acesso, retencao, anonimização, logs, auditoria e IA.
- Itens ja estudados: dados pessoais, consentimento, politicas de acesso, logs sensiveis, auditoria, secrets e Secure SDLC.

Crie uma sessao de estudo com:
1. uma explicacao curta do conceito;
2. 5 perguntas progressivas;
3. 3 exercicios praticos;
4. 1 desafio um pouco mais dificil;
5. exemplos de aplicacao em projetos reais;
6. um checklist para eu avaliar se entendi.

Regras:
- Nao entregue respostas completas antes de eu tentar responder.
- Corrija minhas respostas com explicacoes.
- Se eu errar, de uma dica antes de revelar a solucao.
- Ao final, sugira o proximo assunto que devo estudar.
```
