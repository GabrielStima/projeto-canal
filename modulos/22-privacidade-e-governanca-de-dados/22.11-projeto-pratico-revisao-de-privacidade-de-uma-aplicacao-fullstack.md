# Projeto Pratico: Revisao de Privacidade de uma Aplicacao Fullstack

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- revisar uma aplicacao fullstack do ponto de vista de privacidade;
- criar inventario e classificacao de dados;
- propor melhorias em acesso, logs, retencao, auditoria e IA;
- documentar riscos e decisoes de governanca.

## Onde Esta Aula Entra no Curso

- Modulo: 22. Privacidade e Governanca de Dados
- Aula anterior: 22.10 Dados Enviados Para IA
- Proxima aula: 23. System Design
- Pre-requisitos: todas as aulas anteriores do modulo 22, seguranca em APIs, observabilidade, cloud, SRE e Secure SDLC.

## Ideia Central

O projeto final consolida o modulo com uma revisao pratica. A ideia nao e criar uma aplicacao nova, mas pegar uma aplicacao ja conhecida e olhar para ela como um sistema que trata dados de pessoas.

Uma boa revisao de privacidade nao se limita ao banco. Ela passa por frontend, API, banco, storage, logs, filas, caches, analytics, suporte, CI/CD, backups, prompts de IA e acessos administrativos.

O resultado esperado e um documento tecnico simples: dados tratados, riscos encontrados, melhorias propostas, decisoes aceitas e proximos passos.

## Topicos da Aula

- Escolha da aplicacao base.
- Inventario e classificacao dos dados.
- Revisao de coleta, finalidade e minimizacao.
- Revisao de politicas de acesso e telas administrativas.
- Revisao de logs, auditoria, retencao, descarte e IA.
- Plano de melhoria por prioridade.

## Exemplo Pratico

Revise uma plataforma de estudos:

```text
Fluxos:
- cadastro e login
- progresso em aulas
- comentarios
- pagamentos
- suporte
- dashboard administrativo
- logs e monitoramento
- uso de IA pelo time
```

Para cada fluxo, responda:

- que dados entram?
- onde ficam?
- quem acessa?
- por que sao necessarios?
- aparecem em logs?
- sao enviados para terceiros?
- por quanto tempo ficam?
- existe auditoria para acesso sensivel?
- da para reduzir, mascarar, anonimizar ou descartar?

## Analogias e Intuicoes

- Revisao de privacidade e como threat modeling focado em dados.
- Inventario e como raio-x: mostra onde o dado esta antes de tratar o problema.
- Plano de melhoria deve priorizar risco real, nao perfeicao teorica.

## Erros Comuns

- Revisar apenas formulario de cadastro.
- Ignorar ferramentas de suporte, analytics, logs e IA.
- Criar uma lista enorme sem prioridade.
- Propor "criptografar tudo" como resposta generica.
- Nao diferenciar risco alto, medio e baixo.
- Nao definir dono ou proximo passo para cada melhoria.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Como criar um inventario de dados de uma aplicacao real?
- Como priorizar riscos de privacidade?
- Que melhorias sao rapidas e que melhorias exigem mudanca de arquitetura?
- Como documentar decisoes e riscos aceitos?
- Como este modulo prepara System Design?

## Pratica Recomendada

- Escolha uma aplicacao do curso e crie o inventario de dados.
- Classifique dados em publico, interno, pessoal, sensivel e segredo.
- Revise um endpoint, uma tela administrativa, uma tabela e um log.
- Defina politica de retencao para tres categorias de dados.
- Crie uma lista de melhorias em prioridade alta, media e baixa.
