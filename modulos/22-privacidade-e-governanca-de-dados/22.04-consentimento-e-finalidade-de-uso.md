# Consentimento e Finalidade de Uso

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar consentimento em nível prático para sistemas;
- diferenciar consentimento, finalidade e preferência de produto;
- evitar consentimentos agrupados, obscuros ou desnecessários;
- modelar escolhas do usuário de forma rastreável.

## Onde Esta Aula Entra na Formação

- Módulo: 22. Privacidade e Governança de Dados

- Próxima aula: 22.05 Políticas de Acesso a Dados

## Ideia Central

Consentimento é uma autorização dada pela pessoa para uma finalidade específica. Ele precisa ser claro, informado e livre. Em software, isso vira decisão de interface, texto, banco, API, histórico e revogação.

Mas consentimento não deve ser usado como resposta automática para tudo. Antes de pedir consentimento, o time precisa entender a finalidade do tratamento e a base adequada ao contexto. Também precisa separar o que é necessário para entregar o serviço do que é opcional, como marketing, personalização ou compartilhamento com terceiros.

Finalidade é a resposta para "por que este dado existe?". Se a finalidade muda, o tratamento precisa ser reavaliado.

## Tópicos da Aula

- Consentimento como escolha específica e informada.
- Finalidade, adequação e necessidade.
- Consentimento separado de termos de uso e contrato principal.
- Revogação e histórico de consentimento.
- Preferências de comunicação, cookies, analytics e marketing.

## Exemplo Prático

Em um cadastro, evite um único checkbox como:

```text
[ ] Aceito os termos, a política de privacidade, receber marketing e compartilhar meus dados com parceiros.
```

Uma modelagem melhor separa finalidades:

```ts
type UserConsent = {
  userId: string;
  purpose: "transactional_email" | "marketing_email" | "analytics_personalization";
  granted: boolean;
  grantedAt: string;
  revokedAt?: string;
  source: "signup" | "settings" | "support";
  noticeVersion: string;
};
```

O sistema precisa saber não apenas o estado atual, mas quando, onde e sob qual texto a escolha foi feita.

## Analogias e Intuicoes

- Consentimento é como uma permissão específica, não um cheque em branco.
- Finalidade é como destino em uma viagem: se o destino muda, a autorização anterior pode não servir.
- Preferências são interruptores: a pessoa precisa conseguir mudar depois.

## Erros Comuns

- Misturar consentimento com aceite de termos.
- Fazer checkbox pré-marcado.
- Exigir dado opcional como condição para usar serviço essencial.
- Não guardar versão do texto apresentado.
- Não oferecer caminho claro para revogação quando consentimento for a base.
- Usar uma finalidade genérica como "melhorar a experiência" para tudo.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Consentimento é sempre necessário?
- O que torna um consentimento específico e informado?
- Como finalidade afeta coleta e uso de dados?
- Como armazenar histórico de consentimento?
- Que escolhas do usuário precisam ser reversíveis?

## Prática Recomendada

- Revise um formulário e separe campos necessários de opcionais.
- Escreva finalidades claras para três campos coletados.
- Modele uma tabela simples de consentimentos ou preferências.
- Desenhe um endpoint para revogar uma preferência de marketing.
