# LGPD/GDPR Básico Para Devs

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar LGPD/GDPR em linguagem prática para desenvolvimento;
- reconhecer princípios que afetam produto, banco, API, logs e operação;
- diferenciar consentimento de outras bases legais em alto nível;
- saber quando chamar pessoas de juridico, segurança ou privacidade.

## Onde Esta Aula Entra na Formação

- Módulo: 22. Privacidade e Governança de Dados
- Aula anterior: 22.01 Dados Pessoais, Dados Sensíveis e Ciclo de Vida do Dado
- Próxima aula: 22.03 Inventário, Classificação e Governança de Dados

## Ideia Central

LGPD e GDPR são marcos de proteção de dados pessoais. Para devs, o objetivo não é decorar artigos nem atuar como advogado. O objetivo é entender os princípios que viram decisões técnicas: finalidade clara, coleta necessária, transparência, segurança, prevenção, responsabilização, direitos dos titulares e cuidado especial com dados sensíveis.

Um erro comum é achar que consentimento resolve tudo. Na prática, tratamento de dados precisa de uma hipótese legal adequada ao contexto. Consentimento é uma delas, mas não a única. Em muitos sistemas, existem dados tratados para executar um contrato, cumprir obrigação legal, exercer direito, proteger vida, atender legítimo interesse ou outra base aplicável.

Para engenharia, a pergunta mais útil é: "consigo explicar por que este dado existe, onde ele é usado, quem acessa, como o titular é informado e quando ele deixa de ser necessário?"

## Tópicos da Aula

- O que LGPD/GDPR tentam proteger.
- Princípios práticos para desenvolvimento.
- Tratamento de dados como coleta, uso, armazenamento, compartilhamento e eliminação.
- Bases legais em alto nível.
- Direitos do titular como requisitos de sistema.
- Limites da responsabilidade do dev.

## Exemplo Prático

Uma API de cursos coleta `name`, `email`, `password`, `birthDate` e `marketingOptIn`.

Perguntas que um dev deve levantar:

- `birthDate` é necessário para entregar o curso ou é apenas curiosidade de produto?
- `marketingOptIn` está separado do aceite de termos?
- a resposta `GET /me` retorna apenas o necessário?
- existe caminho para exportar, corrigir ou excluir dados quando aplicável?
- logs de erro registram dados pessoais desnecessários?
- backups têm retenção compatível com as regras do produto?

Essas perguntas não resolvem toda a conformidade, mas evitam que a arquitetura nasça contra os princípios.

## Analogias e Intuicoes

- LGPD/GDPR para devs é como acessibilidade: não é apenas uma regra externa, muda como você desenha a experiência.
- Base legal é como permissão de processamento: você precisa saber por que pode tratar aquele dado.
- Direitos do titular são requisitos de produto que precisam virar fluxo, tela, API, suporte ou processo.

## Erros Comuns

- Tratar LGPD/GDPR como checklist de banner de cookies.
- Usar consentimento para tudo sem avaliar se ele é adequado.
- Coletar dado "porque pode ser útil no futuro".
- Ignorar dados em logs, backups, planilhas e ferramentas de terceiros.
- Prometer exclusão imediata sem considerar obrigações legais, antifraude, backup e auditoria.
- Dar resposta jurídica definitiva sem envolver especialistas.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- O que significa tratar dados pessoais?
- Por que finalidade e necessidade afetam modelagem de dados?
- Consentimento é sempre obrigatório?
- Que direitos do titular podem impactar sistemas?
- Quando o dev deve escalar a decisão para jurídico, segurança ou privacidade?

## Prática Recomendada

- Escolha um formulário e escreva a finalidade de cada campo.
- Revise uma resposta de API e remova campos que não precisam ser expostos.
- Liste quais fluxos do produto precisariam existir para atender pedidos de acesso, correção ou exclusão.
- Escreva cinco perguntas que você faria ao time de produto antes de coletar um novo dado.
