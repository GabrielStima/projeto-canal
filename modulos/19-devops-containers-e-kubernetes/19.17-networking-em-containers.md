## Ideia Central

Containers têm rede própria. Para uma aplicação responder fora do container, portas precisam ser publicadas. Para containers conversarem entre si, eles precisam compartilhar uma rede e usar nomes ou endereços adequados. Entender isso evita erros clássicos com `localhost`.

## Tópicos da Aula

- Rede do host vs rede do container
- Publicação de portas
- DNS interno entre containers
- `localhost` dentro do container
- Redes bridge
- Comunicação com serviços externos

## Exemplo Prático

A Rede Bridge do Hospital:

Você tem dois containers: O Back-End (Node) e o Banco (Redis).
O Back-end do hospital tenta se conectar à `localhost:6379`. Vai dar erro de conexão recusada.
Por quê? O Localhost do Container A aponta pra ele mesmo. Eles estão isolados.
Solução de Rede: O Docker os joga numa 'Rede Bridge' comum. E você configura o Node para conectar na URL `redis-container:6379`. O Docker usa um DNS interno mágico que resolve o nome pra um IP flutuante.

## Analogias e Intuições

- O Backend e o Banco estão em prédios separados num condomínio gigante. Ligar pro próprio quarto (localhost) dá ocupado. Eles precisam falar com a portaria virtual do condomínio (Rede Docker) e dizer o nome do prédio do Banco para conectar os cabos de rede invisíveis.

## Erros Comuns

- Usar `localhost` para acessar outro container.
- Publicar portas desnecessárias.
- Misturar rede interna e exposição pública.
