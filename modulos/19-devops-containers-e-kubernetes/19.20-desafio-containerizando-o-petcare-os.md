# Desafio: Containerizando o PetCare OS

## Objetivo

Ao final deste desafio, você deve ser capaz de:

- containerizar uma aplicação backend real (Node.js + banco de dados);
- justificar quando Docker é suficiente e quando Kubernetes seria overengineering;
- identificar decisões de persistência, networking e segurança em containers;
- aplicar boas práticas de build de imagem (camadas, .dockerignore, imagem mínima);
- reconhecer a diferença entre "funciona no container" e "funciona em produção".

## Onde Esta Aula Entra na Formação

- Módulo: 19. DevOps, Containers e Kubernetes
- Aula anterior: Segurança em Containers
- Próxima aula: Kubernetes
- Pré-requisitos: Docker, build de imagens, Docker CLI, persistência, networking, registries e segurança em containers (aulas 19.10–19.19).

## Ideia Central

Este é um desafio de fixação. Até aqui você estudou containers de forma fragmentada: build, CLI, execução, persistência, networking, registries e segurança. Agora o objetivo é aplicar tudo junto sobre uma aplicação real do PetCare OS.

Containerizar não é apenas criar um `Dockerfile`. É decidir o que entra na imagem, o que fica fora, como o banco persiste dados, como os serviços se comunicam, quais credenciais vazam e quando a solução é proporcional ao problema.

> **A pergunta que este desafio faz:** *se você entregasse o PetCare OS rodando em containers hoje, qual seria o menor conjunto de decisões que garantiria que funcione em outra máquina?*

## O Cenário

O PetCare OS tem um backend Node.js com as seguintes características:

- API REST com Express/Fastify;
- Conexão com PostgreSQL para dados persistentes;
- Variáveis de ambiente para banco, porta e secrets;
- Logs estruturados em stdout;
- Healthcheck no endpoint `GET /health`.

O código já funciona localmente com `npm run dev`. Sua tarefa é transformar esse setup local em uma stack containerizada funcional.

## Desafio

### Etapa 1 — Diagnóstico (não escreva código ainda)

Leia o cenário acima e responda por escrito:

1. Quantos serviços distintos existem nesta arquitetura? (API, banco, mais algum?)
2. O banco de dados deve ficar dentro do container da API ou em um container separado? Por quê?
3. Quais dados precisam persistir além do ciclo de vida do container?
4. Quais variáveis de ambiente são sensíveis e não podem ficar no `Dockerfile`?
5. Se você parar e remover todos os containers, perde todos os dados? Isso é aceitável para um ambiente de desenvolvimento?

### Etapa 2 — Containerização

Crie os artefatos necessários para rodar o PetCare OS em containers:

**Dockerfile da API:**
- Base em imagem Node.js oficial (prefira `node:20-alpine` a `node:latest`);
- Instalação de dependências em camada separada do código-fonte;
- Uso de `.dockerignore` para evitar copiar `node_modules`, `.env`, logs e arquivos desnecessários;
- Execução como usuário não-root;
- Exposção da porta correta;
- Comando de startup que não dependa de `npm run dev`.

**Docker Compose (opcional, mas recomendado):**
- Serviço `api` com build do Dockerfile;
- Serviço `db` com imagem PostgreSQL oficial;
- Volume nomeado para persistir dados do banco;
- Rede interna para comunicação entre serviços;
- Variáveis de ambiente via `.env` ou `environment`;
- Healthcheck básico para o banco.

**Scripts de inicialização:**
- Como o banco é criado/populado na primeira vez?
- Como garantir que a API só inicie depois que o banco aceite conexões?

### Etapa 3 — Decisão de Escala

Após containerizar, responda:

- **Quando Docker + Compose é suficiente?** Liste 3 cenários reais onde essa stack atende sem Kubernetes.
- **Quando Kubernetes se torna necessário?** Liste 3 sinais que indicam que orquestração deixou de ser overengineering.
- **Qual é o custo operacional de adicionar Kubernetes agora?** Considere: complexidade de configuração, necessidade de cluster, custo de infraestrutura, curva de aprendizado do time e observabilidade adicional.

### Etapa 4 — Registro de decisões

Escreva um parágrafo curto respondendo:

- Qual foi a decisão mais difícil de tomar e por quê?
- Alguma prática de segurança você deixou de lado por simplicidade? Qual e qual o risco?
- Se o PetCare OS precisar escalar para 3 instâncias da API daqui a 3 meses, o que você mudaria hoje para facilitar essa transição?

## Critérios de Avaliação

- O `Dockerfile` usa camadas de cache de forma eficiente?
- A imagem final é menor que 200MB? (dica: multi-stage build ajuda)
- Dados do banco persistem entre reinícios do container?
- Credenciais não estão hardcoded no `Dockerfile`?
- A justificativa "Docker vs. Kubernetes" usa argumentos de escala, custo e complexidade — não apenas preferência pessoal?
