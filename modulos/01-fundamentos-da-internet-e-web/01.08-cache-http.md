# Cache HTTP

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que cache HTTP existe;
- entender como cache pode melhorar performance;
- reconhecer riscos de cache mal configurado;
- diferenciar a ideia de cache de uma nova request completa.

## Ideia Central

Cache é uma forma de reutilizar respostas ou partes de respostas para evitar trabalho repetido.

Na Web, muitos recursos não mudam o tempo todo: imagens, arquivos CSS, arquivos JavaScript, fontes e algumas respostas de API. Se o cliente ou algum intermediário puder reaproveitar uma resposta recente, a aplicação pode ficar mais rápida e consumir menos recursos.

Mas cache também pode causar confusão. Uma resposta antiga pode continuar aparecendo quando você esperava ver uma versão nova.

## Exemplo Prático

Imagine que uma página usa um arquivo `styles.css`.

Na primeira visita, o browser baixa esse arquivo. Se o servidor informar que o arquivo pode ser reaproveitado por algum tempo, o browser pode evitar baixar tudo de novo em visitas futuras.

Isso melhora performance. Mas, se o arquivo mudou e o cache ainda acha que a versão antiga é válida, o usuário pode ver estilos desatualizados.

## Analogias e Intuições

- Cache é como guardar uma resposta recente para não repetir a pergunta.
- Cache melhora velocidade quando a resposta ainda é válida.
- Invalidação de cache é decidir quando uma resposta guardada deixou de ser confiável.

## Erros Comuns

- Achar que cache sempre é bom.
- Esquecer que cache pode entregar conteúdo antigo.
- Confundir cache HTTP com banco de dados.
- Ignorar cache ao investigar bugs de frontend.
- Não perceber que browser, servidor e intermediários podem participar do cache.
