# Projeto Prático: Deploy em Kubernetes

## Objetivo

Ao final desta aula, você deve ser capaz de:

- escrever manifestos Kubernetes para executar uma aplicação em um cluster;
- configurar ConfigMaps, Secrets, Services e probes de forma coesa;
- definir requests, limits e políticas de segurança para os pods;
- executar um deploy, verificar o estado da aplicação e simular um rollback.

## Onde Esta Aula Entra na Formação

- Módulo: 19B. Kubernetes
- Aula anterior: Tópicos Avançados de Kubernetes
- Próxima aula: SRE, Operação e Incidentes
- Pré-requisitos: CI/CD, Docker, imagens, containers, registries, networking, persistência, segurança em containers e fundamentos de Kubernetes.

## Ideia Central

O projeto final consolida o módulo: uma imagem publicada no registry é implantada em um cluster Kubernetes com manifestos declarativos. O foco é garantir que a aplicação fique disponível, configurável e observável, sem montar uma plataforma empresarial completa.

## Tópicos da Aula

- Escopo do projeto
- Cluster de laboratório
- Manifestos: Namespace, Deployment, Service, ConfigMap e Secret
- Probes de readiness e liveness
- Requests, limits e políticas de segurança
- Aplicação e verificação
- Rollback conceitual

## Exemplo Prático

Deploy do PetCare OS em Kubernetes:

1. O cluster de laboratório já está configurado com `kubectl` funcional.
2. Você cria um namespace `petcare-staging` para isolar os recursos.
3. O Deployment usa a imagem publicada no registry com a tag do commit.
4. ConfigMap define variáveis não sensíveis; Secret armazena credenciais do banco.
5. O Service do tipo ClusterIP expõe a API dentro do cluster.
6. Probes garantem que o tráfego só chegue quando a aplicação estiver pronta.
7. Você aplica os manifestos, verifica os pods e testa o endpoint.
8. Se algo falhar, você reverte para a revisão anterior do Deployment.

## Analogias e Intuições

- O Deployment é a planta de uma fábrica: você diz quantas máquinas quer e como elas devem ser, e o Kubernetes mantém essa planta funcionando.
- O Service é o telefone da fábrica: independente de qual máquina atenda, o número continua o mesmo.

## Erros Comuns

- Usar a tag `latest` nas imagens do Deployment.
- Ignorar probes e depois descobrir que a aplicação reinicia sem motivo aparente.
- Não definir requests e limits, causando instabilidade no cluster.
- Expor Services desnecessariamente para fora do cluster.
