# README Profissional

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o papel de um README em um projeto profissional;
- escrever um README útil para setup, uso e manutenção;
- documentar comandos, variáveis, dependências e limites sem expor segredos;
- reconhecer quando o README deve apontar para documentos mais específicos.

## Onde esta aula entra no curso

- Módulo: 25. Documentação Técnica
- Aula anterior: [Tipos de Documentação Técnica](25.02-tipos-de-documentacao-tecnica.md)
- Próxima aula: [Documentação para Times](25.04-documentacao-para-times.md)
- Pré-requisitos: documentação mínima de projeto, terminal, package managers, variáveis de ambiente, GitHub e comunicação técnica.

## Ideia Central

README é a porta de entrada do projeto. Ele deve ajudar uma pessoa a entender o que o projeto faz, como rodar, como testar, como configurar o ambiente e onde encontrar informações importantes.

Um README profissional não precisa ser enorme. Ele precisa ser confiável. Comandos precisam existir, variáveis precisam estar documentadas, requisitos precisam ser claros e links precisam apontar para documentos reais.

Quando o projeto cresce, o README deixa de carregar todos os detalhes e passa a orientar navegação: arquitetura em outro documento, decisões em ADRs, propostas em RFCs, operação em runbooks e uso externo em guias específicos.

## Tópicos da Aula

- Objetivo e escopo do projeto.
- Pré-requisitos, instalação e execução local.
- Variáveis de ambiente e `.env.example`.
- Scripts principais: dev, build, test, lint e migrations.
- Links para arquitetura, operação, API e decisões.
- Sinais de README desatualizado.

## Exemplo Prático

Estrutura mínima para uma API TypeScript:

~~~markdown
# API de Cursos

API para gerenciar cursos, aulas e matrículas.

## Requisitos

- Node.js 22
- pnpm
- PostgreSQL

## Como rodar

```bash
pnpm install
cp .env.example .env
pnpm db:migrate
pnpm dev
```

## Scripts

- `pnpm dev`: inicia o servidor local.
- `pnpm test`: roda testes automatizados.
- `pnpm lint`: verifica padrão de código.
~~~

O exemplo é curto, mas responde perguntas essenciais. Em um projeto real, cada comando precisa ser testado.

## Analogias e Intuições

- README é a recepção do projeto.
- Um bom README reduz tempo até a primeira contribuição.
- README não deve tentar ser toda a documentação do sistema.

## Erros Comuns

- Ter comandos que não funcionam mais.
- Omitir versão de runtime ou dependências externas.
- Colocar valores reais de secrets em exemplos.
- Escrever descrição genérica que não explica o projeto.
- Esconder links importantes em documentos soltos.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Outra pessoa consegue rodar o projeto seguindo o README?
- Os comandos foram testados?
- As variáveis obrigatórias estão documentadas?
- O README aponta para documentos mais profundos?
- O que precisa ser atualizado quando o projeto muda?

## Prática Recomendada

- Pegue um projeto pequeno.
- Reescreva o README com objetivo, requisitos, setup, scripts e configuração.
- Teste os comandos em uma instalação limpa ou ambiente novo.
- Adicione links para documentos futuros de arquitetura e decisões.
