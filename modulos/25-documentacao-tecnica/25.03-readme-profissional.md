# README Profissional

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar o papel de um README em um projeto profissional;
- escrever um README util para setup, uso e manutencao;
- documentar comandos, variaveis, dependencias e limites sem expor segredos;
- reconhecer quando o README deve apontar para documentos mais especificos.

## Onde Esta Aula Entra no Curso

- Modulo: 25. Documentacao Tecnica
- Aula anterior: [Tipos de Documentacao Tecnica](25.02-tipos-de-documentacao-tecnica.md)
- Proxima aula: [Documentacao Para Times](25.04-documentacao-para-times.md)

## Ideia Central

README e a porta de entrada do projeto. Ele deve ajudar uma pessoa a entender o que o projeto faz, como rodar, como testar, como configurar o ambiente e onde encontrar informacoes importantes.

Um README profissional nao precisa ser enorme. Ele precisa ser confiavel. Comandos precisam existir, variaveis precisam estar documentadas, requisitos precisam ser claros e links precisam apontar para documentos reais.

Quando o projeto cresce, o README deixa de carregar todos os detalhes e passa a orientar navegacao: arquitetura em outro documento, decisoes em ADRs, propostas em RFCs, operacao em runbooks e uso externo em guias especificos.

## Topicos da Aula

- Objetivo e escopo do projeto.
- Pre-requisitos, instalacao e execucao local.
- Variaveis de ambiente e `.env.example`.
- Scripts principais: dev, build, test, lint e migrations.
- Links para arquitetura, operacao, API e decisoes.
- Sinais de README desatualizado.

## Exemplo Pratico

Estrutura minima para uma API TypeScript:

~~~markdown
# API de Cursos

API para gerenciar cursos, aulas e matriculas.

## Requisitos

- Node.js 22
- pnpm
- PostgreSQL

## Como rodar

```bash
pnpm install
cp .env.example .env
pnpm db:migrate
pnpm dev
```

## Scripts

- `pnpm dev`: inicia o servidor local.
- `pnpm test`: roda testes automatizados.
- `pnpm lint`: verifica padrao de codigo.
~~~

O exemplo e curto, mas responde perguntas essenciais. Em um projeto real, cada comando precisa ser testado.

## Analogias e Intuicoes

- README e a recepcao do projeto.
- Um bom README reduz tempo ate a primeira contribuicao.
- README nao deve tentar ser toda a documentacao do sistema.

## Erros Comuns

- Ter comandos que nao funcionam mais.
- Omitir versao de runtime ou dependencias externas.
- Colocar valores reais de secrets em exemplos.
- Escrever descricao generica que nao explica o projeto.
- Esconder links importantes em documentos soltos.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Outra pessoa consegue rodar o projeto seguindo o README?
- Os comandos foram testados?
- As variaveis obrigatorias estao documentadas?
- O README aponta para documentos mais profundos?
- O que precisa ser atualizado quando o projeto muda?

## Pratica Recomendada

- Pegue um projeto pequeno.
- Reescreva o README com objetivo, requisitos, setup, scripts e configuracao.
- Teste os comandos em uma instalacao limpa ou ambiente novo.
- Adicione links para documentos futuros de arquitetura e decisoes.
