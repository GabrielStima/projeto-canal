# Documentacao Tecnica

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- explicar por que documentacao tecnica importa em projetos reais;
- diferenciar documentacao de codigo, decisao, arquitetura, operacao e produto;
- reconhecer os principais formatos que serao estudados no modulo;
- saber como o modulo sera organizado.

## Ideia Central

Documentacao tecnica e a pratica de transformar conhecimento tecnico em algo consultavel, revisavel e util para outras pessoas. Ela nao substitui codigo funcionando, testes ou conversas. Ela registra contexto que o codigo sozinho nao mostra: por que uma decisao foi tomada, quais riscos foram aceitos, como operar o sistema, como integrar com ele e como evolui-lo.

Um projeto pode ter codigo correto e ainda assim ser dificil de manter se ninguem sabe como rodar, quais comandos sao seguros, por que a arquitetura foi escolhida, onde ficam os limites, quais alternativas foram descartadas e quais partes exigem cuidado.

Este modulo organiza documentacao como ferramenta de engenharia: README para entrada, ADR para decisoes, RFC para propostas, HLD e LLD para arquitetura, C4 e diagramas para comunicacao visual, alem de documentacao interna e externa para publicos diferentes.

## Topicos da Aula

- Documentacao como parte do trabalho de engenharia.
- Diferenca entre explicar, registrar, orientar e propor.
- Documentacao viva vs documento abandonado.
- Publicos: voce no futuro, time, revisores, operadores, consumidores externos.
- Mapa das proximas aulas do modulo.

## Exemplo Pratico

Imagine uma API de pagamentos. O codigo mostra rotas, validacoes e chamadas externas, mas nao explica tudo:

- por que o gateway A foi escolhido em vez do gateway B;
- o que acontece quando o gateway fica indisponivel;
- quais webhooks precisam ser idempotentes;
- como rodar a API localmente;
- quais variaveis de ambiente sao obrigatorias;
- que dados nao podem aparecer em logs;
- como um parceiro externo deve testar a integracao.

Essas respostas pertencem a documentos diferentes. Um README ajuda no setup, um ADR registra a escolha do gateway, um HLD mostra a arquitetura, um LLD detalha idempotencia e uma documentacao externa orienta o parceiro.

## Analogias e Intuicoes

- Documentacao e memoria compartilhada do projeto.
- Um bom documento reduz perguntas repetidas sem tentar substituir conversa.
- Documentar e deixar rastros de raciocinio, nao escrever texto decorativo.

## Erros Comuns

- Escrever documentacao apenas depois que tudo deu errado.
- Documentar como se todo publico tivesse o mesmo contexto.
- Criar documento grande sem dono, data ou objetivo.
- Copiar template sem preencher decisoes reais.
- Confiar em documentacao gerada por IA sem verificar o sistema.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Que problema a documentacao tecnica resolve?
- Quando um README nao e suficiente?
- Por que decisoes precisam ser registradas?
- Como escolher o documento certo para cada publico?
- O que torna uma documentacao perigosa ou inutil?

## Pratica Recomendada

- Escolha um projeto que voce ja conhece.
- Liste cinco perguntas que uma pessoa nova faria para trabalhar nele.
- Classifique cada pergunta como setup, arquitetura, decisao, operacao ou uso externo.
- Escolha o tipo de documento mais adequado para responder cada pergunta.
