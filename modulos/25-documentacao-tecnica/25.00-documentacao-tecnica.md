# Documentação Técnica

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar por que documentação técnica importa em projetos reais;
- diferenciar documentação de código, decisão, arquitetura, operação e produto;
- reconhecer os principais formatos que serão estudados no módulo;
- saber como o módulo será organizado.

## Ideia Central

Documentação técnica é a prática de transformar conhecimento técnico em algo consultável, revisável e útil para outras pessoas. Ela não substitui código funcionando, testes ou conversas. Ela registra contexto que o código sozinho não mostra: por que uma decisão foi tomada, quais riscos foram aceitos, como operar o sistema, como integrar com ele e como evoluí-lo.

Um projeto pode ter código correto e ainda assim ser difícil de manter se ninguém sabe como rodar, quais comandos são seguros, por que a arquitetura foi escolhida, onde ficam os limites, quais alternativas foram descartadas e quais partes exigem cuidado.

Este módulo organiza documentação como ferramenta de engenharia: README para entrada, ADR para decisões, RFC para propostas, HLD e LLD para arquitetura, C4 e diagramas para comunicação visual, além de documentação interna e externa para públicos diferentes.

## Exemplo Prático

O Legado Silencioso do PetCare:

Uma pessoa desenvolvedora construiu o assistente de IA do PetCare (Módulo 24), salvou no GitHub, mas não escreveu documentação suficiente para rodar ou manter o projeto.
Um ano depois, o assistente apresentou problemas. O novo time abriu o repositório e encontrou pastas sem descrição. Ninguém sabia como subir o ambiente local. Ninguém sabia por que havia escolhido `Pinecone` em vez de `Postgres`.
O código estava correto, mas a ponte de conhecimento havia sido perdida. A recuperação custou semanas de trabalho e atrasou correções importantes.

## Analogias e Intuições

- Código sem documentação técnica é como equipamento complexo sem manual: pode funcionar bem, mas outras pessoas terão dificuldade para operar, consertar ou evoluir.

## Erros Comuns

- Escrever documentação apenas depois que tudo deu errado.
- Documentar como se todo publico tivesse o mesmo contexto.
- Criar documento grande sem dono, data ou objetivo.
- Copiar template sem preencher decisões reais.
- Confiar em documentação gerada por IA sem verificar o sistema.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Que problema a documentação técnica resolve?
- Quando um README não é suficiente?
- Por que decisões precisam ser registradas?
- Como escolher o documento certo para cada público?
- O que torna uma documentação perigosa ou inútil?
