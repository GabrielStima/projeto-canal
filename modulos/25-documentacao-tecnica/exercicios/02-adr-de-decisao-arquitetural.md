# Exercício 02 — ADR de Decisão Arquitetural

## O Que Você Já Possui

Você escreveu o README do PetCare OS em `docs/README.md`. Agora você vai registrar uma decisão arquitetural importante para que o time futuro entenda por que o sistema foi construído dessa forma.

## A Tarefa

Escolha uma decisão arquitetural real do PetCare OS, como:

- uso de PostgreSQL como banco principal em vez de MongoDB;
- adoção de filas para notificações em vez de envio síncrono;
- conteinerização com Docker e orquestração com Kubernetes;
- separação da API em serviços por bounded context.

Crie `docs/adr/001-<nome-curto>.md` seguindo a estrutura:

1. **Título:** decisão em uma linha;
2. **Status:** proposto, aceito, substituído ou rejeitado;
3. **Contexto:** qual problema ou restrição levou à decisão;
4. **Decisão:** o que foi escolhido;
5. **Alternativas consideradas:** pelo menos duas, com motivo da descarte;
6. **Consequências:** ganhos, custos, riscos e dívidas técnicas aceitas;
7. **Quando revisitar:** condições que podem fazer a decisão ser reavaliada.

Use dados realistas para o PetCare OS: volume de clínicas, jornadas críticas, restrições de custo, equipe e conformidade.

## O Que Você Vai Produzir

Um ADR que registra uma decisão importante do sistema e será referenciado pelos diagramas e runbooks dos próximos exercícios.

Nos próximos exercícios você desenhará a arquitetura atual e detalhará um fluxo específico.

Critérios de correção:
1. O título resume a decisão em uma linha.
2. O status está claro e condiz com o momento da decisão.
3. O contexto explica problema, restrições e forças em jogo.
4. A decisão é específica e não genérica.
5. As alternativas descartadas possuem justificativa técnica ou de negócio.
6. As consequências incluem ganhos, custos e riscos reais.
7. Existe critério objetivo para revisitar a decisão.
8. A linguagem é direta, sem exageros ou jargão não explicado.
