# RFC

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar o papel de uma RFC em projetos técnicos;
- propor uma mudança com contexto, alternativas, impacto e riscos;
- diferenciar RFC de ADR;
- usar RFC para alinhar pessoas antes de implementar mudanças relevantes.

## Ideia Central

RFC significa Request for Comments. Em projetos de software, uma RFC é um documento usado para propor uma mudança e coletar feedback antes da decisão final.

Enquanto um ADR registra uma decisão tomada, uma RFC organiza uma discussão. Ela deixa explícito o problema, a motivação, os objetivos, o que está fora de escopo, as alternativas, os impactos, os riscos e o plano de implementação.

RFCs ajudam quando a mudança afeta múltiplos times, contratos, arquitetura, dados, segurança, custos, experiência do usuário ou operação. Elas evitam que decisões grandes apareçam de surpresa em um pull request.

## Tópicos da Aula

- Quando escrever uma RFC.
- Problema, contexto e objetivos.
- Fora de escopo.
- Proposta e alternativas.
- Impactos, riscos e plano de migração.
- Como transformar RFC aprovada em ADR ou tarefas.

## Exemplo Prático

Uma RFC para mudar autenticação de sessão para tokens pode incluir:

- problema atual: dificuldade de escalar sessões entre regiões;
- objetivo: permitir autenticação stateless para APIs específicas;
- fora de escopo: login social e painel administrativo;
- proposta: usar access token curto e refresh token rotacionado;
- alternativas: manter sessão centralizada ou usar gateway de autenticação;
- riscos: revogação, vazamento de token, compatibilidade com clientes;
- plano: piloto em uma API, métricas, rollback e documentação.

A RFC não precisa ser perfeita. Ela precisa tornar a discussão concreta.

## Analogias e Intuicoes

- RFC é uma conversa organizada antes da construção.
- Uma boa proposta mostra também o que não será feito.
- Feedback cedo é mais barato que retrabalho tarde.

## Erros Comuns

- Escrever RFC depois que a implementação já está pronta.
- Defender uma solução sem explicar o problema.
- Omitir riscos para facilitar aprovação.
- Misturar decisões aprovadas com perguntas abertas.
- Não registrar o resultado da discussão.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Qual problema a RFC tenta resolver?
- Quem precisa comentar?
- O que está fora de escopo?
- Quais alternativas foram consideradas?
- Que risco ou migração precisa ser planejado?

## Prática Recomendada

- Escolha uma mudança técnica relevante.
- Escreva uma RFC curta com problema, proposta, alternativas, riscos e plano.
- Marque três perguntas que você faria ao time antes de aprovar.
- Transforme a decisão final em um ADR resumido.
