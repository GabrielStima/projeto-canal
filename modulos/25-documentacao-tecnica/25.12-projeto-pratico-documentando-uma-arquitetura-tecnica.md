# Projeto Prático: Documentando Uma Arquitetura Técnica

## Objetivo

Ao final desta aula, você deve ser capaz de:

- integrar os principais documentos do módulo em um pacote coerente;
- documentar uma arquitetura técnica para leitura, revisão e manutenção;
- conectar README, documentação interna, documentação externa, ADR, RFC, HLD, diagramas e LLD;
- revisar se a documentação corresponde ao sistema real.

## Onde Esta Aula Entra na Formação

- Módulo: 25. Documentação Técnica
- Aula anterior: [LLD](25.11-lld.md)
- Próxima aula: Encerramento da trilha macro definida no currículo atual.
- Pré-requisitos: todas as aulas anteriores do módulo 25, system design, arquitetura, APIs, SRE, segurança, privacidade e IA para documentação.

## Ideia Central

O projeto final do módulo deve provar que você consegue transformar raciocínio técnico em documentação útil. A entrega não é um conjunto de textos isolados. É um pacote de documentação que ajuda pessoas diferentes a entender, rodar, revisar, operar, integrar e evoluir um sistema.

Você pode usar uma aplicação já criada na formação, uma arquitetura desenhada no módulo de system design ou o assistente técnico do módulo de IA. O importante é documentar um sistema com decisões reais, limites claros e exemplos verificáveis.

## Tópicos da Aula

- Escolha do sistema a documentar.
- Mapa de publicos e perguntas.
- README e guia interno.
- Guia externo ou referência de uso.
- ADR e RFC.
- HLD, C4, diagramas e LLD.
- Revisão de corretude, links, exemplos e riscos.

## Exemplo Prático

Marco Magistral de Staff Engineer N17 - A Diplomacia Final:

Aqui acaba sua Formação Técnica de Código.
Você construirá o Dossiê Final do PetCare OS.
1. Um HLD brutal e C4 Model mostrando Nuvem, Gateways, Redis e a IA Autônoma.
2. Um ADR Histórico validando a escolha do PostgreSQL sobre o Mongo usando Trade-Offs (CAP Theorem).
3. Um README matador usando Badges e Setup de Imagens OCI Assinadas (Supply Chain Security).
O que resta agora é uma Arquitetura de Alto Nível de Silicon Valley, descrita e documentada como um produto Enterprise Global. Você transcendeu de 'Codificador' para 'Mestre Construtor'.

## Analogias e Intuições

- A Formação de Engenharia Fullstack não formou um digitador rápido de React. Formou o Arquiteto e o Gestor. A Documentação do Marco N17 é a tese de mestrado. O carimbo onde o engenheiro diz: 'Eu sei fazer, eu sei por que fiz e sei explicar pro mundo'.

## Erros Comuns

- Criar documentos que não conversam entre si.
- Gerar texto com IA sem validar comandos, links e exemplos.
- Documentar arquitetura idealizada, não o sistema real.
- Omitir riscos, limites e decisões desconfortáveis.
- Fazer diagramas sem explicar o que eles querem responder.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Cada documento tem público e objetivo claros?
- Os comandos, exemplos e links foram verificados?
- As decisões arquiteturais têm contexto e consequências?
- O HLD e o LLD estão em níveis diferentes?
- Os diagramas ajudam a responder perguntas reais?

## Prática Recomendada

- Escolha um sistema simples, mas realista.
- Produza o pacote mínimo de documentação do projeto.
- Use IA apenas como apoio de organização e revisão, sem aceitar detalhes inventados.
- Peça para outra pessoa seguir o README e revisar uma decisão usando o ADR.
- Atualize os documentos a partir dos problemas encontrados.
