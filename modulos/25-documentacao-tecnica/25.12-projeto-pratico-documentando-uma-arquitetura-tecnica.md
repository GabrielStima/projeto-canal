# Projeto Pratico: Documentando Uma Arquitetura Tecnica

## Objetivo

Ao final desta aula, voce deve ser capaz de:

- integrar os principais documentos do modulo em um pacote coerente;
- documentar uma arquitetura tecnica para leitura, revisao e manutencao;
- conectar README, documentacao interna, documentacao externa, ADR, RFC, HLD, diagramas e LLD;
- revisar se a documentacao corresponde ao sistema real.

## Onde Esta Aula Entra no Curso

- Modulo: 25. Documentacao Tecnica
- Aula anterior: [LLD](25.11-lld.md)
- Proxima aula: Encerramento da trilha macro definida no curriculo atual.
- Pre-requisitos: todas as aulas anteriores do modulo 25, system design, arquitetura, APIs, SRE, seguranca, privacidade e IA para documentacao.

## Ideia Central

O projeto final do modulo deve provar que voce consegue transformar raciocinio tecnico em documentacao util. A entrega nao e um conjunto de textos isolados. E um pacote de documentacao que ajuda pessoas diferentes a entender, rodar, revisar, operar, integrar e evoluir um sistema.

Voce pode usar uma aplicacao ja criada no curso, uma arquitetura desenhada no modulo de system design ou o assistente tecnico do modulo de IA. O importante e documentar um sistema com decisoes reais, limites claros e exemplos verificaveis.

## Topicos da Aula

- Escolha do sistema a documentar.
- Mapa de publicos e perguntas.
- README e guia interno.
- Guia externo ou referencia de uso.
- ADR e RFC.
- HLD, C4, diagramas e LLD.
- Revisao de corretude, links, exemplos e riscos.

## Exemplo Pratico

Documente uma plataforma de aulas com API, frontend, banco, fila e worker:

1. README: objetivo, requisitos, comandos e configuracao.
2. Documentacao interna: ownership, dashboards, deploy e troubleshooting.
3. Documentacao externa: guia de integracao ou referencia de API.
4. ADR: decisao de usar fila para processamento de notificacoes.
5. RFC: proposta para adicionar webhooks para parceiros.
6. HLD: visao de componentes, fluxos, dados, riscos e trade-offs.
7. C4/diagramas: contexto, containers e sequencia do fluxo principal.
8. LLD: detalhe do processamento de webhook ou notificacao.

Ao final, faca uma revisao como se outra pessoa fosse manter o sistema sem conversar com voce.

## Analogias e Intuicoes

- O projeto e um kit de sobrevivencia tecnico para o sistema.
- Documentacao boa permite continuidade.
- A melhor revisao e tentar usar o documento para fazer uma tarefa real.

## Erros Comuns

- Criar documentos que nao conversam entre si.
- Gerar texto com IA sem validar comandos, links e exemplos.
- Documentar arquitetura idealizada, nao o sistema real.
- Omitir riscos, limites e decisoes desconfortaveis.
- Fazer diagramas sem explicar o que eles querem responder.

## Checklist de Entendimento

Antes de seguir, a pessoa deve conseguir responder:

- Cada documento tem publico e objetivo claros?
- Os comandos, exemplos e links foram verificados?
- As decisoes arquiteturais tem contexto e consequencias?
- O HLD e o LLD estao em niveis diferentes?
- Os diagramas ajudam a responder perguntas reais?

## Pratica Recomendada

- Escolha um sistema simples, mas realista.
- Produza o pacote minimo de documentacao do projeto.
- Use IA apenas como apoio de organizacao e revisao, sem aceitar detalhes inventados.
- Peça para outra pessoa seguir o README e revisar uma decisao usando o ADR.
- Atualize os documentos a partir dos problemas encontrados.
