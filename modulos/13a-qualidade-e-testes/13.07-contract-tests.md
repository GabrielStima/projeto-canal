# Contract Tests

## Objetivo

Ao final desta aula, você deve ser capaz de:

- explicar contract tests como proteção de contrato;
- diferenciar contrato de regra de negócio;
- validar respostas e exemplos contra uma expectativa;
- entender como contract tests ajudam consumidores e provedores.

## Onde Esta Aula Entra na Formação

- Módulo: 13. Qualidade, Testes, Observabilidade é Performance
- Aula anterior: Testes de Integração
- Próxima aula: Testes End-to-End
- Pré-requisitos: Contract testing introdutório, contratos de API, OpenAPI, backward compatibility, testes de integração e erros padronizados.

## Ideia Central

Contract tests verificam se uma interface continua cumprindo a promessa combinada. Em APIs, isso envolve campos, tipos, formatos, status codes e exemplos relevantes.

Eles não provam que todo comportamento está correto. Eles protegem consumidores contra quebras inesperadas e ajudam provedores a evoluir sem romper contratos por acidente.

## Tópicos da Aula

- Contrato entre consumidor e provedor.
- Schema, exemplos e interações.
- Backward compatibility.
- Diferença para integração.
- Diferença para E2E.
- Limites do contract testing.

## Exemplo Prático

Proteção no App do Estoque:

Se a API do backend devolve `{ "id": 1, "nomeMedicamento": "Vacina V10" }`, e o time de desenvolvimento renomear a variável no banco para `nome_medicamento`, a integração do App Mobile com a API quebra silenciosamente.

Os Contract Tests (ex: Pact) validam estritamente o formato JSON de saída (`Schema` e chaves esperadas) cruzando a expectativa do frontend com o que a API exibe no Swagger/OpenAPI.

## Analogias e Intuições

- Contract testing é como firmar o formato de um contrato de aluguel em cartório. Se o proprietário mudar as cláusulas sozinho sem notificar a imobiliária, o contrato é anulado (o teste falha e bloqueia o deploy).
*Limite: O contrato valida se os dados retornam na formatação certa, não garante se a lógica interna do aluguel está correta.*

## Erros Comuns

- Validar apenas status code.
- Fazer contrato amplo demais e virar E2E.
- Esquecer exemplos de erro.
- Atualizar teste para aceitar quebra sem discutir compatibilidade.
- Confundir contrato com implementação do banco.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual contrato está sendo protegido?
- Quem consome esse contrato?
- Que mudança deveria quebrar o teste?
- Que mudança deveria continuar compatível?
- O teste cobre sucesso e erro?

## Prática Recomendada

- Acesse mentalmente a rota `GET /api/pets/:id` do PetCare OS.
- Liste as chaves obrigatórias que jamais poderiam mudar de nome sem quebrar a integração com o Portal do Tutor.
- Explique como o controle de versão da API (v1, v2) e Backward Compatibility se alinham com Contratos.
