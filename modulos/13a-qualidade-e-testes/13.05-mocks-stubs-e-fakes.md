# Mocks, Stubs e Fakes

## Objetivo

Ao final desta aula, você deve ser capaz de:

- diferenciar mocks, stubs e fakes em nível prático;
- reconhecer quando substituir uma dependência em teste;
- evitar esconder problemas reais com doubles demais;
- escolher o double mais simples para cada cenário.

## Onde Esta Aula Entra na Formação

- Módulo: 13. Qualidade, Testes, Observabilidade é Performance
- Aula anterior: Testes Unitários
- Próxima aula: Testes de Integração
- Pré-requisitos: Testes unitários, TypeScript, funções, dependências, APIs, banco de dados e estratégia de testes.

## Ideia Central

Doubles de teste substituem dependências para tornar um teste mais controlável. Um stub devolve uma resposta preparada. Um fake implementa uma versão simples, mas funcional, de uma dependência. Um mock também verifica interações esperadas.

Eles são úteis quando uma dependência é lenta, externa, cara, instável ou difícil de controlar. Mas também podem deixar testes irreais se substituírem justamente aquilo que precisava ser validado.

## Tópicos da Aula

- Dependências em testes.
- Stub.
- Fake.
- Mock.
- Interação vs resultado.
- Risco de excesso de doubles.

## Exemplo Prático

```ts
type Mailer = {
  sendWelcomeEmail(email: string): Promise<void>;
};

async function registerUser(email: string, mailer: Mailer) {
  await mailer.sendWelcomeEmail(email);
  return { email, status: "created" };
}
```

No teste, o envio real de email pode ser substituido por um double.

## Analogias e Intuições

- Um double é um dublê: ajuda no ensaio, mas não prova tudo sobre a cena real.
- Quanto mais falso o ambiente, mais cuidado você precisa ter com a conclusão.

## Erros Comuns

- Usar mock para tudo por padrão.
- Testar se uma função interna foi chamada sem validar comportamento.
- Criar fake mais complexo que o código real.
- Substituir banco sempre e nunca testar integração real.
- Deixar o teste passar com contratos irreais.

## Checklist de Entendimento

Antes de seguir, você deve conseguir responder:

- Qual dependência está sendo substituída?
- Por que ela precisa ser substituída neste teste?
- O teste valida resultado ou interação?
- O double representa um comportamento realista?
- Existe um teste de integração cobrindo a dependência real?

## Prática Recomendada

- Escolha uma função que envie email ou chame API externa.
- Crie um stub de sucesso e um stub de falha.
- Liste o que esse teste cobre.
- Liste o que ele não cobre e precisaria de outro nível de teste.
